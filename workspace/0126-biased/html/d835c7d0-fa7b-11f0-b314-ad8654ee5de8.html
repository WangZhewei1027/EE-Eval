Bellman–Ford Algorithm

A comprehensive, textual guide to the Bellman–Ford algorithm: what it does, how it works, why it is correct, and how to use it safely when graphs may have negative-weight edges.

Overview and purpose

The Bellman–Ford algorithm is a classic algorithm for computing single-source shortest paths in weighted directed (or undirected) graphs that can contain negative edge weights. Given a source vertex s, Bellman–Ford computes the shortest path distance from s to every other vertex, or reports that a negative-weight cycle is reachable from s (in which case shortest paths are undefined).

Key features

  • Accepts negative edge weights; unlike Dijkstra, it does not require all weights to be non-negative.
  • Detects negative-weight cycles reachable from the source.
  • Time complexity O(V * E) where V is number of vertices, E is number of edges.
  • Simple to implement and widely used as a building block in other algorithms (e.g., Johnson's algorithm for all-pairs shortest paths).

Problem statement (formal)

Input: A weighted graph G = (V, E) with edge weight function w: E → ℝ, and a source vertex s ∈ V.

Output: For every vertex v ∈ V, the algorithm either:

  • computes δ(s, v) = length of the shortest path from s to v, or
  • reports that there is no finite shortest path because a negative-weight cycle is reachable from s.

Note: A negative-weight cycle reachable from the source implies you can loop and reduce the path length arbitrarily; thus a well-defined finite shortest path does not exist for vertices reachable through or beyond such a cycle.

High-level intuition

The Bellman–Ford algorithm repeatedly relaxes edges. A relaxation of an edge (u, v) with weight w updates the current best-known distance to v by trying to go through u: 

if dist[u] + w < dist[v] then dist[v] := dist[u] + w

If you imagine a shortest path from s to some vertex v has at most k edges, then performing k rounds of relaxation (considering every edge in each round) is enough to discover that path. In a graph with V vertices, any true simple shortest path can have at most V−1 edges (a simple path cannot visit a vertex twice), so doing V−1 full rounds is sufficient to find all shortest paths (if they exist).

Pseudocode

// Bellman-Ford (classic)
procedure BellmanFord(vertices V, edges E, source s)
    // Initialization
    for each vertex v in V
        dist[v] := +∞
        predecessor[v] := null
    dist[s] := 0

    // Relax edges repeatedly
    for i from 1 to |V|-1
        for each edge (u, v) with weight w in E
            if dist[u] + w < dist[v]
                dist[v] := dist[u] + w
                predecessor[v] := u

    // Check for negative-weight cycles
    for each edge (u, v) with weight w in E
        if dist[u] + w < dist[v]
            report "Negative-weight cycle reachable from source"
            return (cycle exists)

    return dist, predecessor

predecessor[] can be used to reconstruct shortest paths when no negative cycle is present.

Proof sketch of correctness

Two core facts explain why Bellman–Ford works:

  1. Any shortest simple path from s to v uses at most V−1 edges. Therefore, after k iterations of relaxing all edges, the algorithm has guaranteed correctness for any shortest path that uses ≤ k edges. After V−1 iterations, all simple shortest paths are thus discovered.
  2. If a further relaxation (after V−1 iterations) can still decrease some dist[v], then there exists a path from s to v with more than V−1 edges that is strictly shorter than the best-known path. That implies a cycle whose total weight is negative is on that path (because you must have repeated some vertex to exceed V−1 edges), and that cycle is reachable from s. Hence you can detect negative cycles with one more pass.

Complexity

  • Time complexity: O(V * E) in the worst case (V-1 iterations × E edges per iteration), plus one extra pass for negative cycle detection.
  • Space complexity: O(V + E) to store graph and O(V) for distances and predecessors.

In dense graphs E = Θ(V^2), so Bellman–Ford becomes O(V^3). For many practical graphs where edges are non-negative, Dijkstra (with a priority queue) is faster: O((E + V) log V) or similar. But Bellman–Ford remains essential whenever negative edges are involved.

Relaxation explained

Relaxation is the basic operation: trying to improve the current best estimate dist[v] using an edge (u, v). It is monotonic: dist values can only decrease (move toward the true shortest distance). The predecessor array is updated alongside to support path reconstruction. 

// relax edge (u, v, w)
if dist[u] != +∞ and dist[u] + w < dist[v]
    dist[v] := dist[u] + w
    predecessor[v] := u

Negative cycles: detection and consequences

After V−1 iterations, run one additional pass over all edges. If any distance can still be reduced, a negative-weight cycle is reachable from the source. What this means practically:

  • There is no well-defined finite shortest path to any vertex that can be reached via that cycle.
  • Many implementations mark such vertices as "−∞" (meaning distance can be made arbitrarily small) or otherwise report the cycle's existence and optionally return nodes on it.

Detecting which vertices are affected by the negative cycle (beyond simply reporting existence) requires an additional traversal: after finding an edge that relaxes, you can BFS/DFS from that edge's target to mark all reachable vertices.

Comparison with Dijkstra

  • Dijkstra only works with non-negative edge weights; it uses a greedy strategy with a priority queue and is faster on non-negative graphs.
  • Bellman–Ford handles negative weights and detects negative cycles, but at the cost of higher time complexity.
  • Choose Bellman–Ford when negative weights are present or when you need negative cycle detection; otherwise prefer Dijkstra for efficiency.

Step-by-step example (manual walkthrough)

We will examine a small directed graph with 5 vertices {A, B, C, D, E}. Edges and weights:

A → B (6)
A → D (7)
B → C (5)
B → D (8)
B → E (-4)
C → B (-2)
D → C (-3)
D → E (9)
E → A (2)

We run Bellman–Ford with source s = A.

Initialization

dist[A] = 0
dist[B] = ∞
dist[C] = ∞
dist[D] = ∞
dist[E] = ∞

Iteration 1 (relax all edges)

Relax edges in any order; we describe one natural order (A→B, A→D, B→C, B→D, B→E, C→B, D→C, D→E, E→A):

After A→B: dist[B] = 6
After A→D: dist[D] = 7
After B→C: dist[C] = 11
After B→D: dist[D] stays 7  (because 6 + 8 = 14 > 7)
After B→E: dist[E] = 2   (6 + -4)
After C→B: dist[B] = 9? No, 11 + -2 = 9 (worse than 6), so keep 6
After D→C: dist[C] = 4   (7 + -3)
After D→E: dist[E] stays 2 (7 + 9 = 16 > 2)
After E→A: dist[A] = 4? 2 + 2 = 4 but dist[A] is 0, so keep 0

Iteration 2

Relaxing again can propagate improvements:
B→C: dist[C] = 6 + 5 = 11 (not better than 4)
C→B: dist[B] = 4 + -2 = 2  (improvement!)
Then B→E: dist[E] = 2 + -4 = -2 (improvement)
E→A: dist[A] = -2 + 2 = 0 (no change)
Now some other relaxations confirm the updated values.

Iteration 3 and 4

By the end of the 4th iteration (V−1 = 4), the shortest distances stabilize:

dist[A] = 0
dist[B] = 2
dist[C] = -1
dist[D] = 7
dist[E] = -2

One more pass detects no negative cycle because no further relaxations are possible. These distances are the correct shortest path lengths from A.

The detailed per-edge bookkeeping illustrates how improvements propagate outward from the source across multiple iterations.

Practical implementation notes and optimizations

  • Early exit: If an entire iteration of relaxing all edges causes no changes, you can stop early. This does not change correctness and often saves time in practice.
  • Edge ordering: The correctness does not depend on edge order, but a "good" order may converge faster in practice when using early exit.
  • Detecting affected vertices: If you need to mark all vertices whose distances can be reduced indefinitely because of negative cycles, after detecting a relaxing edge do a traversal from its target to mark all reachable vertices.
  • Sparse graphs: With sparse graphs using adjacency lists, store edges in a simple edge list for easy iteration; early exit can make Bellman–Ford acceptable for many real problems.
  • Large weights and overflow: Use a numeric type that can represent 'infinity' and sums without overflow. In practice, check for "dist[u] == +INF" before adding.

When to use Bellman–Ford

  • When edges may be negative and you need correct shortest paths.
  • When you must detect negative cycles reachable from the source.
  • When implementing Johnson's algorithm for all-pairs shortest paths: Bellman–Ford is used to compute potential values.

Variants and related algorithms

  • Bellman–Ford–Moore: Another historical name often used interchangeably.
  • Johnson's algorithm: Uses Bellman–Ford once to reweight edges to nonnegative values, then runs Dijkstra from each vertex for all-pairs shortest paths.
  • SPFA (Shortest Path Faster Algorithm): An optimization that uses a queue and often performs faster in practice on many graphs, but worst-case complexity can still be O(VE) and pathological cases can be very slow.

Troubleshooting and common mistakes

  • Forgetting to check dist[u] != +∞ before performing dist[u] + w can cause spurious arithmetic or overflow.
  • Not running the extra pass to detect negative cycles may silently give wrong answers when negative cycles exist.
  • Using Bellman–Ford when all weights are non-negative might be suboptimal in runtime; consider Dijkstra when applicable.
