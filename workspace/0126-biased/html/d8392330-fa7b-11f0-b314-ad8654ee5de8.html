Domain Name System (DNS) — Comprehensive Educational Reference

A thorough textual explanation of DNS concepts, operation, record types, algorithms, security, and operational notes. Includes one simple demonstration.

Overview — What is DNS?

The Domain Name System (DNS) is the distributed hierarchical directory service that translates human-readable domain names (like example.com) into machine-usable network identifiers such as IP addresses (e.g. 93.184.216.34 for IPv4, or 2606:2800:220:1:248:1893:25c8:1946 for IPv6). DNS is an essential internet service that makes it practical for humans to access services by names instead of numeric addresses.

DNS also stores other kinds of information associated with names (mail routing, service records, cryptographic keys, etc.) and supports features for delegation, caching, zone transfers, and security extensions (DNSSEC).

Core Components and Roles

  • Resolver (Client / Stub Resolver): Typically part of the operating system or application. It issues DNS queries on behalf of an application (web browser, mail client).
  • Recursive Resolver (Recursive DNS Server): Accepts queries from stub resolvers and performs the full resolution process (contacting other DNS servers) until it obtains an answer. Many ISPs and public DNS services (like 8.8.8.8) provide these.
  • Authoritative Name Server: Holds DNS records for a domain and can answer queries authoritatively for zones it manages.
  • Root Name Servers: The top of the DNS hierarchy. They know the authoritative servers for top-level domains (TLDs) like .com, .org, .net, and country code TLDs.
  • TLD Name Servers: Authoritative for a top-level domain; they direct queries to authoritative servers for second-level domains (example.com).
  • Zone: A contiguous portion of the namespace served by a set of authoritative servers. A zone file contains resource records (RRs).

Common DNS Record Types

  • A — IPv4 address mapping.
  • AAAA — IPv6 address mapping.
  • CNAME — Canonical name (alias) that points one name to another canonical name.
  • NS — Delegates authority for a zone to specific name servers.
  • MX — Mail exchanger record for routing email.
  • PTR — Pointer for reverse DNS (IP -> name).
  • SOA — Start of authority: metadata about a zone (primary server, serial, refresh, retry, expire, minimum/negative TTL).
  • SRV — Service locator record (services/protocol/priority/weight/port/target).
  • TXT — Arbitrary text. Used for SPF, DKIM, verification tokens.
  • DNSKEY, RRSIG, DS, NSEC/NSEC3 — DNSSEC records for cryptographic authentication of DNS data.

Basic DNS Message and Wire Format (Conceptual)

DNS messages are sent over UDP (usually) or TCP on port 53. Conceptually a DNS message contains:

  • Header — ID, flags (QR, Opcode, AA, TC, RD, RA), response code (RCODE), counts for sections.
  • Question — The name, type, and class being queried.
  • Answer — Resource records answering the question.
  • Authority — Records pointing to authoritative name servers (NS records, SOA for negative answers).
  • Additional — Extra helpful records (glue A/AAAA for name servers, OPT for EDNS0).

Names are represented as a sequence of length-prefixed labels and terminated with a zero-length label. The protocol includes name compression to reduce packet size by referencing previously encoded names inside the message.

Resolution Models — Iterative vs Recursive

There are two key styles of resolver behavior:

  • Recursive resolution (typical for stub resolvers): A client asks a recursive resolver to obtain an answer. The recursive resolver performs all necessary queries to other servers (root, TLD, authoritative) until it finds the answer or an error, then returns that result.
  • Iterative (or non-recursive) resolution: A resolver (often a recursive server querying other servers) asks a server which may return either an authoritative answer or a referral to other servers to try next. The client then contacts the referred servers in sequence. Authoritative servers often respond iteratively (i.e., return referrals).

Step-by-Step Example — Typical Recursive Resolution Flow (Textual)

Let's resolve "www.example.com" with a recursive resolver that has no cached entries. The process is described step-by-step as plain text:

  1. Client (stub resolver) asks its configured recursive resolver: "What's the A record for www.example.com?" (UDP port 53, RD flag set).
  2. Recursive resolver checks its cache for an answer. Cache miss.
  3. Recursive resolver sends a query to a root server: "Who is authoritative for .com?" (Question: NS for com.).
  4. Root server responds with a referral: it returns the NS records for .com and additional glue A/AAAA records for those TLD servers (or the resolver uses root hints to contact them).
  5. Recursive resolver queries a .com TLD server: "Who is authoritative for example.com?"
  6. .com TLD server responds with a referral: NS records for example.com and possibly glue records (A/AAAA) for those name servers.
  7. Recursive resolver queries one of the authoritative name servers for example.com: "What's the A record for www.example.com?"
  8. Authoritative server responds with an answer (A record) and an authoritative answer flag (AA). If there is no www record, the server may return NXDOMAIN or NOERROR with no answers; the SOA in the authority section indicates negative caching TTL rules.
  9. Recursive resolver caches the answer with the TTL provided and returns the A record to the stub resolver.
  10. The client receives the IP and initiates a TCP (or TLS) connection to that IP to contact the service (e.g., web server).

DNS Resolution Algorithm — Pseudocode (Textual)

Below is a high-level textual pseudocode description of a recursive resolver's algorithm.

function resolveRecursive(name, type):
  if cache contains (name, type) and not expired:
    return cached result

  // start at root servers
  servers = rootHints()
  while true:
    for server in servers:
      response = sendQuery(server, name, type, recursionDesired=false)
      if response is authoritative and contains answer:
        cache response records by their TTL
        return response.answer
      if response is authoritative and NXDOMAIN:
        // no such name
        cache negative answer using SOA/minimum rules
        return NXDOMAIN
      // response is a referral: extract next-level NS and glue
      servers = extractGlueOrResolve(response.authority NS records)
      break  // query next set of servers

  // if loop exits without result -> SERVFAIL or timeout

Caching and TTL

Caching is central to DNS performance and scalability. When a resolver receives records, it stores them in cache for their time-to-live (TTL) interval. TTL is specified per resource record by the authoritative server.

Notes and rules:

  • Each RR retains its own TTL; compound answers may have different TTLs per RR.
  • Negative caching (caching NXDOMAIN or no-data responses) is governed by RFC 2308. The SOA's minimum field historically used to convey negative caching TTLs; modern practice uses the Minimum field consistently for negative caching unless an NOERROR + no answers includes per-record TTLs.
  • Resolvers can cap TTLs or ignore very large TTLs, but generally obey TTL until expiry.
  • Caching reduces query traffic and latency but can delay propagation of changes unless TTLs are set low during planned updates.

Delegation and Glue Records

Delegation is the mechanism where a zone is split and authority for a subtree is assigned to a set of name servers. The parent zone stores NS records pointing to the child zone's name servers. If those name servers are inside the child zone (in-bailiwick), the parent must provide glue records (A/AAAA) so that resolvers can learn the IP addresses to contact the child zone's authoritative servers.

Reverse DNS (PTR) — Mapping IP to Name

Reverse DNS uses special domains under in-addr.arpa (IPv4) and ip6.arpa (IPv6). For example, 93.184.216.34 maps to 34.216.184.93.in-addr.arpa, which contains a PTR record pointing back to the canonical name. Reverse DNS is commonly used for logging, spam filtering, and some validation checks.

Zone Files — Example (BIND-style)

$ORIGIN example.com.
$TTL 3600
@   IN SOA  ns1.example.com. hostmaster.example.com. (
        2026012501 ; serial (YYYYMMDDNN)
        7200       ; refresh
        3600       ; retry
        1209600    ; expire
        3600       ; minimum TTL
)
    IN NS   ns1.example.com.
    IN NS   ns2.example.com.

ns1 IN A    198.51.100.10
ns2 IN A    198.51.100.11

www IN A    198.51.100.20
mail IN A   198.51.100.30
    IN MX   10 mail.example.com.

Zone Transfers (AXFR/IXFR)

Zone transfers are the mechanism for copying a zone from a primary (master) to secondary (slave) authoritative servers. AXFR is full transfer, IXFR is incremental. These typically use TCP for reliability and are often protected by access control lists and TSIG (transaction signatures).

Common Operational Practices

  • TTL management: Use higher TTLs for stable records to reduce load; reduce TTL before major planned changes to speed propagation of updates.
  • Monitoring and redundancy: Provide multiple authoritative name servers on separate networks and ideally in different geographic regions.
  • Glue placement: Avoid placing name servers inside the zone being served when possible, or ensure parent provides correct glue.
  • Use of split-horizon (views): Some deployments serve different answers to internal vs external clients for the same name (e.g., internal-only addresses).

Advanced Topics — DNSSEC, EDNS, DoT, DoH

DNSSEC (Domain Name System Security Extensions) adds cryptographic signatures to DNS data to ensure authenticity and integrity. Key record types:

  • DNSKEY — public key used to verify signatures.
  • RRSIG — digital signature for a RRset.
  • DS — Delegation Signer: parent zone stores a hash of the child's DNSKEY to link the chain of trust.
  • NSEC/NSEC3 — records proving non-existence of names or types (authenticated denial of existence).

With DNSSEC, resolvers that validate will follow the chain of trust starting at a configured trust anchor (commonly the root's trust anchor) and verify signatures until they can trust the returned data.

EDNS0 extends the original DNS protocol to allow larger UDP messages, additional flags, and extension mechanisms (OPT records). This enables DNSSEC signatures and larger responses without falling back to TCP as often.

To improve privacy and protect against on-path attackers, DNS can be transported over encrypted channels:

  • DoT — DNS over TLS (port 853).
  • DoH — DNS over HTTPS (uses HTTP/2 or HTTP/3 over TLS), often on port 443.

Security Threats and Mitigations

Major threats:

  • Cache poisoning: An attacker injects false records into a resolver's cache. Mitigations: source port and transaction ID randomization, DNSSEC validation, limiter on acceptance of unsolicited data.
  • Amplification/reflection DDoS: Attackers send small spoofed queries to open recursive DNS servers to generate large responses directed at victims. Mitigations: disable open recursion, response rate limiting (RRL), minimize amplifying resource record types replies to arbitrary queries.
  • MitM / on-path attacks: Prevented by DNSSEC and transport encryption (DoT/DoH).
  • Zone transfer abuse: Restrict AXFR access, use TSIG for authentication.

Debugging Tools and Examples

Common command-line tools (examples with conceptual explanations):

dig +noall +answer www.example.com A
# Shows the answer section only.

dig @8.8.8.8 example.com NS
# Query a specific server for NS records.

dig +trace www.example.com
# Performs iterative resolution locally, showing steps from root down.

nslookup www.example.com 8.8.8.8
# Older tool, interactive and non-interactive modes.

host -t MX example.com
# Query MX records for mail routing details.

Explain dig +trace conceptually: it starts at the root, follows referrals, and shows each step: root -> TLD servers -> authoritative servers.

Detailed Example — What Happens When You Open https://www.example.com/

  1. Your browser asks the OS stub resolver for the A/AAAA records for www.example.com (this may be done by a browser-level asynchronous DNS lookup, or via system APIs).
  2. The OS resolver asks the configured recursive resolver (often your ISP or a public DNS like 1.1.1.1) for the name.
  3. The recursive resolver either answers from cache or performs iterative queries starting from the root servers if the data is not cached, following the referral chain until it contacts the authoritative server and receives the final IP addresses.
  4. The recursive resolver returns the IP(s) to your machine. If both IPv4 and IPv6 are present, most systems decide an order (happy eyeballs algorithm may try both).
  5. The browser opens a TCP (or TCP+TLS) connection to the resolved address on port 443, performs the TLS handshake, and then issues an HTTP request.

Negative Responses and NXDOMAIN

If a name does not exist, the authoritative server responds with NXDOMAIN. Negative caching rules mean resolvers may cache the absence for a period determined by the SOA minimum field and RFC 2308. Properly setting SOA values helps control negative caching impact.

Miscellaneous Concepts

  • Glue vs Non-glue: Glue records are supplied by parent zones to break circular dependency when nameservers are inside the delegated zone.
  • Round-robin DNS: Multiple A/AAAA records can be returned for load distribution; simple rotation often happens at the server or resolver level.
  • Split-horizon DNS (views): Different answers based on client source or view (internal vs external).
  • Dynamic DNS (DDNS): Protocol for clients to update records dynamically (e.g., DHCP integrated updates).
  • Resolver policies: Some DNS servers implement filtering (e.g., blocking malicious domains, parental controls) by returning NXDOMAIN or redirecting queries.

Best Practices Summary

  • Run authoritative services on multiple servers with geographically and topologically diverse networks.
  • Use appropriate TTLs depending on expected change frequency.
  • Protect zone transfers and use TSIG where appropriate.
  • Deploy DNSSEC and configure validators for critical domains to mitigate cache poisoning and forgery.
  • Limit recursion for public resolvers to trusted clients to avoid abuse.
  • Monitor and log queries and resolution performance; use metrics to detect unusual traffic patterns indicating abuse.
