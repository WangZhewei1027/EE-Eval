Big-O Notation — A Comprehensive, Textual Guide

This page explains Big‑O notation in depth: definitions, intuition, worked examples, recurrence relations, rules, common complexities, and practical considerations. The content is focused on textual explanation and pedagogy. Minimal interactivity is included: one small demonstration button that computes and prints growth numbers for a few functions and n values.

1. What Big‑O Is (Intuition)

Big‑O notation describes the asymptotic upper bound of a function. Informally: it tells how a function grows as its input size (conventionally n) becomes very large, ignoring constant factors and lower‑order terms. When we say "an algorithm runs in O(n) time", we mean that for large inputs its running time grows at most linearly with n (up to a constant multiplier).

Big‑O focuses on growth rate, not exact running time. It helps compare algorithms by how they scale, not by absolute speed on a particular machine.

2. Formal Definitions

2.1 Big‑O (Upper bound)

Let f(n) and g(n) be functions mapping positive integers to non‑negative real numbers. We write:

f(n) = O(g(n))

if there exist constants c > 0 and n0 ≥ 1 such that for all n ≥ n0:

f(n) ≤ c · g(n)

Interpretation: beyond some point n0, f(n) is at most a constant times g(n).

2.2 Big‑Omega and Big‑Theta

Big‑Omega is a lower bound:

f(n) = Ω(g(n))  means  ∃ c > 0, n0: for all n ≥ n0, f(n) ≥ c · g(n)

Big‑Theta captures a tight (both upper and lower) bound:

f(n) = Θ(g(n))  means  f(n) = O(g(n)) and f(n) = Ω(g(n))

2.3 Little‑o notation (strictly smaller)

f(n) = o(g(n)) means f grows strictly slower than g. Formally:

limₙ→∞ f(n)/g(n) = 0

Limit-based equivalence

If limsup f(n)/g(n) < ∞ then f = O(g). If lim f(n)/g(n) = c where 0 < c < ∞ then f = Θ(g). These are useful when comparing smooth functions.

3. Common Complexity Classes (from fastest-to-slowest growth)

Below are typical complexity classes; k is a constant > 1, and log is log base 2 unless noted.

  • O(1) — constant time
  • O(log n) — logarithmic
  • O(n^a) — polynomial (e.g., O(n), O(n^2), O(n^3))
  • O(n log n) — "linearithmic"
  • O(k^n) — exponential (k>1)
  • O(n!) — factorial (very fast growth)

Examples of algorithms

  • O(1): access array element by index.
  • O(log n): binary search (in sorted array).
  • O(n): single pass scan—e.g., find max in array.
  • O(n log n): efficient comparison-based sorts (merge sort, heapsort).
  • O(n^2): naive selection/bubble sort, pairwise comparisons.
  • O(2^n): brute-force subsets, naive Fibonacci recursion.

4. Practical Rules and "Algebra" of Big‑O

  • Drop constants: c·f(n) = Θ(f(n)) for any constant c > 0.
  • Drop lower-order terms: f(n)+g(n) = Θ(max(f(n),g(n))). Example: n^2 + 3n + 7 = Θ(n^2).
  • Sums: If f(n)=O(h(n)) and g(n)=O(h(n)), then f(n)+g(n)=O(h(n)).
  • Products with constants or slow functions: If f(n)=O(a(n)) and g(n)=O(b(n)), then f(n)·g(n)=O(a(n)·b(n)).
  • Log rules: log_a(n) = Θ(log_b(n)) for any constants a,b > 1 (bases change is constant factor).
  • Independent inputs: For functions of two inputs, f(n,m), report complexity in both variables (e.g., O(n·m)).

5. Step‑by‑Step Example Analyses

5.1 Single loop

// Pseudocode
for (i = 1; i <= n; i++) {
  doConstantWork();
}

Each iteration does O(1) work; the loop runs n times. So total is O(n). Formally: T(n) = n·O(1) = O(n).

5.2 Two sequential loops

// Pseudocode
for (i = 1; i <= n; i++) { doConstantWork(); }
for (j = 1; j <= n; j++) { doConstantWork(); }

Each loop is O(n); sequential composition sums: O(n) + O(n) = O(n). We drop constant factors (2), so total O(n).

5.3 Nested loops

// Pseudocode
for (i = 1; i <= n; i++) {
  for (j = 1; j <= n; j++) {
    doConstantWork();
  }
}

The inner loop runs n times for each of the n outer iterations: n × n = n^2. So O(n^2).

5.4 Loop with changing step size

// Pseudocode
i = 1;
while (i <= n) {
  doConstantWork();
  i = i * 2;
}

i takes values 1, 2, 4, 8, ..., ≤ n — that's about log2(n)+1 iterations, so O(log n).

5.5 Example with break (best/worst-case)

// Pseudocode: linear search in array
for (i = 1; i <= n; i++) {
  if (arr[i] == key) return i; // break early
}
return -1;

Best case: key at first position → O(1). Worst case: not present or last → O(n). Average case depends on distribution of key (commonly O(n) if uniformly random absent/present).

5.6 Recursive examples

Recurrence: T(n) describes runtime in terms of smaller inputs.

Linear recursion

T(n) = T(n-1) + O(1)  with T(1)=O(1)  ⇒  T(n) = O(n)

Interpretation: work decreases by one each call; total is proportional to n.

Divide-and-conquer (binary split)

T(n) = T(n/2) + O(1)  ⇒  T(n) = O(log n)

Each recursive call halves the input; depth ~ log n and each level does constant work.

Merge sort recurrence

T(n) = 2·T(n/2) + Θ(n)  ⇒  T(n) = Θ(n log n)

At each level there is Θ(n) merging cost; number of levels ~ log n.

Exponential recursion (Fibonacci naive)

T(n) = T(n-1) + T(n-2) + O(1)  ⇒  T(n) = Θ(φ^n) where φ≈1.618

Repeated subproblems cause exponential blowup. Dynamic programming reduces to O(n).

6. The Master Theorem (quick tool for many recurrences)

For recurrences of the form:

T(n) = a·T(n/b) + f(n)

where a ≥ 1 and b > 1 are constants, the asymptotic behavior depends on comparing f(n) to n^{log_b(a)}:

  • Case 1: If f(n) = O(n^{c}) where c < log_b(a), then T(n) = Θ(n^{log_b(a)}).
  • Case 2: If f(n) = Θ(n^{log_b(a)}·log^k n) for some k ≥ 0, then T(n) = Θ(n^{log_b(a)}·log^{k+1} n).
  • Case 3: If f(n) = Ω(n^{c}) where c > log_b(a), and if a·f(n/b) ≤ c'·f(n) for some c' < 1 (regularity), then T(n) = Θ(f(n)).

Example: Merge sort has a=2, b=2, so n^{log_b(a)} = n. f(n)=n so Case 2 with k=0 ⇒ T(n) = Θ(n log n).

7. Amortized Analysis (brief)

Amortized analysis spreads the cost of expensive operations across many cheap ones. Example: dynamic array that doubles capacity when full.

When resizing from capacity k to 2k, you pay O(k) for copying. But resizing happens rarely: after resizing to 2k you get ≈k cheap insertions before next resize. Summing costs across n insertions yields total O(n) cost, so amortized cost per insertion is O(1).

8. Space Complexity

Space complexity measures additional memory used (besides input). Examples:

  • In-place sort like heapsort: O(1) extra space (ignoring recursion).
  • Mergesort (typical array implementation): O(n) extra space for merging.
  • Recursion depth contributes to space: quicksort average depth O(log n) but worst-case O(n).

9. Worked Code Analyses (textual walkthrough)

9.1 Example — Count pairs with sum (naive)

// Pseudocode: count pairs (i,j) with i<j and A[i]+A[j]==S
count = 0
for i from 1 to n:
  for j from i+1 to n:
    if A[i]+A[j] == S:
      count++

Inner loop runs roughly n-i times, so total comparisons ≈ Σ_{i=1..n} (n-i) = O(n^2). So O(n^2) time, O(1) extra space.

9.2 Example — Count pairs with sum (optimized)

// Pseudocode: sort then two-pointer
sort(A)               // O(n log n)
left = 1; right = n
while left < right:
  sum = A[left] + A[right]
  if sum == S: count++; left++; right--;
  else if sum < S: left++;
  else: right--

Sorting dominates: O(n log n). Two-pointer pass is O(n). Total O(n log n) time, O(1) extra space if sorting in-place.

10. Common Pitfalls and Misconceptions

  • Big‑O gives upper bounds, but people often use it as exact runtime — be precise: say "upper bound" or "Θ" for tight bound.
  • Ignoring constants can be misleading for small n. An O(n) algorithm with large constant may be slower than an O(n log n) one for practical input sizes.
  • Average-case vs worst-case: e.g., quicksort average O(n log n) but worst-case O(n^2) unless randomized or carefully implemented.
  • Misuse of Big‑O for multi-variable functions: use O(n·m) for nested loops over two arrays, not O(n^2) unless m = n.
  • Recurrences require careful base cases and integer rounding; always justify transitions when solving recurrences.

12. Short Proof Examples (Formal)

12.1 Show 3n + 4 = O(n)

We must find constants c, n0 such that 3n + 4 ≤ c·n for all n ≥ n0.

For n ≥ 4, 4 ≤ n, so 3n + 4 ≤ 3n + n = 4n. Choose c = 4 and n0 = 4. Thus 3n + 4 = O(n).

12.2 Show n^2 + 100n = Θ(n^2)

Upper bound: n^2 + 100n ≤ n^2 + 100n^2 = 101n^2 for n ≥ 1. So O(n^2) with constant 101.

Lower bound: n^2 + 100n ≥ n^2 for n ≥ 1, so Ω(n^2). Combine to get Θ(n^2).

13. When to Use Which Notation

  • O(g(n)): when you want an upper bound on the growth rate (very common).
  • Θ(g(n)): when you can tightly bound runtime above and below (preferred if available).
  • Ω(g(n)): when you want a lower bound (less common in algorithm descriptions alone).

14. Study Advice and Next Steps

  • Practice analyzing simple loops and recurrences by counting primitive operations (comparisons, assignments).
  • Learn common patterns: divide-and-conquer, dynamic programming, two-pointer technique, sliding window, hashing.
  • Study Master Theorem applications and practice solving recurrences via substitution and recursion trees.
  • When comparing algorithms, consider both time and space; measure constants for real-world inputs if performance matters.
