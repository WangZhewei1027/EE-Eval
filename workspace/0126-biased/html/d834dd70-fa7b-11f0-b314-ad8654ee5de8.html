Counting Sort — Comprehensive Explanation

A detailed, pedagogical exposition of the Counting Sort algorithm: theory, step-by-step walkthroughs, pseudocode, complexity analysis, variants, limitations, and usage guidance.

1. What is Counting Sort?

Counting Sort is a non-comparative integer sorting algorithm. Instead of comparing elements pairwise (like quicksort or mergesort), it counts the number of objects that have distinct key values (often integers) and uses arithmetic to determine each element's position in the final output. It is particularly efficient when the range of input keys (call it k) is not significantly larger than the number of elements (n).

Key idea: Replace comparisons with counting occurrences and use these counts to place items directly into their sorted positions.

2. When to use Counting Sort?

  • When keys are integers (or can be mapped to non-negative integers) in a reasonably small range: k = maxKey - minKey + 1.
  • When you need a stable sort for integer keys (preserving the relative order of equal keys).
  • As an inner stable sorter in Radix Sort (counting sort is commonly used to sort digits).

Not a good choice if keys are arbitrary large integers or if the range k is huge compared to n (memory becomes prohibitive).

3. Core idea and intuition

Suppose you know how many elements are equal to 0, how many are 1, how many are 2, and so on. If you accumulate these counts you know, for each key value v, the index range in the final sorted array where all elements equal to v should appear. Using this information, you can write each element directly into its final position.

The algorithm typically proceeds in three conceptual phases:

  1. Count occurrences of each key value into an array C.
  2. Transform C into a cumulative count array so that C[v] gives the number of elements ≤ v (or the end position for v).
  3. Place items into an output array B using C to determine positions, decrementing counts as you place elements.

4. Pseudocode (stable counting sort)

CountingSort(A, k) // A is input array of integers in [0..k] let B = new array, same length as A let C = array of zeros of length k+1 // 1) Count occurrences for j from 0 to A.length-1: C[A[j]] = C[A[j]] + 1 // 2) Compute cumulative counts for i from 1 to k: C[i] = C[i] + C[i-1] // 3) Place elements in output array (stable: traverse A from right to left) for j from A.length-1 downto 0: B[ C[A[j]] - 1 ] = A[j] C[A[j]] = C[A[j]] - 1 return B

Notes: The pseudocode above assumes zero-based arrays. Cumulative counts are used so C[v] stores the number of items ≤ v; subtract 1 to get a zero-based index.

5. Step-by-step worked example (stable Counting Sort)

We will sort the array A = [4, 2, 2, 8, 3, 3, 1] using Counting Sort. Keys are in the range 1..8 (k = 8).

  1. Initialize count array C of size k+1 = 9 with zeros: C = [0,0,0,0,0,0,0,0,0]
  2. Count occurrences by scanning A left-to-right: After processing all elements we obtain: C = [0, 1, 2, 2, 1, 0, 0, 0, 1] (index 0 through 8; C[1] = 1 (one '1'), C[2] = 2 (two '2's), etc.)
  3. Compute cumulative counts so each C[i] becomes number of elements ≤ i: Starting from i=1 to 8: C = [0, 1, 3, 5, 6, 6, 6, 6, 7] Now C[2] = 3 (so there are three elements ≤ 2); C[3] = 5, etc.
  4. Build output B by scanning A right-to-left (this preserves stability): - Start with an empty B of length 7. Process elements from rightmost to left: A[6]=1 → place at index C[1]-1 = 0 → B[0]=1; decrement C[1] → C[1]=0 A[5]=3 → place at index C[3]-1 = 4 → B[4]=3; decrement C[3] → C[3]=4 A[4]=3 → place at index C[3]-1 = 3 → B[3]=3; decrement C[3] → C[3]=3 A[3]=8 → place at index C[8]-1 = 6 → B[6]=8; decrement C[8] → C[8]=6 A[2]=2 → place at index C[2]-1 = 2 → B[2]=2; decrement C[2] → C[2]=2 A[1]=2 → place at index C[2]-1 = 1 → B[1]=2; decrement C[2] → C[2]=1 A[0]=4 → place at index C[4]-1 = 5 → B[5]=4; decrement C[4] → C[4]=5 Final B = [1, 2, 2, 3, 3, 4, 8]

This demonstrates a stable outcome: when two equal keys (e.g., the two '3's or the two '2's) appear, their relative order in the output is the same as in the input because we traverse A from right to left while using cumulative counts.

6. Complexity analysis

Let n be the number of elements and k be the range of keys (k = maxKey - minKey + 1).

  • Time complexity: O(n + k). - Counting occurrences: O(n). - Prefix-sum over C: O(k). - Placing elements into output: O(n).
  • Space complexity: O(n + k). - Output array B requires O(n). - Count array C requires O(k).

When k = O(n) the algorithm runs in linear time O(n). If k is very large compared to n, counting sort can be inefficient or infeasible due to memory use.

7. Correctness and stability

Correctness is ensured because cumulative counts identify exact index ranges for each key, and placing each element into the slot determined by C guarantees the correct number of smaller keys are before it.

Stability (preserving relative order of equal keys): Achieved by iterating the input array from right to left when placing elements. The last occurrence of a key ends up at the last position reserved for that key; the earlier occurrences then fill earlier positions—this retains input order among equals.

8. Handling variants & practical considerations

8.1 Negative integers

Counting sort as described expects non-negative integers. For arrays containing negative integers, shift the keys by an offset: compute minKey = minimum(A). Use shiftedKey = A[i] - minKey so keys start at 0. After counting and reconstructing, shift back if needed. This increases k to (maxKey - minKey + 1) but the algorithm is otherwise identical.

8.2 Large key ranges

If k is much larger than n, you may waste memory. Alternatives in that case:

  • Use comparison-based sorts (O(n log n)) like mergesort or quicksort.
  • Compress keys by mapping only used values to a dense range (coordinate compression) if the distinct key count is small.
  • Use a hash/dictionary to count frequencies and then iterate keys in sorted order (this still needs sorting of distinct keys or a tree map).

8.3 In-place counting sort?

The standard stable counting sort is not in-place because it needs an output array of size n and an auxiliary count array of size k. There are specialized variants that can reduce extra memory at the cost of complexity, but they are rarely used in practice.

8.4 Counting sort for non-integers

For values that are not integers, you can map them to integer ranks (coordinate compression) if you only care about relative order. For example, map distinct floating values to 0..m-1 by sorting the distinct set, then use counting sort.

9. Use cases and examples

  • Sorting exam scores (0–100), ages, or other small-range integer data.
  • Radix Sort: Counting sort is used as a stable subroutine to sort by digits.
  • Histogram construction and frequency analysis.
  • Counting sort is useful in GPU/parallel implementations because counting can be parallelized.

10. Implementation notes and optimizations

  • Use an integer array (typed arrays in JavaScript) if available for performance and memory predictability.
  • If the input contains few distinct values relative to k, compress keys to reduce k before counting.
  • When k is small and counts fit in cache, counting sort is extremely fast due to low overhead and simple loops.
  • To minimize memory, you may reuse input array if stability isn't required (but the typical stable algorithm uses a separate output array).

12. Further remarks (theory & extensions)

- Counting sort demonstrates a broader algorithm-design principle: when keys have structure (e.g., small integer range), exploit that structure to avoid comparisons and achieve linear time.

- Counting sort can be extended and combined with other techniques: stable counting for digits (radix), parallel counting with prefix-sum algorithms, or external-memory variants that stream inputs and accumulate counts in blocks.

- A formal correctness proof typically shows that after placing elements using cumulative counts, every element ends up in the right position and the order among equal keys is preserved (if right-to-left placement is used).
