Binary Search — A Comprehensive Written Explanation

A detailed, step-by-step guide to understanding the binary search algorithm, its theory, variants, correctness, and common pitfalls.

What is Binary Search?

Binary search is an efficient algorithm for finding an element's position (or determining absence) in a sorted sequence. It repeatedly divides the search interval in half. If the value of the search key is less than the item in the middle of the interval, the algorithm continues on the lower half; otherwise, it continues on the upper half. Because each step halves the search space, the algorithm is logarithmic in time.

Key idea: Reduce the problem size quickly by using the sorted order. Each comparison eliminates roughly half of the remaining candidates.

When can you use binary search?

  • The data (array, list, or function) must have an ordering: typically sorted in non-decreasing (increasing) order.
  • You must be able to compare an element with the target (less than, greater than, equal to).
  • For variants applied to functions, you need monotonicity (the function changes sign or moves monotonically).

Intuition and Visual Metaphor

Imagine a phone book sorted by last name. To find "Smith", you don't check every entry. Instead you open roughly in the middle, check whether it's before or after "Smith", and then continue searching in the appropriate half. Repeat until you find "Smith" or the interval becomes empty.

The core effect is exponential shrinkage of the search space: after k comparisons the remaining candidates are at most n / 2^k. To reduce n candidates to 1 requires about log2(n) comparisons.

Basic Algorithm (Iterative)

Here's a concise description of the standard iterative algorithm for searching an array A of length n for a target T.

function binary_search(A, T):
    low = 0
    high = n - 1
    while low <= high:
        mid = low + (high - low) // 2   // prevents overflow in some languages
        if A[mid] == T:
            return mid     // found, return index
        else if A[mid] < T:
            low = mid + 1 // search right half
        else:
            high = mid - 1 // search left half
    return -1 // not found

Notes: low and high are inclusive bounds. Loop stops when the interval becomes empty (low > high).

Recursive Version (equivalent)

function binary_search_recursive(A, T, low, high):
    if low > high:
        return -1
    mid = low + (high - low) // 2
    if A[mid] == T:
        return mid
    else if A[mid] < T:
        return binary_search_recursive(A, T, mid + 1, high)
    else:
        return binary_search_recursive(A, T, low, mid - 1)

Both iterative and recursive versions maintain the invariant that if the target exists, it lies within indices [low, high]. The recursive calls shrink that interval by at least one element on each call.

Complexity Analysis

  • Time complexity: O(log n) comparisons in the worst case (base-2 logarithm). Because the interval halves each time, after k steps the remaining interval size is at most n / 2^k. Solve for k so that the interval size < 1: k ≈ log2(n) + 1.
  • Space complexity: O(1) iterative, O(log n) due to recursion depth if using the recursive version (each recursive call uses stack).

Why logarithmic?

Because binary search reduces the number of candidates multiplicatively (dividing by 2 each iteration). This results in a logarithmic number of iterations relative to the initial size.

Loop Invariants and Correctness Sketch

A clear invariant: at the start of each loop iteration, if the target exists in the array, then its index lies somewhere in the closed interval [low, high].

Proof sketch of correctness:

  1. Initialization: Before first loop, low = 0 and high = n - 1, so if the target is in the array it must be in [low, high].
  2. Maintenance: If the invariant holds at the start of an iteration, we compute mid. If A[mid] == T we return mid (correct). If A[mid] < T then all indices ≤ mid are too small, so the target (if present) must be in [mid+1, high]; we set low = mid+1 and the invariant holds. The symmetric argument holds when A[mid] > T.
  3. Termination: The interval size strictly decreases in each iteration (either low increases or high decreases). Eventually low > high and the loop stops. At that point the invariant implies the target cannot be present in any index because the candidate interval is empty. So returning "not found" is correct.

Common Pitfalls and How to Avoid Them

1. Off-by-one errors

Carefully define whether low/high are inclusive or exclusive. The algorithms above use inclusive bounds and the loop condition low <= high. Another common style uses half-open intervals [low, high), then the loop condition becomes low < high and updates differ slightly (e.g., high = mid vs high = mid - 1).

2. Integer overflow when computing mid

If indices are large integers, computing mid as (low + high) / 2 can overflow in some languages. Use mid = low + (high - low) // 2 or language-specific safe operations.

3. Incorrect handling of duplicates

If there are multiple equal elements, the simple binary search returns an arbitrary one of them (the one found). To find the first or last occurrence, use specialized variants (see below).

4. Using binary search on unsorted data

Binary search requires a sorted (or monotonic) predicate. Applying it to unsorted arrays yields incorrect results.

Variants and Extensions

Finding first/last occurrence (lower_bound / upper_bound)

To find the first index at which A[i] ≥ T (lower_bound), or the first index with A[i] > T (upper_bound), use a modified binary search that moves the boundary even when you find an equal element. 

// Example: lower_bound returns smallest i such that A[i] >= T, or n if no such index
function lower_bound(A, T):
    low = 0
    high = n           // notice high is n (half-open interval [low, high))
    while low < high:
        mid = low + (high - low) // 2
        if A[mid] < T:
            low = mid + 1
        else:
            high = mid
    return low

These forms use half-open intervals and guarantee termination without infinite loops if implemented properly.

Binary search on monotonic functions (bisection method)

Binary search generalizes to finding a root of a continuous monotonic function f(x) by repeatedly bisecting an interval [a, b] where f(a) and f(b) have opposite signs or where the property you're testing is monotonic. This is the bisection method, commonly used in numerical analysis. 

// Find x where f(x) crosses target/threshold using monotonic predicate P(x)
low = a
high = b
repeat until high - low < tolerance:
    mid = (low + high) / 2
    if P(mid) is true:
        high = mid
    else:
        low = mid
return (low + high) / 2

Representative Code Snippets

Below are short implementations in common languages to help bridge the understanding to actual code.

Python (iterative)

def binary_search(A, T):
    low, high = 0, len(A) - 1
    while low <= high:
        mid = low + (high - low) // 2
        if A[mid] == T:
            return mid
        elif A[mid] < T:
            low = mid + 1
        else:
            high = mid - 1
    return -1

Java (iterative, safe mid)

int binarySearch(int[] A, int T) {
    int low = 0, high = A.length - 1;
    while (low <= high) {
        int mid = low + (high - low) / 2;
        if (A[mid] == T) return mid;
        else if (A[mid] < T) low = mid + 1;
        else high = mid - 1;
    }
    return -1;
}

Notes on library functions

Many languages provide library binary-search functions (e.g., C++ std::lower_bound, Java Arrays.binarySearch). Understand their contract and return value conventions (some return negative insertion points when not found).

Summary and Best Practices

  • Use binary search whenever you can reduce a sorted/monotonic search space by half at each comparison — it is fast: O(log n).
  • Pick a clear indexing convention (inclusive vs half-open) and stick with it to avoid off-by-one bugs.
  • Use mid = low + (high - low) // 2 to avoid overflow in languages with limited integer ranges.
  • When duplicates matter, implement lower_bound and upper_bound variants.
  • Proofs of correctness rely on maintaining a suitable loop invariant and on the fact that each iteration strictly reduces the search interval.

If you want to practice, write the algorithm by hand for small arrays and trace each iteration. That is the clearest way to internalize the invariants and avoid off-by-one errors.
