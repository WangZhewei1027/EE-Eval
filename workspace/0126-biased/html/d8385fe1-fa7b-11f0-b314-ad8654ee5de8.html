B-Tree Index — Comprehensive Explanation

A focused, textual, pedagogical guide to B-Tree indexes as used in databases and file systems. This document explains structure, algorithms (search, insert, delete), performance reasoning, practical tuning, and common implementation notes.

Overview and motivation

A B-Tree is a balanced tree data structure designed for systems that read and write large blocks of data (e.g., disks, SSDs, or other paged storage). In database systems, B-Trees (and closely related variants such as B+Trees) are the standard index structure because they minimize disk I/O by having high branching factors: each node corresponds to a page that contains many keys and child pointers.

Key goals of a B-Tree index:

  • Support search, insertion, and deletion in O(log n) disk accesses.
  • Keep tree height low by maximizing fan-out (many children per internal node).
  • Maintain balance automatically so each leaf is at the same depth.
  • Allow efficient range scans (especially in B+Tree form).

Definitions and invariants

There are two common ways to parameterize a B-Tree:

  • Order m: each node has at most m children and at most m−1 keys. Every node except the root has at least ceil(m/2) children (hence at least ceil(m/2)−1 keys).
  • Minimum degree t (CLRS style): each node except root has between t−1 and 2t−1 keys, and between t and 2t children. Here t = ceil(m/2).

Common invariants:

  • All leaves are at the same depth (tree is height-balanced).
  • Keys within a node are stored in sorted order.
  • An internal node with k keys has k+1 children. Child i contains keys between key i−1 and key i (conceptually).
  • Root is a special case: it may have fewer keys/children than the minimum for internal nodes.

Node layout and page considerations

In database implementations, a B-Tree node is typically one disk page. Inside the page:

  • Header with metadata (isLeaf flag, key count, pointers to siblings for leaves, etc.).
  • Array of keys and array of pointers (or RIDs) to children/records.
  • Sometimes a prefix-compressed key representation to save space.

Practical tip: choose node size close to the storage page size (e.g., 8 KB) and compute the fanout from average key size + pointer size. Higher fanout reduces height and disk I/O.

Example fanout calculation (approximate):

page_size = 8192 bytes
avg_key_size = 20 bytes
pointer_size = 8 bytes
overhead ≈ 128 bytes

available ≈ page_size - overhead = 8064
entry_size ≈ avg_key_size + pointer_size = 28
fanout ≈ floor(available / entry_size) ≈ 288
So each node may have ≈ 288 keys (and ≈ 289 children).

Search algorithm (conceptual)

Searching in a B-Tree follows the same pattern as in a binary search tree but with multi-way branching within nodes:

  1. Start at the root node.
  2. Within the node, perform a (binary) search over the keys stored in the node to find the interval where the search key belongs.
  3. If the key equals a node key, return the record (if the node stores records) or follow the convention used (in B+Trees, leaf stores records).
  4. If the node is a leaf and key not found, the search fails.
  5. Otherwise, follow the child pointer for the chosen interval and repeat.

Complexity: O(h * log B) CPU comparisons where h is height and B is max keys per node. In disk-I/O terms: O(h) page reads (the dominant cost).

PSEUDOCODE (search)
BTree-Search(node, k):
  i = smallest index such that k ≤ node.key[i]  (binary search)
  if i ≤ node.n and k == node.key[i]:
    return (node, i)   // found
  if node.isLeaf:
    return NOT_FOUND
  else:
    return BTree-Search(node.child[i], k)

Insertion — high-level strategy

The canonical insertion strategy keeps nodes from becoming overfull by splitting full nodes on the way down:

  1. If the root is full, split it and increase tree height by creating a new root with two children.
  2. Descend from the root toward the appropriate leaf, ensuring that when you recurse into a child it is not full: if it is full, split it first (propagate the middle key up and reorganize children).
  3. When you reach a non-full leaf, insert the key into the leaf in sorted order.

The approach "split on the way down" (CLRS) guarantees that you never need to split after inserting — the node you insert into will be non-full. 

PSEUDOCODE (insert, CLRS style)
BTree-Insert(T, k):
  r = T.root
  if r.n == 2t-1:
    s = allocate-node()
    T.root = s
    s.isLeaf = false
    s.n = 0
    s.child[1] = r
    BTree-Split-Child(s, 1)
    BTree-Insert-Nonfull(s, k)
  else:
    BTree-Insert-Nonfull(r, k)

BTree-Insert-Nonfull(x, k):
  if x.isLeaf:
    insert k into x.keys in sorted order
    x.n = x.n + 1
  else:
    i = x.n
    while i ≥ 1 and k < x.key[i]:
      i = i - 1
    i = i + 1
    if x.child[i].n == 2t-1:
      BTree-Split-Child(x, i)
      if k > x.key[i]:
        i = i + 1
    BTree-Insert-Nonfull(x.child[i], k)

Important: the split operation moves a middle key up and creates two nodes from one full node. For B+Trees, the middle key behavior differs (keys are duplicated into the parent and leaves retain records).

Deletion — overview and edge cases

Deletion in a B-Tree is more complex because it must preserve the minimum-occupancy invariant (each non-root node must have at least t−1 keys). There are a few main cases when deleting a key k from node x:

  1. k is in a leaf node: simply delete it (if the node still satisfies min keys) or perform rebalancing (borrow from sibling or merge) if underflow occurs.
  2. k is in an internal node: replace k by a predecessor (largest key in left subtree) or successor (smallest key in right subtree) and then delete that key recursively (which reduces it to the previous case).
  3. If the child that should contain the key has only t−1 keys, ensure before recursing that it has at least t keys by either borrowing from an adjacent sibling with ≥ t keys, or by merging the child with a sibling and moving a key down from the parent. Then continue.

Because deletion has many cases, careful implementation and rigorous testing are critical.

PSEUDOCODE (high-level)
BTree-Delete(T, k):
  delete key k starting at root, applying these principles:
    - If deleting from a node would cause it to have fewer than t-1 keys,
      fix the node beforehand by rotation (borrow) or merge so the recursive call
      always has enough keys to proceed.
  ... (see textbooks/CLRS for fully detailed pseudocode) ...

Note: In practice many systems prefer B+Trees which push complexity into leaf-level handling and make range scans easier. Deleting from B+Trees typically affects only leaves and their sibling pointers.

Split right child: median 20 moves up into root: Root becomes [10 | 20] children: leftmost [5|6|7], middle [12], right [30] Now 17 compares: 10 < 17 < 20 → go to middle [12], insert => [12 | 17] Final tree: [10 | 20] / | \ [5|6|7] [12|17] [30]

This example shows splitting of leaf nodes and movement of median keys up to the parent. More complex splits occur when parents are full and must themselves be split (potentially increasing height).

B-Tree vs B+Tree (practical differences)

The B+Tree is the most common variant used in databases:

  • In a B+Tree, all data records (or pointers to them) reside in the leaf nodes. Internal nodes store only keys to guide search.
  • Leaves are typically linked with sibling pointers to support efficient ordered range scans (sequential traversal across leaves).
  • Because internal nodes contain only keys and pointers, fanout is higher, reducing height further.

Pros of B+Tree:

  • Better range query performance (leaf chaining).
  • Uniform leaf-level layout simplifies scans and bulk reads.
  • Higher fanout reduces tree height.

The rest of the document uses the phrase "B-Tree" to mean the general balanced multi-way tree; many implementation details map directly to B+Tree variants.

Complexity and I/O cost

Let n be the number of keys and B be the average number of keys per node (fanout). The height h satisfies roughly: 

h ≈ log_{B} n

Each search or update requires O(h) page (node) reads; since B is large, h is small in practice (often 2–4 for large databases).

CPU cost per node: searching within a node takes O(log B) if you use binary search or O(B) for linear scan. Binary search is common and cheap since B is moderate and fits in cache for typical implementations.

Practical implementation notes

  • Page/Node locking and concurrency: use lock-coupling or latching strategies (crabbing) for safe concurrent modifications. B-Link trees add right-sibling pointers and logical sequence numbers to make concurrency simpler.
  • Write-Ahead Logging (WAL): SPLIT and MERGE operations must be made durable through logging; many systems log high-level logical operations rather than every page change when possible.
  • Bulk loading: to build a large index quickly, sort keys and pack leaves densely then build the upper levels bottom-up — this is much faster than inserting one-by-one.
  • Duplicates: duplicates can be handled by adding RID lists in leaves or by storing (key, RID) pairs as logical keys to make them unique.
  • Clustered vs Non-clustered index: A clustered index means the table's rows are stored in leaf order of the index (physically ordered), improving range query locality but expensive to maintain on random inserts.

Tuning and heuristics

Choose node/page size to match storage system page size; estimate average key size including pointer overhead; aim for 60–80% occupancy after normal operation to balance space utilization and split frequency.

For write-heavy workloads, consider:

  • Using larger pages to reduce height and IO.
  • Allowing some slack (lower fill factor) to reduce split frequency.
  • Consider log-structured storage or LSM trees for write-intensive workloads (trade-off: range-scan cost).

Advanced topics (brief)

  • Prefix compression: store only distinguishing suffixes of keys within nodes to save space and increase fanout.
  • B* Trees: attempt to keep nodes fuller by redistributing keys between siblings before splitting, often requiring 2-node merges or 3-way redistribution.
  • Variable-length keys: handle by packing keys and keeping offsets to entries; be mindful of worst-case node utilization.
  • B-Link trees: add right sibling pointers and a sequence number to nodes to simplify concurrent splits and searching while updates occur.

Common pitfalls and FAQs

  • Q: Are B-Trees always better than hash indexes? A: No. B-Trees support ordered traversal and range queries. Hash indexes are better for equality-only lookups and can be simpler/cheaper for that narrow case.
  • Q: Why not simply use a binary tree? A: Binary trees have height O(n) without balancing; balanced binary trees have more pointers per node and worse locality for disk-based storage. B-Trees are designed for block-based IO.
  • Q: Is implementing deletion difficult? A: Deletion is the trickiest part because of many rebalancing cases; many production systems rely on tested libraries or use variants (like B+Trees) that simplify some aspects.
