Relational Database — Comprehensive Explanations, Theory, and Educational Guide

A detailed, text-first reference covering the relational model, algebra, calculus, normalization, transactions, indexing, query processing, and practical guidance.

1. Overview and Definition

A relational database is a structured collection of data organized according to the relational model, introduced by E. F. Codd in 1970. Data is stored in relations (commonly called tables). Each relation is a set of tuples (rows), and each tuple is an ordered set of attributes (columns). Relations are defined by schemas that enumerate attributes and their domains (data types).

Key ideas: relations (tables), tuples (rows), attributes (columns), keys (unique identifiers), integrity constraints, and operations (SQL, relational algebra).

2. Historical Context

The relational model simplified data independence and logical querying compared to earlier hierarchical or network database models. SQL became the de facto standard for relational DBMSs (RDBMS), first standardized in the 1980s. Modern RDBMSs implement many optimizations, concurrency controls, recovery mechanisms, and extensions beyond the pure mathematical model.

3. Core Concepts and Terminology

Relation, Tuple, Attribute, Domain

- Relation: a named table with a fixed set of attributes. It corresponds to a mathematical relation: a set of tuples sharing the same attributes. - Tuple: a single row in a table (an ordered set of attribute values). - Attribute: a named column with a domain (the set of allowed values). - Domain: a type or value set (e.g., integers, strings, dates).

Schema vs. Instance

- Schema: the structure of the database (table definitions, attribute types, constraints). - Instance: the data present in the database at a particular moment (the current set of tuples for each relation).

Keys and Integrity Constraints

- Primary Key: an attribute (or set) that uniquely identifies tuples in a relation. - Candidate Key: any minimal set of attributes that can serve as a primary key. - Foreign Key: an attribute in one relation that references a primary key in another relation (enforces referential integrity). - Unique, NOT NULL, CHECK constraints: additional rules ensuring data correctness.

4. Relational Algebra (Theory and Operators)

Relational algebra is a procedural set of operators that manipulate relations. It underpins query execution and optimization.

Main operators (with intuitive descriptions)

  • Selection (σ): filters rows by a predicate. Example: σ_{age > 30}(Employees).
  • Projection (π): selects columns (and eliminates duplicates). Example: π_{name,dept}(Employees).
  • Cartesian product (×): all combinations of rows from two relations.
  • Union (∪) and Set difference (−): set operations requiring union-compatible relations (same attributes).
  • Rename (ρ): renames a relation or attribute(s) used in expressions.
  • Join: combines related tuples from two relations (θ-join, equi-join, natural join). Example: Employees ⨝_{Employees.deptId = Departments.id} Departments.
  • Division (÷): used for queries like "find X such that X is related to all Y in some set".

Example (relational algebra expression)

Find names of employees who work in the 'Sales' department:

π_{name}(σ_{Departments.name='Sales'}(Employees ⨝ Employees.deptId = Departments.id Departments))

5. Relational Calculus

A declarative logic-based approach; not procedural. Two variants:

  • Tuple Relational Calculus (TRC): queries expressed by specifying properties of desired tuples. Example: {t | t ∈ Employees ∧ t.salary > 70000}.
  • Domain Relational Calculus (DRC): variables range over attribute domains rather than full tuples.

Relational calculus inspired SQL's declarative style and is central in reasoning about query expressiveness (e.g., Codd's theorem relates algebra and calculus).

6. SQL: Practical Language Overview

SQL (Structured Query Language) includes sublanguages:

  • DDL (Data Definition Language): CREATE TABLE, ALTER TABLE, DROP TABLE.
  • DML (Data Manipulation Language): SELECT, INSERT, UPDATE, DELETE.
  • DCL (Data Control Language): GRANT, REVOKE.
  • TCL (Transaction Control): BEGIN/COMMIT/ROLLBACK.

Typical examples

-- Create a table
CREATE TABLE Customers (
  customer_id INT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  address TEXT
);

-- Query
SELECT c.name, o.order_id, o.order_date
FROM Customers c
JOIN Orders o ON c.customer_id = o.customer_id
WHERE o.order_date > '2024-01-01';

7. Normalization (Theory and Step-by-Step Example)

Normalization organizes relations to reduce redundancy, anomalies (insert/update/delete), and to express functional dependencies (FDs).

Core normal forms

  • 1NF: atomic attribute values; no repeating groups (every attribute value is atomic).
  • 2NF: 1NF + no partial dependency of a non-key attribute on part of a composite key.
  • 3NF: 2NF + no transitive dependency of non-key attributes on the primary key.
  • BCNF: For every non-trivial FD X → Y, X is a superkey (stronger than 3NF).
  • 4NF, 5NF: address multi-valued dependencies and join dependencies respectively (advanced).

Step-by-step normalization example (orders)

Start with a single unnormalized table capturing orders and items (simplified):

OrdersRaw(OrderID, CustomerName, CustomerAddress, ItemID, ItemName, Quantity, OrderDate, PaymentMethod, SalesRep)

Problems and FDs you might observe:

  • FD: OrderID → OrderDate, PaymentMethod, SalesRep, CustomerName, CustomerAddress (an order has one date, payment, rep, and customer).
  • FD: ItemID → ItemName (item id determines name).
  • Repeating groups: an order can have multiple items, so multiple rows per order duplicate order-level data.

Apply 1NF: ensure atomic values (no nested lists). Next decompose to eliminate redundancy:

Orders(OrderID PK, OrderDate, PaymentMethod, SalesRep, CustomerID FK)
OrderItems(OrderID FK, ItemID FK, Quantity, PRIMARY KEY (OrderID, ItemID))
Customers(CustomerID PK, CustomerName, CustomerAddress)
Items(ItemID PK, ItemName)

This decomposition addresses partial and transitive dependencies and generally leads to 3NF. Verify FDs and lossless join + dependency preservation when decomposing (key concerns in theory).

8. Query Processing, Optimization, and Execution

From SQL to result: parsing, planning, optimization, and execution with a cost-based optimizer.

Phases

  1. Parsing and semantic analysis: build a parse tree; type check; validate schema objects.
  2. Logical plan generation: convert query into relational algebra operators.
  3. Optimization:
    • Rule-based rewrites (push down selections, remove redundancies).
    • Cost-based optimization: estimate costs using statistics (table cardinality, column histograms, index selectivity).
    • Join ordering optimization: dynamic programming (Selinger) or heuristic/greedy approaches for large joins.
  4. Physical plan selection: choose join algorithms, index usage, access paths, parallelization strategy.
  5. Execution: run the plan using iterators, pipelines, vectorized execution, etc.

Join algorithms (conceptual steps)

  • Nested loop join: for each tuple in outer, scan inner (naive; optimized with indexing).
  • Index nested loop: use index on inner to find matching tuples quickly for each outer tuple.
  • Sort-merge join: sort both inputs on join key (or exploit existing order), then merge.
  • Hash join: build hash table on smaller relation using join key, probe with other relation (works well for equijoins).

Cost estimation basics

Estimators use cardinality (row counts) and selectivity (fraction passing predicates). Costs model I/O (pages read/written), CPU for processing, and network/communication for distributed systems.

9. Indexing and Physical Storage

Indexes speed up lookups by maintaining auxiliary structures.

Common index structures

  • B+ Tree: balanced tree optimized for range scans and disk pages; nodes correspond to disk pages; leaf level contains pointers to records.
  • Hash index: constant-time lookup for equality; not suitable for range queries in basic form.
  • Composite index: multi-column index where ordering matters for prefix usage.
  • Covering index: index that contains all columns needed by a query, avoiding a table lookup.

Storage primitives

Data is stored in pages/blocks. A record may be fixed-length or variable-length. Systems use mechanisms like slotted pages, page headers, and tuple identifiers (TIDs).

10. Transactions, Concurrency, and Recovery

Transactions provide correctness and durability guarantees across concurrent operations and failures.

ACID properties

  • Atomicity: All or nothing: a transaction either fully commits or has no effect.
  • Consistency: Transactions move database from one valid state to another, preserving constraints.
  • Isolation: Concurrent transactions do not interfere; isolation levels determine allowed anomalies.
  • Durability: Once committed, effects survive crashes (via logging/checkpoints).

Isolation levels and phenomena

  • Read Uncommitted: may see uncommitted changes (dirty read).
  • Read Committed: avoids dirty reads, but non-repeatable reads and phantoms possible.
  • Repeatable Read: prevents non-repeatable reads, but some systems still allow phantoms.
  • Serializable: strictest; transactions appear to run in serial order.

Concurrency control techniques

  • Two-Phase Locking (2PL): acquire locks (shared/exclusive) then release after commit; guarantees serializability if strict 2PL used.
  • Multi-Version Concurrency Control (MVCC): maintain multiple versions of data so readers don't block writers; readers see a snapshot.
  • Optimistic concurrency: validate at commit time; suitable when conflicts are rare.

Recovery (brief)

Most RDBMSs use Write-Ahead Logging (WAL): before a data page is written, the log must record the change (so crash recovery can redo/undo). Checkpoints periodically shorten recovery time. ARIES is a widely used recovery algorithm with repeating history and physiological redo.

11. Database Design Methodology

Entity-Relationship (ER) Modeling

Start with ER diagrams: entities, relationships, cardinalities (one-to-one, one-to-many, many-to-many), and attributes. Map to relations:

  • Entity → table with primary key.
  • One-to-many: foreign key in the "many" table referencing the "one".
  • Many-to-many: introduce a junction (associative) table with foreign keys to both entities.

Design checklist

  • Identify entities and relationships.
  • List attributes and domains; choose primary keys.
  • Specify foreign keys and integrity constraints.
  • Normalize to at least 3NF unless denormalization is justified for performance.
  • Consider indexing and partitioning strategies for large tables.

12. Advanced and Practical Topics

  • Views: virtual tables defined by queries; can simplify access and provide security layers.
  • Materialized Views: cached results that require refresh strategies.
  • Triggers: procedural code executed on data changes — powerful but can complicate reasoning and performance.
  • Stored Procedures: encapsulated business logic within the database.
  • Partitioning & Sharding: split large tables horizontally for manageability and performance; sharding distributes data across nodes.

13. Practical Examples and Guided Walkthroughs

Example: Simple bookstore schema

Authors(author_id PK, name)
Books(book_id PK, title, isbn, pub_year)
BookAuthors(book_id FK, author_id FK, PRIMARY KEY(book_id, author_id))
Sales(sale_id PK, book_id FK, sale_date, quantity, price)

Common queries

Find best-selling books (by total quantity):

SELECT b.title, SUM(s.quantity) AS total_sold
FROM Books b
JOIN Sales s ON b.book_id = s.book_id
GROUP BY b.title
ORDER BY total_sold DESC;

Step: From requirement to relation

  1. Gather requirements (entities: Book, Author, Sale).
  2. Define attributes and keys.
  3. Model relationships (Book-Author many-to-many → BookAuthors).
  4. Normalize and choose indexes on book_id, author_id, and sale_date for common queries.

14. Algorithms and Patterns (Concise Descriptions)

Selinger-style join ordering (dynamic programming)

The optimizer considers small subsets of relations to find cheapest plans for them, then combines plans to form larger join orders. Cost functions combine I/O and CPU estimates. This approach is exponential in worst case but effective for moderate numbers of joins.

Hash join algorithm (high level)

  1. Choose smaller input as build side.
  2. Partition build side into buckets using a hash function on join key; store in memory (or spill to disk if too large).
  3. For each tuple in probe side, hash join key and probe corresponding bucket to find matches.

Sort-merge join (high level)

  1. Sort both relations on join key (or use existing sorted order).
  2. Scan both relations with two pointers and emit matched pairs while advancing pointers appropriately.

15. Common Pitfalls and Best Practices

  • Avoid premature optimization: normalize design first for correctness; denormalize intentionally for measured performance reasons.
  • Use appropriate indexing: too many indexes slow writes; missing indexes slow reads.
  • Choose proper data types and sizes to reduce storage and improve cache behavior.
  • Monitor statistics and vacuum/maintenance to keep optimizer estimates accurate.
  • Prefer set-oriented operations over row-by-row processing where possible.
