Time Complexity — A Comprehensive Textual Guide

Definitions, theory, step-by-step analyses, examples, and practical advice — focused on readable explanations rather than interactive experiments.

1. What is Time Complexity?

Time complexity describes how the running time (or number of basic operations) of an algorithm grows as the size of the input grows. It abstracts away machine-specific details and constant factors to focus on growth trends as input size n becomes large.

Key idea

We model the algorithm by a function f(n) that maps input size n to the number of operations. Complexity classes (Big O, Theta, Omega) describe the asymptotic behavior of f(n) as n → ∞.

Why it matters

  • Predicts performance for large inputs.
  • Helps compare algorithms independently of hardware.
  • Guides data structure and algorithm choices in real systems.

2. Asymptotic Notation — Formal Definitions

Big O (upper bound)

We say f(n) = O(g(n)) if there exist positive constants c and n0 such that for all n ≥ n0: f(n) ≤ c · g(n).

Informally: f grows no faster than g (up to constant factors) for large n.

Big Ω (lower bound)

f(n) = Ω(g(n)) if there exist positive constants c and n0 such that for all n ≥ n0: f(n) ≥ c · g(n).

Informally: g is a lower bound on the growth of f.

Big Θ (tight bound)

f(n) = Θ(g(n)) if f(n) = O(g(n)) and f(n) = Ω(g(n)). That is, g(n) tightly describes f(n) up to constant factors.

Example (simple proof)

Show that 5n^2 + 3n + 7 = O(n^2).

Choose c = 6 and n0 = 1. For n ≥ 1, 5n^2 + 3n + 7 ≤ 5n^2 + 3n^2 + 7n^2 = 15n^2. (A tighter bound is possible: choose c = 15.) Thus the polynomial is O(n^2).

3. Common Growth Classes (from fastest to slowest practical)

  • O(1) — constant time
  • O(log n) — logarithmic
  • O(n) — linear
  • O(n log n) — linearithmic
  • O(n^2) — quadratic
  • O(n^3), O(n^k) — polynomial
  • O(2^n) — exponential
  • O(n!) — factorial

What each means, in words

- O(1): Running time does not increase (significantly) with n; e.g., accessing array[5].
- O(log n): Doubling the input increases time by a constant; typical of binary search and many divide-and-conquer index lookups.
- O(n): Time grows proportionally with input size; e.g., scanning an array once.
- O(n log n): Common for efficient comparison-based sorting (merge, heapsort).
- O(n^2): Time grows quadratically; typical of nested loops over the same data (some naive sorting).
- Exponential/factorial: Growth so fast that even moderate n becomes infeasible.

4. How to Analyze — Step-by-Step Textual Recipe

  1. Identify the basic operation to count (like comparisons or assignments).
  2. Express the number of times that operation executes as a function of n.
  3. Simplify the function: drop lower-order terms and constant factors for asymptotic upper bound.
  4. If recursion appears, form a recurrence and solve it (recursion tree, Master theorem, substitution).
  5. State best-case, worst-case, and average-case if they differ.

Example: simple loop

// Pseudocode
sum = 0
for i = 1 to n:
    sum = sum + A[i]

Count: the loop body runs n times. Each iteration does a constant amount of work (addition and indexing). So total time = c·n + d = Θ(n).

Example: nested loops

// Pseudocode
for i = 1 to n:
    for j = 1 to n:
        do constant work

Inner body executes n·n = n^2 times. So time = Θ(n^2).

Example: triangular nested loops

// Pseudocode
for i = 1 to n:
    for j = i+1 to n:
        do constant work

Number of iterations = sum_{i=1}^{n} (n-i) = n(n-1)/2 = Θ(n^2). Even though the loops are not symmetric, growth is quadratic.

5. Recurrence Relations and Divide-and-Conquer

Many algorithms split a problem into subproblems. Their time is often captured by a recurrence of the form:

T(n) = a · T(n/b) + f(n)

Interpretation:

  • a — number of subproblems
  • b — factor by which problem size reduces
  • f(n) — cost of dividing/merging or non-recursive work

Master Theorem (useful rule of thumb)

Let T(n) = a T(n/b) + f(n) with a ≥ 1, b > 1. Compare f(n) to n^{log_b a}:

  • If f(n) = O(n^{log_b a - ε}) for some ε > 0, then T(n) = Θ(n^{log_b a}).
  • If f(n) = Θ(n^{log_b a} · log^k n) for some k ≥ 0, then T(n) = Θ(n^{log_b a} · log^{k+1} n).
  • If f(n) = Ω(n^{log_b a + ε}) and regularity condition holds, then T(n) = Θ(f(n)).

Example: Merge Sort

Recurrence: T(n) = 2 T(n/2) + Θ(n). Here a = 2, b = 2, n^{log_b a} = n^{1} = n, f(n) = Θ(n) so we are in case 2 ⇒ T(n) = Θ(n log n).

Example: Binary Search

Recurrence: T(n) = T(n/2) + Θ(1). Here a = 1, b = 2, n^{log_b a} = n^0 = 1, f(n) = Θ(1) ⇒ T(n) = Θ(log n).

6. Best, Worst, and Average Case

Complexities can differ depending on the input:

  • Worst-case: upper bound on running time for any input of size n.
  • Best-case: lower bound — often not useful alone.
  • Average-case: expected running time under some distribution of inputs.

Example: Quicksort

Worst-case: O(n^2) (if pivot always picks worst partition).
Average (randomized pivot): O(n log n).
Practical note: randomized quicksort or careful pivot choice gives near-optimal average behavior.

Example: Hash table operations

Average-case (with low load factor and good hashing): O(1). Worst-case (all keys collide): O(n). In practice, average-case dominates but being aware of worst-case is important for adversarial scenarios.

7. Space Complexity

Space complexity measures extra memory used (beyond input): the size of additional data structures allocated by the algorithm.

  • In-place algorithms: O(1) extra space (modulo recursion stack).
  • Merge sort: Θ(n) additional space for merging (typical implementations).
  • Recursion stack: counts toward space; e.g., quicksort worst-case O(n) stack, average O(log n).

Example: Depth-first search

Using recursion on a tree of height h uses O(h) stack space. In a balanced tree h = Θ(log n). In a degenerate tree, h = Θ(n).

8. Amortized Analysis

Amortized analysis averages the cost of operations over a sequence, smoothing occasional expensive operations across many cheap ones.

Dynamic array resizing example

Operation: append to an array that doubles capacity when full.

  1. Most append operations are O(1) — place element into free slot.
  2. Occasionally, array is full; we allocate a new array of size 2·k and copy k elements — cost O(k).
  3. Amortized cost per append: O(1). Reasoning: each element is copied O(1) times over its lifetime because capacity doubles, so total copies over n inserts is O(n).

9. Practical Examples: Algorithm Analyses (Textual Walkthroughs)

Selection Sort (step-by-step)

// Pseudocode
for i = 1 to n-1:
    minIndex = i
    for j = i+1 to n:
        if A[j] < A[minIndex]:
            minIndex = j
    swap A[i] and A[minIndex]

Analysis: Outer loop runs n-1 times. Inner loop runs (n-1) + (n-2) + ... + 1 = n(n-1)/2 checks ⇒ Θ(n^2) comparisons. Swaps: at most n swaps. So time = Θ(n^2), space = O(1).

Binary Search (step-by-step)

// Pseudocode (on sorted array)
low = 1; high = n
while low ≤ high:
    mid = floor((low + high)/2)
    if A[mid] == target: return mid
    else if A[mid] < target: low = mid + 1
    else: high = mid - 1
return not found

Each iteration halves the search range. Number of iterations ≤ log2 n + 1 ⇒ time = Θ(log n), space = O(1).

Merge Sort (step-by-step)

// Pseudocode
function mergesort(A):
    if length(A) ≤ 1: return A
    mid = length(A) / 2
    left = mergesort(A[1..mid])
    right = mergesort(A[mid+1..end])
    return merge(left, right) // linear-time merge

Recurrence: T(n) = 2T(n/2) + Θ(n) ⇒ T(n) = Θ(n log n). Space: Θ(n) for merging.

10. Rules and Tricks for Simplifying Complexity Expressions

  • Drop constant multipliers: 100·n = O(n).
  • Drop lower-order terms: n^2 + n = O(n^2).
  • If f(n) = O(g(n)) and g(n) = O(h(n)), then f(n) = O(h(n)).
  • Multiplying by logarithms: n log n grows faster than n but slower than n^{1+ε} for any ε>0.
  • When in doubt, count dominant loops and recursion depth to find the leading term.

Dimensionless constants can matter in practice

Asymptotic notation ignores constants; however, for small n or over limited real input sizes, constants, cache behavior, memory allocation, and implementation details can dominate. Use complexity to choose algorithms; use profiling to optimize real systems.

11. Advanced Topics (concise explanations)

Average-case analysis

Requires a probability distribution over inputs. E.g., average-case quicksort assumes all pivot positions equally likely, giving expected Θ(n log n).

Randomized algorithms

Randomization can improve expected running time or simplify analyses. Example: randomized quicksort reduces probability of worst-case input behavior.

Probabilistic bounds and high-probability guarantees

Some algorithms guarantee good running time with high probability rather than deterministically. This is common in hashing and randomized selection.

Cache-aware and cache-oblivious analysis

Time complexity refined to consider memory hierarchy and block transfers (I/O complexity). This affects practical performance for large data.

13. Worked Example — Counting Operations Precisely

Consider a loop with conditionals:

// Pseudocode
count = 0
for i = 1 to n:
    if A[i] is even:
        count = count + 1   // O(1)
    else:
        for j = 1 to 10:
            do constant work

Analysis: Outer loop runs n times. For each i, either O(1) or O(1·10) = O(1) work occurs. Both branches are constant time, so whole loop is Θ(n). Although there's a nested loop, its size is constant 10, so it does not increase asymptotic order.

Lesson

Only loops or recursions that depend on n contribute to asymptotic increases; constant-bounded inner loops remain O(1).
