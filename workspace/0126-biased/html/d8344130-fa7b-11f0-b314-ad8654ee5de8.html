Suffix Tree — Comprehensive Explanation

A deep, educational guide to suffix trees: their definition, structure, construction ideas (including Ukkonen's algorithm), common operations, complexity, practical implementation notes, and pedagogical examples.

What is a suffix tree?

A suffix tree for a string S of length n is a compressed trie (prefix tree) that contains every suffix of S. It is a rooted, directed tree where:

  • Each edge is labeled with a non-empty substring of S.
  • No two edges out of the same node begin with the same character.
  • Every suffix of S corresponds to exactly one path from the root to a leaf.
  • Leaf nodes represent suffixes and are often labeled with the starting index of that suffix in S.

Conventionally, S is terminated by a unique character (often written as '$' or '\0') that does not appear elsewhere in S. This guarantees every suffix ends at a different leaf and simplifies reasoning.

Why suffix trees matter

Suffix trees turn many string problems into tree/traversal problems that can be solved in time proportional to the pattern length rather than the text length. Common problems solved efficiently with suffix trees:

  • Exact substring search: O(m) time for a pattern of length m (plus cost to report occurrences).
  • Longest repeated substring (LRS): find the deepest internal node representing a repeated substring.
  • Longest common substring (LCS) of two strings by building a generalized suffix tree.
  • Finding all occurrences of patterns, computing distinct substrings, and many bioinformatics tasks.

Basic properties

  • Number of leaves equals the number of suffixes (n if S includes a terminal marker).
  • Each internal node (except the root) has at least two children.
  • Using edge-label compression (storing substring intervals), a suffix tree can be represented with O(n) nodes and edges and O(n) space (up to constants).

Edge representation — intervals instead of strings

Practical implementations store each edge label as a pair (start, end) pointing into the original string S. For leaves the end is often a global reference that can change (useful in Ukkonen's algorithm) so that leaf edge ends effectively extend as new characters arrive.

Suffix tree vs suffix trie

A suffix trie is a (non-compressed) trie that stores all suffixes character-by-character. It can have Theta(n^2) nodes for a text of length n. A suffix tree is the compressed version (compress chains of single-child nodes into one edge), reducing the size to O(n) nodes and edges.

The suffix tree groups suffixes by their prefixes. The root will have edges for the distinct first characters { 'b', 'a', 'n', '$' } and the subtrees under 'a' and 'n' will reflect the shared continuations.

How a suffix tree helps find substrings

To check if a pattern P appears in S, start at the root and follow edges whose labels match successive characters of P. Because edge labels may be longer than one character, you compare P against the substring indicated by the edge-interval. If you match all characters of P before running out of P, then P appears — and the leaves in the subtree below the current node/point give the starting positions of occurrences.

Time to search is O(m) where m = |P|, assuming character comparisons are O(1) and that the traversal of an edge label takes time proportional to the number of characters of the pattern matched (not the edge label length beyond what's needed).

Common operations

  • Substring search: traverse from root matching pattern characters along edge labels.
  • Report occurrences: collect leaves under the matched node; these leaves are labeled with suffix start positions.
  • Longest repeated substring: find the internal node of maximum string-depth (sum of edge lengths on path from root).
  • Longest common substring of two strings: build a generalized suffix tree for both strings and find deepest internal node whose subtree contains leaves from both strings.

Constructing suffix trees — overview of approaches

There are several construction strategies:

  • Naive: Insert each suffix into a trie and compress. This is O(n^2) time and O(n^2) space in the worst case (if implemented as a trie without compression).
  • Ukkonen's online algorithm: Builds the suffix tree in O(n) time and O(n) space (amortized) by adding text characters one at a time and maintaining an active point and suffix links.
  • McCreight's algorithm: A linear-time offline algorithm that uses suffix links differently.
  • Suffix array based: Build suffix array + LCP (longest common prefix) array in O(n) time; the suffix tree can be simulated or constructed from these arrays in O(n) time. Suffix arrays are often more space-efficient in practice.

Ukkonen's algorithm — high level explanation

Ukkonen introduced an online, left-to-right algorithm that produces the suffix tree in O(n) time. The key ideas:

  • Process the text S one character at a time in phases. After phase i you have built a suffix tree for S[0..i].
  • Each phase performs a number of "extensions" that ensure all suffixes ending at the new character are present.
  • Use an active point (active node, active edge, active length) to remember where the next extension should start; this avoids rescanning from the root each time.
  • Use suffix links between internal nodes to quickly jump to the proper continuation when handling smaller suffixes; suffix links amortize the cost of walking down the tree.
  • Represent leaf edge ends with a shared global end index so extending all leaves by one character is O(1) per phase (conceptually).

Important invariants and terminology

  • Active point: (activeNode, activeEdge, activeLength). ActiveEdge is the first character of the current suffix being extended; activeLength is how deep into that edge we are.
  • Remainder: number of suffixes that still need to be added in the current phase.
  • Suffix link: for an internal node representing string Xα (where α is a character), a suffix link points to the node representing α if it exists; used to move from one internal node to the next smallest suffix quickly.

Ukkonen — conceptual pseudocode (high level)

Let S be the input string with terminal '$'
root = new node
activeNode = root
activeEdge = -1 (none)
activeLength = 0
remainder = 0
for pos from 0 to n-1:
    add character S[pos] to the global end (for leaf edges)
    remainder += 1
    lastNewInternalNode = null
    while remainder > 0:
        if activeLength == 0:
            activeEdge = S[pos]  // start matching from this character
        if there is no edge from activeNode starting with activeEdge:
            // Rule 2: create a new leaf edge
            create leaf edge from activeNode labeled S[pos..]
            if lastNewInternalNode != null:
                set suffix-link(lastNewInternalNode) = activeNode
                lastNewInternalNode = null
        else:
            next = edge starting with activeEdge
            if activeLength >= length(next):
                // walk down (move active point deeper)
                activeNode = next.target
                activeLength -= length(next)
                activeEdge = next.charAt(activeLength)
                continue
            if S[next.start + activeLength] == S[pos]:
                // Rule 3: character already on edge, do nothing (implicit)
                activeLength += 1
                if lastNewInternalNode != null:
                    set suffix-link(lastNewInternalNode) = activeNode
                    lastNewInternalNode = null
                break
            // Rule 2 (split edge and insert internal node)
            splitNode = split edge 'next' at activeLength
            create leaf from splitNode labeled S[pos..]
            if lastNewInternalNode != null:
                set suffix-link(lastNewInternalNode) = splitNode
            lastNewInternalNode = splitNode
        remainder -= 1
        if activeNode == root and activeLength > 0:
            activeLength -= 1
            activeEdge = S[pos - remainder + 1]  // move forward on the current suffix
        else if activeNode != root:
            activeNode = suffix-link(activeNode) or root
    end while
end for

This pseudocode highlights the rules (Ukkonen's rules 1–3), the active point mechanics, and suffix links. Implementations must handle many detail cases carefully to maintain O(n) running time.

Why Ukkonen is linear (informal intuition)

Two main mechanisms enable linearity:

  1. Suffix links let you move between related internal nodes in O(1) time each, avoiding re-traversal from the root when handling successive suffixes.
  2. The active point and the remainder variable prevent repeated insertion work for suffixes that are already implicitly represented in the tree; many extensions are "do nothing" (Rule 3) and cost constant time on average. The number of splits and new nodes is O(n), and each character causes a bounded amount of extra work amortized across phases.

Finding the longest repeated substring (LRS)

The LRS is the substring spelled by the deepest internal node (by string-depth) that has at least two descendant leaves. After building the suffix tree, compute string depth (sum of edge lengths along root→node) during a DFS; the maximum among internal nodes with ≥2 leaf descendants is the LRS.

Generalized suffix tree: longest common substring (LCS) of two strings

Build a generalized suffix tree by concatenating the two strings with distinct terminal markers (e.g., S1 + '#' + S2 + '$', where '#' and '$' are unique). Each leaf is labeled as originating from S1 or S2. A node whose subtree contains leaves from both strings represents a common substring; the deepest such node (maximum string-depth) corresponds to the LCS. This can be found in linear time after building the tree (or linear in total length for the build+scan).

Space and implementation notes

  • Use arrays/objects for nodes; store edge labels as (start, end) indices into S.
  • For leaves in Ukkonen, use an "end" pointer shared across all current leaves so extending them is O(1) per phase.
  • Use hash maps or arrays for outgoing edges keyed by the first character of the edge (alphabet dependent).
  • Practical memory constant matters: suffix arrays or suffix automata are often preferred when memory is constrained.

Comparison to alternative structures

  • Suffix Array: an array of all suffixes in sorted order. With LCP array, many suffix tree queries become possible but some algorithms are simpler on explicit trees. Suffix arrays tend to be more memory-efficient in practice.
  • Suffix Automaton (DAWG): a minimal deterministic automaton that recognizes all substrings. It has at most 2n−1 states and is usually smaller than the suffix tree; it's great for distinct substrings, counting occurrences, and other substring queries.

Applications

  • Exact pattern matching and multiple-pattern search.
  • Finding maximal repeats, tandem repeats, and substring statistics.
  • Bioinformatics: read mapping, genome assembly helpers, indexing sequences.
  • Data compression (finding repeated sequences) and plagiarism detection.

Pitfalls and practical tips

  • Implementing Ukkonen correctly is subtle — many edge cases around active length, exact match vs split, and suffix link creation must be handled carefully.
  • Indexing with 0-based or 1-based intervals and whether end is inclusive/exclusive must be consistent throughout the implementation.
  • Test on simple examples (like "a", "aa", "aba", "banana", "mississippi") to validate behavior (counts, leaf labels, internal nodes).
  • If memory is tight or strings are extremely large, consider suffix array approaches or succinct/sparse representations.

Demonstration: pattern search of "ana" in "banana$"

Text: banana$ (indices 0..6)
Pattern: ana

Key idea: follow characters of the pattern from the root along edge labels. If we consume the whole pattern, every leaf in the subtree below the current node represents an occurrence (the leaf's index is the starting position).

Observation about the tree: The root has children for first characters: 'b', 'a', 'n', '$'. All suffixes starting with 'a' (indices 1, 3, 5) appear under the 'a' branch.

Conceptual root branches:
  root
   ├─ 'b' -> leaf [0]                 // suffix "banana$"
   ├─ 'a' -> subtree with leaves [1,3,5]  // suffixes starting with 'a'
   ├─ 'n' -> subtree with leaves [2,4]
   └─ '$' -> leaf [6]

Step-by-step traversal for "ana":

  1. Pattern first char = 'a'. From the root, choose the edge whose first character is 'a'. We enter the 'a' branch.
  2. Now we must match the next characters "na". Suppose the edge label (or subsequent edge labels down the subtree) provides that prefix "na" on the path to some leaves. We compare 'n' then 'a' sequentially.
  3. We successfully match all three characters "a"→"n"→"a" before the pattern ends, so the pattern occurs in the text.
  4. To report occurrences: enumerate the leaf labels under the current node. They are [1, 3] for the substring "ana". (Leaf [5] corresponds to "a$" and is too short to contain "ana".)

Result: Pattern "ana" occurs at start indices 1 and 3 (0-based).

Note: the explanation above deliberately avoids giving a particular compressed edge decomposition for the 'a' subtree — implementations will store edge labels as intervals (start, end) into the original string so traversal compares pattern characters to S[edge.start + offset]. The algorithmic principle is the same regardless of the exact compressed layout.
