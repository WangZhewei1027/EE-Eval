Sliding Window — A Comprehensive, Textual Guide

Concepts, templates, algorithm descriptions, step‑by‑step reasoning, and annotated examples for the Sliding Window technique.

1. What is the Sliding Window?

The sliding window is a common algorithmic technique used to solve problems that involve contiguous subarrays or substrings of a sequence (array, string, or similar). The key idea is to maintain a "window" (a contiguous range defined by two indices, often called left and right) that moves over the sequence while you update and reuse information efficiently rather than recomputing from scratch.

Intuitively: if two candidate subproblems overlap, there is often shared work you can reuse — sliding window formalizes that reuse by incrementally adjusting the window and updating the result.

  • Window: a contiguous block of elements between indices left and right (inclusive or exclusive depending on convention).
  • Slide: move left and/or right by one (or more) positions and update your maintained state (like current sum, counts, or maximums).
  • Purpose: reduce the overall work from O(n*k) or worse to O(n) or O(n log n) depending on supporting data structures.

2. When to use Sliding Window (Intuition & Recognition)

Look for these signals in a problem statement:

  • You're asked about contiguous segments (subarrays, substrings, intervals).
  • You're asked to compute something across all windows of a certain width k (e.g., maximum sum of k consecutive elements).
  • You're asked to find the longest/shortest window that satisfies a condition (e.g., smallest subarray with sum >= S, longest substring with ≤ K distinct characters).
  • Naive approaches recompute overlapping results; you can update incrementally when the window shifts.

The sliding window is not magical — it is an optimization that exploits overlap between contiguous subproblems and keeps an invariant (e.g., current sum, counts of characters).

3. Types of Sliding Windows

There are two main flavors:

  1. Fixed-size window: The window has a fixed length k. Example: "maximum sum of any subarray of size k". The right pointer moves forward; once the window reaches size k, you update results and move left forward as you expand right further.
  2. Variable-size (expanding/contracting) window: The window expands with the right pointer and contracts with the left pointer to maintain or satisfy some condition (e.g., maintain sum <= S or number of distinct characters <= K). This pattern typically uses a while-loop that moves left as necessary.

4. Sliding Window Templates (Pseudocode + Detailed Explanation)

4.1 Fixed-size window (length = k)

Problem pattern: compute some function over every contiguous subarray of size k.

// Pseudocode: fixed-size window (example: sum of each window of size k)
initialize sum = 0
for right from 0 to n-1:
    sum += arr[right]           // include the new element
    if right >= k - 1:
        // window size is k now: arr[right-k+1 ... right]
        record or use 'sum'
        sum -= arr[right-k+1]   // remove the leftmost element to slide the window forward

Step-by-step reasoning: accumulate into sum as you move right; once you have k elements, you use the rolling sum and then subtract the element that leaves the window when you advance to the next position.

4.2 Variable-size window (expand/contract)

Pattern: expand right to make the window "valid" (or to try to meet a condition); then contract left to restore validity or optimize the result.

// Pseudocode: variable-size window (example: smallest window with sum >= target)
left = 0
sum = 0
best = +infinity (or sentinel)
for right from 0 to n-1:
    sum += arr[right]                     // expand window to the right
    while sum >= target:                  // window is valid — try to shrink from the left
        best = min(best, right-left+1)   // update answer
        sum -= arr[left]                 // remove leftmost element
        left += 1
// result: best (or indicator if none)

Explanation: The while loop maintains an invariant: as long as the current window meets the condition (sum ≥ target) we can try removing left elements to find a smaller valid window. Each element enters and leaves the window at most once, so total time is O(n).

5. Example 1 — Maximum Sum of Subarray of Size k (Detailed Walkthrough)

Problem: Given an array of integers and an integer k, find the maximum sum of any contiguous subarray of size k.

Brute-force idea

Check every start index i and compute the sum of the k elements starting at i. This is O(n*k) and repeats work because overlapping windows share elements.

Sliding window solution (step-by-step)

  1. Maintain a rolling sum of the current window of size ≤ k.
  2. Move right pointer and add arr[right].
  3. When right >= k - 1, you have a full window. Use the current sum to update max. Then subtract arr[right - k + 1] to slide window forward for the next step.

Pseudocode

maxSum = -infinity
windowSum = 0
for right from 0 to n-1:
    windowSum += arr[right]
    if right >= k-1:
        maxSum = max(maxSum, windowSum)
        windowSum -= arr[right-k+1]   // slide: remove leftmost
return maxSum

Correctness (informal)

Every window of size k is visited exactly once when right reaches the window's rightmost index. The windowSum at that moment equals the sum of the k elements because we have been adding each element once and removing each element exactly once when it leaves the window.

Time & Space

O(n) time, O(1) extra space (excluding input).

6. Example 2 — Smallest Subarray with Sum ≥ S (Variable-size window)

Problem: Given an array of positive integers and a target S, find the length of the smallest (shortest) contiguous subarray whose sum is at least S. If none exists, return 0 (or other sentinel).

Why sliding window works

Because all numbers are positive, increasing the right side of the window never decreases the sum; therefore when the sum becomes >= S, shrinking from the left can only decrease size and potentially yield a smaller valid window. The monotonic behavior (sum increases as we add positive numbers) guarantees correctness when we try to contract.

Pseudocode

left = 0
sum = 0
minLen = +infinity
for right from 0 to n-1:
    sum += arr[right]
    while sum >= S:
        minLen = min(minLen, right - left + 1)
        sum -= arr[left]
        left += 1
if minLen == +infinity: return 0
else return minLen

Step-by-step textual walkthrough (example)

Array: [2, 1, 5, 2, 3, 2], S = 7

  1. right=0: sum=2 < 7.
  2. right=1: sum=3 < 7.
  3. right=2: sum=8 ≥ 7 → window length = 3 (indices 0..2). Try shrink: remove arr[0]=2 → sum=6 < 7 → stop shrinking.
  4. right=3: sum=8 ≥ 7 → window length = 3 (indices 1..3). Shrink: remove arr[1]=1 → sum=7 ≥ 7 → window length = 2 (indices 2..3). Shrink: remove arr[2]=5 → sum=2 < 7 → stop. Current best = 2.
  5. right=4: sum=5 < 7.
  6. right=5: sum=7 ≥ 7 → window length = 4 (indices 3..5). Shrink: remove arr[3]=2 → sum=5 < 7 → stop. Final best = 2.

Important: positivity of numbers matters. If array can contain negatives, sum may not behave monotonically when expanding; then this exact technique may fail and requires different approaches (prefix sums + data structures, or Kadane-like variants).

7. Example 3 — Longest Substring with At Most K Distinct Characters

Problem: Given a string s and an integer K, find the length of the longest substring that contains at most K distinct characters.

Approach

Use a variable-size sliding window and a frequency map to count characters in the current window. Expand right until the number of distinct characters exceeds K, then move left to reduce distinct count.

Pseudocode

left = 0
maxLen = 0
freq = empty map  // character -> count
distinct = 0
for right from 0 to n-1:
    c = s[right]
    freq[c] = freq.get(c,0) + 1
    if freq[c] == 1:
        distinct += 1
    while distinct > K:
        d = s[left]
        freq[d] -= 1
        if freq[d] == 0:
            distinct -= 1
        left += 1
    maxLen = max(maxLen, right - left + 1)
return maxLen

Step-by-step (example)

String: "eceba", K = 2

  1. right=0 'e': window="e", distinct=1, max=1
  2. right=1 'c': window="ec", distinct=2, max=2
  3. right=2 'e': window="ece", distinct=2, max=3
  4. right=3 'b': window="eceb", distinct=3 → shrink left: left moves to drop 'e' → window="ceb", distinct=3 (because there is still 'e' at pos 2?) — actually remove counts until distinct=2 → final window could be "eb" or "ceb" depending on counts; result continues

This example shows how counts and distinct tracking allow controlled contraction. Each character enters and leaves the frequency map at most once, so O(n) time.

8. Complexity & Correctness Patterns

Time complexity

  • Fixed-size or variable-size sliding windows typically run in O(n) time because each element is processed a constant number of times (added once when right moves, removed at most once when left moves).
  • If you maintain additional data structures (heaps, balanced trees), complexity may rise to O(n log n) for operations like finding max/min in a window when not using a deque.

Space complexity

  • O(1) for primitive rolling aggregates (sum, count) with fixed-size tasks.
  • O(k) or O(alphabet_size) for frequency maps or when storing window contents.

Correctness reasoning (informal)

In fixed-size windows, each distinct window is visited exactly once when the right pointer reaches its end. In variable-size windows, correctness relies on invariants such as monotonicity (e.g., sums only increase when adding positive numbers) or careful accounting of counts so that contracting maintains the validity of the condition.

9. Common Pitfalls & Practical Tips

  • Assume positivity when you shouldn't. Many sliding window solutions for sums require positive numbers. If negatives are allowed, analyze if the monotonic property holds; if not, consider prefix sums or other methods.
  • Off-by-one errors: decide whether your window is [left, right] inclusive or [left, right) half-open, and be consistent when computing lengths or removing elements.
  • For problems requiring max/min over a window, use a monotonic deque (double-ended queue) to achieve amortized O(1) per operation.
  • Be explicit about the invariant you maintain. Example: "sum is the sum of elements in arr[left..right]" or "freq counts characters in the current window". Writing the invariant helps avoid logic bugs.
  • Remember boundaries: when right reaches n-1, you may still need to contract the window in variable-size approaches.

10. Variations & Extensions

  • Monotonic deque: maintain potential maxima/minima in amortized O(1) to answer max/min in each sliding window.
  • Two pointers generalization: sliding window is a special case of two-pointer techniques where one pointer lags and the other advances.
  • Circular arrays: convert to linear by concatenating or handle wrap-around carefully.
  • 2D sliding window: for matrices you can use prefix sums or cumulative sums to get window sums in O(1) per window after O(n*m) preprocessing.
  • Negatives allowed: you might need prefix sums + data structures (ordered set, deque of candidate prefix indices) or dynamic programming.

11. Practice Problems & Suggested Approaches

  • Maximum sum of subarray of size k — fixed-size sliding window.
  • Smallest subarray with sum ≥ S — variable-size sliding window (positive nums).
  • Longest substring with at most K distinct characters — variable-size with frequency map.
  • Longest substring without repeating characters — variable-size with last index map.
  • Max sliding window (for each window of size k, output max) — use monotonic deque.
  • Subarray sums equals K (with negatives allowed) — use prefix-sum + hashmap (different technique).

Try to identify which problems are sliding-window-friendly by looking for contiguous constraints and whether you can maintain a compact state when the window changes.
