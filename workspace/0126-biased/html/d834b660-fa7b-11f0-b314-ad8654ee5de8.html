Quick Sort — A Comprehensive Explanation

An in-depth, educational guide to Quick Sort: definitions, algorithms, partition strategies, step-by-step walkthrough, complexity analysis, and practical advice. Includes a minimal demonstration you can run once.

1. What is Quick Sort?

Quick Sort is a comparison-based sorting algorithm based on the divide-and-conquer paradigm. It sorts an array (or list) by selecting a value called the pivot, partitioning the other elements into two sub-arrays according to whether they are less than or greater than the pivot, and then recursively sorting the sub-arrays. When implemented well, Quick Sort is typically very fast in practice and is one of the most commonly used sorting algorithms for arrays.

2. Key ideas and terminology

  • Pivot: A value chosen to partition the array.
  • Partitioning: Reordering elements so that values less than pivot come before it and values greater than pivot come after it. The pivot ends up in its final sorted position.
  • Divide and conquer: After partitioning, recursively sort left and right parts.
  • In-place: Many Quick Sort implementations rearrange elements inside the input array, requiring only O(log n) auxiliary stack space on average for recursion (not counting input storage).
  • Stability: Quick Sort is not stable by default: equal elements may change order unless stability measures are specifically implemented.

3. High-level algorithm (textual)

At a high level, Quick Sort works like this:

  1. Choose a pivot element from the array.
  2. Rearrange the array so that elements less than the pivot come before it and elements greater come after. This is the partition step.
  3. Recursively apply Quick Sort to the left and right partitions.
  4. Combine the sorted partitions (no extra work is needed: partitioning put the pivot in its final position).

This algorithm finishes when sub-arrays have zero or one element (base cases), which are already sorted.

4. Partitioning schemes

Partitioning is the heart of Quick Sort. Two widely taught schemes are:

Lomuto partition scheme (simple, intuitive)

Choose a pivot (commonly the last element). Maintain an index i such that elements at indices <= i are less than pivot. Iterate j from start to end-1: if a[j] < pivot, increment i and swap a[i] and a[j]. Finally swap a[i+1] and pivot. The pivot is now at position i+1.

// Lomuto partition (pseudocode)
function lomutoPartition(A, low, high):
    pivot = A[high]
    i = low - 1
    for j from low to high - 1:
        if A[j] < pivot:
            i = i + 1
            swap A[i] with A[j]
    swap A[i + 1] with A[high]
    return i + 1  // pivot index

Pros: easy to understand and implement.
Cons: performs poorly when there are many equal elements and can do more swaps than other schemes.

Hoare partition scheme (efficient, fewer swaps)

Choose a pivot (commonly A[(low+high)/2]). Use two indices i (left) and j (right). Increment i until A[i] ≥ pivot? and decrement j until A[j] ≤ pivot?; if i < j swap A[i] and A[j]. Continue until i ≥ j and return j. Note: Hoare's partition returns an index j such that all elements ≤ j are <= pivot and all > j are ≥ pivot, but pivot a priori may not end up at its final position immediately; recursion boundaries differ from Lomuto's.

// Hoare partition (pseudocode)
function hoarePartition(A, low, high):
    pivot = A[floor((low + high)/2)]
    i = low - 1
    j = high + 1
    while true:
        repeat i = i + 1 until A[i] >= pivot
        repeat j = j - 1 until A[j] <= pivot
        if i >= j:
            return j
        swap A[i] with A[j]

Pros: fewer swaps and comparisons on average, especially good when many equal values exist. Cons: slightly more complex recursion boundaries and careful index handling required.

5. Formal (recursive) pseudocode

One canonical implementation using Lomuto partition:

function quicksort(A, low, high):
    if low < high:
        p = lomutoPartition(A, low, high)
        quicksort(A, low, p - 1)
        quicksort(A, p + 1, high)

With Hoare partition, the recursive calls use partition index j differently:

function quicksort(A, low, high):
    if low < high:
        p = hoarePartition(A, low, high)
        quicksort(A, low, p)
        quicksort(A, p + 1, high)

7. Complexity analysis (time and space)

Time complexity

  • Best case: O(n log n) — good pivot choices that split the array into balanced subproblems (roughly halves).
  • Average case: O(n log n) — with a reasonable pivot strategy (random or median-of-three) expected running time is proportional to n log n.
  • Worst case: O(n^2) — occurs when pivot choices are consistently poor: e.g., always choosing smallest or largest element on already sorted input when using first/last-as-pivot. This degenerates into repeated linear scans of nearly full arrays, producing quadratic work.

A useful heuristic: average number of comparisons is roughly 2n ln n ≈ 1.386 n log2 n.

Space complexity

  • In-place partitioning uses O(1) additional memory for data movement.
  • Recursive call stack: O(log n) on average (balanced splits) but O(n) in worst-case recursion depth if the partitions are maximally unbalanced.
  • Careful iterative implementations or tail-call elimination reduce worst-case stack usage, but the algorithm still may need additional strategies to avoid depth O(n).

8. Stability and in-place discussion

Quick Sort is not stable by default: the relative order of equal elements may change because of swaps. If stability is required, either use a stable sorting algorithm (like Merge Sort) or adapt Quick Sort with extra bookkeeping (which typically increases memory usage).

Many common Quick Sort implementations are in-place (they only use a small amount of extra temporary storage). This in-place property is a major reason Quick Sort is used widely for arrays.

9. Pivot selection strategies

Pivot choice heavily affects performance. Strategies include:

  • First or last element — simple but dangerous on already sorted or reverse-sorted input (produces worst-case behavior).
  • Random pivot — choose a random element to reduce the chance of worst-case input; gives randomized Quick Sort with expected O(n log n) time regardless of input order.
  • Median-of-three — pick median of first, middle, last elements; often reduces bad cases and improves practical performance.
  • True median — choose the actual median (or an approximate median via selection algorithms). This guarantees O(n log n) worst-case but is typically more expensive per partition.

10. Handling many equal keys — three-way partitioning

When the input has many repeated elements, a three-way partition (Dutch National Flag) is effective. It partitions into three regions: <pivot, =pivot, >pivot. This ensures elements equal to the pivot are handled in one pass and avoids repeatedly partitioning equal elements, which improves performance when duplicates are common.

// 3-way partition (Dijkstra's Dutch National Flag)
function quicksort3way(A, low, high):
    if low >= high: return
    lt = low
    i = low
    gt = high
    pivot = A[low]
    while i <= gt:
        if A[i] < pivot:
            swap A[lt] with A[i]; lt++; i++
        else if A[i] > pivot:
            swap A[i] with A[gt]; gt--
        else:
            i++
    quicksort3way(A, low, lt-1)
    quicksort3way(A, gt+1, high)

11. Practical optimizations

  • Randomization: Randomly permute input or pick a random pivot to make worst-case inputs extremely unlikely.
  • Median-of-three: Often speeds up pivot choice with little cost.
  • Hybrid approach: Use Quick Sort for large partitions and switch to Insertion Sort for small partitions (e.g., size < 10). Insertion Sort is faster on tiny arrays due to lower overhead.
  • Tail recursion elimination: Recur into the smaller subarray first and loop for the larger one to bound recursion depth to O(log n) on average.
  • Introsort: Combine Quick Sort with Heap Sort: start with Quick Sort and if recursion depth exceeds a limit, switch to Heap Sort to guarantee O(n log n) worst-case time.

12. When to use Quick Sort

Quick Sort is a strong general-purpose sorter for in-memory arrays when average-case speed and low constant factors matter. It is often preferred for general array sorting when memory is constrained and stable sort is not required. Use Merge Sort when stability or guaranteed O(n log n) worst-case time is essential (e.g., external sorting or linked lists).

13. Common pitfalls and debugging tips

  • Off-by-one errors in partition indices — double-check boundaries for both Lomuto and Hoare schemes.
  • Beware using first/last element as pivot on already sorted data — leads to O(n^2).
  • Stack overflow on deep recursion — use tail-call optimization, iterate on the larger partition or switch to an iterative algorithm when needed.
  • Stability — if the relative ordering of equal items matters, Quick Sort without modifications will not preserve it.
  • Equal elements — use three-way partitioning for best performance when duplicates are frequent.

14. Summary

Quick Sort is a practical, efficient, in-place sorting algorithm with average-case time O(n log n) and simple, compact implementations. Its performance depends heavily on pivot selection and partition strategy. With randomization, median-of-three, or three-way partitioning, Quick Sort is robust for most inputs. For guaranteed worst-case bounds, use Introsort or Merge Sort.
