B-Tree — A Comprehensive Educational Guide

An in-depth, textual explanation of B-trees: definitions, properties, algorithms (search, insert, delete), step-by-step narratives, examples, and practical considerations. This page is primarily explanatory; a single demonstration block is available below to reveal a step-by-step insertion example.

What is a B-Tree?

A B-tree is a self-balancing tree data structure that maintains sorted data and allows searches, sequential access, insertions, and deletions in logarithmic time. It is optimized for systems that read and write large blocks of data (such as databases and file systems) where minimizing disk I/O is important.

The B in B-tree is sometimes said to stand for "balanced", "broad", or "Bayer" (after one of its inventors). In practice, "B-tree" refers to a family of balanced multi-way search trees.

Node structure and core properties

Most textbook treatments use a parameter t, called the minimum degree (CLRS notation). A B-tree of minimum degree t (t ≥ 2) obeys the following:

  • Each node contains between t−1 and 2t−1 keys, inclusive, except the root which can have fewer.
  • Each internal node with k keys has k+1 children (pointers to subtrees).
  • All keys within a node are kept in sorted order.
  • All leaves appear at the same depth (the tree is height-balanced).
  • Keys partition the subtrees: between key[i] and key[i+1] lie all keys in child i+1.

Consequences (quick summary):

  • Maximum keys per node: 2t − 1
  • Minimum keys per non-root node: t − 1
  • Maximum children per node: 2t
  • Minimum children per non-root internal node: t

Implementation note: Some literature uses "order m" with a slightly different meaning: "m" equals maximum number of children. The minimum-degree formulation (t) is standard in algorithm textbooks; be careful about definitions when reading different sources.

Search (how lookups work)

Search in a B-tree is a generalization of binary search tree lookup. At each node, you scan the keys (typically with binary search if keys are many) to find the interval that contains the target key or find that the node contains the key.

High-level description

  1. At a node x, perform binary (or linear) search among x.keys to find index i such that key == x.keys[i] or key < x.keys[i] (first key >= target).
  2. If a match is found, return it (successful lookup).
  3. If x is a leaf and no match: the key is not present.
  4. Otherwise, descend to the child pointer at index i (the interval) and repeat.

Pseudocode (concise)

BTree-Search(x, k)
  i = smallest index in x.keys with x.keys[i] >= k (binary search)
  if i exists and x.keys[i] == k
    return (x, i)      // key found in node x at position i
  else if x.leaf
    return NULL        // not found
  else
    return BTree-Search(x.child[i], k)

Remarks: With node keys stored in arrays, you typically perform binary search inside a node costing O(log B) where B is keys per node. Overall complexity dominated by tree height.

Insertion (how keys are added)

Insertion is performed so that node size limits are never violated. The standard approach (CLRS) inserts while ensuring you never descend into a full node by splitting full children on the way down. Two helper ideas:

  • Split-Child(x, i): splits child x.child[i] that is full (has 2t−1 keys) into two nodes with t−1 keys each, moving the median key up into x.
  • Insert-NonFull(x, k): inserts key k into node x, assuming x is not full; if x is internal, it finds the child to descend into, splits that child if it is full, and then recurses.

Step-by-step narrative

  1. If root is full (has 2t−1 keys), create a new root s (empty), make old root a child of s, and call Split-Child(s, 0). Now the tree height increases by 1 and s is not full.
  2. Call Insert-NonFull(s or the root), which will insert the key into the appropriate leaf without ever encountering a full node on the descent because any full child was split before descending into it.
  3. When inserting into a leaf, just insert the key into the keys array in sorted order (shifting others as needed).

Pseudocode

BTree-Insert(T, k)
  r = T.root
  if r.n == 2t - 1        // root is full
    s = new node
    T.root = s
    s.leaf = false
    s.n = 0
    s.child[0] = r
    Split-Child(s, 0)
    Insert-NonFull(s, k)
  else
    Insert-NonFull(r, k)

Split-Child(x, i)
  y = x.child[i]             // y has 2t - 1 keys (full)
  z = new node
  z.leaf = y.leaf
  z.n = t - 1
  for j = 1 to t-1
    z.key[j] = y.key[j+t]    // move upper half to z
  if not y.leaf
    for j = 1 to t
      z.child[j] = y.child[j+t]
  y.n = t - 1
  for j = x.n+1 down to i+1
    x.child[j+1] = x.child[j]
  x.child[i+1] = z
  for j = x.n down to i
    x.key[j+1] = x.key[j]
  x.key[i] = y.key[t]        // median moves up
  x.n = x.n + 1

Insert-NonFull(x, k)
  i = x.n
  if x.leaf
    insert k into x.keys in correct position (shift keys)
    x.n = x.n + 1
  else
    find child index i such that k should go into x.child[i]
    if x.child[i].n == 2t - 1
      Split-Child(x, i)
      if k > x.key[i]      // after split, decide which of the two children
        i = i + 1
    Insert-NonFull(x.child[i], k)

Note: Pseudocode uses 1-based indices for conceptual clarity; real implementations usually use 0-based arrays.

Deletion (removing keys) — detailed cases

Deletion from a B-tree is more complex than insertion because we must ensure nodes remain above the minimum key threshold (t−1 keys for non-root nodes). The high-level strategy is to ensure that when we descend to a child to delete a key, the child is guaranteed to have at least t keys (so that deletion can be done safely). This may require redistributing or merging nodes on the way down.

Deletion cases (deleting key k from node x)

Suppose we want to delete key k from B-tree rooted at T.root:

  1. If key k is in node x and x is a leaf: remove k from x (simple).
  2. If k is in internal node x at index i:
    1. If the child y = x.child[i] that precedes k has at least t keys, find k's predecessor k' (maximum key in subtree y), replace k with k', and recursively delete k' from y.
    2. Else if the child z = x.child[i+1] that follows k has at least t keys, find k's successor k' (minimum key in subtree z), replace k with k', and recursively delete k' from z.
    3. Else, both y and z have t−1 keys: merge k and z into y (make y contain k and all keys from z), free z, now y has 2t−1 keys; then recursively delete k from y (now contained in y).
  3. If k is not present in internal node x, determine the child c = x.child[i] that must contain k. Before descending:
    1. If c has t keys (minimum t−1 keys but may have exactly t−1 before some adjustments), we need to ensure it has at least t keys before recursing. Use these strategies:
      • If an immediate sibling (left or right) has at least t keys, borrow a key from that sibling via the parent (rotate keys through parent into c).
      • Else, both immediate siblings have t−1 keys: merge c with one sibling and move a key from the parent down into the merged node (now that merged node has 2t−1 keys). Then recurse into the merged node.

Pseudocode sketch (high-level)

BTree-Delete(T, k)
  Delete-From-Node(T.root, k)
  if T.root.n == 0 and not T.root.leaf
    T.root = T.root.child[0]    // shrink height

Delete-From-Node(x, k)
  // many cases as described above; this is a conceptual sketch
  find index i such that k <= x.key[i] (or i = x.n if bigger than all keys)
  if found key k in x at index i
    if x.leaf
      remove k from x.keys
    else
      // key in internal node: use predecessor/successor or merge
      handle internal-node deletion cases (see description)
  else
    if x.leaf
      // key not found
      return NOT-FOUND
    ensure child x.child[i] has at least t keys
      if child has t-1 keys, try borrow from siblings, else merge
    Delete-From-Node(x.child[i], k)

This sketch omits low-level index shifts and pointer movement; actual implementation carefully shifts arrays of keys and child pointers and handles boundary conditions.

Deletion is harder to get right in code than insertion because of the many cases and the need to preserve invariants. Many production systems prefer alternative strategies (lazy deletion, tombstones, periodic reorganization) depending on workload.

Complexity and height analysis

Let n be the number of keys stored and t the minimum degree. The height h (root-to-leaf path length) of a B-tree satisfies:

h ≤ log_t((n + 1) / 2)

This bound follows from the fact that every internal node (except the root) has at least t children, so the number of keys grows at least exponentially in the height.

Operation costs

  • Search: O(h * log B) where B is keys per node (logarithmic in node size) — typically O(log n).
  • Insert/Delete: O(h * log B) for finding the position plus constant number of splits/merges per level on the worst path — overall O(log n).
  • Disk I/O: B-trees are tuned so that a node corresponds to a disk block; height small means fewer disk reads.

Practical considerations and implementation notes

Choosing node size

In disk-backed systems, choose node size to match block or page size (e.g., 4KB). Use the maximum number of keys that fit in a block, considering key and pointer sizes and any overhead. This maximizes fan-out (number of children) and minimizes tree height, reducing disk seeks.

In-memory vs on-disk representation

In-memory implementations often use pointers and dynamic arrays. On-disk implementations store contiguous arrays of keys and child offsets, packed for compactness. When reading from disk, reading an entire node in one I/O is beneficial.

Key ordering and comparisons

Keys must support a total ordering. For composite keys (tuples), lexicographic ordering is common. Comparators must be stable and consistent with equality semantics used by the application.

Concurrency

Implementing concurrent insert/delete requires careful locking or lock-free techniques to allow parallel access. Common strategies: latch-coupling (hand-over-hand locking), optimistic concurrency with validation, or reader-writer locks on node granularity.

Bulk loading

When building a B-tree from a large sorted dataset, bulk-loading algorithms (pack nodes to full capacity) are far more efficient than inserting items one-by-one.

Variants and related structures

  • B+ Tree: Stores all actual records (or pointers to records) in the leaves; internal nodes store only keys for routing. Leaves are usually linked for efficient range scans. Widely used in databases.
  • B* Tree: A variant that keeps nodes fuller by redistributing among siblings before splitting, improving space utilization.
  • LSM Trees: Log-structured merge trees are an alternative index design optimizing write throughput rather than in-place balancing.
  • FM-Index and tries: Other structures useful for specialized workloads (text, prefix search).

Comparison with other balanced trees

- Red-black trees and AVL trees are binary trees (2-way). They keep height low but have smaller branching factor and are optimized for in-memory operations (low per-node overhead). B-trees are multi-way and optimized for minimizing disk or page reads by maximizing branching factor.

- B+ trees improve range scan performance and are commonly used in databases and file systems. Red-black trees may be preferable for purely in-memory sets/maps with many small, frequent updates.

Common pitfalls and advice

  • Confusing "order" vs "minimum degree": check your source's definitions carefully.
  • Array index off-by-one errors are frequent — decide on 0-based or 1-based indexing early and be consistent.
  • Deletion case analysis is error-prone — thorough unit tests and invariants checks (node size ranges, sorted keys, children count) are essential.
  • Tune node size to storage medium (page size) to reduce fragmentation and boost throughput.
  • For workloads dominated by writes, consider alternatives (write buffering, LSM trees) or batching inserts.
