Introduction & Goals

HTTPS stands for HTTP over TLS (historically "HTTP over SSL"). It is the dominant protocol to provide encrypted and authenticated communication between a web client (browser) and a server. When you browse a site with a URL starting with https://, the HTTP-level messages are carried inside an encrypted TLS session that protects confidentiality, integrity and (usually) server authentication.

Primary security goals of HTTPS

  • Confidentiality: Prevent eavesdroppers from reading exchanged data.
  • Integrity: Ensure data isn't modified undetectably in transit.
  • Authentication: Confirm the identity of the server (and optionally the client) using certificates and public key infrastructure (PKI).

HTTPS protects the content of HTTP messages and some headers. It does not hide which IP addresses communicate (network-level metadata) and, historically, the Server Name Indication (SNI) leaked the requested hostname unless encrypted with new features (Encrypted Client Hello / ECH).

TLS vs SSL — The Underlying Protocol

TLS (Transport Layer Security) is the modern protocol that replaced SSL (Secure Sockets Layer). TLS versions evolved to fix design and security issues in SSL. Common versions you will encounter in practice are TLS 1.0, 1.1, 1.2 and 1.3. SSL versions (2.0, 3.0) are obsolete and insecure.

Layers

At a high level, TLS consists of two main parts:

  • Handshake protocol: Negotiation of cryptographic algorithms and exchange of data to establish keys and authenticate endpoints.
  • Record protocol: Fragmentation, compression (rarely used), encryption and integrity protection of application data (e.g., HTTP).

Where HTTPS sits

When you use HTTPS, the typical stack is: TCP (or QUIC/UDP) → TLS → HTTP. Modern protocols like HTTP/3 run over QUIC, which integrates transport and TLS-like security directly into the QUIC protocol.

Cryptographic Primitives Used by TLS

TLS combines different classes of cryptographic primitives. Understanding their roles is essential.

1. Asymmetric (public-key) cryptography

Purpose: authenticate endpoints and (historically) perform key exchange.

  • RSA: widely used historically for signatures and key transport. RSA key transport means the client encrypts a premaster secret with the server's public RSA key. This is not forward-secret unless ephemeral keys are used.
  • Diffie–Hellman (DH) / Elliptic Curve Diffie–Hellman (ECDH/ECDHE): provide key agreement so both sides derive shared secrets without transmitting them. ECDHE is the most common modern choice and provides (perfect) forward secrecy when ephemeral keys are used.

2. Symmetric encryption

Purpose: encrypt bulk data once keys are established.

  • AES: block cipher used in modes such as GCM (AES-GCM, an AEAD) or CBC (older).
  • ChaCha20-Poly1305: an AEAD alternative that is efficient on software-only systems and mobile devices.

3. Hashes and Message Authentication

  • SHA-2 family (SHA-256, SHA-384): used in signatures, HMACs and as components in key derivation.
  • HMAC: keyed-hash for message authentication used in older constructions.
  • AEAD (Authenticated Encryption with Associated Data): provides encryption and integrity in one primitive (e.g., AES-GCM, ChaCha20-Poly1305).

4. Key derivation

TLS uses pseudo-random functions (PRF) and key derivation functions such as HKDF (in TLS 1.3) to expand secrets into keys and IVs. A secure KDF prevents leaking key material even if some inputs are known.

TLS Handshake — Step-by-step (Detailed Narrative)

This section describes the typical handshake flows. TLS 1.2 and TLS 1.3 differ significantly. Both are explained step-by-step so you can follow the decisions and cryptographic operations.

TLS 1.2 (classic long-form handshake)

High-level steps (numbers are message groups):

  1. ClientHello
    • Client sends supported TLS version, a random 32-byte value (ClientRandom), supported cipher suites, compression methods, extensions (SNI, ALPN), and optionally ephemeral key shares (for ECDHE) if supported.
  2. ServerHello
    • Server responds with chosen version, ServerRandom (32 bytes), chosen cipher suite, and selected extensions.
  3. Server Certificate
    • Server sends its certificate chain (leaf certificate, any intermediate CA certificates) to allow the client to verify the server's identity.
  4. ServerKeyExchange (if needed)
    • When the cipher suite requires additional parameters (e.g., ECDHE), the server sends parameters for the key exchange and signs them with its certificate private key to bind the parameters to its identity.
  5. CertificateRequest (optional)
    • Server may request a client certificate (mutual TLS).
  6. ServerHelloDone
    • Server signals it is done with its part of the handshake.
  7. Client Certificate (optional)
    • If requested, the client supplies its certificate and proves possession of its private key by signing handshake messages.
  8. ClientKeyExchange
    • Client sends key exchange material. For RSA key transport it sends an RSA-encrypted premaster secret. For (E)DHE, it sends its ephemeral public key share.
  9. ChangeCipherSpec & Finished
    • Both sides send a ChangeCipherSpec message to switch to the newly negotiated cipher and keys, then each sends a Finished message — a verification value computed over the handshake transcript and master secret. If the Finished messages verify, the handshake completed successfully and application data can be exchanged.

TLS 1.3 (simplified, faster and safer)

TLS 1.3 streamlines the handshake, removes insecure constructs, and mandates forward secrecy in common configurations.

  1. ClientHello (includes key share)
    • Client sends supported versions, ClientRandom, extensions, supported cipher suites, and an ephemeral key share (e.g., X25519).
    • Client may also include a PSK (pre-shared key) identity for session resumption and optionally use 0-RTT for sending early application data (with replay risks).
  2. ServerHello (includes chosen key share)
    • Server selects cipher suite, returns its ServerHello with its ephemeral key share. Both sides compute shared secret from the key shares and derive handshake keys using HKDF.
  3. Encrypted Extensions + Certificate + CertificateVerify
    • Server sends any negotiated extensions in EncryptedExtensions, then its Certificate (signed chain), and a CertificateVerify message: a signature with the server's private key over parts of the handshake to prove possession.
  4. Finished
    • Server sends its Finished using handshake-derived keys. Client verifies, then sends its own Finished. After this, application traffic keys are derived and used.

Key differences in TLS 1.3: no RSA key transport, removed MD5/SHA-1 PRF, no renegotiation, fewer round-trips (can be 1-RTT or 0-RTT with resumption), and mandatory AEAD ciphers in common configurations.

How keys are derived (conceptual)

Both versions use the agreed shared secret (from ECDHE/DH or premaster secret) and combine it with ClientRandom and ServerRandom to produce the master secret using a PRF/KDF. The master secret is expanded into symmetric keys and IVs. TLS 1.3 uses a structured HKDF-based key schedule producing separate secrets for handshake and application data, improving isolation and forward secrecy.

Certificates and the Public Key Infrastructure (PKI)

Certificates are X.509 documents that bind an identity (domain name, organization) to a public key and are digitally signed by a certificate authority (CA). They form the basis for authentication in TLS.

Key certificate fields

  • Subject: the identity the certificate asserts (common name, organization).
  • Subject Alternative Names (SAN): current standard place for hostnames and IPs — browsers rely on SAN for hostname matching.
  • Issuer: the CA that signed the certificate.
  • Validity: Not Before / Not After dates defining the certificate lifetime.
  • Public Key: the public key corresponding to the private key held by the server.
  • Signature Algorithm and Signature: the CA's algorithm and signature over the certificate fields.

Certificate chain and verification

When a server sends its certificate chain it typically includes the leaf certificate and any intermediate CA certificates. The client’s job is to:

  1. Verify the certificate chain up to a trusted root (present in the client's trust store).
  2. Check each signature in the chain is valid and that certificates are within their validity period.
  3. Confirm the certificate matches the requested hostname (SAN/CN check).
  4. Check revocation status (CRL or OCSP) and any relevant policies (e.g., minimum key sizes, unacceptable algorithms).

Types of validation

  • Domain Validation (DV): CA verifies control of the domain (automated issuance like Let’s Encrypt).
  • Organization Validation (OV): CA verifies additional identity data about the organization.
  • Extended Validation (EV): more rigorous checks historically used to display special UI; modern browsers deemphasize EV-indicators.

Revocation

Revocation mechanisms include CRLs (certificate revocation lists) and OCSP (online certificate status protocol). Online checking can be slow or privacy-invasive, so browsers use mechanisms like OCSP stapling and CRLite/OneCRL/soft-fail policies. Revocation is a known practical weakness because of deployment issues.

TLS Versions, Cipher Suites & Best Practices

TLS version guidance

  • Disable SSL 2.0 / SSL 3.0 and TLS 1.0 / 1.1 (obsolete and insecure).
  • TLS 1.2 is widely used and secure when configured with modern cipher suites (use ECDHE, AEAD ciphers, strong hashes).
  • TLS 1.3 is preferred: simpler, faster, removes legacy insecure choices, and provides better forward secrecy and privacy.

Cipher suite selection (principles)

  1. Prefer ECDHE (ephemeral Diffie-Hellman) for key exchange — provides forward secrecy.
  2. Prefer AEAD ciphers (AES-GCM, AES-CCM, ChaCha20-Poly1305) for combined encryption and integrity.
  3. Avoid old constructions: RC4, DES, 3DES, AES-CBC (unless mitigations applied), MD5, SHA-1 signatures.

Example modern TLS 1.3 cipher suites

TLS_AES_128_GCM_SHA256
TLS_AES_256_GCM_SHA384
TLS_CHACHA20_POLY1305_SHA256

Example modern TLS 1.2 cipher suites

TLS_ECDHE_ECDSA_WITH_CHACHA20_POLY1305_SHA256
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384

Note: The names list key exchange, authentication, encryption, and hash used for pseudo-random functions or signatures. In TLS 1.3 the negotiation is simplified.

Deployment, Key Management, and Operational Topics

Ports, SNI and related topics

HTTPS commonly uses TCP port 443. Server Name Indication (SNI) allows hosting multiple TLS names (virtual hosts) on one IP by having the client include the desired hostname in the handshake. Traditional SNI was unencrypted and could be observed by on-path parties; TLS 1.3 and related efforts introduce ECH to encrypt the client hello and SNI.

TLS termination and architecture

Many deployments terminate TLS at load balancers or reverse proxies, then pass traffic to backend servers either in plaintext (internal networks) or over another TLS session. Consider end-to-end encryption and key management responsibilities (where private keys are stored: on the proxy, the origin, or a hardware security module).

Certificate lifecycle

Operational steps:

  1. Generate a private key (keep it secret; ideally in HSM for high security).
  2. Create a CSR (certificate signing request) and submit to a CA.
  3. Install issued certificate and any intermediate chains on the server.
  4. Monitor expiration and renew before expiry. Automate where possible (Let's Encrypt + ACME).

Client certificates (mutual TLS)

Mutual TLS (mTLS) requires clients to present certificates to authenticate themselves to servers. Useful in APIs, internal services, and high-security contexts. mTLS increases complexity in certificate management and revocation, but offers strong client authentication without passwords.

Common Vulnerabilities, Historical Issues and Mitigations

TLS/HTTPS is generally secure when correctly configured. However, past vulnerabilities and misconfigurations can break security.

Notable historical issues (summary)

  • POODLE (SSLv3): exploited SSL 3.0 fallback and padding; led to disabling SSL 3.0.
  • BEAST: exploited CBC mode in TLS 1.0; mitigated by moving to TLS 1.1+ and use of AEAD.
  • Heartbleed: a memory bug in OpenSSL's heartbeat extension allowing secret leakage. Mitigation: patch and reissue keys/certificates if compromised.
  • ROBOT/Bleichenbacher: attacks against RSA PKCS#1 v1.5 padding; mitigate by avoiding RSA key transport and applying mitigations.

Operational pitfalls

  • Using expired/weak certificates: renew and use strong RSA/ECDSA keys and avoid small key sizes.
  • Improper chain configuration: failing to send intermediates causes connection failures or warnings.
  • Weak cipher suites enabled: restrict server to modern ciphers and prioritize them.
  • Improper revocation handling: rely on OCSP stapling and consider short certificate lifetimes.

Active mitigations & hardening

  • Use TLS 1.3 where possible.
  • Configure servers with ECDHE and AEAD cipher suites only.
  • Enable HSTS (HTTP Strict Transport Security) to avoid downgrade attacks; set a long max-age and includeSubDomains when appropriate.
  • Enable OCSP stapling to speed and protect revocation checks.
  • Use certificate transparency (CT) monitoring and log checks.
  • Consider using Public Key Pinning sparingly (HPKP is deprecated due to risks); use alternatives like Certificate Transparency, and short certificate lifetimes.

Handshake: Step-by-step explanation in plain language

Imagine two people, Alice (client) and Bob (server), want to agree on a secret codebook so they can talk privately:

  1. Alice says: "Here are the ways I can communicate (protocols), and here's a random number I made (ClientRandom). I also give you a temporary public key that only I will use for this conversation."
  2. Bob replies: "I choose these options, here is my random number (ServerRandom), and here is my temporary public key."
  3. Using both temporary keys, they mathematically compute the same secret value without sending it across the wire (Diffie-Hellman).
  4. Bob proves who he is by showing a signed certificate signed by someone Alice trusts (a CA). This certificate contains Bob's long-term public key.
  5. They derive symmetric keys from the shared secret and random numbers. From now on, their messages are encrypted and integrity-protected.

RSA vs ECDHE — conceptual difference

RSA key transport: client encrypts a random premaster secret to the server's public RSA key. If the server's private key is ever compromised, past sessions can be decrypted (no forward secrecy).

ECDHE: both sides create ephemeral ECDH keypairs, exchange public keys, and compute a shared secret. Even if the server's long-term key is later compromised, past session keys remain safe (perfect forward secrecy) because ephemeral secrets were used and not stored.

  - shared_secret = X25519(client_priv_A, server_pub_B)
  - handshake_keys = HKDF-Expand(shared_secret, "handshake", client_random|server_random)

Server sends EncryptedExtensions (e.g., ALPN), then:
Server Certificate: [example.com leaf], signed by Intermediate CA
CertificateVerify: signature over transcript with server's private key
Server Finished: MAC over handshake transcript using handshake_keys

Client verifies signatures and Finished, then:
Client Finished: MAC over handshake transcript using handshake_keys_client
Now application keys are derived and HTTP/3 or HTTP/1.1 over TLS can begin with encrypted data.

Notes:
  - If PSK + 0-RTT used, client could send early data after ClientHello, but server must accept carefully due to replay risk.
  - All private keys are kept secret; random nonces ensure freshness.

Glossary & Quick Reference

AEAD
Authenticated Encryption with Associated Data — a cipher mode that provides encryption and integrity together (e.g., AES-GCM).
CA (Certificate Authority)
An entity that issues digital certificates and attests binding between public keys and identities.
ClientHello / ServerHello
Initial TLS handshake messages for negotiating version and parameters.
DH / ECDHE
Diffie–Hellman / Elliptic Curve Diffie–Hellman Ephemeral: key exchange methods for deriving shared secrets.
HSTS
HTTP Strict Transport Security: header telling browsers to only access a site over HTTPS for a given period.
OCSP
Online Certificate Status Protocol — a way to check if a certificate is revoked.
Perfect Forward Secrecy (PFS)
A property that prevents compromised long-term keys from exposing past session keys (commonly via ephemeral DH/ECDHE).
