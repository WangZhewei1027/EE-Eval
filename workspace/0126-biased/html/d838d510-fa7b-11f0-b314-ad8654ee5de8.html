TCP/IP — A Comprehensive Educational Guide

A thorough textual explanation of the TCP/IP Internet protocol suite: architecture, protocols, algorithms, step-by-step narratives, examples and practical notes. Small, simple demonstration included.

1. Overview: What is TCP/IP?

TCP/IP is the common name for the set of network protocols that power the Internet and most modern networks. It stands for Transmission Control Protocol/Internet Protocol. Conceptually it is a layered suite where each layer provides services to the layer above and uses services from the layer below. The suite enables host addressing, packet forwarding, reliable (and unreliable) transport, name resolution, and many application protocols (HTTP, DNS, SMTP, etc.).

Historical context

Developed in the 1970s and standardized in the 1980s, TCP/IP replaced numerous earlier network protocols by providing a robust set of interoperable standards. The "Internet" as we know it grew around TCP/IP because of its scalability, modularity, and flexibility.

Layered architecture (Internet model)

TCP/IP is often explained with four conceptual layers:

  • Link layer (Network Interface) — hardware and media (Ethernet, Wi‑Fi). Responsible for delivering frames on the same physical network.
  • Internet layer — addressing and routing across networks (IPv4, IPv6, ICMP, ARP).
  • Transport layer — end-to-end communication between processes (TCP, UDP, SCTP).
  • Application layer — protocols used by applications (HTTP, DNS, SMTP, DHCP, etc.).

The OSI 7-layer model is a teaching tool; TCP/IP maps differently but the layered idea is the same.

2. The Internet Layer: IP (IPv4 & IPv6)

This layer enables global addressing and the forwarding of packets between networks via routers. The principal protocol is Internet Protocol (IP). Two major versions are in use today: IPv4 and IPv6.

IPv4: packet structure (high-level)

An IPv4 header contains fields that control routing and delivery. Key fields:

  • Version (4), Header Length, Type of Service/DSCP
  • Total Length (header + payload)
  • Identification, Flags, Fragment Offset (fragmentation)
  • Time To Live (TTL) — decremented at each router, prevents loops
  • Protocol — indicates transport (TCP=6, UDP=17)
  • Header Checksum — covers the header only
  • Source and Destination IP addresses

Routers examine destination IP, consult routing tables, and forward packets toward the destination. Fragmentation (when a packet is larger than MTU) may occur at routers (or be avoided by path MTU discovery).

Addressing, CIDR and subnetting (step-by-step example)

IPv4 addresses are 32-bit numbers, usually written as dotted decimals, e.g. 192.168.1.100.

Classless Inter-Domain Routing (CIDR) uses a prefix length to denote a network. Example: 192.168.1.0/24 means the first 24 bits are the network part, leaving 8 bits for host addresses (256 addresses total — 254 usable hosts typically).

Subnetting example — step-by-step:

  1. You have prefix 192.168.0.0/24 and need 4 subnets of equal size.
  2. /24 provides 8 host bits; to make 4 subnets you need 2 additional network bits: /26 (24+2=26).
  3. /26 yields 64 addresses per subnet; usable hosts per subnet: 62 (excluding network and broadcast addresses).
  4. Subnets are: 192.168.0.0/26, 192.168.0.64/26, 192.168.0.128/26, 192.168.0.192/26.

ARP (Address Resolution Protocol)

Within a local link, IP addresses must be mapped to link-layer addresses (MAC addresses for Ethernet). ARP broadcasts a request: "Who has IP X.X.X.X?" The owner replies with its MAC. ARP results are cached for performance.

IPv6: goals and differences

IPv6 provides a 128-bit address space, simplified header, mandatory support for extension headers, improved multicast and neighbor discovery (replaces ARP), and built-in features to reduce fragmentation. IPv6 addresses are written in hexadecimal colon notation (e.g., 2001:0db8::1).

3. Routing: How packets find their way

Routing is the process of selecting paths in a network to send packets from source to destination. This is handled in two complementary ways:

  • Forwarding — the per-router action of moving packets from input to output interfaces using the routing table.
  • Routing — the control-plane protocols that build and maintain routing tables.

Routing protocols and algorithms

Common classes of routing protocols:

  • Distance-vector (e.g., RIP): each router advertises distances to destinations to neighbors; simple but can converge slowly and suffers from count-to-infinity problems.
  • Link-state (e.g., OSPF): each router advertises its local links to all routers; every router builds the same map of the network and computes shortest paths (Dijkstra).
  • Path-vector (BGP): used between autonomous systems on the Internet; advertises full paths and policies; complex but scalable.

Shortest-path example (link-state, conceptual step-by-step)

  1. Each router discovers its neighbors and reports link costs in Link State Advertisements (LSAs).
  2. All routers flood LSAs so every router gets a consistent view of the network graph.
  3. Each router runs Dijkstra's algorithm on the graph to compute a shortest-path tree rooted at itself.
  4. The router installs forwarding entries for next-hop decisions derived from the tree.

4. Transport Layer: TCP and UDP

The transport layer provides host-to-host communication services for applications. The two most important transport protocols are UDP and TCP.

UDP (User Datagram Protocol)

UDP is connectionless and provides minimal framing of data with ports. It offers low overhead and no reliability guarantees — packet loss, duplication, or reordering may occur. Used where low latency or simplicity matters (DNS queries, VoIP, video streaming).

TCP (Transmission Control Protocol) — goals

TCP provides a reliable, ordered, byte-stream service between two endpoints. Its main goals include:

  • Reliable delivery (retransmit lost data)
  • In-order delivery to the application
  • Flow control (avoid overwhelming receiver)
  • Congestion control (avoid overwhelming the network)
  • Multiplexing via port numbers

TCP segment header (selected fields)

  • Source port, Destination port
  • Sequence number, Acknowledgment number
  • Data offset (header length), Flags (SYN, ACK, FIN, RST, PSH, URG)
  • Window size (receiver advertised window)
  • Checksum (covers TCP header and payload)
  • Options (MSS, Window Scale, SACK permitted, Timestamps)

Three-way handshake — step-by-step textual narrative

Establishing a TCP connection between client A and server B:

  1. A -> B: SYN, seq = x. Client indicates it wants to open a connection and sends an initial sequence number (ISN).
  2. B -> A: SYN+ACK, seq = y, ack = x+1. Server replies with its own ISN and acknowledges the client's SYN.
  3. A -> B: ACK, seq = x+1, ack = y+1. Client acknowledges the server's SYN; connection is established.

This exchange establishes initial sequence numbers and synchronizes windows and options. The connection is full-duplex after handshake.

Flow control vs congestion control (conceptual)

Flow control prevents the sender from overwhelming the receiver: the receiver advertises a window that tells the sender how many bytes it can send without further acknowledgment.

Congestion control prevents the network from being overloaded. TCP algorithms adjust the sending rate (congestion window, cwnd) in response to perceived network conditions (loss or explicit signals).

TCP congestion control algorithms (explanations and step-by-step)

Modern TCP uses a combination of algorithms. The typical sequence:

  • Slow Start — when a connection starts (or after severe loss), cwnd begins small (often MSS) and increases exponentially: cwnd += MSS for each ACK received (effectively doubling each RTT).
  • Congestion Avoidance — when cwnd reaches a threshold (ssthresh), TCP switches to additive increase: cwnd += MSS*(MSS/cwnd) per ACK (equivalent to approx +1 MSS per RTT).
  • Fast Retransmit — when the sender receives three duplicate ACKs, it presumes a segment was lost and retransmits without waiting for a timeout.
  • Fast Recovery — after fast retransmit, some TCP variants (like Reno) reduce cwnd and enter congestion avoidance; NewReno and CUBIC have refinements.

Simple AIMD (Additive Increase Multiplicative Decrease) pseudocode:

// On ACK:
if (cwnd < ssthresh) {
  // slow start
  cwnd += MSS;
} else {
  // congestion avoidance
  cwnd += (MSS * MSS) / cwnd; // approx. +1 MSS per RTT
}

// On detected loss:
ssthresh = cwnd / 2;
cwnd = max(MSS, ssthresh);

Different TCP implementations (Reno, NewReno, BIC, CUBIC, DCTCP) vary in details. Understanding the principles (slow start, AIMD, fast retransmit/recovery) is most important.

Reliable delivery mechanisms (steps)

  1. Sender segments data into TCP segments and assigns sequence numbers.
  2. Receiver responds with ACKs indicating the next expected sequence number (cumulative ACK).
  3. If sender observes duplicate ACKs or a timeout, it retransmits segments.
  4. Optional selective acknowledgments (SACK) allow receiver to inform sender about non-contiguous blocks received, enabling efficient retransmission.

5. Application Protocols and Typical Flows

Application protocols sit on top of transport. Typical web (HTTP over TCP) flow, step-by-step:

  1. Client resolves domain with DNS: client -> DNS resolver -> authoritative server -> receives IP address.
  2. Client initiates TCP three-way handshake with server's IP: SYN, SYN-ACK, ACK.
  3. Client sends HTTP request (GET /) in the TCP stream.
  4. Server responds with HTTP headers and payload (HTML, images, etc.).
  5. TCP handles segmentation, retransmission, ordering; HTTP may reuse the connection or close it.

DHCP and IP configuration (narrative)

On many networks, hosts obtain an IP configuration automatically via DHCP:

  1. Client boots and does a DHCPDISCOVER broadcast.
  2. DHCP servers reply with DHCPOFFER (IP address, gateway, DNS servers).
  3. Client selects an offer and sends DHCPREQUEST.
  4. Server confirms with DHCPACK; client configures address and routes.

6. Sockets and Ports (how applications interface)

Sockets are the programming abstraction used by applications to send/receive data. A socket is defined by the 4-tuple: (source IP, source port, destination IP, destination port).

Ports and services

Well-known ports map to services (HTTP=80, HTTPS=443, DNS=53). Servers listen on a port; clients use ephemeral ports assigned by the OS.

Minimal pseudocode examples (conceptual)

// Server (TCP) conceptual:
socket = socket(AF_INET, SOCK_STREAM)
bind(socket, addr="0.0.0.0", port=80)
listen(socket)
conn, peer = accept(socket)
read/write on conn

// Client (TCP) conceptual:
socket = socket(AF_INET, SOCK_STREAM)
connect(socket, addr="example.com", port=80)
send HTTP request
read response
close(socket)

7. Control Protocols and Diagnostics

ICMP (Internet Control Message Protocol) is used for network diagnostics and error reporting (e.g., "destination unreachable", TTL exceeded). Tools built on ICMP and transport help diagnose problems:

  • ping — sends ICMP echo requests and measures round-trip time; useful for basic reachability and latency checks.
  • traceroute — discovers the path to a destination by increasing TTLs; shows intermediate routers and where packets are delayed or blocked.
  • tcpdump/wireshark — packet capture tools; allow inspection of headers and payload (very useful for learning and troubleshooting).

8. Security Considerations

TCP/IP was designed for flexibility and performance; many security mechanisms were added later. Important topics:

  • TLS — provides transport-layer encryption and authentication for application protocols (HTTPS).
  • IPsec — suite for securing IP communications via authentication and encryption (can operate at network layer).
  • NAT — Network Address Translation rewrites IP addresses and/or ports, enabling many hosts to share one public IPv4 address. It complicates end-to-end connectivity and some protocols.
  • Firewalls — filter packets by address, port and state; may perform NAT or application-level inspection.
  • Common attacks — IP spoofing, SYN flood (DoS), man-in-the-middle, DNS spoofing. Mitigations: ingress filtering, SYN cookies, rate limiting, DNSSEC, TLS.

9. End-to-end Example: Fetching a Web Page — Detailed Narrative

Step-by-step sequence when a user types "https://example.com/" and presses Enter (simplified, without caching):

  1. Browser checks cache for DNS and TLS session info. If no cached DNS entry, it queries the OS resolver.
  2. OS resolver queries configured DNS server; DNS lookup returns the server's IP address(es).
  3. Browser selects an IP and initiates TCP handshake to port 443 (HTTPS).
  4. Client and server perform TCP three-way handshake, negotiating MSS and window scaling.
  5. Immediately or shortly after, the TLS handshake begins inside the established TCP connection — client hello, server hello, certificates, key exchange.
  6. Once TLS is established, the client sends an encrypted HTTP request (GET /) over the secure channel.
  7. Server responds with an encrypted HTTP response containing HTML, which the client decrypts and renders. Browser may follow links and fetch additional resources concurrently (new TCP connections or HTTP/2 multiplexing).
  8. When transfers are finished, connections may be kept alive for reuse; otherwise, TCP teardown starts using FIN/ACK (four-way close) or RST if aborted.

10. Troubleshooting Hints and Diagnostics (textual)

Common steps when a network problem occurs:

  1. Check physical/link: is the network interface up? Is the cable connected or Wi‑Fi associated?
  2. Check IP configuration: ip addr / ifconfig / ipconfig — is the host configured with an IP and gateway?
  3. Ping the gateway (first-hop). If it fails, the local link or configuration is usually the problem.
  4. Use traceroute to see where packets stop or where latency spikes occur.
  5. Capture packets (tcpdump/wireshark) to inspect headers and retransmissions; look for ICMP errors and TCP retransmits.
  6. For performance problems, observe retransmissions, duplicate ACKs, throughput limits, and verify MTU/path MTU discovery problems (black hole routers).

11. Learning Exercises and Worked Examples

Subnet calculation

Given network 10.10.0.0/20, how many hosts? /20 = 12 host bits, so 2^12 = 4096 addresses; usable hosts (approx) = 4094.

TCP handshake and sequencing example

Client ISN = 1000, Server ISN = 5000. After handshake the first data byte sent by client will have sequence number 1001 and be acknowledged as ack=1001+N where N is the number of bytes.

Congestion window growth example (numerical)

Assume MSS = 1000 bytes and initial cwnd = 1*MSS, ssthresh = 8*MSS. ACKs per RTT double cwnd while cwnd < ssthresh:

  • RTT1: cwnd = 1 MSS (send 1 MSS)
  • RTT2: cwnd = 2 MSS
  • RTT3: cwnd = 4 MSS
  • RTT4: cwnd = 8 MSS — now cwnd == ssthresh
  • RTT5 onwards: additive increase (+1 MSS per RTT) -> 9 MSS, 10 MSS, ...
