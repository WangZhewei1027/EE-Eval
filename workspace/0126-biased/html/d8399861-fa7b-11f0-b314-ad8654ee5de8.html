What is the SDLC?

The Software Development Life Cycle (SDLC) is a structured sequence of phases that guides the process of developing software systems. It defines what activities should be performed, their order, the artifacts produced, and who is responsible. SDLC provides repeatability, quality controls, and predictable outcomes.

Key objectives of SDLC:

  • Deliver reliable, maintainable software that meets stakeholder needs.
  • Manage cost, schedule and risks systematically.
  • Provide traceability from requirements to tests and code.
  • Support quality assurance and continuous improvement.

Core SDLC Phases — Step-by-step textual descriptions

Below are the canonical SDLC phases with pedagogical step-by-step descriptions. Each phase has objectives, common activities, typical artifacts, and roles involved.

1) Planning & Feasibility

Objective: decide whether to invest in the project and define high-level scope, stakeholders and constraints.

Activities: stakeholder interviews, high-level requirements capture, feasibility analysis (technical, financial, legal), cost-benefit, timeline estimation, resource planning.

Artifacts: project charter, feasibility report, initial budget estimate, stakeholder register.

Roles: sponsor, product owner, project manager, business analyst, architect.

2) Requirements Engineering

Objective: elicit and record what the system must do (functional) and the constraints (non-functional).

Activities (step-by-step):

  1. Elicit requirements through interviews, workshops, domain observation, surveys, and prototypes.
  2. Organize and classify requirements: functional, non-functional (performance, security, usability), and domain rules.
  3. Write clear, testable requirement statements or user stories.
  4. Validate requirements with stakeholders (reviews, sign-off, prototypes).
  5. Establish traceability for each requirement.

Artifacts: requirements specification (SRS), user stories backlog, acceptance criteria, requirements traceability matrix (RTM).

3) System Analysis

Objective: analyze requirements and define the system behavior, data, and interactions at a higher level.

Activities: use cases, data modeling (ER diagrams), sequence diagrams, functional decomposition, risk analysis.

Artifacts: use case models, data models, high-level process flow diagrams.

4) Architecture & Detailed Design

Objective: decide how the system is organized (components, interfaces, data storage) and specify detailed designs for implementation.

Activities: choose architectural patterns (layered, microservices, event-driven), design APIs, design database schemas, create class diagrams and detailed component descriptions, identify cross-cutting concerns (security, logging).

Artifacts: architecture decision records (ADRs), component diagrams, interface specifications, design documents, prototyping results.

Design principles to consider: SOLID, separation of concerns, modularity, encapsulation, least privilege, scalability, observability.

5) Implementation (Coding)

Objective: convert design into working code and build executable artifacts.

Activities (step-by-step):

  1. Set up the development environment, repositories and branching model.
  2. Implement features in small increments aligned with requirements or user stories.
  3. Use code reviews, pair programming, and static analysis tools for quality control.
  4. Write unit tests and integrate into CI pipelines.
  5. Continuously merge and build to detect integration problems early.

Artifacts: source code, build scripts, unit tests, compiled binaries or containers.

6) Verification & Testing

Objective: validate that the software meets requirements and verify that defects are found and removed.

Types of testing:

  • Unit testing — tests for individual functions or classes.
  • Integration testing — tests interactions between modules.
  • System testing — end-to-end testing of the complete system.
  • Acceptance testing — validation against requirements with stakeholders (UAT).
  • Non-functional testing — performance, load, stress, security, usability, accessibility.

Activities: prepare test plans, create test cases, execute tests, record defects, perform regression testing after fixes.

Artifacts: test plan, test cases, test reports, defect logs.

7) Deployment & Release

Objective: deliver the software to production and make it available to end-users.

Activities: packaging, release planning, environment provisioning, running deployment scripts, monitoring initial rollout, rollback planning.

Artifacts: release notes, deployment plan, runbooks, monitoring dashboards.

8) Maintenance & Evolution

Objective: support and evolve software after release: fix defects, adapt to new requirements, improve performance and security.

Activities: triage issues, release patches, plan new versions, technical debt reduction, refactoring.

Artifacts: change requests, patches, updated documentation, maintenance logs.

9) Retirement

Objective: properly decommission software when it's no longer useful: migrate data, inform users, archive artifacts, remove services safely.

Activities: data migration, compliance checks, disabling services, post-mortem/lessons learned.

Common SDLC Models and When to Use Them

SDLC is implemented through different models. Choose a model based on project uncertainty, stakeholder involvement, regulatory requirements, and speed of delivery needed.

Waterfall

Sequential phases without repetition. Works when requirements are stable and well understood (e.g., regulatory projects).

Pros: predictable schedule, clear milestones. Cons: late feedback, expensive change.

Iterative and Incremental

Build and deliver in increments; each iteration includes multiple SDLC phases. Good for medium uncertainty and when early delivery is beneficial.

Spiral

Risk-driven iterative model; each cycle addresses risks and refines requirements. Useful for large, complex, high-risk systems.

V-Model

Emphasizes verification and validation: each design phase has a corresponding testing activity. Good when testing rigor and traceability matter.

Agile (Scrum, Kanban)

Short iterations (sprints), continuous delivery of usable increments, strong stakeholder collaboration. Suitable when requirements evolve and time-to-market matters.

DevOps / Continuous Delivery

Cross-functional teams owning build, test, deployment and operations. Automate everything: CI, CD, monitoring. Ideal for rapid releases and operational feedback loops.

Model selection tip: combine models (e.g., Agile for feature delivery + DevOps for deployment + V-Model testing rigor) according to organizational needs.

Estimation Techniques (Theory and Simple Algorithms)

Good estimation balances accuracy, transparency and stakeholder expectations. Below are common techniques and their conceptual formulas.

COCOMO (Constructive Cost Model) — simplified idea

COCOMO estimates effort from lines of code (LOC) or function points. Basic COCOMO formula (very simplified):

Effort (person-months) = a * (KLOC)^b

Where a and b depend on project class (organic, semi-detached, embedded). This is a high-level model; use tool-supported COCOMO II for practical use. Caution: LOC-based models are sensitive to language and programmer style.

Function Points

Measure functionality from the user's perspective (inputs, outputs, queries, files). Translate function points to effort using historical productivity metrics.

Agile Estimation: Story Points & Velocity

Estimate stories with relative sizes (story points). Team measures velocity (story points per sprint) and forecasts how many sprints are needed.

Delphi / Wideband Delphi

A facilitated expert estimation process where anonymized estimates are iterated until consensus is reached. Reduces anchoring bias.

Simple estimation algorithm (pseudocode)

// Example: combine expert estimates with historical velocity
For each user story:
  collect n expert estimates (best, likely, worst)
  compute three-point estimate: E = (best + 4*likely + worst) / 6
Sum E for all stories to get total expected effort in story points
Estimate number of sprints = ceil(total_story_points / average_velocity)

Notes: use separate calculations for development vs testing effort; maintain a risk contingency.

Quality Assurance and Testing Strategies (Textual Guidance)

Quality is built throughout the SDLC — not only during a final test phase. Adopt multiple techniques:

  • Shift-left testing: write tests and do verification earlier (unit tests, TDD).
  • Automation: automate unit, integration, and regression tests in CI pipelines.
  • Exploratory testing: testers explore the software to find unexpected issues.
  • Performance testing: baseline and load tests under realistic scenarios.
  • Security testing: threat modeling, static analysis, penetration testing.

Test Case Prioritization — simple heuristic algorithm

Prioritize tests by risk, customer usage frequency, and recent code changes. A textual heuristic:

  1. Label tests covering high-risk requirements as priority 1.
  2. Label tests that exercise recent code changes as priority 1 or 2.
  3. Label critical path/user journeys as priority 1.
  4. Run priority 1 tests on every build; run full regression nightly or in scheduled pipelines.

Traceability

Maintain a Requirement Traceability Matrix (RTM) that maps requirements to tests, design elements, and code modules. This is essential for regulated domains and to verify coverage.

Example RTM columns: Requirement ID → Description → Design Item → Source Code Module → Test Case ID → Status

Risk Management, Metrics and KPIs

Identify and manage risks throughout the SDLC. A simple risk process: identify → analyze (probability & impact) → prioritize → treat (mitigate) → monitor.

Useful metrics

  • Defect density = defects / KLOC or per feature size.
  • Mean Time To Recovery (MTTR) / Mean Time Between Failures (MTBF).
  • Test coverage (note: coverage is a heuristic, not a guarantee).
  • Lead time, cycle time (for Agile/DevOps), deployment frequency.
  • Velocity (story points per sprint) and release predictability.

Metric advice: select a small set of actionable metrics and review them regularly; avoid using metrics to micromanage individuals.

Roles and Responsibilities

Business & Product

  • Product Owner / Sponsor — defines vision, prioritizes features.
  • Business Analyst — elicits and documents requirements.
  • UX / Product Designer — defines user interactions and designs.

Delivery & Operations

  • Project Manager / Scrum Master — coordinates and removes impediments.
  • Software Architect — defines architecture and quality attributes.
  • Developers — implement features and unit tests.
  • QA / Testers — design and execute tests and report defects.
  • DevOps / Site Reliability — build pipelines and manage production.

Documentation — What to Keep at Each Phase

SDLC emphasizes artifacts. Conservative list of minimum documents:

  • Planning: Project charter, initial scope and budget.
  • Requirements: SRS or product backlog, acceptance criteria.
  • Design: Architecture decisions, interface specifications.
  • Implementation: Code with inline documentation, API docs.
  • Testing: Test plans, test cases, defect reports.
  • Deployment: Release notes, deployment runbooks, rollback plans.
  • Maintenance: Change logs, incident reports, postmortems.

Documentation tip: prefer concise, living documents stored in version control or collaborative platforms so they stay up-to-date.

Tools and Automation (high-level)

Common tool categories used across SDLC:

  • Version control: Git, repository hosting (GitHub, GitLab, Bitbucket).
  • CI/CD: Jenkins, GitHub Actions, GitLab CI, CircleCI.
  • Issue tracking and planning: Jira, Azure Boards, Trello.
  • Testing frameworks: xUnit family, Selenium, JMeter.
  • Code quality: linters, static analysis (SonarQube), code review platforms.
  • Cloud & containers: Docker, Kubernetes, cloud providers for deployment.

Case Study — Step-by-step Example (Library Web App)

Scenario: build a simple library web application where users can search for books and borrow them.

Planning

Define stakeholders (library staff, users), outline goals (search, borrow, return, track loans), estimate initial budget and timeline (2-4 sprints), and identify constraints (privacy, limited budget).

Requirements

Examples of user stories:

  • "As a user I want to search books by title or author so that I can find items."
  • "As a member I want to borrow a book for 14 days so that I can read it at home."

Define non-functional requirements: system must handle 200 concurrent users, page load < 2s, and comply with data privacy rules.

Design

Choose a layered architecture: web UI → API layer → service layer → database. Sketch database: Books, Members, Loans tables. Decide to use RESTful APIs and token-based authentication.

Implementation

Split into sprints: first sprint implements search API and database schema with unit tests; second sprint implements borrowing workflows and basic UI.

Testing

Write unit tests for search logic; integration tests for API endpoints; system tests for end-to-end borrow/return flow; load tests to validate concurrency requirement.

Deployment

Automate builds and containerize the app. Deploy to a staging environment, perform acceptance testing with the library staff, then roll out to production with monitoring in place.

Maintenance

Monitor logs, fix issues, and plan enhancements (late fees, reservation feature). Maintain documentation and run periodic security reviews.

Practical Tips and Best Practices

  • Start with clear goals — ambiguous projects rarely succeed.
  • Invest in requirements validation and acceptance criteria up front.
  • Integrate testing into daily development (automated tests, CI).
  • Prefer small, frequent releases over infrequent big-bang deployments.
  • Track technical debt and plan explicit refactoring iterations.
  • Keep documentation concise, versioned, and close to the code.
  • Use data and retrospective feedback to continuously improve the process.
