Queue — Concept, Theory, Algorithms, and Educational Explanation

A comprehensive, readable guide to queues: definition, operations, multiple implementations, algorithmic complexity, pseudocode, invariants, concurrency considerations, use cases, and simple step demonstration.

1. What is a Queue?

A queue is a linear abstract data type that models a collection of elements arranged in a strict order. The defining behavioral property is FIFO — "First In, First Out". Items are added to the back (tail) and removed from the front (head). Think of a queue as a real-life line at a checkout: the first person to join is the first person served.

Core operations

  • enqueue(x): Insert element x at the back of the queue.
  • dequeue(): Remove and return the element at the front. Usually undefined or an error when the queue is empty.
  • peek() / front(): Return the front element without removing it.
  • isEmpty(): Test whether the queue contains no elements.
  • size(): Return the number of elements currently in the queue.

Consider these calls and a descriptive step-by-step state (front at left):

// Start: empty

3. Basic invariants and correctness

Every queue maintains a strict ordering invariant: elements are dequeued in the order they were enqueued. Implementation-specific invariants include:

  • If using indices (head/tail), the invariant "head points to the current front element (or to the position where the next element would be read when empty)" must be maintained.
  • When using a bounded circular buffer, the invariant distinguishing empty from full must be clear (common approaches: use a size counter, track capacity-1, or a flag).

4. Implementations

4.1 Simple array (shift on dequeue)

Represent the queue as a dynamic array. Enqueue: append at the end. Dequeue: remove from index 0 (shifting all remaining elements).

// Conceptual
enqueue(x): array.push(x)
dequeue():  if array.empty throw; else return array.shift()

Complexity: enqueue O(1) average (amortized, for dynamic array append), dequeue O(n) because shift moves many elements. Simple and easy but inefficient for frequent dequeues.

4.2 Circular buffer (ring buffer) — recommended for bounded queues

Use a fixed-size array with two indices, head and tail, and a count. When tail reaches end, it "wraps" to 0. This supports O(1) enqueue and dequeue. 

// Using array buffer[capacity], head=0, tail=0, size=0
enqueue(x):
  if size == capacity: error or resize
  buffer[tail] = x
  tail = (tail + 1) mod capacity
  size = size + 1

dequeue():
  if size == 0: error
  value = buffer[head]
  head = (head + 1) mod capacity
  size = size - 1
  return value

Key points: handle wrap-around correctly, and decide how to represent full vs empty. Many implementations keep a size counter to avoid ambiguity.

4.3 Singly linked list with head and tail pointers

Maintain pointers to the head (front) and tail (back). Enqueue at tail, dequeue from head — both O(1) time. 

// Node { value, next }
enqueue(x):
  node = new Node(x)
  if tail != null: tail.next = node
  tail = node
  if head == null: head = node
  size++

dequeue():
  if head == null: error
  value = head.value
  head = head.next
  if head == null: tail = null
  size--
  return value

Memory overhead: one node per element plus pointer fields. No fixed capacity unless artificially constrained.

6. Variants and related concepts

  • Deque (double-ended queue): allows inserts/removals at both ends.
  • Priority queue: not FIFO — elements are dequeued by priority value.
  • Blocking queue: used in concurrent programming: enqueue may block if full, dequeue may block if empty.
  • Concurrent queues: lock-based or lock-free implementations to support multiple producers/consumers.

7. Common use cases

  • Breadth-first search (BFS) in graphs and trees.
  • Task scheduling, event handling, message passing.
  • I/O buffering, print spoolers, pipelines.
  • Producer-consumer patterns where producers enqueue work and consumers dequeue work.

8. Example: BFS (textual walkthrough)

Given a graph and a start node S, BFS uses a queue to process nodes in layers.

// Pseudocode (high level)
queue.enqueue(S)
visited.add(S)
while not queue.isEmpty():
  v = queue.dequeue()
  for w in neighbors(v):
    if w not visited:
      visited.add(w)
      queue.enqueue(w)

Step-by-step: enqueue the start S; then repeatedly remove a node, explore its neighbors, and enqueue any unvisited neighbors. This naturally visits nodes in increasing distance (number of edges) from S.

9. Concurrency considerations

When multiple threads access a queue concurrently, race conditions arise unless access is synchronized. Common approaches:

  • Mutex/lock: Protect enqueue/dequeue operations with a lock.
  • Lock-free algorithms: Use atomic instructions (CAS) to implement thread-safe queues with high concurrency (e.g., Michael-Scott queue).
  • Blocking semantics: Use condition variables or semaphores so consumers wait when empty and producers wait when full (bounded buffer problem).

Bounded buffer (producer-consumer)

// Classic solution sketch:
mutex.lock()
while buffer.full():
  notFull.wait(mutex)      // producer sleeps until not full
enqueue(item)
notEmpty.signal()
mutex.unlock()

mutex.lock()
while buffer.empty():
  notEmpty.wait(mutex)     // consumer sleeps until not empty
item = dequeue()
notFull.signal()
mutex.unlock()

10. Edge cases and pitfalls

  • Off-by-one in circular buffers: correctly wrap indices and choose a representation for full vs empty.
  • When resizing a circular buffer, be careful to copy elements in logical order to new array.
  • Iteration over a queue usually should be snapshot-based or documented to avoid concurrent modification surprises.
  • Memory leakage in some languages: ensure removed nodes can be garbage-collected or freed.

11. Pseudocode: circular buffer with automatic resizing

// buffer: array of capacity
// head: index of front element
// size: number of elements
// tail = (head + size) mod capacity

ensureCapacity(required):
  if required <= capacity: return
  newCap = max(capacity*2, required)
  newBuf = new array[newCap]
  // copy logical order: elements i=0..size-1
  for i in 0..size-1:
    newBuf[i] = buffer[(head + i) mod capacity]
  buffer = newBuf
  capacity = newCap
  head = 0

enqueue(x):
  ensureCapacity(size + 1)
  buffer[(head + size) mod capacity] = x
  size++

dequeue():
  if size == 0: error
  value = buffer[head]
  buffer[head] = null // help GC
  head = (head + 1) mod capacity
  size--
  return value

13. Frequently asked questions (FAQ)

Q: When should I use a linked list vs an array-based queue?

A: Use a linked list when you need unbounded size without costly resizing and when memory overhead per element is acceptable. Use a circular buffer when you know an upper bound on size or care about compactness and contiguous memory (better cache performance).

Q: Is a priority queue a queue?

A: It is a related abstract data type but not FIFO: it dequeues the element with the highest (or lowest) priority. It is a different ADT with different semantics and performance trade-offs.

Tip: When designing systems with queues, choose an implementation based on expected operation mix (many dequeues vs many enqueues), memory constraints, and concurrency model.
For further study: read about Michael-Scott lock-free queues, concurrent ring buffers (LMAX Disruptor), and queueing theory (M/M/1 queues) for performance/latency analysis.
