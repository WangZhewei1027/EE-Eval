Topological Sort — A Comprehensive Explanation

What it is, why it works, algorithms, proofs, examples, practice exercises and notes for implementation. This page is primarily textual and educational; a single button near the bottom triggers a small step-by-step demonstration of Kahn’s algorithm for a fixed example.

Overview and definition

A topological sort (or topological ordering) of a directed graph is a linear ordering of its vertices such that for every directed edge u → v, vertex u appears before v in the ordering.

Important constraint: A topological ordering exists iff the graph is a directed acyclic graph (DAG) — that is, it has no directed cycles.

When and why we use it

Topological sort is used whenever you need to schedule or order items subject to precedence constraints:

  • Build systems (compile targets with dependencies).
  • Task scheduling with dependency constraints.
  • Course prerequisite planning (order courses so prerequisites come first).
  • Dependency resolution in package managers and compilers.
  • Many algorithms on DAGs (e.g., longest path in a DAG, dynamic programming on DAGs) begin by computing a topological ordering.

Formal properties

Existence
A topological ordering exists exactly when the directed graph has no directed cycle (i.e., is a DAG).
Non-uniqueness
A DAG can have multiple valid topological orders. The ordering is unique only when the DAG defines a total order (for example, each consecutive pair is constrained) — equivalently, when the DAG has a Hamiltonian path that respects all edges or when every consecutive pair of vertices in the ordering is connected by a directed edge in that order.
Partial orders
Topological ordering corresponds to any linear extension of the partial order defined by reachability in the DAG.

Intuition

Consider tasks that depend on other tasks. A topological order is simply any valid sequence in which you can perform all tasks so that no task is started before its prerequisites are complete. The simplest way to think about it:

  • Find nodes that have no prerequisites (in-degree 0) — you can do these first.
  • Remove them from the system (or mark them done), and update the prerequisites counts for remaining nodes.
  • Repeat.

Two classic algorithms

There are two common ways to compute a topological order:

  1. Kahn’s algorithm (BFS-like, using in-degrees).
  2. DFS-based algorithm using reverse finishing times (postorder).

Kahn's algorithm — step-by-step description

Idea: Repeatedly remove vertices with in-degree zero and append them to the output list.

High-level steps:

  1. Compute the in-degree (number of incoming edges) for every vertex.
  2. Place all vertices with in-degree 0 into a queue (or another container).
  3. While the queue is not empty:
    1. Remove a vertex u from the queue and append u to the output list.
    2. For each edge u → v, reduce the in-degree of v by 1. If in-degree(v) becomes 0, add v to the queue.
  4. If you output all vertices, the output is a topological ordering. If some vertices remain (you cannot reach in-degree 0), the graph has a cycle.
// Pseudocode: Kahn's algorithm
compute in-degree[v] for all v
Q = queue of all v with in-degree[v] == 0
output = empty list
while Q is not empty:
    u = Q.pop()
    output.append(u)
    for each edge (u -> v):
        in-degree[v] -= 1
        if in-degree[v] == 0:
            Q.push(v)
if output.size() != number_of_vertices:
    report "Graph has a cycle (no topological ordering)"
else:
    output is a valid topological order

DFS-based algorithm — step-by-step description

Idea: Perform a depth-first search and record the vertices in the order of finishing times. Reverse that finish-order to obtain a topological order.

  1. Perform a DFS over the graph. Track visited nodes and recursion stack to detect cycles if you want an explicit cycle check.
  2. When a vertex finishes (i.e., after all its outgoing neighbors have been visited), push it onto a stack or append it to the front of a list.
  3. When DFS is complete, the stack (or the reversed finish time list) contains a topological ordering.
// Pseudocode: DFS-based topological order
visited[v] = false for all v
stack = empty (or list)
function dfs(u):
    visited[u] = true
    for each edge (u -> v):
        if not visited[v]:
            dfs(v)
    // after exploring all descendants
    stack.push(u)

for each vertex v:
    if not visited[v]:
        dfs(v)

output = pop all vertices from stack (last pushed first) // or reverse the list of finishing times

Why these algorithms are correct (sketches)

Kahn's algorithm correctness sketch

In-degree zero nodes have no incoming edges, so they have no prerequisites and can safely appear first. Removing a node with in-degree 0 and its outgoing edges cannot create new cycles, and when in-degree of another node drops to 0, it means all its prerequisites have been output. If the algorithm outputs all vertices, each edge u → v has u output before v because the edge is removed at the moment u is processed. If a cycle exists, no node in the cycle can ever get in-degree 0, so the algorithm will output fewer than n nodes and correctly report the cycle.

DFS-based algorithm correctness sketch

In DFS, when you finish a node u you have already visited all nodes reachable from u (its descendants). Therefore, u should appear before none of its descendants in the final linear order — in fact, descendants must come after u in the reversed finishing sequence. Thus reversing finishing times yields an order where every edge u → v satisfies u before v. If a back edge exists (edge to an ancestor in recursion stack), then there is a cycle and DFS can detect it using a recursion stack marker.

Complexity analysis

  • Both Kahn's algorithm and DFS-based algorithm run in O(V + E) time, where V is the number of vertices and E the number of edges, because each vertex and edge is processed a constant number of times.
  • Space complexity: O(V + E) to store the adjacency list plus O(V) for auxiliary arrays (in-degree, visited, stack, queue).

Example 1 — Simple DAG (unique-ish ordering)

Graph (edges):

A -> C B -> C C -> D B -> E D -> F E -> F

There are multiple valid topological orders; one valid order is: A, B, C, E, D, F (or A, B, E, C, D, F depending on queue order).

Example 2 — Multiple valid orders

Graph: two independent chains 1→2→3 and 4→5. A topological order can interleave items from both chains, e.g. 1,2,3,4,5 or 4,1,2,3,5 or 1,4,2,3,5 etc. As long as the local order of each chain is preserved, the merged order is valid.

When the topological ordering is unique

A DAG has a unique topological ordering iff every pair of consecutive vertices in the ordering is connected by a directed edge (equivalently, the DAG has a Hamiltonian path that respects all precedence relations), or more usefully:

During Kahn's algorithm, a sufficient and necessary condition for uniqueness is that at every step the queue contains exactly one vertex. If at any step there are two or more choices, then you can construct at least two different valid orders by choosing them in different orders.

Practical implementation notes

  • Represent graphs using adjacency lists (vector/list per node) for O(V+E) performance.
  • If you need a deterministic smallest-lexicographic topological order, use a priority queue (min-heap) instead of a queue in Kahn's algorithm. This raises the runtime to O((V+E) log V) because of heap operations.
  • For cycle detection: Kahn will detect if it outputs fewer than V nodes. DFS can detect back edges using a recursion stack marker.
  • Beware of 1-based vs 0-based indices and input parsing when implementing in contest or production settings.

Variations and related problems

  • Lexicographically smallest topological order (use a priority queue in Kahn).
  • Count the number of topological orderings — a #P-complete problem in general.
  • Find a topological ordering consistent with additional constraints (e.g., tie-breakers): still solvable by adjusting data structures.
  • Longest path in a DAG — compute distances by relaxing edges in topological order (single pass).

Common pitfalls and misconceptions

  • Topological sorting only applies to directed graphs. For undirected graphs, the notion doesn't make sense in this form.
  • Cycles prevent topological orderings. Even a single directed cycle anywhere in the graph blocks any global ordering that respects all edges.
  • Disconnected DAGs are fine — topological orderings interleave nodes from different components arbitrarily (as long as local constraints are satisfied).
  • Assuming uniqueness without checking the queue size during Kahn’s algorithm can be misleading.

Exercises (with guidance)

Exercise 1: Given edges: 1→3, 2→3, 3→4, 2→5. Find two different topological orderings.
Two valid orders: 1,2,3,5,4 or 2,1,3,5,4 (or 1,2,5,3,4 — many variants). Notice 3 must come before 4; 2 must come before 3 and 5; 1 only constrained to be before 3.
Exercise 2: Is topological ordering possible for a graph with edges 1→2, 2→3, 3→1? Explain.
No. The graph contains the directed cycle 1→2→3→1, so no topological order exists.

Sample implementations (concise notes)

Both algorithms are short to implement. For Kahn’s algorithm use an array of in-degrees, a queue and an adjacency list. For DFS, a recursive DFS with a vector for order and a cycle detection marker suffices.
