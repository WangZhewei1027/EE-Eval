Definition

A Binary Search Tree (BST) is a binary tree with an ordering property that enables efficient search, insertion, and deletion of comparable keys. Formally, each node contains a key (and possibly associated value). For every node:

  • All keys in the left subtree are strictly less than the node's key (assuming no duplicates). If duplicates are allowed, a convention is adopted (e.g., duplicates go to the right).
  • All keys in the right subtree are strictly greater than the node's key.

This ordering property is what distinguishes a BST from an arbitrary binary tree and is the basis for logarithmic-time operations in balanced trees.

Key properties

  • BST is ordered: in-order traversal visits nodes in non-decreasing sorted order.
  • Height (h): length of the longest root-to-leaf path. Performance depends on h.
  • If the tree is balanced, h = O(log n). In the worst case (degenerate), h = O(n) and operations degrade to linear time.
  • A BST does not define balancing rules by itself — balancing is provided by variants like AVL and Red-Black trees.

Basic operations and algorithms

We describe the core operations: search, insert, and find-min/find-max. For clarity, pseudocode and step-by-step narratives are provided.

Search (recursive and iterative)

Goal: determine whether a key k is present in the BST and return its node.

// Recursive search
function BST_Search(node, k):
    if node == null:
        return null       // not found
    if k == node.key:
        return node
    else if k < node.key:
        return BST_Search(node.left, k)
    else:
        return BST_Search(node.right, k)

// Iterative search
function BST_Search_Iter(root, k):
    current = root
    while current != null:
        if k == current.key:
            return current
        else if k < current.key:
            current = current.left
        else:
            current = current.right
    return null

Narrative: Start at the root. Compare the target with current node's key. If equal, found. If smaller, go left; if larger, go right. Continue until you find it or reach a null pointer (not found).

Insert

Insert maintains the BST invariant. You locate the correct null child where the new node should be attached.

// Recursive insert (returns new subtree root)
function BST_Insert(node, key):
    if node == null:
        return new Node(key)
    if key < node.key:
        node.left = BST_Insert(node.left, key)
    else:
        node.right = BST_Insert(node.right, key)
    return node

// Iterative insert (parent pointer)
function BST_Insert_Iter(root, key):
    parent = null
    current = root
    while current != null:
        parent = current
        if key < current.key:
            current = current.left
        else:
            current = current.right
    newNode = new Node(key)
    if parent == null:
        return newNode     // tree was empty
    else if key < parent.key:
        parent.left = newNode
    else:
        parent.right = newNode
    return root

Narrative: Walk down from the root as if searching for the key; when a null child is reached, add the new node there. The structure remains a BST by construction.

Find-min and find-max

Find the minimum key by following left children; find maximum by following right children.

function BST_Min(node):
    if node == null:
        return null
    while node.left != null:
        node = node.left
    return node

function BST_Max(node):
    if node == null:
        return null
    while node.right != null:
        node = node.right
    return node

Deletion — the trickiest operation

Deleting a node must preserve the BST property. There are three cases to consider for the node to be deleted (call it z):

  1. z has no children (leaf): remove it simply by replacing it with null.
  2. z has one child: replace z with its child (connect parent of z to z's child).
  3. z has two children: this is the subtle case. Replace z's key with either the in-order predecessor (maximum in z's left subtree) or the in-order successor (minimum in z's right subtree), and then delete that predecessor/successor node (which will have at most one child). Commonly, the successor is used.

Using the in-order successor ensures that when you replace z's key with successor's key, the BST property remains valid because the successor is the smallest key larger than z.key.

// Delete using successor
function BST_Delete(root, key):
    if root == null:
        return null
    if key < root.key:
        root.left = BST_Delete(root.left, key)
    else if key > root.key:
        root.right = BST_Delete(root.right, key)
    else: // root.key == key -> delete this node
        if root.left == null:
            return root.right
        else if root.right == null:
            return root.left
        else:
            // both children exist: find successor (min in right subtree)
            succ = BST_Min(root.right)
            root.key = succ.key
            // delete successor node from right subtree
            root.right = BST_Delete(root.right, succ.key)
    return root

Narrative (two-child case): Suppose z has both children. Find the successor s (the left-most node in z's right subtree). s's key is guaranteed to be > all keys in z.left and < all other keys in z.right after s. Copy s.key into z. Then recursively delete s from z's right subtree. Because s is minimum in that subtree, s has no left child; this reduces deletion to case 1 or 2.

Example deletion walkthrough

Delete 50 from this tree:

        50
       /  \
      30  70
     / \  / \
    20 40 60 80
  1. 50 has two children. Find successor: minimum in right subtree → 60.
  2. Replace 50's key with 60: root now holds 60.
  3. Delete the original node containing 60 (which is in the right subtree). Node 60 is a leaf or has at most one child, so removal is simple.

Tree traversals and their uses

Traversals visit all nodes in a particular order; they are fundamental for many algorithms.

In-order (Left, Node, Right)

Produces the keys in sorted increasing order for a BST. Useful for extracting sorted sequences.

Pre-order (Node, Left, Right)

Useful for copying the tree structure or for prefix expressions.

Post-order (Left, Right, Node)

Useful for deleting/freeing nodes or postfix expressions.

// In-order recursive
function InOrder(node):
    if node == null: return
    InOrder(node.left)
    visit(node)
    InOrder(node.right)

// Pre-order recursive
function PreOrder(node):
    if node == null: return
    visit(node)
    PreOrder(node.left)
    PreOrder(node.right)

// Post-order recursive
function PostOrder(node):
    if node == null: return
    PostOrder(node.left)
    PostOrder(node.right)
    visit(node)

Note: Iterative traversals use explicit stacks; in-order iterative is often used to avoid recursion depth issues.

Time and space complexity

Let n be the number of nodes and h the height of the tree.

  • Search, Insert, Delete: O(h) time. In a balanced BST, h = O(log n), so operations are O(log n). In the worst-case (degenerate linked-list-like tree), h = O(n), so operations are O(n).
  • Find-min/Find-max: O(h).
  • Traversal visiting all nodes: O(n) time and O(h) auxiliary space for recursive stack (or O(n) if explicit container used for traversal results).
  • Space: O(n) for storing nodes. Auxiliary recursion stack O(h).

Why search complexity depends on height

Each step of search descends one level in the tree; the number of steps is at most the depth of the found node (or the height if not found). Therefore, the time is proportional to the path length from root to a leaf, i.e., O(h).

Balanced variants and augmentations

BSTs can be extended to provide guaranteed logarithmic operations:

  • AVL trees — strict balancing using node heights and rotations; guarantee O(log n) operations with a slightly higher constant factor for rotations.
  • Red-Black trees — color-based balancing that offers amortized O(log n) operations and is commonly used in language libraries (e.g., Java TreeMap uses Red-Black).
  • Splay trees — self-adjusting trees that move recently accessed elements near the root (good for locality); amortized guarantees.

Order-statistics tree (augmented BST)

A BST can be augmented to store the size of each subtree in every node, enabling operations like "select(k)" (find the k-th smallest element) and "rank(x)" (find the number of elements less than x) in O(h) time.

// Maintain node.size = 1 + size(left) + size(right)
// Select(k): find k-th smallest
function Select(node, k):
    if node == null: return null
    leftSize = size(node.left)
    if k == leftSize + 1:
        return node
    else if k <= leftSize:
        return Select(node.left, k)
    else:
        return Select(node.right, k - leftSize - 1)

Common pitfalls and practical tips

  • Unbalanced trees: naive insertion order (sorted inputs) can degrade performance. Use balancing or randomized insertion (e.g., treaps) if you need guarantees.
  • Duplicate keys: decide a policy up-front (disallow duplicates, or store counts, or place duplicates consistently left or right). Inconsistent handling leads to bugs.
  • Recursive depth / stack overflow: deep trees cause recursion stack overflow. Use iterative implementations for production code or ensure tree is balanced.
  • Pointer management and parent pointers: many textbook algorithms assume parent pointers. If omitted, handle parent updates carefully during delete/rotate operations.
  • When deleting a node and replacing it with successor/predecessor, ensure you handle attachment of children correctly and update any augmented fields (like subtree sizes or heights).

Guided answers (brief)
