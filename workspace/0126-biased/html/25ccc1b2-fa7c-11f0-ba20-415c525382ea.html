Congestion Control in Computer Networks

Introduction

Congestion Control is a fundamental concept in computer networking that refers to techniques and algorithms designed to prevent network congestion — a state where network resources become overloaded with too much traffic, leading to packet losses, delays, and degradation of overall network performance.

Modern networks, especially the Internet, rely heavily on congestion control mechanisms to maintain efficient data transmission and fair sharing of limited bandwidth among users and applications.

What is Congestion?

When the demand for network resources (such as link bandwidth, buffer space) exceeds available capacity, the network experiences congestion.

This overloading causes buffers in routers or switches to fill up, forcing packet drops and retransmissions, which worsen the congestion problem in a feedback loop. Congestion can degrade network throughput and increase latency, jitter, and packet loss.

"Congestion is the phenomenon in which too many packets are present in the subnet, and performance degrades due to loss and delay."

Key symptoms of congestion:

  • Increased packet loss rates
  • Higher end-to-end delay for data transmission
  • Reduced throughput
  • Increased jitter (variation in delay)

Importance of Congestion Control

Without congestion control, a network can become unstable and inefficient, leading to:

  • Collapse of network throughput as retransmissions overload the network
  • Unfair bandwidth sharing among users or applications
  • Severe delays and unreliable communication, especially harmful for real-time services

Congestion control ensures:

  • Stable and efficient network performance
  • Fairness among multiple simultaneous data flows
  • Adaption of sending rates to network conditions

How Does Congestion Control Work?

Congestion control mechanisms typically function end-to-end between communicating hosts (e.g., computers, servers). They adjust the rate of data transmission based on feedback about current network conditions.

The key idea is to detect signs of congestion early and then reduce the transmission rate to allow the network to recover.

Approaches to congestion control can be divided into:

  • Open-loop control: Designed to avoid congestion before it happens using admission control policies or traffic shaping.
  • Closed-loop control: Monitors network feedback (like packet loss, delay changes) and adjusts transmission rates dynamically.

Most modern protocols (such as TCP) use closed-loop feedback mechanisms.

Congestion Control in TCP (Transmission Control Protocol)

TCP, the dominant transport protocol on the Internet, contains built-in congestion control mechanisms to regulate data flow between sender and receiver. TCP congestion control adapts the rate of packet transmission to avoid overwhelming intermediate routers.

Key Parameters and Concepts

  • Congestion Window (cwnd): A sender-side variable limiting the amount of data the sender can inject into the network before receiving an acknowledgment (ACK).
  • Slow Start Threshold (ssthresh): The point at which TCP shifts from slow start to congestion avoidance.
  • Round-Trip Time (RTT): The time it takes a packet to go from sender to receiver and back; crucial for timing retransmissions.

TCP Congestion Control Algorithms

TCP controls congestion through four major phases/algorithms:

  1. Slow Start (SS):

    At the start of a connection, TCP begins with a small congestion window (usually 1 or 2 segments) and increases it exponentially with each acknowledgment received. This rapid growth helps discover the available network capacity quickly.

    Specifically, the cwnd doubles every RTT until it reaches ssthresh or packet loss is detected.

  2. Congestion Avoidance (CA):

    After reaching ssthresh, TCP grows the congestion window more cautiously, increasing it linearly (by roughly one segment per RTT). This tries to find the maximum sustainable transmission rate without causing congestion.

  3. Fast Retransmit:

    When the sender notices multiple duplicate ACKs (indicating packet loss but not enough to timeout), it retransmits the missing packet immediately without waiting for a retransmission timer to expire.

  4. Fast Recovery:

    Following a fast retransmit, TCP reduces the cwnd (often to ssthresh) and enters the congestion avoidance phase rather than slow start. This allows faster recovery while being cautious.

Example of TCP Congestion Window Evolution:

Start with cwnd = 1, ssthresh = 16

  • After 1 RTT with no loss: cwnd = 2 (doubling)
  • After 2 RTTs: cwnd = 4
  • After 3 RTTs: cwnd = 8
  • After 4 RTTs: cwnd = 16 → Switch to congestion avoidance
  • After 5 RTTs: cwnd = 17 (linear increase)
  • …until packet loss → set ssthresh = cwnd/2, reset cwnd

Other Congestion Control Algorithms and Techniques

Crowded networks have inspired specialized congestion control mechanisms beyond traditional TCP:

Random Early Detection (RED)

RED is a router-level algorithm that monitors average queue sizes. Instead of waiting until buffers overflow, RED preemptively drops packets probabilistically when congestion begins to build, signaling senders to slow down before severe congestion sets in.

Explicit Congestion Notification (ECN)

ECN allows routers to mark packets instead of dropping them to indicate impending congestion. End hosts that receive marked packets react by reducing transmission rates, thus preventing loss.

Rate-Based and Window-Based Controls

  • Rate-Based: Sending data at a controlled rate (bits per second), often used in real-time multimedia streaming.
  • Window-Based: Limiting the number of outstanding packets (such as TCP’s cwnd).

High-Speed TCP Variants

For high bandwidth-delay product networks (e.g., long-distance fiber links), traditional TCP is inefficient. Variants like:

  • TCP Cubic (default in many Linux kernels)
  • TCP Vegas (delay-based congestion control)
  • TCP BBR (model-based approach focused on bottleneck bandwidth and RTT)

improve performance by using alternative strategies to increase throughput and reduce congestion.

Step-by-Step Explanation: How TCP Detects and Responds to Congestion

  1. Start Transmission: TCP begins sending packets with a small congestion window.
  2. Monitor ACKs: The sender counts acknowledgments marking delivered packets.
  3. Increase cwnd: If ACKs arrive without issues, cwnd increases according to slow start or congestion avoidance rules.
  4. Detect Packet Loss:
    • If a retransmission timeout expires without ACK - signals loss.
    • If three or more duplicate ACKs are received - fast retransmit triggers.
  5. Reduce cwnd and ssthresh: On detecting congestion, the sender reduces cwnd and sets ssthresh to half of the current window size.
  6. Resume Transmission: Depending on the event (timeout or fast retransmit), TCP re-enters slow start or congestion avoidance to cautiously ramp up sending rate.

Summary

  

    Congestion control is vital for maintaining network stability and efficiency. Through mechanisms such as TCP slow start, congestion avoidance, and loss detection, networks can adapt dynamically to traffic demands and avoid collapse under heavy load.
  

  

    Understanding congestion control is important for network engineers, developers of network protocols, and anyone interested in how data moves reliably across the Internet.
