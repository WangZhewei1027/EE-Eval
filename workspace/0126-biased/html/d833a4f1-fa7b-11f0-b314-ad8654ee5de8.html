Red‑Black Tree — A Comprehensive Educational Guide

A self-contained, text-first explanation of the structure, invariants, algorithms, and reasoning behind Red‑Black Trees. Detailed theory, step-by-step cases, pseudocode, proofs sketch, and examples are provided for learning and teaching.

Overview — What is a Red‑Black Tree?

A Red‑Black Tree is a balanced binary search tree (BST) that guarantees O(log n) time for fundamental operations (search, insertion, deletion). It enforces additional constraints (coloring and structural rules) to keep the tree approximately balanced without requiring full rebalancing on each operation.

Key ideas

  • Each node has a color: red or black.
  • Extra rules (invariants) on colors and structure ensure that the longest path is no more than twice the shortest path from root to leaf, providing balance.
  • Operations (insert/delete) change colors and perform local rotations to restore the invariants.

Fundamental Invariants (Rules)

These are the canonical properties every valid Red‑Black Tree must satisfy:

  1. Each node is colored either red or black.
  2. The root is always black.
  3. All leaves (NIL pointers) are considered black. In textbooks, NIL is a sentinel black node.
  4. If a node is red, then both its children are black (no two red nodes can be adjacent).
  5. Every path from a node to any of its descendant NIL leaves contains the same number of black nodes (the black-height).

Notes: The rules about NIL leaves being black and black-height equality produce the balancing effect. The black-height of the tree is used in proofs about height bounds.

Why Red‑Black Trees Work — Intuition and Height Bound

Intuition: By disallowing consecutive red nodes and ensuring equal black-height on every root-to-leaf path, the tree avoids extremely skewed shapes. Red nodes act as "extra" nodes that can be placed on a path without increasing the black-height; at most one red between blacks preserves near balance.

Height bound (sketch)

Let bh be the black-height of the root (number of black nodes from root to leaf excluding the root's parent). Any root-to-leaf path contains bh black nodes and at most bh red nodes (because reds cannot be consecutive). Thus the maximum path length is at most 2*bh. The minimum number of internal nodes n in such a tree with black-height bh satisfies: 

Minimum nodes n(bh) ≥ 2^bh - 1
So bh ≤ log2(n+1)
Thus height h ≤ 2*bh ≤ 2*log2(n+1)

Conclusion: height is O(log n), yielding logarithmic-time operations.

Basic Building Blocks: Rotations

Rotations change the local structure while preserving the binary search tree order. They are elementary and used to restore balance after inserts/deletes.

Right Rotation (visual)

    y                       x
   / \        right        / \
  x   C     ------>       A   y
 / \                         / \
A   B                       B   C

Left rotation is symmetric: rotate left at x to move right child up.

Rotations only rearrange pointers and do not change colors (colors may be updated separately as algorithm demands).

Insertion — Algorithm and Cases

Insertion follows two phases: standard BST insert with color red for the new node, and then a fix-up phase to restore Red‑Black properties.

Why insert a new node as red?

Inserting red avoids immediately increasing black-height of paths; we only recolor or rotate as needed to fix red-parent violations.

Pseudocode (high level)

RB-Insert(root, z):
  1. Insert z like a normal BST node; set z.color = RED.
  2. Call RB-Insert-Fixup(root, z) to restore properties.
  3. Ensure root.color = BLACK.

Fix-up cases (textual step-by-step)

Consider node z (just inserted and colored red). While z's parent is red, we have a violation (two reds in a row). There are two symmetric situations depending on whether z's parent is a left or right child.

Assume P = parent(z), G = grandparent(z), U = uncle(z).

  1. Case 1 — Uncle U is red:
    • Action: Recolor P and U to black, G to red, and set z = G (move up) to continue fixing.
    • Effect: Moves the red violation up; preserves black-height. Possibly root recoloring at end.
  2. Case 2 — Uncle U is black and z is a right child (assuming P is left child):
    • Action: Perform left rotation on P (z becomes P), converting to Case 3.
    • Effect: Prepares configuration for Case 3 by making z a left child.
  3. Case 3 — Uncle U is black and z is a left child (P is left child):
    • Action: Recolor P black, G red, then do a right rotation on G.
    • Effect: Restores properties locally; termination occurs because a red parent above may now be resolved.

Symmetric steps are applied when P is the right child. After the loop, recolor the root black.

Example: Insert sequence (textual)

  1. Start with empty tree. Insert 10 → becomes black root.
  2. Insert 20 → inserted red as right child of 10. Parent is black → no fix needed.
  3. Insert 30 → inserted red as right child of 20. Parent 20 is red → violation. Uncle is NIL (black). Since z is right child and parent is right child, rotate left at 10 and recolor appropriately. Result: 20 becomes root (black), 10 left (red), 30 right (red) — but then recolor may enforce parent black/children red etc.

Deletion — Overview and Core Ideas

Deletion is more complex than insertion because removing a node may change black-heights of paths. The deletion algorithm removes a node as in BST deletion, then performs a fix-up to restore invariants.

High-level approach

  • Perform standard BST deletion: if node has two children, swap with successor and remove (like in ordinary BST).
  • Track a node x which replaces the deleted node position; if the removed node was black, x may have a "double-black" problem (a deficit in black-height) that must be fixed by recoloring and rotations.

Fix-up cases (textual summary)

Let x be the current node potentially carrying an extra black. Let w be x's sibling.

  1. Case 1 — w is red:
    • Action: Recolor w black, parent red, rotate at parent (left if x is left child). This transforms into another case with a black sibling.
  2. Case 2 — w is black and both w's children are black:
    • Action: Recolor w red, move x up to parent (removes one black from both x and sibling paths); continue loop.
  3. Case 3 — w is black, w's left child is red, right child is black (for x left child):
    • Action: Recolor w left child black, w red, rotate right at w. Now in Case 4.
  4. Case 4 — w is black, w's right child is red (for x left child):
    • Action: Copy parent's color to w, set parent black, set w's right child black, rotate left at parent. Clear x's extra black and finish.

There are symmetric versions when x is the right child. This set of cases ensures the double-black is resolved in O(log n) operations.

Complete Pseudocode (Concise)

-- Insertion
RB-Insert(T, z):
  Insert z like BST, z.color = RED
  while z.parent.color == RED:
    if z.parent == z.parent.parent.left:
      y = z.parent.parent.right  -- uncle
      if y.color == RED:
        z.parent.color = BLACK
        y.color = BLACK
        z.parent.parent.color = RED
        z = z.parent.parent
      else:
        if z == z.parent.right:
          z = z.parent
          Left-Rotate(T, z)
        z.parent.color = BLACK
        z.parent.parent.color = RED
        Right-Rotate(T, z.parent.parent)
    else: (mirror image)
  T.root.color = BLACK

-- Deletion (sketch)
RB-Delete(T, z):
  y = z (node to be removed)
  original_color = y.color
  if z.left == NIL:
    x = z.right
    Transplant(T, z, z.right)
  else if z.right == NIL:
    x = z.left
    Transplant(T, z, z.left)
  else:
    y = Tree-Minimum(z.right)
    original_color = y.color
    x = y.right
    if y.parent == z:
      x.parent = y
    else:
      Transplant(T, y, y.right)
      y.right = z.right; y.right.parent = y
    Transplant(T, z, y)
    y.left = z.left; y.left.parent = y
    y.color = z.color
  if original_color == BLACK:
    RB-Delete-Fixup(T, x)

-- RB-Delete-Fixup: uses cases with sibling w to fix double-black.

This pseudocode captures the control flow but each function (Transplant, rotations, Tree-Minimum) is the usual BST helper.

Sequence: 10, 20, 30, 15

  1. Insert 10:
    • Tree empty → new node 10 colored RED initially then root recolored BLACK. Final: 10(B).
  2. Insert 20:
    • 20 inserted as right child of 10, colored RED. Parent 10 is BLACK — no fix. Final: 10(B) with right child 20(R).
  3. Insert 30:
    • 30 inserted as right child of 20 (color RED).
    • Parent P=20 is RED → violation. Uncle U is NIL (black).
    • Since z is right child and P is right child (mirror of left-case), perform left rotation at G=10, then recolor P black and G red.
    • Result: 20(B) root, 10(R) left, 30(R) right. Finally recolor root black if necessary (it already is black in this case).
  4. Insert 15:
    • 15 inserted as left child of 20? Actually 15 < 20, > 10, so it is right child of 10.
    • 15 colored RED. Parent 10 is RED → violation. Uncle 30 is RED.
    • Case 1 applies: recolor parent(10) and uncle(30) to BLACK, grandparent (20) to RED. Set z = 20 and continue.
    • Now z is root (20). Root recolored BLACK at end. Final tree: 20(B), left child 10(B) with right child 15(R), right child 30(B).

This walkthrough demonstrates cases 1/2/3 interplay and how rotations + recoloring restore invariants.

Complexity and Practical Notes

  • Time complexity:
    • Search: O(log n)
    • Insert: O(log n) amortized (actually worst-case O(log n) too)
    • Delete: O(log n)
  • Space: O(n) for node storage; commonly each node stores color, key, pointers to left, right, parent.
  • Practical considerations:
    • Red‑Black Trees are widely used in libraries (e.g., std::map in several standard libs, some implementations of balanced associative containers).
    • They provide good worst-case guarantees and relatively simple constant-time fixes (rotations and recolors).
    • Alternative self-balancing BSTs include AVL trees (stricter balance, faster lookups but more rotations on inserts/deletes) and B-trees (for disk-based storage).

Formal Proof Sketches (Why Invariants Hold and Terminate)

Termination of fix-up

For insertion fix-up: each loop iteration either performs recoloring that moves the problem up to the grandparent (reducing the number of red-red violations but possibly increasing depth of inspection), or performs a rotation that directly resolves the violation locally. The root eventually becomes black, and since each recoloring moves up in the tree, the loop can run at most O(height) times → O(log n).

Preservation of BST order

Rotations preserve in-order ordering of nodes: left‑rotation and right‑rotation only rewire nodes while keeping the left-subtree keys less than the parent and right-subtree keys greater than the parent. Recoloring changes only colors, not keys or structure.

Black‑height invariant maintenance

Insert fix-up ensures that black-heights remain equal by careful recoloring and, when necessary, rotating which moves black nodes without changing counts on paths inappropriately. Delete fix-up addresses deficits by redistributing blacks through the sibling and parent.

Compact Reference: Quick Reminders

  • New nodes are placed red.
  • Fix insert when parent is red: check uncle color → recolor or rotate.
  • Delete is more involved: track the double-black and use sibling cases.
  • Root must always be black.
  • No red node may have a red child.
