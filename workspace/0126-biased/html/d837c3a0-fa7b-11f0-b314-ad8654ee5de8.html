CPU Scheduling — Comprehensive Guide

A thorough, pedagogical explanation of CPU scheduling: concepts, objectives, common algorithms, examples, pseudocode, trade-offs, and practical considerations.

What is CPU Scheduling?

CPU scheduling is the activity performed by an operating system kernel to decide which of the processes in the ready queue will be assigned to the CPU. The scheduler's job is to maximize desirable performance metrics (throughput, CPU utilization, fairness) while minimizing undesirable metrics (waiting time, turnaround time, response time, and missed deadlines in real-time systems).

Key terms and simple definitions

  • Process/Thread: Execution entities that require the CPU to run code; threads are light-weight processes.
  • CPU burst: A continuous period during which a process uses the CPU.
  • I/O burst: Time a process spends waiting for I/O; alternating with CPU bursts.
  • Ready queue: Set of processes that are ready to run but waiting for the CPU.
  • Context switch: Overhead work where the kernel saves the state of one process and loads the state of another.
  • Preemptive scheduling: A running process can be interrupted (preempted) to schedule another process.
  • Non-preemptive scheduling: A running process continues until it blocks or finishes; it cannot be preempted by the scheduler.

Objectives and Metrics

Schedulers are judged by measurable outcomes. Important metrics include:

  • CPU Utilization: Fraction of time the CPU is busy. Higher is better (but context switching costs exist).
  • Throughput: Number of processes completed per time unit.
  • Turnaround time: Completion time − arrival time. Measures how long a process takes from submission to completion.
  • Waiting time: Time spent in ready queue waiting for the CPU. Turnaround minus CPU burst times.
  • Response time: Time from submission until the first response (important for interactive systems).
  • Fairness: No process should be starved indefinitely (unless policy explicitly allows).
  • Deadline miss rate: For real-time systems, ratio of deadlines missed.

Formulas (common):

Turnaround = Completion − Arrival Waiting = Turnaround − Total CPU time Response = Time of first scheduling − Arrival

Common Scheduling Algorithms

Below are widely taught algorithms with conceptual descriptions, pros and cons, and pseudocode sketches.

1. First-Come, First-Served (FCFS)

Non-preemptive. Processes are scheduled in order of arrival. Simple to implement (queue).

  • Advantages: Easy, fair in arrival order, minimal scheduling overhead.
  • Disadvantages: Convoy effect — long jobs block shorter ones, can yield large average waiting time.

Pseudocode (non-preemptive):

readyQueue.enqueue(processes by arrival order) while readyQueue not empty: p = readyQueue.dequeue() run p to completion
Example (FCFS):
Consider processes with (Arrival, Burst): P1(0, 5), P2(1, 3), P3(2, 8), P4(3, 6). FCFS order: P1 → P2 → P3 → P4. Gantt (textual):
|P1|P2|P3|P4| 0 5 8 16 22
Turnaround times: P1: 5 − 0 = 5 P2: 8 − 1 = 7 P3: 16 − 2 = 14 P4: 22 − 3 = 19 Average turnaround = (5+7+14+19)/4 = 11.25

2. Shortest Job First (SJF) — Non-preemptive

Select the process with the smallest next CPU burst. Non-preemptive SJF minimizes average waiting time if all bursts are known in advance (optimal under that assumption).

  • Pros: Low average waiting time.
  • Cons: Requires prior knowledge of burst lengths; can starve long jobs.

Pseudocode idea:

while processes remain: select ready process with smallest burst run it to completion
Example (SJF):
Same processes: P1(0,5), P2(1,3), P3(2,8), P4(3,6). Timeline selection: At t=0: only P1 available → run P1 (0–5). At t=5: P2(3), P3(8), P4(6) ready → choose P2 (5–8). Next choose P4 (8–14), then P3 (14–22). Gantt:
|P1|P2|P4|P3| 0 5 8 14 22
Turnaround: P1:5, P2:8−1=7, P3:22−2=20, P4:14−3=11. Average = (5+7+20+11)/4 = 10.75

3. Shortest Remaining Time First (SRTF) — Preemptive SJF

At any arrival, if a new process has less remaining CPU time than the current running process's remaining time, preempt and run the new one.

  • Pros: Lower average waiting time than non-preemptive SJF.
  • Cons: Frequent context switches possible; still requires burst knowledge.

4. Round Robin (RR)

Preemptive algorithm giving each ready process a fixed time slice (quantum). After a quantum expires the process is placed at the end of the ready queue if it still needs CPU.

  • Pros: Fair and good for interactive systems (bounded response time).
  • Cons: Quantum selection is critical; too small → lots of context switching; too big → degenerates to FCFS.

Pseudocode:

readyQueue.enqueue(processes in arrival order) while readyQueue not empty: p = readyQueue.dequeue() run p for min(quantum, p.remaining) if p.remaining > 0: readyQueue.enqueue(p)

5. Priority Scheduling

Each process has a priority; scheduler selects the highest priority ready process. Can be preemptive or non-preemptive.

  • Problems: Starvation of low-priority processes.
  • Mitigation: Aging — gradually increase priority of waiting processes.

6. Multilevel Queue and Multilevel Feedback Queue (MLFQ)

Multilevel Queue: multiple queues each with its own scheduling policy (e.g., foreground RR, background FCFS). Processes are permanently assigned to one queue based on attributes (interactive vs batch).

Multilevel Feedback Queue: allows processes to move between queues based on observed behavior (CPU bursts, IO patterns). It is adaptive and aims to favor interactive short jobs while demoting CPU-bound jobs.

  • MLFQ policies are used by many general-purpose OSes to balance responsiveness and throughput.

7. Real-time Scheduling (RMS, EDF)

Real-time systems require meeting deadlines. Two classical approaches:

  • Rate Monotonic Scheduling (RMS): static priorities assigned by period (shorter period → higher priority). Works for periodic tasks. There is a utilization bound: U = sum(Ci/Ti). For n tasks, guaranteed schedulable if U ≤ n(2^{1/n} − 1). For large n this approaches ln 2 (~0.693).
  • Earliest Deadline First (EDF): dynamic scheduling picks task with earliest absolute deadline; optimal on a uniprocessor for preemptive scheduling: feasibility test is U ≤ 1 for independent periodic tasks.

Hard real-time vs soft real-time: hard means missed deadlines are catastrophic; soft means occasional misses are acceptable. Scheduling guarantees differ accordingly.

Illustrative Comparisons and Trade-offs

No single scheduler is best for every goal. Common trade-offs:

  • Minimize average waiting time (SJF) vs. guarantee fairness and responsiveness (RR).
  • Minimize context-switching overhead (FCFS) vs. responsiveness to new short jobs (SRTF).
  • Static priorities are efficient but inflexible; dynamic/adaptive schedulers handle diverse workloads better.

Starvation and Aging

Some algorithms can starve processes (e.g., low-priority processes in priority scheduling). Aging gradually increases a waiting process's effective priority to eventually allow it to run.

Priority Inversion

Priority inversion occurs when a low-priority task holds a resource needed by a high-priority task, and a medium-priority task preempts the low-priority holder. Solutions include priority inheritance (low-priority temporarily inherits higher priority) and priority ceiling.

Implementation Considerations

Data structures

  • Simple queue for FCFS and RR.
  • Priority queue / heap (min-heap) for SJF/SRTF/priority scheduling.
  • Multiple queues for MLFQ.
  • Efficient lock-free structures and per-CPU run queues for multiprocessor systems to reduce contention.

Preemption and Context Switch Costs

Every preemption implies a context switch (saving registers, kernel bookkeeping). Scheduling policies must weigh the benefit of preemption against its cost.

Multiprocessor and Multicore Scheduling

Scheduling across multiple CPUs introduces additional concerns:

  • Affinity: Keeping a process on the same CPU (cache warmth).
  • Load balancing: Moving tasks to less-busy CPUs.
  • Global vs. Partitioned: Global scheduling shares one ready pool across CPUs; partitioned assigns tasks to specific CPUs (local scheduler per CPU).

Advanced: Linux Completely Fair Scheduler (CFS) — Conceptual)

Modern kernels use variants of fair-share scheduling. The CFS uses a "virtual runtime" and a red-black tree to approximate processor fairness across tasks. It gives each task a share of CPU proportional to its weight (priority).

Round Robin (manual example)

Consider P1(0, 5), P2(1, 4), P3(2, 2). Quantum = 2.

Timeline by hand:
t=0: P1 runs 0–2 (remaining 3) t=2: P2 arrived; queue: P2,P3,P1 t=2: P2 runs 2–4 (remaining 2) t=4: P3 runs 4–6 (remaining 0) — completes at 6 t=6: P1 runs 6–8 (remaining 1) t=8: P2 runs 8–10 (remaining 0) — completes at 10 t=10: P1 runs 10–11 (remaining 0) — completes at 11
Completion times: P1=11, P2=10, P3=6 Turnaround: P1:11−0=11, P2:10−1=9, P3:6−2=4 Waiting: P1:11−5=6, P2:9−4=5, P3:4−2=2

SRTF (preemptive) short example

Processes: P1(0,7), P2(2,4), P3(4,1). SRTF behavior:

At t=0: P1 runs (remaining 7). t=2: P2 arrives (remaining 4) < P1(5) → preempt P1, run P2. t=4: P3 arrives (1) < P2(2) → preempt P2, run P3 (t=4–5) finish. Then resume P2 (t=5–7), finish at 7. Then P1 (t=7–12). Completion: P3=5,P2=7,P1=12.

Pseudocode Summaries

FCFS

enqueue arriving processes by arrival time while readyQueue not empty: p = readyQueue.dequeue() dispatch(p) until p finishes or blocks

Round Robin

quantum = Q readyQueue contains arrived processes in FIFO order while readyQueue not empty: p = readyQueue.dequeue() run p for min(Q, p.remaining) if p.remaining > 0: readyQueue.enqueue(p) on arrival of new process: enqueue it

SRTF

use a min-priority queue keyed by remaining time on arrival of new process: insert into queue on CPU idle or process preemption: select process with minimum remaining time

Practical Notes

  • In practice, the OS scheduler must consider kernel threads, interrupts, device drivers, and priority inversion situations.
  • Schedulers are frequently hybrid: e.g., interactive tasks promoted, batch tasks demoted, and heuristics tuned for workload.
  • For multi-core machines, per-CPU run queues and work-stealing are common approaches to reduce contention and maintain cache affinity.

When to use which algorithm — guidance

  • Interactive, user-facing systems: favor low response time (RR, MLFQ, fair-share schedulers).
  • Batch throughput: favor short average completion time (SJF-like policies, but avoid starvation).
  • Hard real-time: use RMS or EDF with feasibility analysis and bounded latencies.
