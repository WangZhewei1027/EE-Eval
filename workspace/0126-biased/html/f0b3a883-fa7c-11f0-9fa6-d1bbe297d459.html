Structured Query Language (SQL) - Comprehensive Guide

Introduction to SQL

Structured Query Language (SQL) is a standardized programming language used for managing and manipulating relational databases. Developed in the 1970s at IBM, SQL has become the de facto standard for database operations across various database management systems (DBMS) including MySQL, PostgreSQL, Oracle, SQL Server, and SQLite.

Note: SQL is pronounced as "ess-que-el" or "sequel". While the language is standardized, different database systems may implement slight variations or extensions to the standard SQL.

SQL enables users to:

  • Create and modify database structures
  • Insert, update, and delete data
  • Query and retrieve data with complex conditions
  • Control access to data
  • Ensure data integrity

Basic SQL Concepts

Relational Database Model

SQL operates on relational databases, which organize data into one or more tables (or "relations") of columns and rows. Key concepts include:

  • Tables: Collections of related data organized in rows and columns
  • Rows (Records): Individual entries in a table
  • Columns (Fields): Specific attributes of the data
  • Primary Key: A unique identifier for each record
  • Foreign Key: A field that links to a primary key in another table

Example Table Structure:

employee_id (PK) first_name last_name department_id (FK) hire_date
101 John Smith 5 2020-06-15
102 Jane Doe 3 2019-11-22

SQL Sublanguages

SQL commands can be categorized into several sublanguages:

Sublanguage Purpose Common Commands
Data Definition Language (DDL) Define and modify database structure CREATE, ALTER, DROP, TRUNCATE
Data Manipulation Language (DML) Manipulate data within tables SELECT, INSERT, UPDATE, DELETE
Data Control Language (DCL) Control access to data GRANT, REVOKE
Transaction Control Language (TCL) Manage transactions COMMIT, ROLLBACK, SAVEPOINT

Data Definition Language (DDL)

DDL commands are used to define, modify, and delete database objects like tables, indexes, and views.

CREATE Statement

The CREATE statement is used to create new database objects.

CREATE TABLE employees ( employee_id INT PRIMARY KEY, first_name VARCHAR(50) NOT NULL, last_name VARCHAR(50) NOT NULL, email VARCHAR(100) UNIQUE, hire_date DATE DEFAULT CURRENT_DATE, salary DECIMAL(10,2) CHECK (salary > 0), department_id INT, FOREIGN KEY (department_id) REFERENCES departments(department_id) );

Key elements in table creation:

  • Data Types: INT, VARCHAR, DATE, DECIMAL, etc.
  • Constraints: PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL, CHECK, DEFAULT

ALTER Statement

The ALTER statement modifies an existing database object.

-- Add a new column ALTER TABLE employees ADD COLUMN phone_number VARCHAR(15); -- Modify a column ALTER TABLE employees ALTER COLUMN salary SET DATA TYPE DECIMAL(12,2); -- Drop a column ALTER TABLE employees DROP COLUMN phone_number;

DROP Statement

The DROP statement removes database objects.

DROP TABLE employees;

Warning: DROP statements are irreversible and will permanently delete the object and its data. Always ensure you have backups before executing DROP commands.

Data Manipulation Language (DML)

DML commands are used to retrieve, insert, update, and delete data in database tables.

SELECT Statement

The SELECT statement retrieves data from one or more tables.

-- Basic SELECT SELECT first_name, last_name FROM employees; -- SELECT with WHERE clause SELECT * FROM employees WHERE department_id = 5 AND salary > 50000; -- SELECT with ORDER BY SELECT first_name, last_name, salary FROM employees ORDER BY salary DESC; -- SELECT with GROUP BY SELECT department_id, AVG(salary) as avg_salary FROM employees GROUP BY department_id HAVING AVG(salary) > 60000;

Key SELECT components:

  • WHERE: Filters rows before grouping
  • GROUP BY: Groups rows sharing a property
  • HAVING: Filters groups after grouping
  • ORDER BY: Sorts the result set
  • Aggregate functions: COUNT, SUM, AVG, MIN, MAX

INSERT Statement

The INSERT statement adds new rows to a table.

-- Insert single row INSERT INTO employees (employee_id, first_name, last_name, email, hire_date, salary, department_id) VALUES (103, 'Robert', 'Johnson', 'rjohnson@example.com', '2021-03-10', 75000.00, 3); -- Insert multiple rows INSERT INTO employees VALUES (104, 'Sarah', 'Williams', 'swilliams@example.com', '2020-08-15', 68000.00, 5), (105, 'Michael', 'Brown', 'mbrown@example.com', '2019-05-22', 82000.00, 3);

UPDATE Statement

The UPDATE statement modifies existing data in a table.

-- Update specific rows UPDATE employees SET salary = salary * 1.05 WHERE department_id = 5; -- Update with subquery UPDATE employees SET salary = salary * 1.10 WHERE employee_id IN ( SELECT employee_id FROM top_performers );

Warning: Always include a WHERE clause in UPDATE statements unless you intend to update all rows in the table.

DELETE Statement

The DELETE statement removes rows from a table.

-- Delete specific rows DELETE FROM employees WHERE hire_date < '2010-01-01'; -- Delete all rows (be careful!) DELETE FROM employees;

SQL Joins

Joins combine rows from two or more tables based on related columns.

INNER JOIN

Returns only the rows that have matching values in both tables.

SELECT e.first_name, e.last_name, d.department_name FROM employees e INNER JOIN departments d ON e.department_id = d.department_id;

LEFT JOIN (or LEFT OUTER JOIN)

Returns all rows from the left table and matched rows from the right table (NULL if no match).

SELECT e.first_name, e.last_name, d.department_name FROM employees e LEFT JOIN departments d ON e.department_id = d.department_id;

RIGHT JOIN (or RIGHT OUTER JOIN)

Returns all rows from the right table and matched rows from the left table (NULL if no match).

SELECT e.first_name, e.last_name, d.department_name FROM employees e RIGHT JOIN departments d ON e.department_id = d.department_id;

FULL JOIN (or FULL OUTER JOIN)

Returns all rows when there is a match in either left or right table.

SELECT e.first_name, e.last_name, d.department_name FROM employees e FULL JOIN departments d ON e.department_id = d.department_id;

CROSS JOIN

Returns the Cartesian product of rows from tables (all possible combinations).

SELECT e.first_name, d.department_name FROM employees e CROSS JOIN departments d;

Note: In practice, INNER JOIN is the most commonly used join type, followed by LEFT JOIN. RIGHT JOIN and FULL JOIN are used less frequently.

Subqueries

A subquery is a query nested inside another query. Subqueries can be used in SELECT, FROM, WHERE, and HAVING clauses.

WHERE Clause Subquery

SELECT first_name, last_name, salary FROM employees WHERE salary > ( SELECT AVG(salary) FROM employees );

FROM Clause Subquery (Derived Table)

SELECT dept_avg.department_id, dept_avg.avg_salary FROM ( SELECT department_id, AVG(salary) as avg_salary FROM employees GROUP BY department_id ) dept_avg WHERE dept_avg.avg_salary > 70000;

Correlated Subquery

A subquery that references columns from the outer query.

SELECT e.first_name, e.last_name, e.salary FROM employees e WHERE e.salary > ( SELECT AVG(salary) FROM employees WHERE department_id = e.department_id );

EXISTS Operator

Tests for the existence of rows in a subquery.

SELECT d.department_name FROM departments d WHERE EXISTS ( SELECT 1 FROM employees e WHERE e.department_id = d.department_id AND e.salary > 100000 );

Transactions

A transaction is a sequence of operations performed as a single logical unit of work.

ACID Properties

  • Atomicity: All operations complete successfully or none do
  • Consistency: Database remains in a consistent state
  • Isolation: Concurrent transactions don't interfere
  • Durability: Completed transactions persist even after system failure

Transaction Control

BEGIN TRANSACTION; UPDATE accounts SET balance = balance - 1000 WHERE account_id = 1; UPDATE accounts SET balance = balance + 1000 WHERE account_id = 2; -- If everything is OK COMMIT; -- If something went wrong -- ROLLBACK;

Example Scenario: A bank transfer between two accounts must be atomic - either both the debit and credit succeed, or neither does.

Indexes

Indexes are database objects that improve the speed of data retrieval operations.

Creating Indexes

-- Single column index CREATE INDEX idx_last_name ON employees(last_name); -- Composite index CREATE INDEX idx_dept_salary ON employees(department_id, salary); -- Unique index CREATE UNIQUE INDEX idx_email ON employees(email);

When to Use Indexes

  • Columns frequently used in WHERE clauses
  • Columns used in JOIN conditions
  • Columns used in ORDER BY clauses
  • Columns with high selectivity (many distinct values)

When to Avoid Indexes

  • Tables with frequent write operations (indexes slow down INSERT, UPDATE, DELETE)
  • Columns with few distinct values (low selectivity)
  • Very small tables

Views

Views are virtual tables based on the result set of a SQL query.

Creating Views

CREATE VIEW employee_directory AS SELECT e.employee_id, e.first_name, e.last_name, d.department_name FROM employees e JOIN departments d ON e.department_id = d.department_id;

Using Views

SELECT * FROM employee_directory WHERE department_name = 'Engineering';

Benefits of Views

  • Simplify complex queries
  • Provide security by restricting access to specific columns
  • Present consistent, unchanged interfaces even if underlying tables change
  • Summarize data from multiple tables

Stored Procedures

Stored procedures are prepared SQL code that can be saved and reused.

Creating a Stored Procedure

CREATE PROCEDURE get_employees_by_department(IN dept_id INT) BEGIN SELECT first_name, last_name, hire_date FROM employees WHERE department_id = dept_id ORDER BY last_name; END;

Executing a Stored Procedure

CALL get_employees_by_department(5);

Benefits of Stored Procedures

  • Improved performance (precompiled execution plan)
  • Reduced network traffic
  • Code reusability
  • Enhanced security (can restrict direct table access)

Database Normalization

Normalization is the process of organizing data to minimize redundancy and dependency.

Normal Forms

Normal Form Description
1NF Each table cell should contain a single value, and each record needs to be unique
2NF Be in 1NF and all non-key attributes are fully dependent on the primary key
3NF Be in 2NF and has no transitive dependencies (non-key attributes don't depend on other non-key attributes)
BCNF Boyce-Codd Normal Form - stricter version of 3NF
4NF No multi-valued dependencies
5NF No join dependencies

Note: In practice, most databases are normalized to 3NF or BCNF. Higher normal forms are used in specialized cases.

Conclusion

SQL is a powerful language for managing and querying relational databases. This guide has introduced the fundamental concepts including:

  • Database structure and design principles
  • Basic and advanced query techniques
  • Data manipulation and definition
  • Joins, subqueries, and transactions
  • Performance optimization with indexes
  • Database objects like views and stored procedures

  • Exploring advanced topics like window functions and recursive queries
  • Learning about database-specific features in your chosen DBMS
  • Studying query optimization and execution plans
