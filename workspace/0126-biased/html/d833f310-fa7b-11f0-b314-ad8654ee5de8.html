Min‑Heap — Comprehensive Explanation

A self-contained explanation of the min-heap data structure: definition, theory, algorithms, step-by-step examples, complexity analysis, and a tiny demonstration. This page is primarily textual and educational.

1. What is a Min-Heap?

A min-heap is a specialized complete binary tree data structure that satisfies the heap property:

  • For every node v (except the root), the value at v is greater than or equal to the value at its parent.
  • Equivalently: the smallest element of the tree is always at the root.

Two key characteristics:

  • Shape property (completeness): the tree is complete — every level, except possibly the last, is completely filled, and nodes in the last level are left-aligned.
  • Heap property: parent's key ≤ children's keys (for a min-heap).

Why use a min-heap?

The min-heap supports extracting the smallest element efficiently, and is commonly used to implement a priority queue. Many algorithms — for example Dijkstra's shortest paths and Prim's minimum spanning tree — rely on an efficient priority queue. Heaps are also the basis of heapsort.

2. Representation and Indexing

The most common practical representation of a binary heap is a contiguous array. If the array is 0-indexed and an element is at index i:

  • Parent index: parent(i) = floor((i - 1) / 2)
  • Left child: left(i) = 2 * i + 1
  • Right child: right(i) = 2 * i + 2

Using an array avoids pointers and allocations for each node, is cache-friendly, and is the standard approach for binary heaps.

Array: [1, 3, 6, 5, 9, 8]

Interpreted as a tree:

               / \

            5  9 8

3. Core Operations (Algorithms and Textual Step-by-Step)

3.1 Insert (push)

To insert a new key x into a min-heap stored in array A:

  1. Append x to the end of A (maintains completeness).
  2. "Heapify-up" (also called sift-up or bubble-up): while x is smaller than its parent, swap it with its parent. Continue until either the root is reached or the parent is ≤ x.
// Pseudocode (0-indexed):
procedure insert(A, x)
    i := size(A)
    append x to A
    while i > 0 and A[parent(i)] > A[i]
        swap A[i] and A[parent(i)]
        i := parent(i)
end

Complexity: Each swap moves the element up one level. The heap height is O(log n), so insert is O(log n) worst-case.

3.2 Extract-Min (pop)

To remove and return the smallest element (the root):

  1. Save the root value to return later.
  2. Move the last element in the array to the root position and remove the last slot (reduces size by 1).
  3. "Heapify-down" (sift-down): while the moved element is greater than one of its children, swap it with the smaller of the two children. Continue until both children are ≥ element or there are no children.
// Pseudocode:
procedure extractMin(A)
    if size(A) == 0 then error or return null
    min := A[0]
    A[0] := A[last]
    remove last element of A
    heapifyDown(A, 0)
    return min
end

procedure heapifyDown(A, i)
    while left(i) < size(A)
        smallest := left(i)
        if right(i) < size(A) and A[right(i)] < A[left(i)]
            smallest := right(i)
        if A[i] <= A[smallest] break
        swap A[i], A[smallest]
        i := smallest
end

Complexity: Heapify-down travels at most the height of the tree, O(log n).

3.3 Build-Heap (Floyd's algorithm)

Given an arbitrary array of n elements, you can turn it into a valid min-heap in-place using:

  1. For i from floor(n/2) - 1 downto 0, call heapifyDown(A, i).
// Pseudocode:
procedure buildMinHeap(A)
    n := size(A)
    for i := floor(n/2)-1 downto 0
        heapifyDown(A, i)
end

Intuition: nodes at indices ≥ floor(n/2) are leaves (no children) and already heaps. Sifting internal nodes from bottom-up fixes local violations; the bottom-up approach is more efficient than inserting each element individually.

3.4 Decrease-Key / Delete

Decrease-Key (reducing the value of A[i]) — update A[i] := newKey then heapify-up from i. Delete can be implemented by decrease-key to -∞ then extract-min, or by replacing A[i] with last element then heapify-down or up depending on the value.

3.5 Heap-Sort (brief)

Heap-sort: build a max-heap (for ascending sort) in-place, then repeatedly swap the root with the last element, reduce heap size by 1, and heapify-down the root. Running time O(n log n), not stable, but in-place.

4. Complexity Analysis and Proofs

4.1 Height of a heap

A binary heap with n elements has height ⌊log2 n⌋. Reason: a complete binary tree of height h can have between 2^h and 2^{h+1}-1 nodes. Solving 2^h ≤ n yields h ≤ log2 n.

4.2 Time complexities

  • Insert: O(log n) worst-case (sift-up at most h levels)
  • Extract-min: O(log n) worst-case (sift-down at most h levels)
  • Peek-min: O(1)
  • Build-heap (bottom-up): O(n) — see explanation below

4.3 Why build-heap is O(n)

A naive bound that each heapify could cost O(log n) would suggest O(n log n), but the bottom-up build is tighter because most nodes are near the leaves and have small heights to repair. A common proof uses summation of work per node:

  1. Number of nodes at height i is at most ⌈n / 2^{i+1}⌉.
  2. Each node at height i costs O(i) to heapify (in the worst case it may move down i levels).

Total work ≤ Σ_{i=0}^{⌊log n⌋} (number of nodes at height i) × O(i) = O(n Σ i/2^i) = O(n). The infinite sum Σ i/2^i converges to a constant (2), so total is O(n).

4.4 Correctness sketch (invariant-based)

For heapify-down at index i: precondition — children subtrees rooted at left(i) and right(i) are already heaps. The procedure places A[i] into correct position so the subtree rooted at i becomes a heap. The correctness follows by observing that each swap moves the violating item downwards to a position where both children are ≥ item; since children were heaps before, the invariant holds.

5. Step-by-Step Example (Textual Narrative)

Example: Build a min-heap from array

Start with array:

[7, 2, 5, 3, 1, 8, 4]

n = 7. We will call heapifyDown for indices floor(7/2)-1 = 2, then 1, then 0.

  1. i = 2. A[2] = 5. Its children: left = index 5 → 8, right = index 6 → 4. The smallest child is 4 (index 6). Since 4 < 5, swap A[2] and A[6]. Array becomes: 
    [7, 2, 4, 3, 1, 8, 5]
    Now consider index 6 -- it's a leaf. Done.
  2. i = 1. A[1] = 2. Children: left index 3 → 3, right index 4 → 1. Smallest child is 1 (index 4). Since 1 < 2, swap index 1 and 4: 
    [7, 1, 4, 3, 2, 8, 5]
    Now at index 4 (value 2) — it's a leaf. Done.
  3. i = 0. A[0] = 7. Children: left index 1 → 1, right index 2 → 4. Smallest child is 1 (index 1). Swap: 
    [1, 7, 4, 3, 2, 8, 5]
    Continue heapifyDown at index 1 (value 7). Children: left index 3 → 3, right index 4 → 2. Smallest child is 2 (index 4). Swap: 
    [1, 2, 4, 3, 7, 8, 5]
    Index 4 is a leaf now. Done.

Final min-heap array: [1, 2, 4, 3, 7, 8, 5].

Extract-min on this heap (step-by-step)

Initial: [1, 2, 4, 3, 7, 8, 5]

  1. Remove root (1). Move last element 5 to root: 
    [5, 2, 4, 3, 7, 8]
  2. Heapify-down from root: children 2 and 4 → smallest 2 at index 1. Swap: 
    [2, 5, 4, 3, 7, 8]
  3. Continue at index 1: children 3 and 7 → smallest 3 at index 3. Since 3 < 5, swap: 
    [2, 3, 4, 5, 7, 8]
  4. Index 3 is leaf — done. Extracted value: 1. New heap root: 2.

6. When to Use a Min-Heap — Use Cases and Comparisons

Use cases

  • Priority queue where the smallest key must be extracted quickly.
  • Dijkstra's algorithm (using heap as priority queue for tentative distances).
  • Prim's algorithm for minimum spanning tree.
  • Event-driven simulation where the next event in time must be processed first.
  • Heapsort and streaming top-k queries (by maintaining a heap of size k).

Compared to other structures

  • Balanced binary search trees (BSTs) provide ordered traversal and support search by key; heaps do not support fast search for an arbitrary key (no ordering across siblings).
  • Fibonacci heaps offer better amortized decrease-key for some graph algorithms (amortized O(1) decrease-key), which can improve theoretical complexity of Dijkstra's algorithm, but they are more complex in practice and have larger constants.
  • An array-based binary heap is usually the best practical choice for implementing a simple priority queue due to simplicity and cache-friendliness.

7. Variants and Extensions

  • Max-heap: symmetric variant where parent's key ≥ children's keys. Used when largest element is needed at root.
  • d-ary heaps: each node has d children. Increasing d reduces heap height (good for decrease-key operations) but increases branching cost per step. A common choice is d = 4 or 8 in practice.
  • Pairing, binomial, Fibonacci heaps: pointer-based structures with better amortized bounds for some operations (e.g., decrease-key), used in theoretical improvements of graph algorithms.
  • Min-heap with keyed items: often stores key-value pairs (priority, payload). When keys are equal, payload tie-breaking may be applied.

8. Common Pitfalls and Practical Tips

  • Remember to use the correct parent/child formulas for 0-indexed vs 1-indexed arrays.
  • When implementing decrease-key or delete, carefully choose whether to heapify-up or heapify-down based on the new value compared to the old one.
  • Heaps do not provide efficient contains(x) or remove(x) without additional index mapping. If you need arbitrary removal or decrease-key by value, maintain an auxiliary map from item → index.
  • Heapsort destroys the heap: the array is overwritten with the sorted contents.
  • Floating point comparisons and custom comparator functions must be consistent and transitive to maintain heap invariants.
