HTTP — Hypertext Transfer Protocol

A thorough textual guide: concepts, mechanisms, message formats, versions, security, caching, examples, and clear step-by-step explanations.

What is HTTP?

HTTP stands for Hypertext Transfer Protocol. It is the foundational application-layer protocol used on the World Wide Web to transfer hypermedia documents (HTML), images, JSON, and other content types between clients (usually browsers or programs) and servers. HTTP defines the structure of requests and responses, semantics of methods (like GET or POST), how headers and bodies are composed, and how connections are managed. It is a stateless protocol at its core — each request-response pair stands alone unless higher-level mechanisms (cookies, tokens) add continuity.

Key roles

  • Client: initiates requests (web browser, API client, mobile app).
  • Server: listens for connections and responds with status codes, headers, and a message body.
  • Intermediaries: proxies, caches, gateways, and load balancers that can inspect, cache, forward, or transform traffic.

Design principles

  • Statelessness: each request contains all information needed for the server to process it.
  • Uniform interface: consistent method semantics, resource-oriented identification via URLs.
  • Extensibility through headers: new features often introduced via headers without changing structure.

HTTP message format (textual, high-level)

At the application layer, HTTP messages are plain text (pre-HTTP/2 framing differences ignored). There are two message types:

Request (example: HTTP/1.1)

GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: demo-client/1.0
Accept: text/html,application/xhtml+xml
Accept-Encoding: gzip
Connection: keep-alive

[optional message body for methods like POST]

Response (example)

HTTP/1.1 200 OK
Date: Tue, 26 Jan 2026 12:00:00 GMT
Content-Type: text/html; charset=utf-8
Content-Length: 137
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
Cache-Control: max-age=3600

<!doctype html>
<html>
<head><title>Example</title></head>
<body>Hello world</body>
</html>

A request starts with a request-line (method, request-target, and HTTP version), followed by headers, a blank line, and an optional message body. Responses start with a status-line (version, status-code, reason-phrase), headers, blank line, and optional body.

Important header categories

  • Request headers: Host, User-Agent, Accept, Accept-Encoding, Authorization, Cookie, If-Modified-Since, If-None-Match, etc.
  • Response headers: Content-Type, Content-Length, Cache-Control, ETag, Set-Cookie, Location (for redirects), WWW-Authenticate, etc.
  • General/connection: Date, Connection, Transfer-Encoding.

HTTP methods — semantics and examples

HTTP methods define intended actions. Two important properties are:

  • Safe: method does not modify server state (e.g., GET, HEAD).
  • Idempotent: repeated identical requests produce the same effect on the server (e.g., PUT, DELETE, GET are idempotent; POST is generally not).

Common methods

  • GET — retrieve a representation of a resource. Safe, idempotent. No body in the request by convention.
  • HEAD — same as GET but server returns headers only (no body).
  • POST — submit data to the server (create a resource, run an action). Not idempotent by default.
  • PUT — replace state of a resource with provided entity. Idempotent.
  • DELETE — remove a resource. Idempotent in semantics.
  • PATCH — partial modification of a resource (semantics depend on patch format).
  • OPTIONS — query the server for supported methods/communication options for a resource.

Example: POST form submission (simplified)

POST /submit-form HTTP/1.1
Host: www.example.org
Content-Type: application/x-www-form-urlencoded
Content-Length: 27

name=Alice+Smith&age=28

HTTP status codes — categories and common codes

Status codes are three-digit numbers grouped by leading digit:

  • 1xx (Informational): provisional responses (rare in practice).
  • 2xx (Success): request succeeded (200 OK, 201 Created, 204 No Content).
  • 3xx (Redirection): client must take additional action (301 Moved Permanently, 302 Found, 304 Not Modified).
  • 4xx (Client error): client made a mistake (400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found).
  • 5xx (Server error): server failed to fulfill a valid request (500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable).

Example: conditional cache hit uses 304 Not Modified — server sends no body and instructs the client to use its cached copy.

Step-by-step: What happens when you load a URL (high-level)

  1. DNS resolution: client resolves domain name to IP address (via local cache or DNS servers).
  2. TCP connection: client performs TCP three-way handshake to the server IP and port (typically port 80 for HTTP, 443 for HTTPS).
  3. TLS handshake (if HTTPS): client and server negotiate secure parameters, exchange certificates, and establish a shared symmetric key.
  4. Send HTTP request: client composes an HTTP request and sends over the established (optionally encrypted) connection. For HTTP/1.1, this is textual; for HTTP/2/3 there is binary framing.
  5. Server processes request: server parses request, performs routing, authentication/authorization, invokes application logic, and composes a response.
  6. Response sent: server returns status, headers, and body. Server may include Cache-Control or ETag headers to assist clients and intermediaries.
  7. Client renders/handles response: browser may parse HTML, perform additional resource requests (CSS, JS, images), etc. Intermediaries (like caches) may store the response.
  8. Connection close or reuse: connection may be kept alive for further requests (keep-alive), or closed.

This high-level flow abstracts many details (redirects, 401 challenges, multipart responses, etc.).

Caching and validation (algorithms & headers)

Caching improves performance and reduces load. There are two broad caching behaviors:

  • Freshness-based caching: controlled by Cache-Control: max-age=X or Expires header. If a cached response is still fresh, the client or intermediary can reuse it without contacting origin.
  • Validation-based caching: cached response becomes stale and client issues a conditional request using If-Modified-Since or If-None-Match with the resource's Last-Modified or ETag. Server replies with 304 Not Modified if content unchanged.

ETag and conditional requests (step-by-step)

  1. Server responds with ETag header: ETag: "abc123".
  2. Client caches response and later requests the resource again adding If-None-Match: "abc123".
  3. If resource unchanged, server replies 304 Not Modified (no body). Client uses cached body. If changed, server returns 200 with new body and ETag.

Cache replacement policies (LRU, LFU) are used by caches to evict entries. The HTTP specification doesn't mandate a particular eviction algorithm — implementers choose policies appropriate to memory and traffic patterns.

Content negotiation and representations

HTTP separates resource identity (URL) from representations (HTML, JSON, XML, image formats). Clients indicate preferences:

  • Accept: media types (e.g., application/json, text/html)
  • Accept-Language: language preferences
  • Accept-Encoding: compression algorithms (gzip, br)

Server chooses the best representation for the client's preferences and responds with a Content-Type header to indicate the chosen type.

Security: TLS and HTTPS (high-level explanation)

HTTPS is HTTP over TLS. TLS provides confidentiality, integrity, and optional client authentication. The TLS handshake (simplified) performs:

  1. ClientHello: client proposes protocol version, cipher suites, and sends random nonce.
  2. ServerHello: server selects version/cipher, sends its certificate chain, and sends random nonce.
  3. Client verifies certificate, establishes pre-master secret, and both sides derive symmetric keys.
  4. Subsequent HTTP messages are encrypted and integrity-protected using symmetric algorithms.

TLS details (certificate validation, certificate authorities, revocation, session resumption) are complex. Best practice: always use current TLS versions and keep servers properly configured (no weak cipher suites, no expired certificates).

HTTP versions: evolution in brief

  • HTTP/0.9: very simple, only GET and body, no headers.
  • HTTP/1.0: introduced headers and status codes; each request often opened a new TCP connection.
  • HTTP/1.1: persistent connections by default, chunked transfer encoding, Host header required, pipelining (poorly used in practice).
  • HTTP/2: binary framing, multiplexed streams over a single TCP connection, header compression (HPACK), server push (less used nowadays).
  • HTTP/3: runs over QUIC (UDP-based), provides multiplexing without head-of-line blocking at transport level, faster connection establishment via 0-RTT in some cases.

Examples of important mechanisms

Chunked transfer encoding (HTTP/1.1 streaming)

Server can send data in chunks without knowing total size in advance. Each chunk is prefixed by hex length.

HTTP/1.1 200 OK
Transfer-Encoding: chunked
Content-Type: text/plain

5
Hello
6
 world
0

Range requests (partial content)

GET /video.mp4 HTTP/1.1
Range: bytes=1000-1999

-- server replies with 206 Partial Content and Content-Range header --

Redirect (301 example)

HTTP/1.1 301 Moved Permanently
Location: https://www.example.com/new-path/
Content-Length: 0

Simple authentication flow (WWW-Authenticate)

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Basic realm="example"

-- client sends Authorization: Basic base64(user:pass) --

REST, APIs, and best practices

REST (Representational State Transfer) is an architectural style that commonly uses HTTP. RESTful services emphasize:

  • Resources identified by URIs
  • Use of standard HTTP methods to express intent
  • Stateless interactions
  • Use of proper status codes

API design suggestions

  • Use nouns, not verbs, in URIs (e.g., /orders/123, not /getOrder).
  • Use appropriate status codes (201 for created, 204 when no content, 400 for client errors).
  • Support pagination and filtering via query parameters for collections.
  • Document content types, authentication, error formats, and rate limits.

Common pitfalls and threats

  • Mixing secure and insecure content: loading HTTP subresources on HTTPS pages breaks security expectations.
  • Improper cache control: sensitive responses must not be cached publicly (use Cache-Control: no-store or private).
  • CSRF (Cross-Site Request Forgery): protect state-changing endpoints using anti-CSRF tokens or same-site cookies.
  • XSS (Cross-Site Scripting): sanitize or encode user-provided content returned in responses.
  • Header injection or open redirects: validate inputs used to build headers (like Location) to avoid security issues.

Tools, debugging, and testing

Useful techniques:

  • Use curl for raw requests and quick experiments: curl -v -I https://example.com
  • Use browser developer tools (Network tab) to inspect headers, timing, and payloads.
  • Use tcpdump/wireshark for low-level packet capture (encrypted traffic requires session keys to decrypt HTTPS).
  • Logs on server side (access/error logs) help diagnose server responses and errors.

Simple debugging checklist

  1. Is the Host header correct? (HTTP/1.1 requires it.)
  2. Are required authentication headers present?
  3. Check Content-Type and Content-Length when posting bodies.
  4. Examine status codes and any response body for error messages.
Quick reference
  • Default ports: 80 (HTTP), 443 (HTTPS)
  • Idempotent methods: GET, PUT, DELETE, HEAD
  • Common headers: Host, Content-Type, Authorization, Cookie
  • Cache control: Cache-Control, ETag, If-None-Match
  • Compression: Accept-Encoding, Content-Encoding

Protocols and performance optimizations

Performance techniques shaped by HTTP features:

  • Keep-alive / persistent connections: reuse TCP/TLS connection for multiple requests to reduce handshake overhead.
  • HTTP/2 multiplexing: multiple simultaneous requests and responses on a single connection without head-of-line blocking at the application level.
  • HTTP/3 (QUIC): reduces latency by combining transport and cryptographic handshakes and avoiding TCP head-of-line blocking.
  • Compression: compress responses (gzip, br) negotiated by Accept-Encoding to reduce bytes transferred.
  • Minimize round trips: use resource bundling, lazy-loading, and caching strategies to reduce request counts.

Advanced topics (concise pointers)

  • Server Push (HTTP/2): server proactively pushes resources; use carefully to avoid wasted bandwidth.
  • CORS (Cross-Origin Resource Sharing): browser-enforced mechanism using Access-Control-Allow-Origin headers and preflight OPTIONS requests for non-simple requests.
  • Content-Security-Policy: header to reduce XSS by restricting resource origins.
  • HSTS (Strict-Transport-Security): instructs browsers to use HTTPS for the site for a period of time.
  • Rate limiting and pagination: important for API scalability; communicate limits via headers (e.g., X-RateLimit-*).

Raw example: full basic interaction (textual illustration)

Below is a textual illustration of a simple HTTP/1.1 exchange over TLS (HTTPS). It omits TCP and TLS packet detail but shows the order of messages your client and server exchange.


-- Client sends request -->
GET /resource/data HTTP/1.1
Host: api.example.com
User-Agent: demo-client/1.0
Accept: application/json
Accept-Encoding: gzip
Authorization: Bearer 

-- Server processes and replies -->
HTTP/1.1 200 OK
Date: Tue, 26 Jan 2026 12:05:00 GMT
Content-Type: application/json; charset=utf-8
Content-Encoding: gzip
Cache-Control: max-age=60
ETag: "v42"

{"data":{"message":"Hello, client"}}

If the client later requests the same resource with If-None-Match: "v42" and the server has not changed the data, the server will return:

HTTP/1.1 304 Not Modified
Date: Tue, 26 Jan 2026 12:06:00 GMT
Cache-Control: max-age=60
ETag: "v42"
