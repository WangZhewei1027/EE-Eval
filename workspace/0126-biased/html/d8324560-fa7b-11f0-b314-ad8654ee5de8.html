1. Definition and Motivation

A dynamic array is an array-like abstract data type that allows elements to be added or removed and that automatically resizes itself as needed. Unlike a fixed-size array (where the capacity is set at creation), a dynamic array grows (and sometimes shrinks) to accommodate changing numbers of elements.

Key motivations:

  • Provide O(1) random access by index (constant-time element retrieval).
  • Allow variable-size storage without requiring the programmer to manage raw memory growth.
  • Offer amortized constant-time appends while keeping elements in contiguous memory (cache-friendly).

2. Size, Capacity, Storage

Important terms:

  • Size (n): number of elements currently stored.
  • Capacity (C): number of elements that can be stored without reallocating (space allocated in memory).
  • Load factor: size / capacity; how full the array is.

A dynamic array keeps an underlying contiguous block of memory (like a C array). When size exceeds capacity, a new larger block is allocated and existing elements are copied to the new block.

3. Core Operations and Complexity

Common operations and their typical time complexities for a well-designed dynamic array:

  • Access by index (get/set): O(1) — direct pointer arithmetic.
  • Append / push_back: Amortized O(1) — occasional O(n) when resizing is required.
  • Pop from end: O(1) — typically just decrement size (may optionally shrink capacity).
  • Insert or remove in middle / front: O(n) — requires shifting elements.
  • Resize (allocate + copy): O(n) for that single operation.

"Amortized O(1)" means that over a sequence of operations the average cost per operation is constant, even though individual operations can be expensive.

4. Resizing Strategies and Trade-offs

When the array is full and a new element must be added, the array's capacity must increase. The strategy for choosing the new capacity determines performance and memory overhead.

Common strategies

  • Doubling (grow by factor 2): When full, new capacity = max(1, capacity * 2). Most common; gives good amortized bounds and simple implementation.
  • Geometric growth with factor α (1 < α ≤ 2): e.g., factor 1.5. Using any constant factor α > 1 yields amortized O(1) append operations. Smaller α reduces wasted space but increases frequency of copies.
  • Additive growth: increase capacity by a fixed amount (e.g., +100). This yields O(n) amortized append cost over many appends and is generally inefficient for large n.

Shrinking policies

When elements are removed, capacity can be reduced to free memory. Straightforward but naive shrinking (e.g., halving capacity when size < capacity/2) can cause many oscillating resizes when alternating push/pop. A safer policy: shrink only when size < capacity / k for some k (commonly k = 4), and optionally only if capacity is above some minimum threshold.

Trade-offs

  • Larger growth factor (e.g., 2): fewer reallocations, more extra (unused) memory on average.
  • Smaller growth factor (e.g., 1.125): less wasted memory, more frequent copies.
  • Frequent shrink/grow: risks repeated expensive copying and poor performance in pathological usage patterns.

5. Amortized Analysis (Proof that push is amortized O(1))

We'll outline two common proofs: the aggregate method and the accounting (banker's) method. Both show that repeated appends with geometric growth cost O(1) amortized per append.

Assumptions

  • Initial capacity is 1 (or some constant).
  • On resize, new capacity = 2 * old capacity.
  • Copying an element costs 1 unit, writing the new element costs 1 unit.

Aggregate method

Consider performing n append operations starting from an empty dynamic array. Each time the array grows from capacity C to 2C, C elements must be copied. How many times is an element copied over the whole sequence?

  • When capacity grows to 1, 1 copy. To 2, 2 copies. To 4, 4 copies. ... up to the largest capacity <= n.
  • Total number of copies <= 1 + 2 + 4 + ... + highest power of two <= n ≈ 2n.

Therefore total copy work is O(n). Adding the n constant-time insert operations gives O(n) total cost for n appends, or O(1) amortized per append.

Accounting (banker) method

Charge each append operation a constant amortized cost, e.g., 3 units:

  • 1 unit pays for the actual insertion.
  • Save 2 units (credit) to pay for future copies when a resize happens.

Each element that exists in the array contributes credits that will be used when that element is copied during a future resize. When doubling, at most one copy per element is needed at each doubling step, but the credits accumulate so there are always enough saved credits to pay for required copies. With 3 charged per insertion, credits never go negative, showing an amortized cost O(1).

Conclusion

With geometric growth (factor > 1), appends are amortized O(1).

6. Pseudocode

push_back (append)

function push_back(array, value):
  if array.size == array.capacity:
    grow(array)
  array.data[array.size] = value
  array.size += 1

function grow(array):
  new_capacity = max(1, array.capacity * 2)
  new_data = allocate(new_capacity)
  for i from 0 to array.size - 1:
    new_data[i] = array.data[i]   // copy or move
  free(array.data)
  array.data = new_data
  array.capacity = new_capacity

pop_back

function pop_back(array):
  if array.size == 0:
    error "underflow"
  array.size -= 1
  // optionally call shrink if array.size is much smaller than capacity

insert at index

function insert(array, index, value):
  if index < 0 or index > array.size: error
  if array.size == array.capacity: grow(array)
  // shift elements to the right
  for i from array.size-1 downto index:
    array.data[i+1] = array.data[i]
  array.data[index] = value
  array.size += 1

7. Implementation Notes and Language-Specific Behavior

C (manual memory management)

/* simple outline in C */
typedef struct {
  void *data;        // pointer to memory
  size_t size;
  size_t capacity;
} DynArr;

/* push_back for int values */
void push_back_int(DynArr* a, int v) {
  if (a->size == a->capacity) {
    size_t newcap = a->capacity ? a->capacity*2 : 1;
    int *newdata = malloc(newcap * sizeof(int));
    if (a->data) {
      memcpy(newdata, a->data, a->size * sizeof(int));
      free(a->data);
    }
    a->data = newdata;
    a->capacity = newcap;
  }
  ((int*)a->data)[a->size++] = v;
}

C++ (std::vector)

std::vector is a standard library dynamic array. It offers strong guarantees about contiguous storage and exposes capacity(), size(), reserve(), and shrink_to_fit() methods. std::vector uses move constructors when available and uses copy constructors otherwise when reallocating.

Java (ArrayList)

ArrayList is a dynamic array implementation in java.util. Java arrays cannot be resized; ArrayList manages an Object[] internally and replaces it with a larger one when needed. Java relies on garbage collection to reclaim old arrays.

Python (list)

Python's list is a dynamic array under the hood. CPython overallocates to reduce the number of reallocations. The exact growth policy changed over revisions; in practice it grows by a factor slightly above 1 (to balance memory and speed). Python lists also store pointers to PyObject*, so the list resizing copies pointers but does not copy the objects themselves.

Rust (Vec)

Vec in Rust is a dynamic array with ownership and move semantics. When reallocating, values are moved (or copied if Copy). Rust ensures memory safety through ownership rules.

Go (slice)

Go's slice is a descriptor holding pointer, length, capacity. append may allocate a new underlying array and return a new slice referencing it. Growth policy is implementation-dependent but geometric.

8. Practical Considerations and Pitfalls

Iterator invalidation

Because reallocations can move the underlying block, pointers/iterators into the array may become invalid after a grow (or shrink). Many languages document this: e.g., C++ iterators are invalidated on reallocation; Java's references to elements remain valid because objects are not copied (only references are copied).

Non-trivial element types

When elements have costly copy or move operations, resizing cost increases. Modern implementations try to move instead of copy when possible. In some languages, relocating objects requires calling constructors/destructors or user-defined copy/move operations.

Memory overhead

With geometric growth the average wasted space is a constant fraction of usage. For doubling, at most ~50% wasted on average; with factor 1.5, less waste.

Fragmentation

Large contiguous allocations can be hard to satisfy in systems with fragmented memory. For extremely large arrays, allocation may fail due to lack of contiguous memory even when total free memory is sufficient.

Multithreading

Dynamic arrays are typically not thread-safe. Concurrent modification must be synchronized; otherwise reallocation and pointer updates can cause race conditions.

9. Examples (short)

C++ append example

std::vector v;
for (int i = 0; i < 10; ++i) {
  v.push_back(i);  // amortized O(1)
}
int x = v[5];  // O(1) random access

Python list

lst = []
for i in range(10):
    lst.append(i)  # list grows automatically
x = lst[5]         # O(1) access

10. When to use a dynamic array

  • You need random access by index very fast (O(1)).
  • You mostly append to the end (amortized O(1)).
  • Memory locality matters (cache-friendly iteration).

When frequent inserts/removals at arbitrary positions are required, consider other data structures (linked lists, balanced trees, rope data structures, or specialized sequence containers) — but note these often sacrifice random access speed and locality.

11. Exercises (with answers)

Exercise 1

Suppose you start with capacity 1 and double capacity whenever full. What is the total number of element copies performed over n append operations?

Answer: O(n). More precisely, < 2n copies (roughly <= 2n - 1).

Exercise 2

If you increase capacity by +100 each time the array is full, what is the amortized cost per append?

Answer: O(n) amortized in worst case per append across many appends, because you copy O(n) elements roughly every (100) appends, making average cost linear if n grows large. In other words, this is not an amortized O(1) strategy.

Exercise 3

Why does doubling capacity give O(1) amortized append cost? Provide an intuitive explanation.

Answer: Doubling causes the number of copies to form a geometric series (1 + 2 + 4 + ...), summing to at most about 2n across n appends, so average cost per append is constant.
