Indexing — Comprehensive Explanation, Theory, and Algorithms

A thorough textual guide to indexing: definitions, types, algorithms, design trade-offs, examples, and implementation notes. Emphasis on clear step-by-step explanation rather than interactive experimentation.

What is an Index?

An index is a data structure that enables faster lookup of records or data items by one or more attributes (keys). The index organizes references to the underlying data so that search, range queries, and sometimes ordering operations can be executed efficiently without scanning all stored items.

Why use an index?

  • Reduce search time from linear to logarithmic or constant (depending on structure).
  • Support fast range queries and ordering (for sorted structures).
  • Enable efficient full-text search, spatial queries, and multidimensional lookup.

High-level classification

  • Primary vs Secondary: Primary indexing links to the physical ordering of data; secondary indexes are additional access paths.
  • In-memory vs On-disk: Some index structures are optimized for memory (hash tables, skip lists), others for disk pages (B-trees/B+ trees, R-trees).
  • Exact-match vs Range vs Full-text vs Spatial: Different query patterns favor different index types.

Core Concepts and Definitions

Key
The attribute or set of attributes used to index records (e.g., a column value).
Record / Posting
An entry in the index that references a document or data row. In information retrieval, this is often called a posting.
Posting List
A list of references (document IDs, row pointers) associated with a term or key.
Node / Page
In tree-based indexes, nodes are logical blocks (often disk pages) containing keys and pointers.
Selectivity
How many records share a key value; high selectivity means few records per key — good for indexing efficiency.

Complexity terms

  • Lookup time: time to find the record(s) for a key.
  • Insert/Delete time: time to update the index.
  • Space overhead: additional storage beyond raw data (pointers, metadata).

Common Index Types — Theory and Algorithms

B-Tree and B+Tree (Disk-oriented sorted index)

B-Trees and B+Trees are balanced tree structures optimized for block (page) I/O. B+Tree is a variant where internal nodes store keys and pointers, and all records (or pointers to records) are stored only at the leaf level, often linked for efficient range scans.

Key properties:

  • Nodes have variable number of children within a range: between ceil(m/2) and m, where m is the order.
  • Tree is height-balanced: all leaves are at the same depth.
  • Insertions and deletions may cause node splits or merges, keeping balance.

Basic B+Tree insert (step-by-step, textual):

  1. Find the correct leaf node where the new key should reside by descending from the root (binary search in each node).
  2. If the leaf has space, insert the key and pointer in sorted order.
  3. If the leaf is full, split the leaf into two nodes: move the upper half of entries to the new node. Promote the smallest key of the new node to the parent as a separator.
  4. If the parent becomes full, recursively split up to the root; if the root splits, create a new root increasing the tree height by one.

Complexities (typical): lookup, insert, delete: O(log_m N) where m is node capacity, N is number of entries. Disk-oriented so m is large: fan-out is high and height is small.

// Simplified pseudocode: B+Tree lookup
function BPTreeLookup(root, key):
    node = root
    while node is not leaf:
        i = findIndexInNode(node.keys, key)   // binary search
        node = node.child[i]
    // node is leaf
    i = findIndexInNode(node.keys, key)
    if node.keys[i] == key:
        return node.pointers[i]   // record or pointer to record
    else:
        return NOT_FOUND

Hash-based Index

Hash indexes map keys to buckets using a hash function. They are excellent for exact-match queries (equality) but poor for range queries.

  • Static hashing: fixed number of buckets; may require overflow chains.
  • Dynamic hashing (extendible/linear hashing): buckets split dynamically to grow without global rehashing.

Lookup algorithm (conceptual): compute hash(key), find bucket, scan bucket for key; return pointer(s).

Complexities: average O(1) expected lookup; worst-case O(n) if many collisions or degenerate hash.

Inverted Index (Information Retrieval — Full-text)

An inverted index maps terms (words, tokens) to postings lists of documents where they occur. It's the core of search engines and document retrieval systems.

Components:

  • Dictionary (term -> postings list metadata)
  • Postings lists: sorted sequences of document IDs, often with term frequency and positional information.

Basic inverted index creation (conceptual steps):

  1. Tokenize documents into terms (apply normalization: lowercasing, stemming, stopword removal as required).
  2. For each term occurrence, append the document ID (and optionally position) to a working postings structure.
  3. After processing documents, sort postings lists by document ID and compress them for storage (delta encoding + variable-length encoding).
// High-level pseudocode for inverted index build
function BuildInvertedIndex(documents):
    dictionary = empty map
    for each docID, text in documents:
        tokens = Tokenize(text)  // sequence of normalized terms with positions
        for position, term in tokens:
            if term not in dictionary:
                dictionary[term] = empty postings list
            append (docID, position) to dictionary[term]
    for each term in dictionary:
        sort dictionary[term] by docID (and optionally positions)
        compress postings list
    return dictionary

Query evaluation examples:

  • Boolean AND of terms: intersect sorted postings lists using a two-pointer merge.
  • Phrase query: use positional postings to check adjacent positions across documents.
// Intersect two sorted postings lists A and B
function Intersect(A, B):
    i = 0; j = 0
    result = []
    while i < length(A) and j < length(B):
        if A[i] == B[j]:
            append A[i] to result
            i++; j++
        else if A[i] < B[j]:
            i++
        else:
            j++
    return result

Suffix Arrays and Suffix Trees

These structures index substrings of a text. A suffix array is a sorted array of all suffixes; a suffix tree is a compressed trie of all suffixes.

Use cases: exact substring search, longest repeated substring, compressed pattern matching. Building a suffix array can be done in O(n) (complex algorithms) or O(n log n) with simpler approaches.

Spatial Indexes (R-tree, Quadtree, KD-tree)

Spatial indexes are designed for multidimensional data (points, rectangles). R-tree organizes bounding rectangles hierarchically. KD-tree partitions space by splitting alternately on dimensions.

R-tree insertion (conceptual): choose a leaf whose bounding rectangle requires least enlargement to include the new object; insert; if overflow, split node and adjust ancestors.

Bitmap Indexes

For columns with low cardinality (few distinct values), bitmap indexes store a bit vector per distinct value. Logical operations (AND, OR) implemented as bitwise operations are extremely fast.

Index Compression and Storage Optimization

Indexes can be large; compression reduces space and I/O. Inverted index postings are commonly compressed with techniques that exploit sorted docIDs:

  • Delta (gap) encoding: store differences between successive docIDs rather than absolute IDs.
  • Variable-length integer encodings: variable-byte encoding (VByte), Elias gamma, Golomb/Rice coding — trade time vs space.
  • Block-based compression: compress blocks of postings using simpler per-block methods to allow skipping.

Compression considerations:

  • Compression reduces I/O: saved bytes transfer faster from disk.
  • Decompression CPU cost: choose algorithms balancing CPU vs I/O.
  • Support for skipping: ability to jump forward in compressed postings (for intersection) improves query speed.

Updates, Maintenance, and Consistency

Indexes must reflect data changes. Strategies differ by index type:

  • Immediate updates: update index synchronously on each insert/delete (common in databases).
  • Batch/incremental: buffer changes in a write-log or memory and merge into the main index periodically (common in search engines). This reduces random I/O at the cost of temporarily stale results.
  • Rebuilds: for large-scale changes, sometimes rebuild the index from scratch.

Concurrency and transactions:

  • Database systems use locking protocols or MVCC to ensure consistent index views.
  • Search systems often use immutable segment-based indexes plus merging; readers see a consistent snapshot while writers create new segments.

Garbage collection and compaction

Periodically remove deleted entries, merge small index fragments, and defragment to reduce space and improve locality.

Index Selection and Design Guidelines

Choosing the right index depends on workload and data characteristics. Key questions to ask:

  • What queries are most frequent? Equality, range, prefix, substring, spatial, or full-text?
  • What are the cardinalities of the attributes? Low-cardinality columns favor bitmap indexes.
  • How often does the data change? Frequent writes favor structures with cheap updates or batching strategies.
  • What are latency and throughput requirements? CPU-bound decompression vs I/O-bound reads matter.

Common heuristics:

  • For small tables, full table scans can be faster than maintaining complex indexes.
  • Use composite indexes when queries involve multiple columns in the same predicate order; beware of order sensitivity.
  • Avoid over-indexing: each index adds write overhead.

Index tuning

  • Collect statistics (histograms, cardinality estimates) to guide query planners.
  • Monitor slow queries and examine execution plans to determine missing indexes or suboptimal usage.
  • Consider partial indexes (index only rows satisfying a predicate) to reduce index size when appropriate.

Practical Example — Inverted Index (Step-by-step, textual)

This example walks through building a small inverted index for three short documents, and then explains how to answer a simple AND query.

Documents (docID : text)

1: "the quick brown fox"
2: "the quick fox jumps"
3: "quick fox brown"

Step 1 — Tokenization and normalization

We remove punctuation and lowercase; we keep stopwords for clarity here.

// tokens per doc (term positions starting at 1)
doc 1: [ (the,1), (quick,2), (brown,3), (fox,4) ]
doc 2: [ (the,1), (quick,2), (fox,3), (jumps,4) ]
doc 3: [ (quick,1), (fox,2), (brown,3) ]

Step 2 — Collect postings

For each term, append (docID, position):

"the"   -> [(1,1), (2,1)]
"quick" -> [(1,2), (2,2), (3,1)]
"brown" -> [(1,3), (3,3)]
"fox"   -> [(1,4), (2,3), (3,2)]
"jumps" -> [(2,4)]

Step 3 — Sort postings by docID and optionally compress

They are already sorted; gap-encoding example for "quick": docIDs [1,2,3] -> gaps [1,1,1]

Query example — "quick AND fox"

Intersect postings for "quick" [1,2,3] and "fox" [1,2,3]. Using two-pointer merge: 

i=0,j=0 -> 1==1 -> add 1 -> i=1,j=1
i=1,j=1 -> 2==2 -> add 2 -> i=2,j=2
i=2,j=2 -> 3==3 -> add 3 -> i=3,j=3
Result: [1,2,3]

All three documents contain both terms; if we required a phrase "quick fox", we would use positions to require position(term2) = position(term1)+1. For doc1: quick@2 and fox@4 -> not adjacent; doc2: quick@2 and fox@3 -> match; doc3: quick@1 and fox@2 -> match. Phrase result: documents 2 and 3.

Note: This example uses a tiny dataset. In real systems, inverted indexes contain millions or billions of postings and are stored compressed on disk in segments for scalability.

Algorithms — Pseudocode and Descriptions

Intersect k sorted postings lists (galloping / exponential search optimization)

When lists have very different lengths, use the shortest list as driver and search longer lists by "galloping" (exponential) to reduce comparisons. 

function IntersectK(lists):
    sort lists by length ascending
    result = lists[0]
    for i = 1 to k-1:
        result = IntersectWithGallop(result, lists[i])
        if result is empty: break
    return result

function IntersectWithGallop(A,B):
    // A is shorter
    i = 0
    j = 0
    out = []
    while i < len(A) and j < len(B):
        if A[i] == B[j]:
            append A[i] to out; i++; j++
        else if A[i] < B[j]:
            // gallop forward in A or advance i
            i = advanceWithExponentialSearch(A, i, B[j])
        else:
            j = advanceWithExponentialSearch(B, j, A[i])
    return out

B+Tree split (node capacity = m) — textual explanation

  1. Let leaf has m entries; inserting new key yields m+1 entries.
  2. Split into left with ceil((m+1)/2) entries and right with remaining entries.
  3. Copy the smallest key of right to parent as separator. Update sibling pointers for leaf-level linked list.
  4. If parent is full, split parent likewise (may propagate up).

Dynamic hashing (extendible hashing) — idea

Use a directory of pointers indexed by first d bits of hash. Each bucket has local depth. When a bucket overflows, if local depth < global depth, split the bucket and adjust directory pointers; if local depth == global depth, double directory size (increment global depth).

Trade-offs and Practical Considerations

No single index is best for every workload. Consider these trade-offs:

  • Read vs write: indexes speed reads but cost writes. Batch updates reduce write cost but increase staleness.
  • Space vs performance: more auxiliary information (positions, frequencies) improves query capabilities but increases size.
  • CPU vs I/O: heavy compression saves I/O at expense of CPU decompression. Modern systems balance CPU and I/O throughput.
  • Complexity vs simplicity: robust, simple structures (hash tables) are easy to implement; complex structures (suffix trees, B+ variants) require more care but support richer queries.

Monitoring and benchmarking

  • Measure query latencies, throughput, and index build/merge times.
  • Profile I/O patterns and CPU usage during queries.
  • Use representative workloads for tuning (not only synthetic tests).

Glossary

Posting
Reference to a document or record associated with a term.
Delta/Gaps
Differences between sorted docIDs to enable smaller integers for compression.
Fan-out
Average number of children of internal nodes in a tree; higher fan-out reduces height.
Segment
Immutable chunk of index; common in search systems to allow fast concurrent reads/writes.

Quick Tips

  • Index selective columns (low estimated cost per query) first.
  • Favor B+Tree for range queries; hash for equality; inverted for full-text; R-tree for rectangles.
  • Keep statistics up to date; they guide planners to use indexes correctly.
