Hash Functions — Comprehensive Explanation

A deep, pedagogical guide exploring the concept, theory, algorithms, properties, common uses, and practical considerations for cryptographic and non-cryptographic hash functions.

Overview: What is a hash function?

A hash function is a deterministic procedure that maps input data of arbitrary size to a fixed-size output, called the hash value, digest, or simply "hash". The mapping is typically written as: 

hash = H(message)

A hash function can be used to create succinct representations of data. Depending on design intent, hashes may be general-purpose (checksums, hash tables) or cryptographic (integrity, authentication, digital signatures).

Core properties and terms

In the context of cryptographic hash functions the important properties are:

  • Deterministic: the same input always produces the same output.
  • Fixed output size: regardless of input length, the digest length is fixed (e.g. 256 bits).
  • Preimage resistance: given a hash value h, it should be computationally infeasible to find any input m such that H(m) = h.
  • Second-preimage resistance: given an input m1, it should be infeasible to find a different input m2 (m2 ≠ m1) such that H(m1) = H(m2).
  • Collision resistance: it should be infeasible to find any two distinct inputs m1, m2 such that H(m1) = H(m2). Note: collision-finding generally exploits the birthday paradox.
  • Avalanche effect: a small change in input (e.g., flipping one bit) produces a substantially different hash.

Why hashes are useful: common applications

  • Data integrity: verify that content was not altered (e.g., downloads accompanied by a published hash).
  • Digital signatures: sign the hash of a large message instead of the message itself; reduces computational cost and preserves integrity.
  • Password storage: store a salted and iterated hash of a password rather than plaintext.
  • Authentication (HMAC): keyed-hash message authentication codes provide message authentication using hashes.
  • Merkle trees: hierarchical hashing enables compact verification of large datasets (used in blockchains, version control, etc.).
  • Hash tables: non-cryptographic hashing maps keys to buckets for efficient lookup.
  • Proof-of-work: miners search for inputs that produce a hash with certain properties (e.g., many leading zeros).

Distinguishing hashes and encryption

Hashing is one-way (non-invertible by design), while encryption is two-way (decryptable with a key). Use hashing when you need a fixed-size fingerprint or irreversible storage. Use encryption when you need confidentiality and the ability to recover original data with the correct key.

Basic theory: collision probability and the birthday paradox

For an n-bit hash, the total number of distinct outputs is 2^n. Two important complexity figures:

  • Preimage attacks: brute force requires ~2^n operations to find any preimage mapping to a given hash.
  • Collision attacks: because of the birthday paradox, collisions are expected after about 2^(n/2) randomly chosen inputs. This halves the effective strength for collision resistance.

Examples:

  • n = 128 bits → collision effort ≈ 2^64 (feasible if 2^64 can be computed); preimage effort ≈ 2^128 (far harder).
  • n = 256 bits → collision effort ≈ 2^128; preimage effort ≈ 2^256 (both infeasible with current technology).

High-level algorithmic families

Two major structural approaches used in cryptographic hashes:

  • Merkle–Damgård construction (used by MD5, SHA-1, SHA-2): compress message blocks iteratively with a compression function; final output is the last state. It includes padding and length encoding to prevent simple extension ambiguities.
  • Sponge construction (used by SHA-3 / Keccak): uses an internal state split into "rate" and "capacity" portions; data is absorbed into the state then squeezed to produce output. It can produce variable-length output and resists some attacks that affect Merkle–Damgård.

Pseudocode: Merkle–Damgård (conceptual)

// Input: message (arbitrary length)
    // Output: fixed-size digest
    function H(message):
      // 1. Padding: append a single '1' bit, then '0' bits, then a representation of the message length
      padded = pad(message)

      // 2. Initialize chaining variable (IV)
      state = IV

      // 3. Process each fixed-size block with compression function f
      for each block in padded:
        state = f(state, block)

      // 4. Output the final state (maybe truncated)
      return output_transform(state)

Compression f maps (state, block) → new state. The security depends on a strong f and correct padding/length treatment.

Pseudocode: Sponge (conceptual)

// Keccak-style sponge
    function sponge_hash(message, rate, capacity):
      state = zeros(rate + capacity)
      // Absorb phase:
      for each block of 'rate' bits in message:
        state[0:rate] = state[0:rate] XOR block
        state = permutation(state)
      // Squeeze phase:
      output = empty
      while length(output) < desired:
        output += state[0:rate]
        state = permutation(state)
      return truncated(output)

Common hash algorithms (brief)

  • MD5: 128-bit digest, fast, now considered broken for collision resistance (do not use for security).
  • SHA-1: 160-bit digest, collisions demonstrated—deprecated for security.
  • SHA-2 family (SHA-224, SHA-256, SHA-384, SHA-512): widely used, secure when used properly (SHA-256 and SHA-512 common).
  • SHA-3 (Keccak): sponge-based, has different internal structure than SHA-2; provides diversification and resistance to certain potential attack classes.
  • BLAKE2 / BLAKE3: modern, fast, and secure; designed as alternatives to SHA-2/3 with high performance.

Password hashing and key derivation

For password storage you must use a deliberately slow, memory-hard, salt-based function to thwart brute-force and parallelized attacks. Examples:

  • PBKDF2: applies HMAC iteratively; parameterized by iterations.
  • bcrypt: cost factor and internal Blowfish work; designed for password hashing.
  • scrypt: memory-hard design to reduce GPU/ASIC advantage.
  • Argon2: winner of the password hashing competition; supports tunable memory, time, and parallelism.

Rule of thumb: use Argon2 (or at least bcrypt/scrypt) with a unique salt per password; store salt and parameters with the resulting hash. Don’t use plain SHA-256 with a salt and a few iterations — it is too fast and easier to brute-force.

HMAC: making a keyed hash for authentication

HMAC (Hash-based Message Authentication Code) builds a secure MAC around an existing hash function. It resists certain extension and collision weaknesses when used properly. Standard formula: 

HMAC(K, m) = H( (K ⊕ opad) || H( (K ⊕ ipad) || m ) )

Here K is a key (possibly zero-padded), ipad/opad are fixed byte constants, and || denotes concatenation. HMAC is widely used for message authentication (TLS, IPsec, APIs).

Attacks and practical considerations

  • Collision attacks: when collisions are found faster than 2^(n/2), collisions can undermine digital signatures and other schemes relying on uniqueness.
  • Length extension attacks: certain Merkle–Damgård-based constructions (e.g., naive use of H(m)) allow an attacker who knows H(m) to compute H(m || pad || extra) without knowing m. Use HMAC or a sponge-based hash to avoid this flaw.
  • Hash truncation: truncating a hash reduces collision resistance; choose digest sizes according to target security (e.g., 256-bit digest for 128-bit collision resistance).
  • Side-channels: implementations must avoid timing and memory side-channels in sensitive contexts (e.g., password comparisons must be constant-time).

Merkle trees: an illustrative example

Merkle trees let you authenticate many items with a single small root digest. Create leaves = H(data_i), then iteratively hash pairs: 

// Four items: A,B,C,D
    L0 = H(A), L1 = H(B), L2 = H(C), L3 = H(D)
    Node0 = H(L0 || L1)
    Node1 = H(L2 || L3)
    Root = H(Node0 || Node1)

To prove A is in the tree you provide L1 and Node1; the verifier recomputes Node0 = H(L0 || L1) and Root and checks equality against the published root. This is widely used in blockchains and distributed systems.

A simple step-by-step toy hash (educational)

Below is a deliberately simple, insecure "toy" hash algorithm presented for learning purposes. It is NOT cryptographically secure. The page includes an optional demonstration that shows how the algorithm mixes data and produces a digest.

// SimpleHash (toy example - not secure)
    // 1. Initialize four 32-bit words with constants
    state[0..3] = IV0, IV1, IV2, IV3

    // 2. For each input byte b:
      i = index of byte (starting at 0)
      j = i mod 4
      state[j] = ROTL32(state[j] XOR b, (b mod 16) + 1)
      state[j] = state[j] + 0x9e3779b9  // golden ratio constant

    // 3. After processing all bytes, finalize:
    //    mix the four words with a few rounds of a simple mixing function,
    //    then concatenate to produce a 128-bit (16-byte) digest.

The toy algorithm highlights core ideas: an internal state, per-block (or per-byte) mixing, non-linearity (XOR + rotation), and a finalization/mixing stage. Real cryptographic hashes use carefully analyzed, larger permutations and more rounds.

Security recommendations (practical)

  • For data integrity and general purpose hashing: use SHA-256, SHA-3-256, or BLAKE3 (choose modern algorithms and keep libraries up-to-date).
  • For password hashing: use Argon2 (tunable memory/time) or bcrypt/scrypt; never use fast hashes (SHA-1/SHA-256) directly without a KDF designed for passwords.
  • For message authentication: use HMAC with a secure hash function or an AEAD construction (e.g., AES-GCM, ChaCha20-Poly1305).
  • For long-term systems: prefer algorithms with active community analysis (SHA-3, BLAKE2/3) and be prepared to transition if weaknesses are discovered.

Quick reference: hash sizes vs collision work

Collision cost ≈ 2^(n/2). Example table (approximate):

n-bit digest   collision effort
128 bits       → 2^64  (weak)
160 bits       → 2^80  (moderate)
256 bits       → 2^128 (strong today)
512 bits       → 2^256 (very strong)
