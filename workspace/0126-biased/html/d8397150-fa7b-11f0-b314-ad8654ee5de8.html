What is REST?

REST stands for Representational State Transfer. It is an architectural style for distributed systems, published by Roy Fielding in his 2000 doctoral dissertation. REST defines principles and constraints that, if followed, lead to scalable, loosely coupled, and evolvable web services that make use of the HTTP protocol and its semantics.

Core idea in a single sentence

Model your system as a set of resources (identifiable things), expose those resources through a uniform interface (HTTP methods and URIs), and exchange representations (typically JSON or XML) of resource state over stateless requests and responses.

REST Constraints and Principles

Fielding described a set of architectural constraints. A server that adheres to them is said to be RESTful.

  • Client–Server: Separation of concerns — client handles user interface, server handles data storage and business logic.
  • Statelessness: Each request from client to server must contain all information necessary to understand and process the request. The server must not store client-specific session state between requests.
  • Cacheable: Responses must define themselves as cacheable or not to improve network efficiency.
  • Uniform Interface: A standard set of operations (usually mapped to HTTP methods) applied consistently across resources.
  • Layered System: System can be composed of hierarchical layers (e.g., load balancers, proxies) without client awareness.
  • Code on Demand (optional): Servers may provide executable code (e.g., JavaScript) to clients to extend functionality.
  • Hypermedia as the Engine of Application State (HATEOAS): Responses should provide links allowing clients to discover actions at runtime (less emphasized in many practical APIs but conceptually important).

HTTP & Uniform Interface

A RESTful API typically uses HTTP verbs to express the desired action on a resource. Common mapping patterns:

Common HTTP Methods

  • GET: Retrieve a representation of a resource. Safe and idempotent.
  • POST: Create a new resource or execute a server-side operation. Not idempotent (usually).
  • PUT: Replace a resource entirely. Idempotent.
  • PATCH: Apply a partial update to a resource. May or may not be idempotent depending on semantics.
  • DELETE: Remove a resource. Idempotent.
  • HEAD: Same as GET but only headers (no body).
  • OPTIONS: Discover allowed methods or CORS policies for a resource.

Idempotency and Safety

  • Idempotent: Making the same request multiple times results in the same server state as making it once (e.g., PUT, DELETE, GET).
  • Safe: Request does not modify resources (e.g., GET, HEAD).

Resources, URIs, and Representations

Resources are conceptual entities (e.g., user, article, order). They are identified by URIs (Uniform Resource Identifiers). A representation is a document that captures the resource state at a point in time (commonly JSON).

URI design guidelines

  • Use nouns, not verbs: /users, /orders/123/items
  • Hierarchy expresses containment: /users/{userId}/posts
  • Use plural resource names consistently: /books, not /book
  • Use query parameters for filtering, pagination, sorting: /books?author=doe&limit=20
  • Keep URIs discoverable and logical; avoid encoding actions into the path (prefer HTTP method + resource).

Example: Simple Book Resource

Example endpoints:

GET  /books              -> list books
GET  /books/{id}         -> get book
POST /books              -> create book
PUT  /books/{id}         -> replace book
PATCH /books/{id}        -> modify book
DELETE /books/{id}       -> delete book

Request / Response Semantics

RESTful communication is request-response oriented. Requests include method, URL, headers, and optionally a body. Responses include a status code, headers, and optionally a body.

Status codes — guidance

  • 2xx — Success (200 OK, 201 Created, 204 No Content)
  • 3xx — Redirection (rarely used in APIs)
  • 4xx — Client errors (400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity)
  • 5xx — Server errors (500 Internal Server Error, 503 Service Unavailable)

Content negotiation

Clients may request a representation format using the Accept header. Servers return the Content-Type header. Common format: application/json. For APIs that support multiple representations, content negotiation allows flexible clients.

Designing a REST API: Step-by-step narrative

The following is a recommended sequence for designing a REST API from scratch.

Step 1 — Identify resources and their relations

Write down the nouns in your domain: users, projects, tasks, invoices. For each resource, note attributes and relationships (one-to-many, many-to-many).

Step 2 — Model resource representations

Decide what fields are returned in responses and accepted in requests. Include metadata (timestamps, e.g., createdAt, updatedAt) and link relations (self, related resources).

Step 3 — Choose URIs and methods

Map operations to HTTP verbs and design clear endpoints. Favor predictable patterns like /resources and /resources/{id}/subresources.

Step 4 — Define behaviors for edge cases

Decide how to handle missing resources, idempotency of updates, partial updates (PATCH semantics), and conflicts (HTTP 409). Define validation rules and error formats.

Step 5 — Consider performance and scale

Design pagination (offset vs cursor), filtering, sorting, and selective field projection. Design caching strategy with Cache-Control and ETag headers. Plan rate limiting and quotas.

Step 6 — Security

Choose authentication (OAuth2, API keys, JWT), transport security (TLS), and authorization model (role-based, resource-based). Validate inputs and sanitize outputs to prevent injection attacks.

Algorithms and Strategies Commonly Used

APIs operate in complex environments; several algorithms or patterns are commonly used. This section explains them in an algorithmic or stepwise manner where applicable.

Pagination

Two common approaches with trade-offs:

  • Offset-based pagination (page & offset)
    • Implementation: SELECT ... LIMIT {limit} OFFSET {offset}
    • Pros: simple and intuitive
    • Cons: performance degrades with large offsets, unstable when data is changing (inserts/deletes)
  • Cursor-based pagination (seek method)
    • Implementation: SELECT ... WHERE (sort_key > cursor_value) ORDER BY sort_key LIMIT {limit}
    • Pros: more efficient for large datasets and stable under concurrent modifications
    • Cons: more complex client logic; cursors can leak internal sorting values unless encoded

Optimistic Concurrency Control (ETags)

Use ETag and If-Match headers to prevent lost updates:

Client GET /items/42 -> receives ETag: "v3"

Client modifies object and sends:
PUT /items/42
If-Match: "v3"
{ ... updated representation ... }

If ETag matches server's current version -> accept update
If not -> 412 Precondition Failed (client must reconcile)

Rate Limiting

Common approaches:

  • Leaky bucket / token bucket: limit requests per time window with burst allowance.
  • Fixed window: simple count per window (can spike at boundaries).
  • Responses include headers like X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset to help clients.

Caching

Design cacheability using standard HTTP headers:

  • Cache-Control: public,max-age=60
  • ETag and If-None-Match allow conditional GETs and 304 Not Modified responses.
  • Vary header indicates what request headers affect representation.

Error Handling and Standard Formats

Define a consistent error response schema. Example (application/problem+json):

HTTP/1.1 400 Bad Request
Content-Type: application/problem+json

{
  "type": "https://example.com/probs/invalid-input",
  "title": "Invalid input",
  "status": 400,
  "detail": "Field 'email' must be a valid email address.",
  "instance": "/users/registration"
}

Include machine-readable error codes, human-readable messages, and optionally links to documentation or remediation steps.

Versioning Strategies

Common versioning approaches:

  • URI versioning: /v1/items — explicit and discoverable, but mixes versioning into identifier.
  • Header versioning: Accept: application/vnd.example.v1+json — less visible but cleaner URIs.
  • Semantic/feature negotiation: prefer to design APIs that don't require frequent version bumps by adding backward-compatible changes where possible.

Security Considerations

  • Always use TLS (HTTPS) for transport.
  • Use secure authentication and authorization (OAuth2, JWT with short lifetimes, rotating credentials).
  • Validate and sanitize inputs server-side; do not trust client validation alone.
  • Use rate limiting and anomaly detection to protect against abuse.
  • Avoid exposing internal implementation details in error messages.
  • Keep secrets (API keys) out of URIs (they can leak in logs). Prefer headers for credentials.

Testing, Documentation, and Tooling

Good documentation and tests are essential. Use OpenAPI/Swagger to describe endpoints, parameters, request/response schemas, and example payloads. Automate tests at multiple levels:

  • Unit tests for business logic
  • Integration tests for controllers/handlers
  • Contract tests checking request/response semantics
  • End-to-end tests in a staging environment

Documentation practices

  • Provide examples for common flows (create, update, retrieve, delete)
  • Document error codes and expected responses
  • Provide code samples in popular languages (curl, JavaScript, Python)
  • Keep docs versioned alongside API versions

Practical Examples and Patterns

Filtering, sorting, and field selection

Common query parameter patterns:

GET /products?category=books&price_lt=30&sort=-rating&fields=id,title,price

Explaination:

  • price_lt=30 means price less than 30
  • sort=-rating means sort by rating descending (prefix -)
  • fields=... selects a projection to reduce payload size

Batch operations

When supported, batch endpoints can reduce round trips. Design carefully to maintain idempotency guarantees and clear status reporting for partial failures.

Common Anti-Patterns to Avoid

  • Using verbs in URIs (e.g., /getUser, /createOrder) instead of HTTP methods.
  • Embedding excessive business logic in URIs or using RPC-style endpoints without good reason.
  • Returning 200 OK for error conditions with an error object in the body — prefer appropriate 4xx/5xx codes.
  • Exposing internal implementation IDs and structure that might change.
  • Breaking backward compatibility without a clear versioning strategy.

Lifecycle of a Typical Request (Step-by-step)

  1. Client constructs an HTTP request with method, URL, headers, and optional body.
  2. Request traverses network and any intermediary proxies or load balancers (layered system).
  3. Server receives request and authenticates the client (e.g., check token in Authorization header).
  4. Server authorizes the action (permission checks, ACLs).
  5. Server validates and parses inputs (query params, body schema).
  6. Server executes business logic, accesses persistence, possibly emits events.
  7. Server prepares a representation of the result, sets appropriate headers (Content-Type, Cache-Control, ETag), and a status code.
  8. Response is returned to client; client handles it and may follow hypermedia links or trigger further requests.

  • Secure authentication and authorization
  • Versioning strategy and backward compatibility plan
