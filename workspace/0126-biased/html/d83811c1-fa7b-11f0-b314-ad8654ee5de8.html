Paging — A Comprehensive, Textual Explanation

Concepts, theory, algorithms, worked examples, and step-by-step narratives for operating systems and memory management

Overview — What is Paging?

Paging is a memory-management technique used by operating systems to map virtual addresses used by a program to physical addresses in main memory (RAM). It divides both virtual address space and physical memory into fixed-size blocks. The blocks in virtual space are called pages, and the blocks in physical memory are called frames.

Key idea: by splitting memory into fixed-size units and using an indirection table, the OS can place a program's pages noncontiguously in physical memory and provide isolation, sharing, and efficient use of RAM.

Core Definitions

  • Virtual address: An address used by a program. It is typically split into a page number and an offset.
  • Page: A fixed-size block of virtual memory (e.g., 4 KiB, 2 MiB).
  • Frame: A fixed-size block of physical memory of the same size as a page.
  • Page table: A per-process data structure that maps virtual page numbers to frame numbers (and contains metadata).
  • Translation Lookaside Buffer (TLB): A small hardware cache of recent virtual-to-physical page translations to accelerate address translation.
  • Page fault: An exception raised when the CPU references a virtual page that is not currently resident in a frame (or lacks permissions).

Address Translation — Step by Step

Suppose the system uses a page size of 4 KiB (4096 bytes) and a 32-bit virtual address. The address is split:

  • Top bits: virtual page number (VPN)
  • Low bits: page offset (0..4095)

Translation steps when the CPU issues a virtual memory access (read/write):

  1. CPU extracts the VPN and offset from the virtual address.
  2. CPU checks the TLB for a cached mapping of VPN → frame number (PFN). If found (TLB hit), compute physical address = PFN × page_size + offset, and access memory.
  3. If TLB miss, consult the process's page table in memory (or multi-level page tables). If the page table entry says the page is resident (has a PFN), load PFN into the TLB and proceed.
  4. If the page table indicates the page is not resident (page not present), the CPU triggers a page fault exception.
  5. The OS page fault handler determines why: maybe the page is on disk (swap or file), or this is an invalid access (segfault), or a protection violation. If valid but swapped out, the OS selects a free frame (or frees one by evicting another page), reads the page from disk into the frame, updates page table and TLB, and restarts the faulting instruction.

Note: the runtime cost of a page fault is orders of magnitude higher than a normal memory access because it may involve disk I/O.

Page Table Structures

Several page-table organizations are used to save space and accelerate translation:

Single-level page table

Simple array indexed by VPN. Works but size grows with the virtual address space, often wasteful for sparse address spaces.

Multi-level page table

Break the VPN into multiple parts. Each part indexes a level of tables. Only the levels needed for the process's used pages are allocated, saving memory. Modern 64-bit systems commonly use 4–5 levels.

Inverted page table

One global table indexed by physical frames rather than virtual pages. Each entry notes what (process, VPN) is stored in that frame. Reduces memory for page tables but makes lookup slower (often requires hashing or TLB reliance).

Hashed page table

Use a hash table keyed by (process id, VPN) to find page table entries; used in some 64-bit systems.

Page Replacement — Algorithms and Rationale

When all physical frames are occupied and the OS needs a free frame to load a page, it must evict (replace) an existing page. The policy selecting which page to evict is a page replacement algorithm. Goals: minimize page faults and maximize throughput.

Desirable metadata in page table entries

  • Present bit: is the page resident?
  • Reference/Accessed bit (R): set when the page is referenced (read/write).
  • Dirty/Modified bit (M): set when the page is written to — determines whether the OS must write it back to disk before eviction.
  • Protection bits: read/write/execute permission.

Classical algorithms

Optimal (Belady's algorithm)

Evict the page whose next use is farthest in the future. It is optimal (minimizes page faults) but requires future knowledge and is not implementable in general. Useful as a benchmark.

FIFO (First-In, First-Out)

Evict the oldest loaded page. Simple but ignores recency and can perform poorly (Belady's anomaly: more frames can lead to more faults).

LRU (Least Recently Used)

Evict the page that was least recently referenced. Good heuristic assuming recent past predicts near future. Approximations exist due to hardware cost.

Approximations to LRU

  • Clock (Second-Chance): use a circular list and an accessed bit. When choosing victim, sweep: if accessed bit = 1, clear it and skip; if 0, evict.
  • NRU (Not Recently Used): periodically clear R bits; classify pages into four classes based on (R,M) and evict from lowest class.

Working Set and WSClock

The working set is the set of pages a process actively uses within a time window. WSClock is an algorithm combining clock and working-set ideas to select victims whose last reference is older than a threshold and are not dirty (or can be written back). It adapts to process locality.

Complexity and implementation notes

Exact LRU requires updating timestamps or moving pages on every reference — expensive in software. Hardware support (like R and M bits) enables approximations. The TLB reduces the frequency of page-table lookups but must be handled carefully when page table entries change.

Working Through a Small Example (Textual)

Consider a reference string of page numbers (CPU accesses): [7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2] and 3 physical frames. We'll reason about how many page faults different algorithms produce.

High-level steps for manual simulation:

  1. Start with empty frames (all frames contain no pages).
  2. For each reference in sequence, check if the requested page is in any of the frames. If yes, it's a "hit" (no page fault). If no, it's a page fault: load the page into a free frame, or if none free, choose a page to evict per the algorithm and replace it.
  3. Update metadata (for LRU, record time of last use; for Clock, set the accessed bit when referenced).

Detailed Algorithms — Pseudocode and Explanation

LRU (Logical description)

LRU concept: on a page fault when replacement is needed, choose the page with the oldest last-reference time.

Data structures:
 - frames: set of frame slots
 - last_used[page]: timestamp or sequence counter of most recent reference

On reference to page p at time t:
 - if p in frames: last_used[p] = t; hit
 - else:
     page fault
     if free frame exists:
         load p into free frame
         last_used[p] = t
     else:
         victim = page q in frames with smallest last_used[q]
         evict q (write back if dirty)
         load p into its frame
         last_used[p] = t

Clock (approximate LRU)

Clock maintains a circular list of frames and a single "hand". Each reference sets the R bit on that page. To replace, advance the hand until you find a page with R=0; while R=1, clear R and advance.

On page fault:
  while true:
    if hand.R == 0:
      choose page at hand as victim
      break
    else:
      hand.R = 0
      advance hand
  evict victim, load new page into that frame, set its R=1, advance hand

Optimal (for analysis only)

To compute optimal replacement at time of fault, choose the resident page whose next reference is farthest in the future (or never referenced again).

Performance Metrics

  • Page fault rate: fraction of memory accesses that cause a page fault.
  • Access time: effective access time (EAT) = (1 - fault_rate) × memory_access + fault_rate × page_fault_service_time. Page fault service time dominates and includes disk I/O time.
  • TLB hit rate: percentage of translations serviced by the TLB. A low TLB miss rate is critical for performance.

Example EAT calculation (illustrative): if memory access = 100 ns, page_fault_service_time = 10 ms = 10,000,000 ns, and fault_rate = 0.00001 (1e-5), then EAT ≈ 100 ns + 0.00001 × 10,000,000 ns = 100 ns + 100 ns = 200 ns. Even tiny fault rates can double access time.

Practical Considerations & Variants

  • Page size tradeoffs: larger pages reduce page-table size and TLB pressure but increase internal fragmentation and I/O time when loading pages. Systems often support multiple page sizes (4 KiB, 2 MiB, 1 GiB).
  • Copy-on-write (COW): when forking a process, the OS initially shares pages between parent and child as read-only; on write, a page fault creates a private copy. This avoids copying eagerly.
  • Memory-mapped files: pages can be filled from files, not just anonymous swap space. Faulting loads the page from the file.
  • Shared pages: multiple processes may map the same physical frames (e.g., shared libraries).
  • Global vs. local allocation: when multiple processes share frames, the OS can use global replacement (any process's pages can be evicted) or local replacement (limit frames available per process). Each policy affects fairness and thrashing risk.
  • Thrashing: occurs when a process spends more time servicing page faults than executing useful work. Common causes: too few frames for the workload, poor locality, or swapping loops.

Study Tips and Typical Exam Questions

To internalize paging concepts: practice hand-simulating small reference strings for different algorithms and frame counts; draw page tables and TLB interactions; compute effective access time for scenarios with given TLB hit rates and page-fault rates.

  • Explain the steps the OS takes when a process issues a memory reference that causes a page fault.
  • Simulate LRU and FIFO on a short reference string and count page faults.
  • Compare multi-level and inverted page tables: when would each be preferred?
  • Describe how copy-on-write reduces duplication after fork().
