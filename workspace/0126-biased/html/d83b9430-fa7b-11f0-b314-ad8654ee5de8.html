Encryption — A Comprehensive Textual Guide

Definitions, theory, algorithms, step-by-step explanations, examples, pitfalls, and best practices — primarily textual and pedagogical.

1. High-level Overview: What Is Encryption?

Encryption is the process of transforming readable data (plaintext) into an unreadable form (ciphertext) using an algorithm and a secret (a key), so that only authorized parties can reverse the process (decrypt) and recover the original data.

Primary goals of encryption in modern systems:

  • Confidentiality: Prevent unauthorized reading of data.
  • Integrity: Detect if data was altered.
  • Authentication: Confirm the identity of parties involved.
  • Non-repudiation: Prevent denial of actions when digital signatures are used.

Basic terms

  • Plaintext: Original readable message or data.
  • Ciphertext: Encrypted, unreadable output produced by an encryption algorithm.
  • Key: Secret parameter that controls the operation of the cipher.
  • Encryption algorithm (cipher): Mathematical procedure for encryption and decryption.
  • Nonce / IV: A number or value used once (or with limited reuse) to ensure non-determinism.
  • Authenticated encryption: Encryption that provides both confidentiality and integrity in a single operation (e.g., AES-GCM).

2. Historical Context and Classical Ciphers

Cryptography has an ancient history. Early systems were classical substitution and transposition ciphers — easy to describe and break by frequency analysis, but historically important for understanding modern designs.

Caesar cipher (very simple)

Shift each letter by a fixed number. Example: shift 3 — A → D, B → E, ... Z → C. 

Encrypt(plaintext, shift):
  For each letter L in plaintext:
    If L is alphabetic:
      Replace L with letter (L + shift) modulo 26
    Else:
      Keep L unchanged
  Return ciphertext
Example: "HELLO" with shift 3 → "KHOOR".

Vigenère cipher (polyalphabetic)

Uses a repeating key word to apply different shifts across positions. More resistant to frequency analysis than Caesar, but still breakable with statistical tools.

Why classical ciphers are insufficient

  • Predictable structure and small key spaces make them vulnerable to frequency analysis and brute-force.
  • They do not provide integrity or authentication.

3. Modern Symmetric Encryption

Symmetric encryption uses a single shared key for both encryption and decryption. It is efficient and used for bulk data encryption.

Block ciphers vs. stream ciphers

  • Block cipher: Encrypts fixed-size blocks (e.g., 128 bits). Examples: AES, DES. Operates in modes to handle data larger than the block size.
  • Stream cipher: Produces a keystream that is XORed with plaintext (e.g., RC4 historically, modern: ChaCha20).

Advanced Encryption Standard (AES)

AES is the dominant block cipher in use today. It operates on 128-bit blocks and supports 128, 192, and 256-bit keys. AES is evaluated for resistance to known attacks and widely standardized.

High-level AES operations (conceptual only):

  • SubBytes — non-linear substitution using an S-box.
  • ShiftRows — permutation of bytes.
  • MixColumns — linear mixing across bytes.
  • AddRoundKey — XOR with a round-derived key.

Block cipher modes of operation (why they matter)

A block cipher by itself only encrypts fixed-size blocks. Modes define how to chain block cipher operations to encrypt arbitrary-length messages and provide additional properties like randomness and integrity.

ModeProperties / Use
ECB (Electronic Codebook)Deterministic per-block — insecure because identical plaintext blocks produce identical ciphertext blocks.
CBC (Cipher Block Chaining)Chaining blocks with IV — hides patterns better, but requires correct IV handling and separate integrity protection (MAC).
CTR (Counter)Transforms block cipher into stream cipher using a counter/nonce. Provides random-access encryption. Requires unique nonce per key.
GCM (Galois/Counter Mode)CTR + authentication tag (AEAD). Provides confidentiality and integrity. Commonly used: AES-GCM.

Padding

Block ciphers require full blocks. Padding schemes (e.g., PKCS#7) add predictable bytes so that the last block is full. Improper padding validation can lead to padding-oracle attacks.

Best practice (symmetric): Use an authenticated encryption mode (AEAD) like AES-GCM, AES-CCM, or ChaCha20-Poly1305 rather than separate encryption and MAC unless you fully understand the implications.

4. Modern Asymmetric (Public-Key) Cryptography

Asymmetric cryptography uses a key pair: a public key (widely shared) and a private key (kept secret). It enables secure key exchange, digital signatures, and encryption without prior shared secrets.

RSA (Rivest–Shamir–Adleman)

RSA security relies on the difficulty of factoring large integers. Keys are typically 2048 bits or larger in modern deployments. RSA supports encryption (with padding) and signatures.

Simplified RSA key idea (conceptual):

  1. Select two large primes p and q, compute n = p * q.
  2. Compute φ(n) = (p−1)(q−1).
  3. Choose e such that 1 < e < φ(n) and gcd(e, φ(n)) = 1 (common choice e = 65537).
  4. Compute d = e^(-1) mod φ(n). Public key = (n, e), private key = d.

Diffie–Hellman (DH) and Ephemeral DH

DH enables two parties to derive a shared secret over an insecure channel. Ephemeral DH (using temporary keys per session) achieves forward secrecy: compromise of long-term keys does not reveal past session keys.

Elliptic Curve Cryptography (ECC)

ECC provides similar security to RSA with smaller key sizes (e.g., a 256-bit ECC key approximates the security of a 3072-bit RSA key). Widely used curves: secp256r1 (P-256), Curve25519 (X25519), etc.

Hybrid Encryption

Asymmetric ciphers are computationally expensive and not used for large data directly. Instead, systems use hybrid encryption: use asymmetric algorithms to securely exchange a symmetric session key, and then use fast symmetric encryption (e.g., AES) for the data.

5. Message Authentication, Hashing, and Authenticated Encryption

Encryption alone does not guarantee integrity (that the message was not altered). Complementary primitives include MACs, digital signatures, and cryptographic hash functions.

Cryptographic hash functions

A hash function maps input data of arbitrary length to a fixed-size digest. Desirable properties: pre-image resistance, second pre-image resistance, and collision resistance. Examples: SHA-256, SHA-3.

Message Authentication Code (MAC)

A MAC is a short piece of information generated from a message and a secret key, used to verify integrity and authenticity. Example: HMAC-SHA256.

Digital Signatures

Signatures use asymmetric keys. The signer uses their private key to sign data (or a hash of data), and anyone with the public key can verify the signature. Signatures provide non-repudiation and authenticity.

Authenticated encryption (AE/AEAD)

Authenticated Encryption with Associated Data (AEAD) provides confidentiality and integrity in one operation. Examples: AES-GCM, ChaCha20-Poly1305. AEAD modes produce a ciphertext and an authentication tag that must be verified during decryption.

6. Typical Attack Types and Vulnerabilities

Understanding attacks helps design countermeasures and choose appropriate algorithms and usage patterns.

Common attack models

  • Ciphertext-only: Attacker only sees ciphertexts.
  • Known-plaintext: Attacker knows some plaintext-ciphertext pairs.
  • Chosen-plaintext / chosen-ciphertext: Attacker can cause encryptions/decryptions of chosen messages (powerful model used in proofs).
  • Side-channel attacks: Exploit physical or timing information leaked during computation (timing attacks, power analysis).

Implementation and misuse vulnerabilities

  • Reusing nonces/IVs in modes like CTR or GCM can catastrophically break security.
  • Using ECB leaks structure of repeated plaintext blocks.
  • Rolling your own cryptography or inventing custom padding/substitution schemes is extremely risky.
  • Poor random number generation undermines keys and nonces.
Do not design or implement cryptographic protocols without expert review. Small mistakes (nonce reuse, weak RNG, improper padding checks) can make systems trivially breakable.

7. Key Management and Randomness

Strong encryption relies on careful key management and high-quality randomness.

Key generation

Generate keys using cryptographically secure random number generators (CSPRNG). Protect keys in storage (use hardware security modules, OS-provided key stores, or encrypted key vaults).

Key lifecycle

  1. Key generation (strong RNG).
  2. Secure distribution (avoid transmitting in plaintext).
  3. Rotation and expiration (limit exposure time).
  4. Revocation and destruction when compromised.

Entropy and randomness

Sources of randomness must be unpredictable. Inadequate entropy (especially during initial system startup) leads to predictable keys. Use system CSPRNGs (e.g., /dev/urandom on modern UNIX systems, or dedicated APIs such as crypto.getRandomValues in browsers).

8. Step-by-Step Example: How a Secure Session Is Established (Hybrid)

This example shows conceptually how two parties establish an encrypted session using asymmetric key exchange and symmetric encryption for data transport.

  1. The server has an asymmetric key pair and a certificate signed by a trusted authority. The client has the authority's root certificate.
  2. The client connects and requests the server's certificate.
  3. The client verifies the certificate and extracts the server's public key.
  4. The client uses ephemeral Diffie–Hellman (or ECDH) to generate an ephemeral key pair and sends the public part to the server.
  5. The server uses its ephemeral key pair and computes the same shared secret. Both sides derive session keys (e.g., symmetric encryption key, MAC key) from the shared secret using a key derivation function (KDF).
  6. From now on, both sides use the symmetric session key for fast encryption (e.g., AES-GCM) of application data. Ephemeral keys are discarded to provide forward secrecy.
This pattern is broadly used in protocols like TLS: use public-key techniques for authentication and secure key exchange, then switch to symmetric encryption for data.

9. Small Worked Numerical Example of RSA (Educational, tiny numbers)

Note: This example uses tiny values for illustration only — real keys are much larger.

  1. Choose primes p = 61 and q = 53. Then n = p*q = 3233.
  2. Compute φ(n) = (61 − 1)(53 − 1) = 3120.
  3. Choose e = 17 (common small choice). Ensure gcd(17,3120) = 1.
  4. Compute d such that d*e ≡ 1 (mod 3120). Here d = 2753 because 17 * 2753 = 46801 ≡ 1 (mod 3120).

Public key = (n=3233, e=17). Private key = d=2753.

Encrypt message m = 65:

c = m^e mod n = 65^17 mod 3233 = 2790

Decrypt:

m = c^d mod n = 2790^2753 mod 3233 = 65

This demonstrates the mathematical principle. Real RSA uses sources of padding (OAEP) to make encryption semantically secure and avoid deterministic outputs.

10. Best Practices, Checklist, and Defensive Guidance

Summary checklist for safe, modern encryption use:

  • Use well-studied, standardized algorithms (AES, ChaCha20-Poly1305, RSA with OAEP, ECDSA, Ed25519, X25519).
  • Prefer AEAD modes (AES-GCM, AES-CCM, ChaCha20-Poly1305) over separate encrypt-then-MAC unless justified.
  • Ensure unique nonces/IVs as required by the mode (in CTR/GCM a nonce per key is mandatory).
  • Use strong key sizes: e.g., AES-128+ is acceptable, AES-256 for higher margin; 2048+ bit RSA (better 3072+) or ECC 256-bit+ for public-key.
  • Avoid rolling your own crypto. Use vetted libraries and follow their recommended APIs.
  • Use secure random sources from the platform (CSPRNG). Avoid predictable seeds.
  • Defend against side channels by using constant-time operations when handling secret-dependent control flow or branches.
  • Use modern protocol frameworks (e.g., TLS 1.3, Signal protocol) when possible rather than ad-hoc designs.
  • Plan key rotation, revocation, and secure key storage from the beginning.

11. Frequently Asked Questions (FAQ)

Q: Is hashing the same as encryption?

A: No. Hashing is a one-way function that produces a fixed-size digest; it is not reversible. Encryption is reversible given the key (decryption).

Q: Can I use base64 to “encrypt” data?

A: No. Base64 is an encoding, not encryption. It only maps binary to ASCII but provides no confidentiality.

Q: What is the difference between authentication and encryption?

A: Encryption provides secrecy; authentication provides assurance of origin and integrity (who created or sent the message). Digital signatures and MACs are used for authentication.

Q: Are longer keys always better?

A: Longer keys generally increase the work required for brute-force attacks, but beyond certain sizes the performance costs and compatibility issues may not be worth it. Choose recommended sizes from standards and balance performance, interoperability, and long-term security needs.

12. Glossary of Important Terms

  • AEAD: Authenticated Encryption with Associated Data.
  • PKI: Public Key Infrastructure — system of certificates, authorities, and policies for public keys.
  • Nonce: Number used once to ensure uniqueness.
  • KDF: Key Derivation Function — expands or derives keys from shared secrets (e.g., HKDF).
  • MAC: Message Authentication Code.
  • OAEP: Padding scheme for RSA encryption that provides semantic security.
