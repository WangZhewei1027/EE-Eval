Interpolation Search — A comprehensive explanation

Theory, algorithm, step-by-step examples, pseudocode, complexity analysis, practical notes and subtle edge-cases. This page focuses on textual teaching: read to understand how interpolation search works and when to use it.

What is Interpolation Search?

Interpolation Search is a searching algorithm for sorted arrays (or similarly indexed ordered collections) that attempts to estimate where a target value is likely to be inside the search interval using the values at the interval's endpoints. It uses a formula — usually linear interpolation — to compute a probable position ("probe") rather than always probing the middle (as Binary Search does).

Key idea: If keys are uniformly (or approximately uniformly) distributed, the target is likely to be close to the proportional position implied by its value relative to the endpoints. Therefore, interpolation can reduce the number of probes drastically compared with binary halving.

Preconditions and assumptions

  • The data must be sorted in ascending (or descending) order.
  • Interpolation Search performs best when keys are roughly uniformly distributed across the value range.
  • Works with numeric keys (integers, floats) or ordered types that support meaningful arithmetic interpolation. For arbitrary comparable objects (strings etc.) it's not directly applicable.

Interpolation formula (1D linear interpolation)

Let: low and high be the current indices bounding the search interval, A[low] and A[high] be the values at these indices, and x be the target value.

The typical probe position is computed as: 

pos = low + floor( (high - low) * (x - A[low]) / (A[high] - A[low]) )

This is exactly the same formula used to linearly map value x's relative position within the interval [A[low],A[high]] to the index interval [low,high].

High-level algorithm (iterative)

  1. Start with low = 0 and high = n - 1 (the array bounds).
  2. While low <= high and x is within A[low]..A[high] do:
    1. If A[low] == A[high], we cannot interpolate (division by zero). If A[low] == x then return any index between low and high; otherwise x is not present.
    2. Compute pos using the interpolation formula above.
    3. Compare A[pos] with x:
      • If A[pos] == x, return pos.
      • If A[pos] < x, set low = pos + 1 (search right subarray).
      • If A[pos] > x, set high = pos - 1 (search left subarray).
  3. If loop ends without finding x, return "not found".

Pseudocode (iterative)

function interpolationSearch(A, n, x):
    low = 0
    high = n - 1

    while low <= high and x >= A[low] and x <= A[high]:
        if A[high] == A[low]:
            if A[low] == x:
                return low  // any index between low and high
            else:
                break  // not found

        pos = low + floor((high - low) * (x - A[low]) / (A[high] - A[low]))

        if A[pos] == x:
            return pos
        elseif A[pos] < x:
            low = pos + 1
        else:
            high = pos - 1

    return -1  // not found

Step-by-step worked example (textual narrative)

Example array (sorted ascending):

A = [10, 12, 13, 16, 18, 19, 21, 24, 29, 31]

Target: x = 18

Initial state

low = 0, high = 9 (indices), A[low]=10, A[high]=31.

Iteration 1 (interpolate)

Compute:

pos = 0 + floor((9 - 0) * (18 - 10) / (31 - 10))
    = floor(9 * 8 / 21) = floor(72 / 21) = floor(3.428...) = 3

Probe position 3: A[3]=16. Compare 16 vs 18: 16 < 18 so update low = pos + 1 = 4.

Iteration 2

Now low = 4, high = 9. A[4]=18, A[9]=31.

pos = 4 + floor((9 - 4) * (18 - 18) / (31 - 18))
    = 4 + floor(5 * 0 / 13)
    = 4 + 0 = 4

Probe position 4: A[4] == 18 — found. Return index 4.

Observations from the example

  • The first probe jumped to index 3 (not the middle, as Binary Search would), based on value proportionality.
  • After one probe, the algorithm landed very close to the correct index and finished quickly.

Complexity

- Worst-case time complexity: O(n). Example worst cases include highly skewed distributions (all equal elements except one, or exponential distributions) or repeated values that make interpolation ineffective. When interpolation picks positions that do not reduce the search interval sufficiently, the algorithm can degrade to linear.

- Average-case time complexity (for uniformly distributed keys): O(log log n). This is asymptotically better than binary search's O(log n) in the ideal model and is one of the main theoretical attractions of interpolation search.

- Space complexity: O(1) extra space for typical iterative implementations.

Why O(log log n) on average for uniform data? (Intuition)

If keys are uniformly distributed, each interpolation probe is likely to land near the target, shrinking the remaining interval significantly — much more than halving. One can show that the expected number of probes grows like log log n for uniform random keys because the size of the remaining interval drops super-exponentially with high probability.

Practical considerations and caveats

  • Interpolation uses division and multiplication; on modern hardware a division may be more expensive than a comparison. For small arrays or when CPU cycles are costly, binary search can be faster despite higher asymptotic complexity.
  • If A[low] == A[high], interpolation formula divides by zero; handle this explicitly (often by a simple equality check).
  • In the presence of many duplicate values, interpolation often performs poorly.
  • Interpolation assumes the mapping from value to index is approximately linear. If the distribution is skewed (e.g., exponential, clustered), interpolation will give bad probes that cause many iterations.
  • Careful implementation must avoid integer overflow when computing pos. Use larger integer types (e.g., 64-bit) or compute fractions in floating-point if safe.

Variants and improvements

  • Hybrid strategies: Combine interpolation search with binary search. For example, when interpolation chooses a probe but the interval does not shrink fast enough, fall back to binary search. This avoids worst-case behavior and retains benefits on uniform data.
  • Exponential + Interpolation: When the array size is not known or for unbounded search, exponential search can find a bounded range and then interpolation can be applied inside that range.
  • Multi-dimensional interpolation (not covered here): For multidimensional sorted structures, more complex interpolation can be attempted but complexity and cost usually outweigh benefits.

Handling duplicates

If the array contains duplicate values equal to A[low] and/or A[high], and A[low] == A[high], interpolation cannot estimate position. A common strategy:

  1. If A[low] == A[high]:
    • If x == A[low], the algorithm can return any index in [low,high] that matches x (or scan for first/last occurrence if required).
    • If x != A[low], x is not present — stop.
  2. Otherwise proceed normally.

Pseudocode: recursive variant (brief)

function interpSearchRecursive(A, low, high, x):
    if low > high or x < A[low] or x > A[high]:
        return -1

    if A[high] == A[low]:
        if A[low] == x: return low
        else: return -1

    pos = low + floor((high - low) * (x - A[low]) / (A[high] - A[low]))

    if A[pos] == x:
        return pos
    elseif A[pos] < x:
        return interpSearchRecursive(A, pos + 1, high, x)
    else:
        return interpSearchRecursive(A, low, pos - 1, x)

Implementations (examples)

JavaScript (iterative):

function interpolationSearch(A, x) {
  let low = 0, high = A.length - 1;
  while (low <= high && x >= A[low] && x <= A[high]) {
    if (A[high] === A[low]) {
      return A[low] === x ? low : -1;
    }
    // Use Math.floor for integer index
    let pos = low + Math.floor((high - low) * (x - A[low]) / (A[high] - A[low]));
    if (A[pos] === x) return pos;
    if (A[pos] < x) low = pos + 1;
    else high = pos - 1;
  }
  return -1;
}

Python (iterative):

def interpolation_search(A, x):
    low, high = 0, len(A) - 1
    while low <= high and A[low] <= x <= A[high]:
        if A[high] == A[low]:
            return low if A[low] == x else -1
        pos = low + (high - low) * (x - A[low]) // (A[high] - A[low])
        if A[pos] == x:
            return pos
        elif A[pos] < x:
            low = pos + 1
        else:
            high = pos - 1
    return -1

Choosing between Binary and Interpolation Search

AspectBinary SearchInterpolation Search
PreconditionSorted arraySorted array; preferably uniformly distributed numeric keys
Average timeO(log n)O(log log n) for uniform data
Worst-case timeO(log n)O(n)
OperationsComparisons, no divisionComparisons + arithmetic (division/multiplication)
Use whenGeneral purpose; reliable worst-case boundLarge arrays with roughly uniform numeric distribution

Numerical stability and integer overflow

When computing pos, the intermediate multiplication (high - low) * (x - A[low]) can overflow fixed-width integer types. Use a wider type (e.g., 64-bit long) or compute as floating-point and clamp to [low,high]. Alternatively compute as: 

pos = low + (int)((double)(x - A[low]) / (A[high] - A[low]) * (high - low));

When using floating-point arithmetic, be careful to clamp the computed pos to the inclusive index interval [low, high] because rounding or floating error may produce off-by-one invalid indices.

When interpolation search is not appropriate

  • Unsorted data — obviously.
  • Non-numeric or non-arithmetic keys where linear interpolation has no meaning.
  • Very small arrays — overhead of interpolation may not justify use; binary or linear search are often simpler.
  • Data with strong clustering or heavy skew — interpolation may repeatedly choose poor probes leading to O(n) time.

Proof sketch of worst case O(n)

Worst-case arises if interpolation badly estimates positions every time and only eliminates a constant or very small number of elements per probe (for example, if elements increase exponentially, the formula will push probes near the low end for many searches). In such cases each probe might move the bounds by only O(1), leading to linear scans of the array.

Practical notes and benchmarking advice

  • Benchmark on realistic data distributions that match your application. Interpolation shines on near-uniform data ranges.
  • In practice, hybrid methods (interpolation + binary fallback) are robust: attempt interpolation but if the interval doesn't shrink quickly, revert to binary or stop after a small number of failed interpolations.
  • Memory/cache effects: interpolation may jump to indices far from current, so on large arrays cache performance can be worse or better than binary depending on access pattern and CPU. Empirical tests are necessary.

Common interview / exam questions

  • Explain when interpolation search has sub-logarithmic complexity and why.
  • Write the interpolation search algorithm and show its behavior on a given array and key.
  • Describe a witness distribution where interpolation search performs as badly as O(n).
Important: Interpolation Search is a specialized tool — powerful for the right data distributions but unsafe as a universal substitute for binary search. Understand distributional assumptions before selecting it.

Demonstration — search x = 29 in A = [3, 5, 8, 12, 14, 18, 21, 24, 29, 33, 37]
  1. Initial: low=0 (A[0]=3), high=10 (A[10]=37).
  2. Compute pos = 0 + floor( (10 - 0) * (29 - 3) / (37 - 3) ) = floor(10 * 26 / 34) = floor(260/34) = floor(7.647) = 7. Probe A[7] = 24. Since 24 < 29, set low = 8.
