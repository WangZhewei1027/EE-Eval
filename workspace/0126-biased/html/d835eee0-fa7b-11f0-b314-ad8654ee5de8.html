Kruskal's Algorithm

A comprehensive, textual guide: definitions, theory, algorithm, proof ideas, complexity, example, and exercises.

What is the problem?

Given an undirected, connected graph where each edge has a numeric weight (a cost), the Minimum Spanning Tree (MST) problem asks: find a subset of edges that connects all vertices (a spanning tree) and has the minimum possible total weight.

Formal definitions

  • Graph (G): G = (V, E) where V is a set of vertices and E is a set of undirected edges.
  • Weighted graph: each edge e in E has a weight w(e) (real number, often non-negative).
  • Spanning tree: a subgraph that is a tree (connected, acyclic) that includes all vertices of G.
  • Minimum Spanning Tree (MST): among all spanning trees, the one with the smallest sum of edge weights.

Why MSTs matter

MSTs are a fundamental structure in combinatorial optimization and appear in applications such as: designing efficient communication or road networks, clustering (single-linkage hierarchical clustering), approximate solutions to other NP-hard problems, and as a building block in algorithms.

Overview of Kruskal's Algorithm

Kruskal's algorithm is a greedy method to compute an MST. The high-level idea:

  1. Sort all edges in non-decreasing order by weight.
  2. Process edges from smallest to largest. For each edge, add it to the growing forest if it connects two different components (i.e., it does not create a cycle).
  3. Stop when you have added (|V| − 1) edges (the number of edges in any spanning tree) or processed all edges.

Pseudocode (informal)

KRUSKAL(G):
  Input: Graph G = (V, E) with weight function w
  Output: Set T of edges forming an MST (or minimum spanning forest if G is disconnected)

  T ← ∅
  Make-Set for every v ∈ V           // use Disjoint Set Union (Union-Find)
  Sort edges E by non-decreasing weight
  for each edge (u, v) in sorted E:
    if Find(u) ≠ Find(v):           // u and v are in different components
      T ← T ∪ {(u,v)}
      Union(u, v)
  return T

Note: If the graph is disconnected, Kruskal returns a Minimum Spanning Forest — one MST per connected component.

Key data structure: Disjoint Set Union (Union-Find)

Kruskal relies on a structure that maintains a partition of the vertex set into disjoint components and supports:

  • Make-Set(x): create a set containing x.
  • Find(x): return a representative (the set id) for the set containing x.
  • Union(x, y): merge the sets containing x and y.

Optimizations

Two standard and essential optimizations make Union-Find nearly constant time per operation:

  • Union by rank/size: attach the smaller tree under the larger so that height grows slowly.
  • Path compression: during Find, make nodes on the path point directly to the representative.

Time complexity of Union-Find

With both optimizations, m operations on n elements take O(m α(n)) time where α(n) is the inverse Ackermann function, which grows extremely slowly (practically ≤ 5).

Complexity of Kruskal's algorithm

  • Sorting edges: O(E log E) = O(E log V) (since E ≤ V^2).
  • Union-Find operations: O(E α(V)).
  • Total: O(E log E + E α(V)) which is O(E log V) for connected graphs.

When to use Kruskal

Kruskal is often preferred on sparse graphs because sorting E edges is relatively cheap. Prim's algorithm with a binary heap is O(E log V), but in practice choices depend on graph representation and constants.

Quick facts

  • MST edge count: |V| − 1 (for connected graphs).
  • Kruskal works on disconnected graphs, producing a minimum spanning forest.
  • Edge weights can be equal—ties don't break correctness (may change which MST is returned).
  • Union-Find is the core that prevents cycles efficiently.

Common variants

  • Use stable sort if you need deterministic tie-breaking.
  • Parallel/distributed Kruskal variants exist for very large graphs (sorting and merging steps parallelized).

Why Kruskal is correct (sketch of proof)

The correctness of Kruskal follows from two core properties of MSTs: the cut property and the cycle property.

Cut property

For any cut (S, V \ S) of the vertices, the minimum-weight edge crossing that cut belongs to some MST. Intuitively, if you split the graph into two parts, the lightest edge that connects these parts is a safe choice: keeping it never prevents obtaining a minimum spanning tree.

Cycle property

For any cycle in the graph, the heaviest edge in that cycle does not belong to any MST. If you have a cycle, you can remove the heaviest edge without disconnecting the graph and reduce cost or keep it optimal.

Application to Kruskal

Kruskal always picks the next smallest edge that does not create a cycle. When it picks an edge across some cut, that edge is a minimum-weight edge for that cut among edges not yet considered — which is a safe choice by the cut property. Iterating this produces an MST.

A formal proof uses induction: assume the partial forest selected by Kruskal can be extended to an MST; show that adding the next safe edge maintains the invariant.

Graph description

A symmetric, undirected graph with 6 vertices: A, B, C, D, E, F.

Edge set and weights

We will list edges as (u, v) : w.

(A, B): 4   (A, C): 3   (B, C): 2   (B, D): 5   (C, D): 7
(D, E): 6   (C, E): 4   (D, F): 2   (E, F): 1

Sorted edges by weight (non-decreasing)

  1. (E, F): 1
  2. (B, C): 2
  3. (D, F): 2
  4. (A, C): 3
  5. (A, B): 4
  6. (C, E): 4
  7. (B, D): 5
  8. (D, E): 6
  9. (C, D): 7

Step-by-step narrative of Kruskal

  1. Initialize: each vertex is its own component: {A}, {B}, {C}, {D}, {E}, {F}. MST edge set T = ∅.
  2. Consider (E, F) weight 1. E and F are in different components → add (E, F) to T. Components: {E,F}.
  3. Consider (B, C) weight 2. B and C different → add (B, C). Components: {B,C}.
  4. Consider (D, F) weight 2. D in {D}, F in {E,F} → add (D, F). Components: {D,E,F}.
  5. Consider (A, C) weight 3. A in {A}, C in {B,C} → add (A, C). Components: {A,B,C}.
  6. Consider (A, B) weight 4. Now A and B are in the same component ({A,B,C}) → adding (A,B) would create a cycle, so skip it.
  7. Consider (C, E) weight 4. C is in {A,B,C}, E is in {D,E,F} → different components → add (C, E). After adding, components merge: {A,B,C,D,E,F} — all vertices are connected.
  8. Stop: we have |V|−1 = 5 edges in T: edges chosen are (E,F):1, (B,C):2, (D,F):2, (A,C):3, (C,E):4.

Result

MST edges (in the order added): (E,F) 1, (B,C) 2, (D,F) 2, (A,C) 3, (C,E) 4.

Total MST weight = 1 + 2 + 2 + 3 + 4 = 12.

Note: Some other MSTs may exist if edges have equal weights; here the edges chosen are consistent with the sorted order and the union-find choices.

Implementation notes and practical tips

  • Store edges as a list/array of triples (u, v, w) and sort by w. Use stable sort if deterministic tie-breaking is required.
  • Implement Union-Find carefully: iterative path compression is simple and efficient.
  • If memory is tight and graph is huge, consider external sorting or streaming strategies; Kruskal requires sorting edges (or at least producing them in sorted order).
  • For dense graphs (E ~ V^2), Prim's algorithm with a Fibonacci heap (or optimized implementation) may be asymptotically better.
  • Negative weights are allowed; Kruskal still works (greedy picks smallest weights first), but be careful of overflow if weights are large magnitude.

Comparisons and variants

Kruskal vs Prim

Both compute an MST but take different approaches:

  • Kruskal sorts edges globally and adds smallest edges avoiding cycles — good for sparse graphs and when edge list is readily available.
  • Prim grows a single tree starting from a root, adding minimal edge crossing the cut between tree and non-tree — efficient with adjacency structures and appropriate priority queue.

Other variants

  • Reverse-Delete: Sort edges by decreasing weight and delete an edge if its deletion does not disconnect the graph — less common in practice but conceptually dual to Kruskal.
  • Distributed/parallel Kruskal: partition edges, compute partial MSTs, and merge — used for very large datasets.

Applications of MST

  • Designing least-cost network backbones (telecom, power distribution).
  • Approximation for NP-hard problems (e.g., approximating TSP using MST).
  • Clustering (single linkage): MST edges can be removed to create clusters.
  • Image processing and computer vision: segmentation, minimal connections.

Common mistakes and pitfalls

  • Attempting to detect cycles by DFS/BFS on the fly can be inefficient — use Union-Find for O(α(n)) per operation.
  • For disconnected graphs, expecting a single MST is incorrect — you get a minimum spanning forest.
  • Not handling equal-weight edges deterministically when required — can lead to different valid MSTs across implementations.
  • Mixing up indices or storing duplicate edges inconsistently (for undirected graphs store each edge once).

Exercises (with brief solutions)

Exercise 1

Given a connected graph with |V| = 7 and |E| = 12, what is the number of edges in any spanning tree? What is the size of a minimum spanning forest if the graph were split into 3 connected components with 7 vertices total?

Answer: Any spanning tree on 7 vertices has 6 edges. If the graph is disconnected into 3 components with sizes n1, n2, n3 (n1+n2+n3=7), a minimum spanning forest has (n1−1)+(n2−1)+(n3−1) = 7 − 3 = 4 edges.

Exercise 2

Explain in one sentence why adding the minimum-weight edge crossing any cut is a safe move toward an MST.

Answer: Because by the cut property, any MST must include some minimum-weight edge crossing that cut; selecting it cannot prevent reaching an overall minimum.
