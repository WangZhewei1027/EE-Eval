Amortized Analysis

Introduction

In the study of algorithms and data structures, amortized analysis is a powerful technique used to understand the average running time of an operation over a sequence of operations, even if some individual operations might be expensive. It helps provide a more realistic performance guarantee than simple worst-case analysis.

Unlike average-case analysis which depends on probabilistic assumptions about inputs, amortized analysis gives a deterministic guarantee on the average cost per operation taken over a worst-case sequence.

What is Amortized Analysis?

Amortized analysis calculates the average time per operation over a worst-case sequence of operations, spread evenly across all the operations. This does not mean every operation is cheap, but rather that the expensive operations are infrequent enough that their cost is "amortized" and averaged out in the long run.

The key idea is:

Even if some operations are costly, when performed over a long sequence of many operations, the average cost per operation is small.

This approach is especially useful when analyzing data structures where occasional expensive operations are performed to maintain efficient overall performance.

Why Do We Need Amortized Analysis?

Some algorithms or data structures have operations that vary drastically in cost. For example:

  • Inserting an element into a dynamic array is usually O(1), but resizing the array requires O(n) time.
  • A binary counter increment can have multiple bit flips occasionally but often flips only one bit.

Worst-case analysis suggests these operations might be expensive. But in reality, the expensive steps happen rarely. Amortized analysis captures this "rare but costly" phenomenon by averaging the cost over multiple operations.

Basic Methods of Amortized Analysis

There are three popular methods to perform amortized analysis:

  1. Aggregate Method: Compute the total cost of a sequence of n operations and then take the average cost per operation.
  2. Accounting Method: Assign an artificial "charge" or "cost" to each operation (amortized cost) that may differ from actual cost, saving surplus charges as "credits" to pay for expensive operations later.
  3. Potential Method: Define a potential function that maps the data structure's state to a numeric potential energy, representing stored work that can pay for future operations.

Each method leads to the same asymptotic amortized costs but differs in style and intuition.

Example 1: Binary Counter

Consider a simple binary counter represented as an array of bits. We want to increment the counter repeatedly.

Operations

The increment() operation flips some bits:

  • Bit 0 toggles every time (0 → 1 or 1 → 0).
  • If bit 0 flips from 1 → 0, bit 1 is toggled next, and so on.
  • The number of bit flips per increment varies: sometimes 1 bit flips, sometimes many.

Worst-case Cost

In the worst case, incrementing flips all bits (e.g., 1111 → 0000). For an n-bit counter, worst-case cost is O(n).

Amortized Analysis (Aggregate Method)

Let’s analyze the amortized cost over k increments:

  • Each bit toggles from 0 → 1 (and vice versa) several times over k increments.
  • Specifically, bit i toggles once every 2^i increments.

Total bit flips over k increments: 

≤ k × (1/2 + 1/4 + 1/8 + ... ) < 2k

Thus, total flips are at most 2k. The amortized cost per increment is: 

total flips / k ≤ 2

This means on average, each increment flips at most 2 bits → amortized cost is O(1).

Summary

Although some increments are costly, the average number of bits flipped per increment is constant, meaning the amortized cost is O(1).

Example 2: Dynamic Array (Array List) Expansion

When using a dynamic array (e.g., array list) which doubles its size when full, insertions usually take O(1) time, but occasionally take O(n) time when resizing.

How does resizing work?

  • Array initially has capacity n.
  • When full (contains n elements), inserting another element triggers resizing:
    • Create a new array of size 2n (doubling capacity)
    • Copy all n elements over (cost O(n))
    • Add the new element (cost O(1))

Worst-case cost of insert

The insert that triggers resizing costs O(n)—which seems expensive.

Amortized Cost Calculation (Aggregate Method)

Consider inserting k elements starting from an empty array:

  • When array size doubles to 2^i, previous 2^{i-1} elements were copied.
  • Total cost for all copies during all resizes up to size k is: 
      ∑i=0^{log k} 2^i = 2^{log k+1} - 1 = 2k - 1 = O(k)

In addition to copy costs, every insertion costs O(1).

Total cost for k insertions is therefore O(k) plus copying overhead O(k) → O(k) total.

Amortized cost per insertion = total cost divided by k = O(1).

Accounting Method Explanation

Alternatively, charge each insertion an amortized cost of 3 units:

  • Use 1 unit to pay actual insertion.
  • Save 2 units as credit to pay for future copying.

When resizing occurs, the credits accumulated from previous insertions pay for the copying, keeping the amortized cost constant.

Summary of Amortized Analysis

  • Amortized cost describes the average cost per operation over a sequence, providing a worst-case average.
  • It is particularly valuable for data structures with occasional expensive operations.
  • The three methods — aggregate, accounting, and potential — provide frameworks for rigorous analysis.
  • Many common data structures such as binary counters, stacks with multi-pop, and dynamic arrays rely on amortized analysis for performance guarantees.

In short:

“Amortized analysis shows that while some operations may be expensive, their cost spread over many cheap operations is small.”
