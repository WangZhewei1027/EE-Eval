Asymmetric Cryptography — A Comprehensive Educational Guide

Detailed conceptual explanations, mathematical foundations, algorithm descriptions, example walkthroughs, security considerations, and best practices for public-key (asymmetric) cryptography.

Overview and Key Concepts

Asymmetric cryptography (also called public-key cryptography) is a branch of cryptography where cryptographic operations use a pair of related, but distinct, keys: a public key (which may be shared openly) and a private key (which must be kept secret). The two keys are mathematically linked: what one key encrypts (or signs), only the other can decrypt (or verify), depending on the operation.

Primary goals

  • Confidentiality: Using a recipient's public key to encrypt data so only the recipient (holding the private key) can decrypt.
  • Authentication: Proving the origin of a message (digital signatures).
  • Integrity: Ensuring data hasn't been altered (signatures + hashing).
  • Non-repudiation: Preventing a sender from denying they sent a signed message.
  • Key exchange and agreement: Securely establishing shared secrets between parties (e.g., Diffie–Hellman).

Why asymmetric vs symmetric?

Symmetric cryptography uses the same key for encryption and decryption. Asymmetric cryptography solves several key distribution problems: you can publish a public key widely while keeping the private key secret, enabling secure communication without pre-sharing secrets. Practical systems typically combine both: asymmetric cryptography for key exchange and authentication, and symmetric cryptography for bulk encryption (hybrid encryption).

Mathematical Foundations (Intuition and Formal Notes)

This section gives the essential mathematics and conceptual building blocks that underlie popular asymmetric algorithms.

1. Modular arithmetic

Operations performed modulo an integer n (written a mod n) are fundamental. For example, exponentiation modulo n (a^k mod n) is the core operation in many public-key systems.

2. Prime numbers and factorization

Some systems rely on the difficulty of factoring a large composite number into primes (for example RSA). If n = p * q where p and q are large primes, recovering p and q from n is computationally hard for appropriately chosen sizes.

3. Euler's totient and multiplicative order

Euler's totient function φ(n) counts integers less than n that are coprime to n. In RSA, φ(n) = (p-1)(q-1) for n = p * q (p,q primes). Euler's theorem generalizes Fermat's little theorem for modulus not necessarily prime and is used to justify RSA's correctness.

4. Discrete logarithm problem (DLP)

The DLP in a chosen group (integers modulo a prime, or points on an elliptic curve) is: given g and h, find x such that g^x = h (in the group). This problem's assumed hardness underpins Diffie–Hellman, DSA, ElGamal, and many elliptic-curve schemes.

5. Elliptic curves

Elliptic Curve Cryptography (ECC) uses algebraic structures of elliptic curves over finite fields. ECC offers comparable security with smaller key sizes compared to multiplicative-group (integer mod p) systems, because the elliptic curve discrete log problem (ECDLP) is harder per bit.

Core Algorithms: How They Work (Step-by-step)

RSA (Rivest–Shamir–Adleman)

RSA is the classic public-key algorithm for encryption and signatures based on integer factorization hardness. Below is a step-by-step outline for RSA key generation, encryption, and decryption:

RSA key generation (conceptual steps)

  1. Choose two large distinct primes p and q (kept secret).
  2. Compute n = p × q (public modulus).
  3. Compute φ(n) = (p − 1) × (q − 1) (kept secret).
  4. Choose an integer e such that 1 < e < φ(n) and gcd(e, φ(n)) = 1. This is the public exponent (often 65537).
  5. Compute d, the multiplicative inverse of e modulo φ(n). That is, find d such that e × d ≡ 1 (mod φ(n)). d is the private exponent.
  6. Public key = (n, e). Private key = (n, d) and optionally (p,q) for faster decryption.

RSA encryption and decryption

  1. To encrypt a numeric message m (0 ≤ m < n): compute c = m^e mod n.
  2. To decrypt ciphertext c: compute m = c^d mod n.
  3. Correctness follows from Euler's theorem: m^{ed} ≡ m (mod n) under the key relationships.

Important: Real RSA rarely encrypts raw messages directly. It uses padding schemes (like OAEP) and hybrid constructions to be secure against modern attacks.

Diffie–Hellman key exchange (classic DH)

DH allows two parties to establish a shared secret over an insecure channel.

  1. Public parameters: a large prime p and a generator g of a subgroup (both public).
  2. A chooses secret a, computes A = g^a mod p and sends A to B.
  3. B chooses secret b, computes B = g^b mod p and sends B to A.
  4. A computes s = B^a mod p = g^{ba} mod p. B computes s = A^b mod p = g^{ab} mod p. Both obtain same s (shared secret).

Elliptic-Curve Diffie–Hellman (ECDH)

ECDH is the elliptic-curve analog of DH: replace exponentiation in multiplicative groups with scalar multiplication of elliptic-curve points. The shared secret is the x-coordinate (or a KDF applied to it) of the resulting point.

Digital signatures (RSA, DSA, ECDSA)

Signatures authenticate origin and integrity. A typical flow:

  1. The signer hashes the message to a fixed-size digest using a cryptographic hash (e.g., SHA-256).
  2. Using the private key, the signer computes a signature over the digest (algorithm-specific).
  3. The verifier uses the signer's public key to check the signature; success means the signature is valid for that digest and public key.

ElGamal

ElGamal works in groups where DLP is hard. It provides both encryption and signature variants. Encryption produces two values (related to ephemeral randomness) and requires randomness per-message to be secure.

Hybrid encryption

Because asymmetric operations are computationally slower, most practical systems use hybrid encryption:

  1. Generate a random symmetric session key (e.g., AES key).
  2. Encrypt bulk data with the symmetric key.
  3. Encrypt the symmetric key with the recipient's public key (asymmetric).
  4. Send the encrypted symmetric key + ciphertext together.

Algorithm Features, Strengths, Weaknesses, and Choices

RSA

  • Security assumption: Integer factorization is hard for chosen key sizes.
  • Key sizes typically: 2048-bit minimum for new deployments (as of the 2020s), 3072 or 4096 for longer term security.
  • Signatures and encryption possible; must use padding (PKCS#1 v1.5, OAEP for encryption; PSS for signatures).
  • Faster verification (with public exponent small) and slower signing if d is large (can use CRT optimizations).

Diffie–Hellman & DSA

  • Security based on discrete log in multiplicative groups modulo primes.
  • Requires careful parameter selection (group sizes, safe primes) and ephemeral randomness for certain signatures.
  • DH parameter reuse can be dangerous if groups are weak; ephemeral Diffie–Hellman (DHE) provides forward secrecy.

Elliptic Curve Cryptography (ECC)

  • Offers similar security with smaller key sizes (e.g., 256-bit ECC ≈ 3072-bit RSA).
  • Popular curves: secp256r1 (P-256), secp384r1, Curve25519 (X25519 for ECDH, Ed25519 for signatures).
  • Resistance against quantum attacks is similar to classical discrete-log systems: both are vulnerable to quantum algorithms like Shor's algorithm.

Quantum considerations

Asymmetric algorithms based on factoring or discrete logarithm (including ECC) are vulnerable to quantum attacks. Post-quantum cryptography (lattice-based, hash-based, code-based, multivariate) is an active area for future-proofing.

Security Properties and Real-World Considerations

Padding and chosen-ciphertext security

Naive use of RSA (raw modular exponentiation) is insecure. Padding schemes (OAEP for encryption, PSS for signatures) are required to achieve provable security against adaptive chosen-ciphertext or chosen-message attacks.

Key sizes and entropy

Security depends on key length and the randomness used during key generation. Use secure random number generators and recommended key sizes. Smaller keys (e.g., 1024-bit RSA) are no longer considered secure.

Side-channel attacks

Timing, power analysis, cache attacks, and fault injection can leak keys even if algorithms are mathematically secure. Implementations must use constant-time algorithms and hardware protections as appropriate.

Certificate Authorities and PKI

In many systems, public keys are distributed and vouched for by Certificate Authorities (CAs). The security of the PKI depends on the trustworthiness of CAs, certificate validation, and revocation mechanisms.

Key management and lifecycle

  • Key generation: use secure hardware or software, with audited randomness sources.
  • Key storage: protect private keys with hardware security modules (HSMs) or secure enclaves if possible.
  • Key rotation: plan for periodic key renewal.
  • Revocation: use certificate revocation lists (CRLs), OCSP, or modern approaches like short-lived certificates.

Step-by-step Walkthroughs and Examples (Textual)

RSA numeric example (classic educational toy)

Below is an illustrative, intentionally small RSA example often used to learn the steps. Do not use these numbers for real security.

1) Choose primes:
   p = 61
   q = 53

2) Compute n = p * q:
   n = 61 * 53 = 3233

3) Compute φ(n) = (p-1)*(q-1):
   φ(n) = 60 * 52 = 3120

4) Choose e such that 1 < e < φ(n) and gcd(e, φ(n))=1:
   e = 17  (common choice for small examples)

5) Compute d, the modular inverse of e modulo φ(n):
   d = 2753  (because 17 * 2753 ≡ 1 (mod 3120))

Public key: (n=3233, e=17)
Private key: (n=3233, d=2753)

Encryption:
   To encrypt message m = 65:
   c = m^e mod n = 65^17 mod 3233 = 2790

Decryption:
   m = c^d mod n = 2790^2753 mod 3233 = 65
          (recovers the original message)

This illustrates the core exponentiation operations and modular arithmetic used in RSA.

Diffie–Hellman example (small, illustrative)

Public parameters:
  p = 23 (prime)
  g = 5 (a generator modulo 23)

Alice picks a = 6, computes A = g^a mod p = 5^6 mod 23 = 8
Bob picks b = 15, computes B = 5^15 mod 23 = 19

Shared secret:
  Alice computes s = B^a mod p = 19^6 mod 23 = 2
  Bob computes s = A^b mod p = 8^15 mod 23 = 2

Both obtain s = 2 (shared secret).

Again: these tiny numbers are only educational. Real parameters are hundreds or thousands of bits.

Common Attacks, Mitigations, and Best Practices

Attacks

  • Mathematical attacks: factoring algorithms (for RSA), index calculus and Pollard's algorithms (for DLP).
  • Implementation attacks: timing, cache, power analysis, fault injection.
  • Protocol-based attacks: downgrade attacks, man-in-the-middle, replay attacks.
  • Cryptographic misuse: reusing ephemeral nonces in DSA/ElGamal/EC-based signatures can leak keys.

Mitigations and best practices

  • Use recommended algorithms and parameters from trusted standards (NIST, IETF, IANA recommendations).
  • Use authenticated encryption and combine signatures with hashing properly.
  • Use OAEP with RSA for encryption and PSS for signatures.
  • Prefer modern curves and libraries (e.g., Curve25519/Ed25519) for new systems.
  • Implement constant-time primitives and use hardware protections where available.
  • Use TLS with ephemeral key exchange (DHE/ECDHE) for forward secrecy.
  • Rotate keys, monitor for compromise, and have clear revocation practices.

Glossary

Public key
A key that can be distributed openly and is used for encryption or signature verification.
Private key
A secret key that must be kept confidential; used for decryption or signature creation.
Trapdoor function
A function easy to compute one way, but hard to invert unless you have special information (the "trapdoor"). A core concept behind RSA.
Padding
Structured randomness added to messages before asymmetric encryption to prevent attacks that exploit deterministic algebraic structure.
Forward secrecy
Property where compromising long-term keys does not compromise past session keys (achieved using ephemeral DH/ECDH).
