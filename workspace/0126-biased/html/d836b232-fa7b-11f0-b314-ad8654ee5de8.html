Overview and Motivation

Branch and Bound (B&B) is a general algorithmic framework for solving discrete and combinatorial optimization problems, including integer programming, mixed integer programming (MIP), and many NP-hard problems such as the traveling salesman problem (TSP) and 0–1 knapsack.

At its heart, B&B systematically explores a tree of subproblems (nodes). Each node represents a restricted version of the original problem. The algorithm maintains global best-known solutions and uses bounds computed from relaxations to eliminate (prune) subtrees that cannot contain better solutions. This trade-off between exhaustive search and intelligent pruning is the defining characteristic of B&B.

Core Idea: Branching, Bounding, Pruning

1. Branching

Branching splits a problem into two (or more) smaller subproblems by imposing additional constraints. For integer problems, typical branching is on a variable x_i that is fractional in a relaxation: create one child with x_i ≤ floor(value) and another with x_i ≥ ceil(value). For combinatorial problems, branching may assign a decision (include/exclude an edge or item).

2. Bounding

For each node/subproblem, compute a bound (a guarantee) on the best possible objective value that can be obtained within that subtree. For maximum problems, we compute an upper bound; for minimization, a lower bound. Common bounds come from relaxations: drop integrality to solve an LP relaxation, or compute heuristic bounds like fractional knapsack.

3. Pruning (Fathoming)

If the bound for a node indicates it cannot lead to an improvement over the best known feasible solution, the node (and its subtree) can be discarded (pruned). If a node's subproblem yields a feasible integer solution that improves the incumbent, update the incumbent. If the node's subproblem is infeasible, it is also discarded.

Status types for nodes

  • Active (live) node: generated but not yet processed
  • Processed node: its children generated, or pruned / solved
  • Pruned node: removed by bounding (cannot improve incumbent)
  • Fathomed node: either infeasible, solved to integer optimality, or pruned

Canonical Algorithm and Pseudocode

Below is a high-level textual pseudocode for Branch and Bound for a maximization problem. This description intentionally emphasizes clarity over implementation minutiae. 

Input: An optimization problem P (maximize f(x) subject to constraints)
Output: best-known integer feasible solution incumbent and its value UB_incumbent

1. Initialize:
   - Create root node representing original problem P.
   - incumbent <- "no solution" (or a feasible solution from a heuristic)
   - UB_incumbent <- value(incumbent) if exists, else -∞ (for maximization)
   - NodeList <- {root} (a list/queue/priority structure of active nodes)

2. While NodeList not empty:
   a. Select a node N from NodeList (selection policy: depth-first, best-bound, etc.)
   b. Compute a bound b(N) for N by solving a relaxation
   c. If b(N) ≤ UB_incumbent then
        prune N (nothing to do)
      else
        try to obtain an integer feasible solution for N (through direct check or heuristic)
        if a feasible integer solution s is found and value(s) > UB_incumbent:
           incumbent <- s; UB_incumbent <- value(s)
        if the relaxation solution is integer (or N is solved exactly) then
           fathom N (no children)
        else
           choose a branching rule to create child nodes N1, N2, ...
           add children to NodeList

3. Terminate: return incumbent and UB_incumbent

Notes: - For minimization, reverse inequality directions. - Node selection and branching policy dramatically affect performance. - Bound tightness determines pruning effectiveness.

Designing Bounds and Relaxations

The quality and cost of computing bounds are central. A tighter bound reduces the search tree size but may be expensive to compute. Cheap bounds are fast but may yield less pruning.

Common bounding techniques

  • LP relaxation: Relax integrality constraints; solve the resulting linear program. Widely used in integer programming; gives strong bounds and can provide branching guidance (fractional variables).
  • Greedy bounds: For knapsack, fractional knapsack (allow item fractions) gives an upper bound for 0–1 knapsack.
  • Constraint propagation: Use problem structure to deduce variable fixings and simple bounds.
  • Dual bounds: In minimization, compute feasible dual solutions to produce lower bounds.
  • Relax-and-round heuristics: Use relaxation solutions rounded to an integer feasible solution to improve incumbent quickly; not a bound but useful.

Trade-offs

- Compute strong bounds at the root and gradually cheaper bounds deeper in the tree. - Use incremental or warm-start LP solves: when children are created from parent LP, reuse the parent's basis to speed up solving.

Branching Strategies and Node Selection

Branching rules

  • Variable branching: Pick a fractional variable x_i and branch x_i=0 and x_i=1 (for binary variables), or floor/ceil for general integers.
  • Strong branching: Tentatively branch on several candidates and evaluate their bounds to choose the best branching variable. Expensive but often reduces tree size.
  • Constraint branching: Branch on assignments or on more complex constraints, e.g., choose to include/exclude an edge in TSP.
  • Inference-based branching: Choose branches that allow strong constraint propagation (fix many variables quickly).

Node selection strategies

  • Depth-first search (DFS): Follows a path to a leaf quickly, often finds feasible incumbents early (low memory), good for finding heuristics.
  • Breadth-first search (BFS): Explores tree level by level, not common due to high memory.
  • Best-bound (best-first): Select node with best bound (promising node) — helps reduce search tree by pruning weaker branches earlier.
  • Hybrid strategies: Start DFS to find an incumbent, then switch to best-bound to prune aggressively.

Practical solvers (CPLEX, Gurobi, SCIP) use sophisticated combinations: heuristics, strong branching on high-impact variables, pseudo-costs (learned estimates of branching effect), and node selection tuned for performance.

Step-by-Step Example: 0–1 Knapsack (Textual Walkthrough)

Problem statement (0–1 knapsack): Given n items with weights w_i and values v_i and capacity W, choose a subset maximizing total value subject to total weight ≤ W. Each item either taken (1) or not (0).

Instance (small for demonstration)

  • Items: A( weight 2, value 6 ), B( weight 2, value 10 ), C( weight 3, value 12 )
  • Capacity W = 5

Root node: relaxation (fractional knapsack)

Compute fractional knapsack upper bound by ordering items by value/weight ratio: - A: 6/2 = 3.0 - B: 10/2 = 5.0 - C: 12/3 = 4.0 Sort: B, C, A. Fill until capacity: - Take B fully (weight 2, value 10). Remaining capacity 3. - Take C fully (weight 3, value 12). Remaining capacity 0. Total fractional value = 22. Since fractional solution is integer here (we took whole items), UB_root = 22 and feasible integer solution {B,C} with weight 5 exists.

In this tiny instance, the relaxation yields an optimal integer solution; B&B would terminate immediately with {B,C} value 22. For demonstration we will imagine a slightly different fractional case where fractional allocation is needed.

Illustrative fractional case (slightly altered values)

Suppose items are: A(2,6), B(2,10), C(3,7). Ratios: A=3.0, B=5.0, C≈2.333. Fractional knapsack: - Take B (v=10, w=2), remain 3. - Take A (v=6, w=2), remain 1. - Take 1/3 of C (v ≈ 7*(1/3)≈2.333), total UB ≈ 18.333. The fractional solution is not integer, so UB_root≈18.333.

Branching and bounding steps (textual)

1. Root: UB ≈ 18.333; no incumbent known yet. Choose variable to branch on: pick C because it was fractionally chosen in relaxation (value not integral).

2. Branch 1 (C = 0): Solve relaxation on remaining items (A,B only). Fractional knapsack with A and B fully fits weight 4 ≤ 5, total value 16. UB_C0 = 16. This gives an incumbent by rounding (A and B both integer) → feasible solution value 16. Update incumbent: 16. Because UB_C0 (16) equals the best solution under C=0, the subtree is fathomed.

3. Branch 2 (C = 1): Fix C selected, remaining capacity = 2. Relaxation on remaining items A and B with capacity 2: choose the best by ratio -> B (value 10, weight 2) fits exactly. UB_C1 = value(C)+value(B) = 7+10 = 17. A feasible integer solution (C and B) exists with value 17; update incumbent to 17. Node is fathomed because relaxation is integral.

4. Compare UB_root (≈18.333), but both children produced incumbents 16 and 17. The root bound was higher than best-found integer 17, but since children are fathomed and no other branching is needed, final optimal integer solution is value 17 (C and B), which is optimal.

This textual example shows how bounds guide pruning and how early heuristics (simple rounding) can find incumbents quickly. With larger instances, many more nodes and branching levels are involved.

Key teaching points from the example
  • Relaxations provide upper bounds that may or may not be integer.
  • Branch on fractional variables to force integrality.
  • A good incumbent reduces search by pruning nodes whose bounds are worse.
  • Efficient B&B balances time spent computing bounds versus time exploring nodes.

Extensions and Variants

Branch-and-Cut

Branch-and-cut integrates cutting planes (valid inequalities) into B&B. At each node, before branching, the solver may add cuts to the relaxation to tighten bounds and potentially produce integer solutions earlier. Widely used for MIP.

Branch-and-Price

For problems with an exponential number of variables (column generation), branch-and-price combines B&B with column generation. At each node, a master problem (restricted LP) is solved and columns are generated via a pricing subproblem.

Hybrid approaches

Modern solvers use hybrids: pseudo-costs (learned impact of branching decisions), heuristics (feasibility pumps, diving heuristics), presolve (simplify problem before search), symmetry breaking, and parallel B&B across processors.

Complexity, Performance, and Practical Tips

Complexity

Worst-case time complexity is exponential in problem size for NP-hard problems. However, B&B often performs well in practice due to effective bounding and heuristics, which prune large portions of the search space.

Memory and implementation considerations

  • Store nodes with only information necessary to re-create relaxations (e.g., variable bounds) to reduce memory cost.
  • Use depth-first when memory is limited; best-first often requires large memory to maintain open nodes sorted by bound.
  • Warm-start LP solvers with the parent basis for faster child solves.
  • Implement reliable floating-point tolerances and deal carefully with rounding when checking integrality and feasibility.

Heuristics to improve performance

  • Start with a strong incumbent found by heuristics to increase pruning early.
  • Use strong branching sparingly (only at top levels) to choose more effective branching variables.
  • Use cut generation early to tighten the root relaxation and reduce tree size.
  • Track pseudo-costs to approximate the benefit of branching on variables without full trial branching.

Parallelization

Parallel B&B distributes nodes across workers. Challenges include load balancing, sharing incumbents and bounds efficiently, and avoiding redundant work. Sophisticated solvers use work stealing and asynchronous incumbent updates.
