Breadth-First Search (BFS)

A comprehensive textual guide to the algorithm, theory, examples, proofs, and practical notes. Minimal interactivity — one simple demo included.

What is BFS? (High-level)

Breadth-First Search (BFS) is a fundamental graph traversal algorithm that explores a graph layer by layer. Starting from a source vertex, it first visits all nodes at distance 1, then all nodes at distance 2, and so on. BFS uses a queue to ensure that nodes are processed in non-decreasing order of distance from the source.

Core idea and intuition

Imagine dropping a pebble at the source vertex. Ripples expand outward uniformly; nodes reached by the first ripple are at distance 1, those by the next ripple are at distance 2. BFS simulates these ripples: it discovers nodes by increasing distance and records predecessors to reconstruct shortest paths in an unweighted graph.

Definitions and model

  • Graph G = (V, E): set of vertices V and edges E. Edges can be directed or undirected.
  • Unweighted graph: every edge has equal cost (implicitly 1). BFS finds shortest paths measured in number of edges.
  • Distance d[u]: the minimum number of edges between source s and vertex u. BFS computes d[u] for every reachable u.
  • Predecessor p[u]: a parent pointer used to reconstruct a shortest path from s to u.

Data structures used

  • Queue (FIFO): holds vertices discovered but not fully processed. This is the heart of BFS.
  • Visited/Color marker: to avoid re-processing nodes (commonly boolean visited[], or color set {white, gray, black}).
  • Distance array: distance[] initialized to infinity (or -1) except distance[source] = 0.
  • Predecessor array: predecessor[] initially null (or -1).

Classic pseudocode (imperative description)

BFS(G, s):
  for each vertex v in G:
    distance[v] = INFINITY
    predecessor[v] = NIL
    visited[v] = false

  distance[s] = 0
  visited[s] = true
  queue Q = empty
  enqueue(Q, s)

  while Q not empty:
    u = dequeue(Q)
    for each neighbor v of u:
      if not visited[v]:
        visited[v] = true
        distance[v] = distance[u] + 1
        predecessor[v] = u
        enqueue(Q, v)
    // u is now fully explored

Step-by-step textual walkthrough (small example)

Consider an undirected graph with nodes: A, B, C, D, E, F and edges:

A: B, C
B: A, D, E
C: A, F
D: B
E: B, F
F: C, E

Start BFS from source A.

Initial state:

  • Queue: [A]
  • Visited: {A}
  • Distance(A)=0
  • Order visited (so far): [ A ]

Iteration 1: Dequeue A, examine neighbors B and C.

  • Visit B: mark visited, distance(B)=1, predecessor(B)=A, enqueue B.
  • Visit C: mark visited, distance(C)=1, predecessor(C)=A, enqueue C.
  • Queue after processing A: [B, C]
  • Order visited: [A, B, C]

Iteration 2: Dequeue B, examine neighbors A, D, E.

  • A already visited → skip.
  • Visit D: visited, distance(D)=2, pred(D)=B, enqueue D.
  • Visit E: visited, distance(E)=2, pred(E)=B, enqueue E.
  • Queue: [C, D, E]
  • Order visited: [A, B, C, D, E]

Iteration 3: Dequeue C, examine neighbors A, F.

  • A visited → skip.
  • Visit F: visited, distance(F)=2, pred(F)=C, enqueue F.
  • Queue: [D, E, F]
  • Order visited: [A, B, C, D, E, F]

Continue: Dequeue D, E, F; each neighbor is already visited — no new enqueues. BFS terminates.

Final BFS tree (parent pointers):

A
├─ B (parent A)
│  ├─ D (parent B)
│  └─ E (parent B)
└─ C (parent A)
   └─ F (parent C)

Distances from A: A=0, B=1, C=1, D=2, E=2, F=2

Correctness (why BFS finds shortest paths in unweighted graphs)

Two important invariants of BFS:

  • Vertices are discovered (become visited and enqueued) in non-decreasing order of their distance from the source.
  • When a vertex v is dequeued (removed from the front of the queue), distance[v] is final (it equals the shortest path length from the source to v).

Sketch of a formal proof: Assume for contradiction that when BFS dequeues a vertex u, distance[u] is not the shortest possible. Then there exists a shorter path from s to u of length k < distance[u]. Consider the first vertex x on that path that is dequeued out of the queue before u; the BFS queue order ensures nodes closer to s are dequeued earlier, which contradicts the assumption. The queue ensures layer-by-layer exploration, so all vertices at distance < distance[u] are discovered and dequeued before u.

Time and space complexity

  • Time (adjacency list): O(|V| + |E|). Each vertex is enqueued and dequeued at most once; each edge is examined once (undirected counted twice in adjacency lists).
  • Time (adjacency matrix): O(|V|^2) because scanning neighbors costs O(|V|) per vertex.
  • Space: O(|V|) for the queue, visited, distance, and predecessor arrays; plus adjacency representation size.

When to use BFS (applications)

  • Shortest path in an unweighted graph.
  • Compute connected components (run BFS from unvisited nodes).
  • Check bipartiteness: try to 2-color while BFSing; if an edge connects same-colored vertices → not bipartite.
  • Layered problems (minimum number of moves, levels, shortest reach in grids).
  • Multi-source BFS: initialize queue with multiple sources at distance 0 to compute shortest distance to the nearest source.
  • Web crawlers (breadth-first crawl), level-order traversal of trees (special case of graphs).

Common variations and extensions

  • Bidirectional BFS: for shortest path between two vertices, run BFS from both ends and meet in the middle—reduces search work on many graphs.
  • 0-1 BFS: for graphs with edge weights 0 or 1, use a deque to achieve shortest paths in O(|V|+|E|).
  • Multi-source BFS: push multiple starting vertices into the initial queue.
  • Layer BFS: maintain list of nodes by level (level-order arrays) for algorithms that process level-by-level.

Reconstructing paths

The predecessor array records, for each discovered vertex v, the vertex pred[v] from which v was first discovered. To reconstruct the shortest path from source s to target t:

  1. If pred[t] is NIL, no path exists.
  2. Otherwise, follow pred[t], pred[pred[t]], ... back to s, collecting vertices. Reverse the collected sequence to get the path s → ... → t.

Implementation notes (pseudocode -> real code)

Common pitfalls and tips:

  • Mark vertices as visited when they are enqueued, not when dequeued. This prevents multiple enqueues of the same vertex in graphs with multiple paths.
  • For huge graphs, prefer adjacency lists to save memory.
  • If you need distances only up to a limit (k), you can stop exploring beyond that level.
  • Use integers or sentinel values (e.g., -1 or INF) for distance initialization.

Code examples (concise)

Python (adjacency list):

from collections import deque

def bfs(adj, s):
    n = len(adj)
    dist = [-1]*n
    pred = [-1]*n
    q = deque()
    dist[s] = 0
    q.append(s)
    while q:
        u = q.popleft()
        for v in adj[u]:
            if dist[v] == -1:
                dist[v] = dist[u] + 1
                pred[v] = u
                q.append(v)
    return dist, pred

JavaScript (adjacency list):

function bfs(adj, s) {
  const n = adj.length;
  const dist = Array(n).fill(-1);
  const pred = Array(n).fill(null);
  const q = [];
  q.push(s);
  dist[s] = 0;
  while (q.length) {
    const u = q.shift();
    for (const v of adj[u]) {
      if (dist[v] === -1) {
        dist[v] = dist[u] + 1;
        pred[v] = u;
        q.push(v);
      }
    }
  }
  return {dist, pred};
}

Edge cases and special considerations

  • Disconnected graphs: run BFS from each unvisited vertex to get all components.
  • Directed graphs: BFS behavior is the same; edges are only followed in their directed direction.
  • Graphs with self-loops or parallel edges: ignore self-loops in shortest path logic; parallel edges do not change distances but may cause redundant neighbor scans (still correct).

Comparison with DFS

DFS explores as deep as possible before backtracking; BFS explores breadth-wise. Use BFS when you need shortest paths in unweighted graphs or level-order exploration. Use DFS when exploring connectivity, topological sort (with additional processing), or searching for a path with less memory in some scenarios (but DFS can consume O(|V|) stack as well).

Proof of runtime O(|V| + |E|) (sketch)

Every vertex is enqueued and dequeued at most once → O(|V|) queue operations. The loop over neighbors examines each adjacency list exactly once; for an adjacency list representation the total work over all vertices is proportional to the sum of the lengths of adjacency lists, which is O(|E|). Summed together: O(|V| + |E|).

Study questions (quick)

  1. Why do we mark visited when enqueuing instead of when dequeuing?
  2. How would you adapt BFS to return the nodes level-by-level as a list of lists?
  3. What changes if edges have positive weights other than 1?

Answers (brief): 1) To avoid repeated enqueues and exponential blow-up; 2) Track level boundaries (use sentinel or store level with node); 3) Weighted edges require Dijkstra's algorithm (BFS only works for unit weights).

Practical tips and performance tuning

  • In performance-critical code, avoid using queue operations that are O(n) (e.g., .shift() on JS arrays). Use a true queue implementation or pointers into an array.
  • For very large sparse graphs, compress adjacency structures and use efficient iterators to reduce memory overhead.
  • When running many BFS searches (e.g., from many sources), consider reusing arrays and resetting using version counters instead of reinitializing large arrays repeatedly.
  • In concurrent or distributed settings, BFS can be parallelized per layer (all nodes in the current frontier can be expanded in parallel), but careful synchronization and duplicate avoidance are required.
