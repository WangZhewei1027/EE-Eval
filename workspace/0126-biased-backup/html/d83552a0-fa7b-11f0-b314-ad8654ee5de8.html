Jump Search

A focused, pedagogical guide to Jump Search: definitions, algorithm, complexity, examples, derivation of optimal block size, comparisons and variations.

What is Jump Search?

Jump Search (also called block search) is a searching algorithm for sorted arrays. It balances between linear and block (jump) scanning: instead of checking every element sequentially (linear search) or dividing the search space in half each time (binary search), Jump Search moves ahead by fixed-size blocks (jumps) until it finds a block that could contain the target, and then performs a linear search inside that block.

Key idea: reduce the number of comparisons by skipping ahead a fixed number of elements, then finish with linear scan within a small range.

When to use Jump Search?

  • Data must be sorted (required).
  • Useful when random access is expensive but you can still skip in blocks efficiently.
  • Particularly relevant for data stored on media where jumps are cheaper than many small consecutive accesses (or where binary search is impractical, e.g., singly linked lists).
  • Not typically the fastest for arrays in RAM—binary search is usually asymptotically better (O(log n)). But jump search has simplicity and can be better for some memory access patterns.

High-level algorithm (textual)

Given a sorted array A of length n and a target value x:

  1. Choose a fixed jump length m (block size). A common choice is m = floor(sqrt(n)).
  2. Begin at index 0. Compare A[0] with x. If equal, return index 0.
  3. Jump ahead by m elements repeatedly: check A[m - 1], A[2m - 1], A[3m - 1], ... (or check A[m], A[2m], etc. — indexing conventions vary). Continue until you find a block where A[j] >= x (or reach the end of the array).
  4. When the block containing x is found (between indices prev and j), perform a linear search from index prev up to j (inclusive) to locate x.
  5. If x is found return its index; otherwise report "not found".

Note: you can decide whether to check the block-end or block-start when jumping. Implementation differences are minor; the fundamental pattern remains the same.

Pseudocode

// Jump Search (one common variant)
// Input: sorted array A[0..n-1], target x
// Output: index of x in A or -1 if not found

function jumpSearch(A, n, x):
    m = floor(sqrt(n))            // block size
    prev = 0
    // jump in blocks until the block that may contain x is found
    while A[min(m, n)-1] < x:
        prev = m
        m = m + floor(sqrt(n))
        if prev >= n:
            return -1            // reached beyond array -> not found

    // linear search within the block starting from prev
    for i from prev to min(m, n)-1:
        if A[i] == x:
            return i
    return -1

This pseudocode uses comparisons at block endpoints. Some variants compare A[m] directly and then linear-scan backward or forward accordingly. Indexing details may be adapted to 0-based or 1-based arrays.

Complexity analysis and derivation of optimal block size

Let the block size be m. In the worst case:

  • Number of block jumps: at most roughly n / m.
  • After a block is selected, the linear search inside the block may inspect up to m elements.

Therefore, the total number of element comparisons (worst-case) is approximately: 

f(m) = n / m + m

To minimize this expression with respect to m (treating m as continuous for calculus), set the derivative to zero: 

f'(m) = -n / m^2 + 1 = 0  =>  m^2 = n  =>  m = sqrt(n)

Thus the block size that minimizes the worst-case number of comparisons is approximately sqrt(n). Substituting m = sqrt(n) gives f(m) ≈ 2 * sqrt(n), so the time complexity is O(sqrt(n)) comparisons (worst-case).

Note: If you analyze average comparisons inside a block as m/2 (i.e., assuming the target is uniformly likely in the block), then the expected comparisons become n / m + m / 2. Minimizing that gives m = sqrt(2n). The commonly cited and simple rule of thumb is m ≈ sqrt(n).

Summary of complexities:

  • Worst-case time complexity: O(n) but, with optimal block size, O(sqrt(n)) comparisons dominates (i.e., O(sqrt(n))).
  • Average-case (for uniformly random position and fixed m ≈ sqrt(n)): O(sqrt(n)).
  • Best-case: O(1) when the element is in the first checked position.
  • Space complexity: O(1) additional space.

Step-by-step worked example

Consider the sorted array A with n = 16 elements:

A = [2, 4, 7, 10, 13, 18, 21, 26, 30, 33, 39, 42, 47, 50, 58, 64]

Let us search for x = 33. Choose m = floor(sqrt(16)) = 4.

We will use the block-end-check convention (check A[m-1] first):

  1. Compare A[3] (index 3: value 10) with 33 → 10 < 33. Not in first block. prev = 4, m = 8.
  2. Compare A[7] (index 7: value 26) with 33 → 26 < 33. Not in second block. prev = 8, m = 12.
  3. Compare A[11] (index 11: value 42) with 33 → 42 >= 33. We've found the block that may contain 33: indices [8..11].
  4. Linear search inside this block from index 8 to 11:
    • i = 8: A[8] = 30 → not 33.
    • i = 9: A[9] = 33 → found! Return index 9.

Total comparisons: block checks at indices 3, 7, 11 (3 comparisons) plus linear checks at indices 8, 9 (2 comparisons) = 5 comparisons. With m = 4 we did about n/m + m = 16/4 + 4 = 8 comparisons in the worst case; in this example we found it faster than that worst-case bound.

Correctness reasoning (informal)

Jump Search is correct because it preserves the invariant that elements in all fully-skimmed blocks are strictly less than the target (when we jump forward only after verifying the block-end is < x). Once we find the first block whose end is ≥ x, if the array is sorted then any element equal to x (if it exists) must lie in the current block or earlier; but earlier blocks were explicitly confirmed to have all elements < x, so x can only lie in the current block. Thus a linear scan inside that small block finds x if it exists.

Variants and practical notes

  • Variant: check A[m] at jumps and then perform linear backward scan in the previous block. Equivalent in spirit; indexing adjustments change which indices are scanned.
  • Average-block optimization: if you assume the element is uniformly distributed, you may choose block size ≈ sqrt(2n) to minimize expected comparisons.
  • Linked lists: Jump Search can be adapted to linked lists by performing m − 1 sequential steps between checkpoints (maintain a pointer to the block start). Binary search is unsuitable for singly linked lists because random access is not O(1); jump search is a reasonable compromise if you can efficiently move in fixed steps.
  • Indexed sequential search: Jump Search idea is related to using a small index of block boundaries to avoid scanning the entire structure.
  • Practical arrays in RAM: Binary Search is often faster (O(log n) vs O(sqrt(n))) if random access is cheap and comparisons are the bottleneck. Jump Search may be more efficient when cache behaviour or expensive random disc seeks change the cost model.

Comparison with other searching methods (summary)

  • Linear Search: O(n) worst-case. Jump Search improves over linear by skipping elements — expected O(sqrt(n)).
  • Binary Search: O(log n). Asymptotically better for arrays with O(1) random access, but requires random access and sorted data (same as jump search). Binary search makes log2(n) comparisons; jump search with sqrt(n) block size makes about 2 sqrt(n) comparisons.
  • Interpolation Search: average-case O(log log n) for uniformly distributed keys but can degrade. Jump Search is distribution-agnostic but slower asymptotically on random-access memory.

Implementation tips and pitfalls

  • Be careful with indexing conventions and array bounds: when you compute the block end, clamp it to n-1.
  • The initial jump should consider the first block correctly: some pseudo-implementations check A[m - 1], others check A[m]; be consistent.
  • Remember the array must be sorted. If not sorted, Jump Search is not correct.
  • If the data structure is mutable and elements are inserted/deleted frequently, maintaining sorted order may be more expensive than searching—consider balanced trees or hash-based methods instead.

Short FAQ

Q: Does Jump Search work for duplicates?
A: Yes — it will find one occurrence of the target if present. If you need the first or last occurrence, you may have to scan within the block accordingly or post-process.

Q: Is there an iterative and recursive version?
A: Yes. The algorithm is usually implemented iteratively because it is naturally index-driven; a recursive form would simulate jumping via recursion but offers no practical advantage.
