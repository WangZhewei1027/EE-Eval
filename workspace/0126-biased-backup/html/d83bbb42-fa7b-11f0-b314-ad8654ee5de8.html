What is a Digital Signature?

A digital signature is a cryptographic construct that allows someone to attach evidence to a digital message or document that proves three key things:

  • Authenticity: the message genuinely originates from the claimed signer (or at least someone holding the signer's private key);
  • Integrity: the message has not been altered since it was signed; any change to the message will cause the signature verification to fail;
  • Non-repudiation: under defined assumptions, the signer cannot plausibly deny having signed the message (this depends on legal/trust frameworks and secure key management).

Note: "Non-repudiation" requires careful definition — cryptography provides technical evidence, but legal and operational systems determine whether that evidence is binding.

Components and Building Blocks

Digital signatures are typically built from a few fundamental primitives:

  • Asymmetric (public-key) cryptography: A private key used for signing and a corresponding public key used for verification. Common families include RSA, DSA, and elliptic-curve schemes (ECDSA, EdDSA).
  • Cryptographic hash functions: Secure one-way functions (e.g., SHA-256) used to compress a message to a fixed-size digest before signing. Hashing is essential for performance and security.
  • Signature scheme: The algorithm that converts a message digest and a private key into a signature (and verification routine that checks it using the public key).
  • Optional padding or randomness: Secure padding schemes (e.g., RSA-PSS) or randomized components in some algorithms (e.g., DSA/ECDSA nonce) to prevent certain attacks.
  • Certificates and trust infrastructure: Public keys are often distributed inside certificates (X.509, OpenPGP) that bind keys to identities via a certification authority (CA) or trust network.

How a Digital Signature Works — Step-by-step Explanation

Here is a generic, stepwise narrative of creating and verifying a digital signature:

  1. Key generation: The signer generates a key pair (private key, public key). The private key is kept secret; the public key is shared.
  2. Message preparation: Take the message to be signed (can be a document, software binary, transaction, etc.). Usually, the message is hashed using a cryptographic hash function to produce a fixed-size digest.
  3. Signing: The signer runs a signing algorithm that takes the private key and the message digest and outputs a signature (a small piece of data bound to both the message digest and the private key).
  4. Transmission: The message is sent along with the signature (or the signature is attached to the message in a container format).
  5. Verification: The verifier uses the signer’s public key, computes the message digest from the received message, and runs the verification algorithm on the signature and the digest. If verification succeeds, the verifier accepts that the message originates from the holder of the private key and has not been altered.

Key points: the private key is never shared, and hashing before signing is standard to ensure fixed-size input and protect against certain attacks.

Major Signature Algorithms

RSA Signatures

RSA can be used for both encryption and signing. In the classic RSA signature scheme (textbook RSA), signing computes s = m^d mod n, where (n, e) is the public key and d is the private exponent. However, textbook RSA is insecure in practice. Practical RSA signatures always combine hashing and a padding scheme:

  • PKCS#1 v1.5: an older standardized padding for signatures — historically popular but has subtle vulnerabilities when poorly implemented.
  • RSA-PSS (Probabilistic Signature Scheme): a modern randomized padding recommended for new systems because it has stronger provable security properties.

DSA (Digital Signature Algorithm)

DSA is a discrete-logarithm-based signature algorithm specified in FIPS 186. DSA relies on secure generation of a per-signature random value called the nonce (k). Reuse or leakage of k breaks the private key.

ECDSA

Elliptic Curve DSA applies the DSA idea on elliptic curves. ECDSA keys are much smaller than equivalent-security integer-field DSA keys (e.g., 256-bit EC keys provide security comparable to ~3072-bit RSA). ECDSA also requires a fresh unpredictable nonce per signature.

EdDSA (ed25519, ed448)

EdDSA (Edwards-curve Digital Signature Algorithm) is a modern elliptic-curve signature scheme designed for speed, safety, and ease-of-use. ed25519 is widely adopted and avoids the need for external randomness during signing by deriving per-message values deterministically from the private key and the message (reducing the risk of poor random-number generators).

Comparative Notes

  • RSA: widely used, simple concept, but larger keys and slower per-operation costs compared to curves.
  • DSA/ECDSA: proven and standardized, but nonce management is critical.
  • EdDSA: modern, fast, resilient to common RNG problems (for deterministic variants), smaller signatures and keys.

Hashing, Padding, and Randomness

These details matter more than many developers expect.

Why hash before signing?

  • Performance: you sign a fixed-length digest instead of arbitrarily long messages.
  • Security: hashing provides domain separation and protects the signature scheme from structural attacks from long messages.
  • Compatibility: standard schemes expect a digest (e.g., SHA-256) as input.

Padding and its purpose

Padding transforms a hash into a value suitable for the signature algorithm and prevents existential forgery. Examples: RSA-PSS uses a randomized mask generation step, whereas PKCS#1 v1.5 uses deterministic structure to encode algorithm identifiers and the digest.

Randomness (nonces)

Some signature algorithms require fresh randomness per signature (eg., DSA, ECDSA). If the randomness is repeated (k reused) or leaked, the private key can be recovered. Deterministic variants (RFC 6979) derive k deterministically from private key and message to avoid RNG pitfalls.

Security Considerations

Threat models and attacks

  • Key compromise: If the private key is stolen, signatures are no longer trustworthy. Protect private keys with hardware modules (HSMs) or secure enclaves where possible.
  • Nonce reuse: For DSA/ECDSA, reuse of nonce k reveals the private key. This is one of the most common real-world mistakes.
  • Hash collisions: If the hash function used is broken (collisions possible), an attacker might craft two messages with same digest and get a valid signature to apply to a different message. Use collision-resistant hash functions (SHA-256, SHA-3 family as appropriate).
  • Fault and side-channel attacks: Timing, power analysis, and fault injection can leak private keys. Constant-time implementations and protective hardware mitigate these risks.
  • Algorithm-specific issues: Plaintext RSA signing without proper padding is insecure; ECDSA/ECDH curve parameter choices must be safe.
  • Quantum threats: Quantum computers running Shor’s algorithm would break RSA and ECC. Post-quantum signature schemes (lattice-based, hash-based, multivariate) are being standardized to address this future risk.

Key sizes and recommendations (as of mid-2020s)

  • RSA: 2048 bits is minimum for many applications; 3072 bits for higher security
  • Elliptic curve: 256-bit curve (e.g., secp256r1, curve25519 family) for mainstream use
  • Ed25519: recommended modern choice for many applications for performance and safety

Implementation Best Practices

  • Always use well-established, well-reviewed cryptographic libraries. Do not write your own primitives.
  • Use deterministic signing variants (where available) or ensure high-quality entropy for nonces.
  • Always hash before signing; be explicit about which hash is used and include algorithm identifiers in signatures or protocol messages so verifiers know what was intended.
  • Prefer modern constructions (e.g., RSA-PSS over PKCS#1 v1.5; Ed25519 over raw ECDSA when protocol-compatible).
  • Protect private keys with hardware-backed stores and rotate keys according to policy.
  • Be mindful of canonicalization: when signing formatted documents (JSON, XML, PDF), canonicalize first to avoid subtle differences between signer and verifier.
  • Consider backwards compatibility, but plan for algorithm agility — design your systems so signatures can move to new algorithms in the future.

Worked Examples, Pseudocode, and Intuition

Simple RSA signature (intuitive math)

RSA uses operations modulo an integer n = p * q. The signer has a private exponent d and public exponent e. Conceptually:

// To sign:
digest = Hash(message)
signature = (digest^d) mod n

// To verify:
digest = Hash(message)
check = (signature^e) mod n
accept if check == digest

In practice, padding transforms digest into a larger formatted value before exponentiation, and verification checks that the padded structure matches expectations.

ECDSA (high-level)

ECDSA signing steps (conceptual):

  1. Compute z = Hash(message).
  2. Pick random nonce k (unique per signature).
  3. Compute r = (k * G).x mod n, where G is the curve base point and n is curve order.
  4. Compute s = k^{-1} * (z + r * d) mod n, where d is the private key.
  5. Signature is (r, s).

Verification uses public key Q = d * G and checks a relation involving s^{-1}, z, r, and Q. If k is reused or leaked, d can be recovered via simple algebra.

Pseudocode for signing and verifying (generic)

// Generic signing flow
function sign(privateKey, message):
    digest = Hash(message)
    signature = SignAlgorithm(privateKey, digest)
    return signature

function verify(publicKey, message, signature):
    digest = Hash(message)
    return VerifyAlgorithm(publicKey, digest, signature)

Remember: the exact shape of SignAlgorithm and VerifyAlgorithm depends entirely on the chosen signature scheme (RSA, ECDSA, EdDSA, etc.).

Applications and Operational Notes

Digital signatures are everywhere in modern computing:

  • Code signing (operating system updates, drivers, mobile apps)
  • Email authentication and signing (PGP, S/MIME)
  • Secure sessions and TLS certificate validation
  • Blockchain transactions and cryptocurrency
  • Electronic documents, contracts, and legal signatures (often combined with audit trails and timestamping)

Certificates, PKI, and trust models

Public keys are commonly distributed inside certificates that assert the identity associated with a public key. Trust models vary: centralized certificate authorities (CA), web-of-trust (PGP), or distributed ledger approaches. Managing trust involves revocation, expiry, and key rotation policies.

Admissibility and legal context

Whether a signature is legally binding depends on jurisdiction and how the signing process is implemented (e.g., whether private keys are controlled in a way that associates the signature with a person). Cryptographic signatures provide evidence; legal weight requires policy, identity assurance, and possibly audited key custody.
