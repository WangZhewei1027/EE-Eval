Process — Comprehensive Explanation

A detailed, pedagogical overview of the computing concept "process": definition, theory, algorithms, lifecycle, scheduling, synchronization, and practical examples — written as an educational reference.

What is a Process? — Definition and Intuition

A process is a fundamental abstraction used by operating systems to represent a running program. It encapsulates the program's executable code, its current activity (program counter, registers), resource usage (open files, memory), and the execution context required to manage, schedule, and isolate execution.

Intuitively, a process is "a program in execution." A single program can create multiple processes (examples: web server worker processes, child processes created for tasks). Processes provide isolation, resource accounting, and a unit of scheduling in the OS.

Key Terms and Concepts

  • Program: static code and data on disk.
  • Process: dynamic state of a running program (memory, registers, open files, etc.).
  • Thread: a schedulable execution context within a process (shares process memory).
  • Address space: the memory layout allocated to a process (stack, heap, code, data).
  • Context switch: the act of saving and restoring CPU/OS state to switch between processes/threads.
  • Process Control Block (PCB): kernel structure holding a process's metadata.
  • Inter-process communication (IPC): mechanisms for processes to exchange data (pipes, sockets, shared memory, messages).

Process Model and the Process Control Block (PCB)

The OS represents each process with a PCB (or task structure). Typical PCB fields:

  • Process identifier (PID)
  • Process state (running, ready, blocked, etc.)
  • Program counter and CPU registers
  • Memory-management information (page tables, base/limit)
  • Open file descriptors and I/O state
  • Accounting information (CPU time used, priority)
  • Parent/child relationships, security credentials

The PCB is essential for context switching: the kernel saves a process's CPU registers and other state into its PCB, then restores another process's PCB to resume it.

Process States and Lifecycle

Common process states (simplified):

  • New: being created.
  • Ready: prepared to run, waiting for CPU.
  • Running: currently executing on the CPU.
  • Blocked (Waiting): waiting for I/O or an event.
  • Terminated (Zombie): finished execution, awaiting parent cleanup.

Step-by-step: Typical lifecycle (high-level)

  1. Create process control structures (allocate PCB and address space).
  2. Load program or set up to execute code (in exec-like models).
  3. Place process in Ready queue.
  4. Scheduler dispatches process to Running state.
  5. Process may block (I/O) or run until time slice ends; on blocking it moves to Blocked; on preemption it moves back to Ready.
  6. On completion, process transitions to Terminated; resources are reclaimed by the OS and parent may collect exit status.

Process Creation and Termination

Most OSes provide primitives to create and execute processes. Two common operations:

  • fork() (Unix): creates a new process by duplicating the calling process (child gets a copy of parent's memory—copy-on-write often used).
  • exec() (Unix): replaces a process's memory with a new program (usually used after fork to run a new program in the child).

Typical sequence in Unix-style systems (step-by-step)

  1. Parent calls fork(). Kernel duplicates PCB (copy-on-write for efficiency), assigns new PID to child.
  2. Both processes continue; fork() returns 0 in the child and child's PID in the parent.
  3. Child may call exec() to load a different program into its address space; child's PID remains the same.
  4. When a process exits, it calls exit(); the kernel sets its state to terminated and records exit status.
  5. Parent calls wait() (or similar) to collect child's exit status; until waited on, child may remain a 'zombie' occupying a small PCB record.
// Simplified pseudocode of fork-exec pattern
pid = fork()
if pid == 0:
    // child
    exec("/bin/myprogram", args)
else:
    // parent
    waitpid(pid)  // collect child's exit status

Context Switching — What and Why

A context switch saves the running process's CPU state and restores another process's state so the CPU can run it. This includes registers, program counter, and possibly memory-management state (page tables).

Context switches allow multiple processes to share a single CPU. They are not free — switching has overhead: CPU cycles for saving/restoring and cache effects (TLB/CPU caches may lose locality).

Practical note: Minimizing unnecessary context switches improves throughput and latency. Choosing appropriate scheduling time slice is a trade-off: shorter slices reduce latency but increase switching overhead.

CPU Scheduling — Goals and Metrics

Scheduling determines which process runs when. Goals vary: maximize CPU utilization, maximize throughput (jobs completed per time), minimize turnaround time, minimize waiting time, or minimize response time (important for interactive systems).

  • Turnaround time: finish time - arrival time.
  • Waiting time: total time spent in Ready queue.
  • Response time: time from request to first response (important for interactive).
  • Throughput: jobs completed per unit time.

Different scheduling algorithms target different goals.

Common scheduling algorithms — descriptions and pseudocode

First-Come, First-Served (FCFS)

Non-preemptive. Processes run in arrival order.

Algorithm FCFS:
Sort processes by arrival time.
For each process in order:
    schedule process to completion
end

Shortest Job First (SJF)

Choose the process with smallest estimated run time. Can be non-preemptive or preemptive (Shortest Remaining Time First, SRTF).

Algorithm SJF (non-preemptive):
While uncompleted processes exist:
    from ready processes, pick the one with smallest burst time
    run it to completion
end

Round Robin (RR)

Preemptive. Each process gets a fixed time quantum in a circular queue. Good for interactive systems.

Algorithm RR:
time_quantum = Q
ready_queue = FIFO queue
while ready_queue not empty:
    p = dequeue(ready_queue)
    run p for min(remaining_time(p), Q)
    if p not finished:
        enqueue(ready_queue, p)
end

Priority Scheduling

Each process has a priority. Lower numbers = higher priority (convention varies). Can be preemptive or non-preemptive. Must handle starvation (aging).

Example: 4 processes

Processes with arrival time and CPU burst (in ms):

P1: arrival 0, burst 8
P2: arrival 1, burst 4
P3: arrival 2, burst 9
P4: arrival 3, burst 5
FCFS

Order: P1, P2, P3, P4

Completion times:
P1: 0+8 = 8
P2: 8+4 = 12
P3: 12+9 = 21
P4: 21+5 = 26

Turnaround times = completion - arrival:
P1: 8-0=8
P2: 12-1=11
P3: 21-2=19
P4: 26-3=23

Average turnaround = (8+11+19+23)/4 = 15.25
SJF (non-preemptive)

At time 0 only P1 ready → run P1 to completion. When P1 finishes at 8, P2,P3,P4 are all ready; pick shortest burst: P2 (4), then P4 (5), then P3 (9).

Order: P1, P2, P4, P3
Completion: P1@8, P2@12, P4@17, P3@26

Turnaround:
P1:8, P2:12-1=11, P4:17-3=14, P3:26-2=24
Average = (8+11+14+24)/4 = 14.25

Time slices of 4 ms. Execution timeline (abbreviated):

t=0-4: P1 (remaining 4)
t=4-8: P1 (remaining 0) -> P1 done at t=8
t=8-12: P2 (burst 4) -> done at t=12
t=12-16: P3 (remaining 5)

Inter-Process Communication (IPC)

Processes are isolated, so IPC is necessary for cooperation. Common IPC mechanisms:

  • Pipes: ordered byte streams between related processes (unnamed or named/FIFOs).
  • Sockets: network-style communication, can be local (AF_UNIX) or remote (TCP/IP).
  • Shared memory: processes map the same memory region; fastest but requires synchronization.
  • Message passing: structured messages exchanged via OS or middleware (e.g., message queues).
  • Remote procedure calls (RPC): abstraction to invoke procedures in remote processes.

Choice of IPC depends on performance, complexity, and whether processes are related or on different machines. Shared memory is fastest for bulk data, while message passing and sockets are safer and better for distributed systems.

Concurrency, Synchronization, and Race Conditions

When processes (or threads) access shared resources concurrently, synchronization is necessary to avoid inconsistent state or races. Common synchronization primitives:

  • Mutex (exclusive lock)
  • Semaphore (counting for resource units)
  • Monitor (higher-level construct combining lock and condition variables)
  • Condition variables (wait/signal coordination)

Classic example — Producer/Consumer

Producer produces items, consumer consumes. Use a bounded buffer, a mutex to protect buffer state, and two semaphores: empty (count of empty slots) and full (count of filled slots). Producers wait on empty, then lock the mutex, place item, unlock mutex, signal full. Consumers wait on full, lock, remove item, unlock, signal empty. 

// Pseudocode for producer
while true:
    item = produce()
    wait(empty)
    mutex.lock()
    put(item)
    mutex.unlock()
    signal(full)

// Pseudocode for consumer
while true:
    wait(full)
    mutex.lock()
    item = get()
    mutex.unlock()
    signal(empty)
    consume(item)

Deadlock — Conditions and Strategies

Deadlock occurs when a set of processes are each waiting for resources held by others in the set. Necessary conditions (Coffman conditions):

  • Mutual exclusion (resources non-shareable)
  • Hold and wait (process holds resources while waiting for others)
  • No preemption (resources cannot be forcibly taken)
  • Circular wait (cycle of processes each waiting for next)

Strategies

  • Prevention: prevent one or more Coffman conditions (e.g., require processes to request all resources at once, or allow preemption).
  • Avoidance: dynamically avoid unsafe states using algorithms like Banker's algorithm (process must declare maximum resource needs).
  • Detection and recovery: let deadlocks occur, detect them (resource allocation graphs), and recover (kill/rollback processes or preempt resources).
// Very short description: Banker's algorithm (sketch)
At each allocation request:
    simulate granting the request
    run safety algorithm to see if system would remain in safe state
    if safe: grant; else: block process

Processes vs Threads — Trade-offs

Threads are lighter-weight units of execution within a process. Threads share the process address space and resources; processes have separate address spaces.

  • Threads: cheaper to create and switch, but require careful synchronization (shared memory).
  • Processes: stronger isolation (faults less likely to corrupt others), more overhead for communication.

Modern systems use combinations: multi-process architectures for isolation (e.g., multiple processes for web browser tabs) and multithreading inside long-running services for concurrency.

Practical Examples: Linux and Windows

Linux: processes represented by task_struct in the kernel. Tools: ps, top, pstree, kill. Common primitives: fork, exec, clone (creates processes or threads depending on flags).

Windows: processes are represented by executive objects (EPROCESS), threads by ETHREAD. Tools: Task Manager, Process Explorer. Windows provides CreateProcess and threading APIs.

Design Considerations and Best Practices

  • Favor appropriate isolation: use processes for fault isolation, threads for performance-sensitive shared-memory tasks.
  • Keep critical sections small to reduce contention and blocking time.
  • Avoid holding locks while performing blocking I/O.
  • Record and monitor process resource usage to detect leaks and runaway processes.
  • Apply aging or priority adjustments to avoid starvation in priority schedulers.

Summary

Processes are the operating system's primary abstraction for running programs. Understanding processes involves knowing their lifecycle, how the kernel represents them, scheduling algorithms and metrics, inter-process communication, synchronization primitives, and techniques for dealing with deadlocks and concurrency issues. Processes and threads are complementary tools; choosing between them depends on isolation, performance, and communication requirements.
