Arrays — A Comprehensive Educational Guide

Definitions, theory, algorithms, step-by-step examples, and practical notes on arrays as a fundamental data structure.

What is an Array?

An array is a collection of elements (values or variables), all of the same type in many typed languages, stored in contiguous memory locations. Arrays provide direct access to elements using an index — typically an integer — that represents the position of the element in the sequence.

Key characteristics

  • Indexed access: Elements are accessed by index e.g. A[i]. Indexing is usually zero-based (first element at index 0) in many languages, though some use one-based indexing.
  • Contiguous memory: Elements are laid out next to each other in memory, enabling constant-time address computation for access.
  • Fixed size (static arrays): In low-level languages, the declared size is fixed. Many high-level languages provide dynamic arrays (resizable).
  • Uniform element type: Arrays store items of the same type (or logically treated as such), which simplifies memory layout and access.

Why arrays are important

Arrays are one of the most important and frequently used data structures because:

  • They provide fast O(1) access to elements by index (constant-time random access).
  • They are space-efficient and map naturally to CPU cache lines because of contiguous layout.
  • They form the building block for many higher-level data structures (stacks, queues, heaps, hash tables internally often use arrays).
  • Understanding arrays helps with memory reasoning, algorithmic complexity, and performance tuning.

Common operations and complexity

Below are the typical operations defined on arrays and their time complexity in the average and worst case for a static or dynamic array backing (random-access array).

  • Access (read/write): A[i] — O(1) time.
  • Traversal: Visit every element — O(n) time.
  • Search (unsorted): Linear search — O(n) time.
  • Search (sorted): Binary search — O(log n) time (requires random access).
  • Insert at end (dynamic array amortized): Append — amortized O(1); worst-case O(n) when resizing occurs.
  • Insert/Delete at arbitrary position: Requires shifting elements — O(n) time.
  • Memory: O(n) space for n elements (plus possible extra slack for dynamic arrays).

Why insertion in middle costs O(n)

Because elements to the right of the insertion index must be shifted one position to make room. This involves moving O(n) items in the worst case (inserting at start).

Example: Insert at index 2 (step-by-step)

Given array A = [3, 8, 1, 6, 0] and we want to insert value 9 at index 2:

  1. Ensure there is capacity. If the internal buffer is full, allocate a larger buffer and copy all elements (resizing).
  2. Starting from the last valid index, shift each element one step to the right until index 2 is free: 
    Before: [3, 8, 1, 6, 0]
Step 1:  [3, 8, 1, 6, 0, _] (grow if needed)
Shift index 4 -> 5: [3,8,1,6,0,0]
Shift index 3 -> 4: [3,8,1,6,6]
Shift index 2 -> 3: [3,8,1,1,6]  (now index 2 free)
Insert 9 at index 2:
After: [3,8,9,1,6,0]
  3. Update the length counter.

Searching in arrays

Linear search (sequential scan)

Algorithm: Start at the first element and compare each element to the target until it is found or the array ends.

// Pseudocode: linearSearch(A, target)
for i from 0 to length(A)-1:
    if A[i] == target:
        return i   // index found
return -1  // not found

Complexity: O(n) time, O(1) space. No assumptions about ordering required.

Binary search (requires sorted array)

Algorithm: Repeatedly divide the search interval in half. Compare the target with the midpoint element to decide which half to keep.

// Pseudocode: binarySearch(A, target)
low = 0
high = length(A) - 1
while low <= high:
    mid = floor((low + high) / 2)
    if A[mid] == target: return mid
    else if A[mid] < target: low = mid + 1
    else: high = mid - 1
return -1

Complexity: O(log n) time, O(1) space. Requires random access, i.e., arrays are ideal for binary search.

Binary search pitfalls

  • Careful with integer overflow when computing mid as (low + high) / 2 in some languages; prefer low + (high - low) // 2.
  • Off-by-one errors: ensure while loop condition and boundary updates are correct for your chosen variant.
  • Binary search finds an index of a matching element, but if duplicates exist, you may need variants to find the first or last occurrence.

Sorting and arrays

Arrays are commonly sorted in-place. Common sorting algorithms and their typical complexity (for arrays):

  • Bubble sort / Selection sort / Insertion sort: O(n^2) time, simple, useful for small arrays or partly-sorted arrays (insertion sort).
  • Merge sort: O(n log n) time, stable, requires O(n) extra memory (not in-place without complex variants).
  • Quick sort: Average O(n log n) time, O(log n) expected stack depth, in-place but unstable in standard form; worst-case O(n^2) but mitigated by randomization or median-of-three pivot selection.
  • Heap sort: O(n log n) time, in-place, not stable.

Choice of sorting algorithm depends on array size, memory constraints, stability requirements, and whether data is partly sorted.

Dynamic arrays (resizable arrays)

High-level languages usually provide arrays that grow as needed (e.g., Python list, Java ArrayList, C++ std::vector, JavaScript Array). These implement a dynamic array by keeping an internal contiguous buffer and resizing when full.

Typical strategy for resizing

When the buffer is full, allocate a new buffer with larger capacity (often 1.5x or 2x the old capacity), copy elements to the new buffer, and free the old buffer. This yields amortized O(1) time for append operations.

Amortized analysis (intuitive)

Although a single append may cost O(n) when resizing, resizing happens infrequently. If capacity doubles, each element is copied O(1) times on average over many appends. Thus the average time per append is O(1).

Multidimensional arrays

Arrays can have more than one dimension. A 2D array is conceptually a grid (matrix). In memory, a 2D array can be implemented in two common ways:

  • Row-major order: rows stored consecutively (C-style). Element A[i][j] address computed as base + ((i * columns) + j) * element_size.
  • Column-major order: columns stored consecutively (Fortran, MATLAB-style).

Multi-dimensional arrays allow direct index-based access but the memory layout determines cache behavior and stride when traversing rows vs columns.

Arrays vs other linear data structures

  • Array vs Linked List:
    • Arrays: O(1) random access, O(n) insert/delete at arbitrary positions (due to shifts), good cache locality.
    • Linked lists: O(n) random access, O(1) insert/delete given a pointer to a node, poor locality, more memory overhead per element (pointers).
  • Array vs Hash Table: Arrays are indexed by numeric position; hash tables map keys to values with average O(1) lookup but no ordering and require hash functions and extra memory.

Practical examples in common languages

C (static array)

int A[5] = {3, 8, 1, 6, 0};
int value = A[2]; // direct access (value = 1)

Java (static-like, arrays are objects)

int[] A = {3, 8, 1, 6, 0};
int value = A[2]; // value = 1

Python (list as dynamic array)

A = [3, 8, 1, 6, 0]
value = A[2]  # value = 1
A.append(9)   # amortized O(1)

JavaScript (Array)

const A = [3, 8, 1, 6, 0];
const value = A[2]; // 1
A.push(9); // append

Step-by-step algorithm walkthroughs (textual)

1) Linear Search Example (step-by-step)

Array: [3, 8, 1, 6, 0, 9, 4], target = 6

  1. Compare A[0] (3) with 6 → not equal.
  2. Compare A[1] (8) with 6 → not equal.
  3. Compare A[2] (1) with 6 → not equal.
  4. Compare A[3] (6) with 6 → equal → return index 3.

2) Binary Search Example (step-by-step)

Sorted array: [0, 1, 3, 4, 6, 8, 9], target = 6

  1. low=0, high=6, mid=floor((0+6)/2)=3 → A[3]=4. 4 < 6, so search right half.
  2. low=4, high=6, mid=floor((4+6)/2)=5 → A[5]=8. 8 > 6, so search left half of this segment.
  3. low=4, high=4, mid=4 → A[4]=6. Found at index 4.

Common pitfalls and best practices

  • Off-by-one errors: Carefully consider whether your language uses zero-based indices and whether loops should use < or <= in conditions.
  • Bounds checking: Many languages (Python, Java) check bounds and will raise exceptions if indexing out of range. In C, accessing out-of-bounds is undefined behavior.
  • Mutable vs immutable elements: Storing references vs values affects modifications: modifying a referenced object inside an array changes the shared object.
  • Be mindful of performance: Repeated insertions at the front of a long dynamic array are costly — consider deque/linked-list or specialized data structures if front inserts are frequent.

Advanced topics (concise)

  • Circular buffers: Use a fixed-size array as a ring buffer for queues with head/tail indices.
  • Slicing and views: Some languages provide cheap views into arrays (e.g., numpy arrays in Python) to avoid copying.
  • Packed and typed arrays: For performance and memory efficiency, languages provide typed arrays (TypedArray in JS, arrays of primitive in Java).
  • Cache considerations: Access arrays sequentially to leverage spatial locality and CPU cache lines for better performance.
