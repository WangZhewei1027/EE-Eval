Trie (Prefix Tree) — A Comprehensive Explanation

Also called a prefix tree. This page focuses on careful, textual teaching: definitions, algorithms, step-by-step examples, variants, complexity, and practical notes. Minimal interactivity.

1. What is a Trie?

A trie (pronounced "try") is a tree-like data structure that stores a dynamic set of strings, where each node represents a prefix shared by some strings. Unlike binary search trees keyed by entire values, tries break keys into components (usually characters) and store them across levels of the tree: the path from the root to a node spells a prefix.

Key idea

Strings with a common prefix share the same initial path. Each edge is labeled by a character (or group of characters in compressed variants). A node typically contains links to children (one per possible next character) and a boolean (or counter) indicating whether the prefix represented by that node is also a complete stored key (word).

Minimal example

Insert "to", "tea", "ted", "ten", "A" into a trie. The shared prefixes (t, te) are reused, reducing redundant storage of prefix characters and enabling fast prefix queries like "all words starting with 'te'".

2. Basic trie node structure

At minimum, a node stores:

  • child pointers (one per symbol in the alphabet, or an associative map)
  • a flag like is_end to mark that the path to this node is a complete stored string

Common implementations for children

  • Fixed-size array indexed by character (good for small fixed alphabets like lowercase letters).
  • Hash map or dictionary from character to child node (saves memory when branching is sparse).
  • Sorted vector of (character, child) pairs or linked list (trades memory for slower lookups).

Example node (conceptually)

// conceptual structure
Node {
  children: map;
  is_end: boolean;        // true if node marks end of a stored word
  // optional: count, frequency, stored index(es), parent pointer, etc.
}

3. Core operations (theory + step-by-step text)

Insertion (Insert word)

Goal: add a word to the trie so later queries can find it or use it as a prefix.

Step-by-step textual algorithm:

  1. Start at the root node.
  2. For each character c in the word, in order:
    1. If the current node has a child for c, follow that child.
    2. Otherwise, create a new node for c, attach it as the child for c, then follow it.
  3. After processing all characters, mark the final node's is_end = true to indicate the entire word is stored.

Example: Insert "cat".

  1. root → no child 'c': create node 'c'.
  2. node 'c' → no child 'a': create node 'a'.
  3. node 'a' → no child 't': create node 't'.
  4. mark node 't'.is_end = true.

Search (Exact match)

Goal: determine if a full word exists in the trie.

Step-by-step:

  1. Start at root.
  2. For each character c in the query word:
    1. If current node has child c: move to that child.
    2. Otherwise: the word is not in the trie — stop and return false.
  3. After processing all characters, check final node's is_end. If true, word exists; otherwise it's only a prefix and not a stored full word.

Prefix search (startsWith)

Goal: determine if any stored word has a given prefix (or retrieve all words starting with prefix).

Step-by-step to check existence:

  1. Follow characters of the prefix from root exactly like the search algorithm.
  2. If you can follow all letters successfully, the prefix exists. Whether any complete word extends it depends on descendant nodes (the final node or some descendant must be marked is_end).

To collect all words under the prefix: from the final node after following the prefix, perform a depth-first traversal, collecting characters when encountering is_end nodes — this yields all matching words.

Deletion (Remove word)

Goal: remove a word while preserving other words and shared prefixes.

Two common approaches:

  • Lazy deletion: find the terminal node and unset is_end = false. This is simple but leaves nodes that may no longer be needed.
  • Physical cleanup: unset is_end, then walk back towards the root removing nodes that have no children and are not is_end. This frees memory and ensures no dangling single-child nodes remain unnecessarily.

Step-by-step physical cleanup:

  1. Follow the word; record the path as a list of nodes.
  2. If the final node's is_end is false: word does not exist -> nothing to do.
  3. Unset final node's is_end.
  4. Walk the recorded path backwards:
    1. If a node has no children and is_end == false, delete it from its parent.
    2. Stop once you find a node that is_end==true or has other children.

4. Complexity analysis

Let m be the length of the key (number of characters) and n be the number of keys (words) in the trie. Let σ be the alphabet size.

  • Time — insertion/search/delete: O(m) in the number of character-level steps (each step may be O(1) if children are array-indexed or O(log σ) or O(1) average if using hash tables).
  • Space — naive: O(sum of lengths of keys × overhead). In the worst case, each character is a separate node; memory cost depends on node representation and branching. Fixed arrays of size σ at each node cost O(σ × nodes), which can be wasteful for large σ.

Tries offer runtime that depends on key length, not the number of keys. This is especially valuable for prefix operations and operations where ordering by prefix matters.

5. Variants and optimizations

Compressed trie / Patricia trie / Radix tree

Instead of storing one character per edge, compressed tries collapse chains of single-child nodes into a single edge labeled with a string. This reduces depth and node overhead, often dramatically reducing memory usage and improving cache behavior.

Ternary Search Tree

A space-efficient alternative that stores 3 pointers per node (left, equal, right) and compares characters like a binary search tree at each character position. It uses less memory than a naive array-of-σ implementation and is often faster than maps for many practical alphabets.

Bitwise trie (for integers)

Consider integers as fixed-length bit strings; a trie over bits is an efficient structure for problems like maximum XOR pair in a set, or longest prefix match in routing (with binary prefixes).

Suffix trie / Suffix tree

A suffix trie stores all suffixes of a text and supports substring queries. A suffix tree (compressed suffix trie) stores suffixes in compressed form and often supports advanced string queries in linear time.

Space-time trade-offs

Choices for child storage (arrays vs maps vs vectors) and compression affect memory and speed. Array children: fast but memory-hungry for large alphabets. Maps: memory-efficient for sparse branching but incur pointer and map costs. Compression reduces nodes but adds complexity to updates.

6. Typical use cases

  • Autocomplete and search suggestions (use prefix traversal + DFS to collect candidates).
  • Spell-checking and approximate matching (with modifications or additional dynamic programming to allow edits).
  • IP routing: longest prefix match (bitwise trie or Patricia trie).
  • Dictionary compression and fast lookups for fixed alphabets.
  • Bioinformatics: indexing k-mers or substrings (suffix tries/trees, suffix arrays are related alternatives).
  • Text algorithms: pattern matching and substring queries with suffix trees/tries.

7. Implementation sketch (Python-like pseudocode)

class TrieNode:
    def __init__(self):
        self.children = {}      # map
        self.is_end = False

class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):
        node = self.root
        for c in word:
            if c not in node.children:
                node.children[c] = TrieNode()
            node = node.children[c]
        node.is_end = True

    def search(self, word):
        node = self.root
        for c in word:
            if c not in node.children:
                return False
            node = node.children[c]
        return node.is_end

    def starts_with(self, prefix):
        node = self.root
        for c in prefix:
            if c not in node.children:
                return False
            node = node.children[c]
        return True

    def collect_words(self, prefix):
        # returns list of words starting with prefix
        node = self.root
        for c in prefix:
            if c not in node.children:
                return []
            node = node.children[c]
        results = []
        self._dfs(node, prefix, results)
        return results

    def _dfs(self, node, path, results):
        if node.is_end:
            results.append(path)
        for c, child in node.children.items():
            self._dfs(child, path + c, results)

This code favors clarity over performance and uses a dictionary for children. For production, tune representation based on alphabet and performance needs.

8. Deletion pseudocode (textual)

def delete(word):
    # Walk while recording (parent, char) pairs
    node = root
    stack = []  # (node, char)
    for c in word:
        if c not in node.children:
            return False  # not found
        stack.append((node, c))
        node = node.children[c]

    if not node.is_end:
        return False  # not stored as a full word

    node.is_end = False

    # Cleanup: remove nodes that became unnecessary
    while stack:
        parent, ch = stack.pop()
        child = parent.children[ch]
        if child.is_end or child.children:
            break  # someone else depends on this node
        del parent.children[ch]  # safe to remove
    return True

9. Practical tips, pitfalls, and extensions

  • Empty string: decide if you allow storing the empty string as a valid key — treat root.is_end accordingly.
  • Character normalization: ensure consistent case handling (lowercase everything) and handle unicode thoughtfully if needed.
  • Memory pools: for large tries, custom node pools and object reuse reduce allocation overhead and fragmentation.
  • Storing extra data: nodes can store counts (how many words use this prefix), indices of words, frequencies for ranking suggestions, or pointers to whole words to speed returns.
  • Concurrency: make updates thread-safe (locks or lock-free structures). For read-mostly workloads, copy-on-write or immutable/persistent tries can be beneficial.
  • Testing: check edge cases — repeated inserts, deleting non-existing words, prefixes that are also words, and large alphabets.
