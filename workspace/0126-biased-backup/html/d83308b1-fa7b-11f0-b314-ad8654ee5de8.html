Hash Table — A Comprehensive Explanation

This document explains the concept, theory, algorithms, and practical considerations behind hash tables. The emphasis is textual: definitions, intuitions, step-by-step examples, and well-documented pseudocode. A single, small demonstration button at the end shows a compact example illustrating insertion into two common collision-resolution strategies.

What is a Hash Table?

A hash table (or hash map) is a data structure that implements an associative array: it maps keys to values. The defining feature is that it uses a hash function to compute an index (or bucket) from a key, enabling expected constant-time (O(1)) average-time operations for insertion, lookup, and deletion.

Key Concepts and Vocabulary

  • Key: The item you use to look up a value (e.g., string, integer, tuple).
  • Value: The data associated with a key.
  • Hash function h(k): A deterministic function that transforms a key k into an integer (usually then mapped into a table index).
  • Bucket / slot: An array entry in the table where values (or lists of values) can be stored.
  • Collision: When two different keys produce the same table index; collision resolution strategies are required.
  • Load factor (α): Defined as the number of stored elements n divided by the number of buckets m: α = n / m. It measures table fullness.
  • Rehashing / resizing: When the table grows or shrinks and keys are re-inserted into a new array with a different size to maintain performance.

Goals of a Good Hash Function

A hash function for a general-purpose hash table should aim to:

  • Be deterministic: the same key always maps to the same hash value.
  • Distribute keys uniformly across buckets to minimize collisions.
  • Be fast to compute (constant time, minimal overhead).
  • Use the full input bits when possible (avoid ignoring significant bits).
  • Not be cryptographic unless strong adversarial resistance is required (cryptographic hashes are slow and unnecessary for typical hash tables).

Typical Hashing Process

Commonly, the process is two-stage: (1) compute a raw integer hash code h_raw = hashCode(key) (this may be large), then (2) apply a compression to map into array indices: index = compress(h_raw) (e.g., index = h_raw mod m or using multiplication).

Common Hash Function Methods

  • Division method: index = k mod m. Choose m prime (not too close to a power of two) to reduce poor distributions.
  • Multiplication method: index = floor(m * frac(k * A)) where 0 < A < 1 is a constant (Knuth recommends A ≈ (sqrt(5)-1)/2).
  • Universal hashing: Choose a hash function randomly from a family of functions to make collision behavior independent of the input distribution.
  • String/object hashing: Combine component hash codes (e.g., polynomial rolling hash) and then compress to the table size.

Collision Resolution Strategies

Collisions are unavoidable when the key space is larger than the table. Two broad strategies exist: separate chaining and open addressing.

Separate Chaining

Each table slot stores a pointer to a (linked) list of entries whose keys hash to that bucket. To insert, compute the bucket and prepend or append the item to the list. To search, traverse the list comparing keys.

Advantages:

  • Simple to implement.
  • Table can hold more than m items comfortably—load factor may exceed 1.
  • Deletion is straightforward: remove the node from the list.

Disadvantages:

  • Extra memory overhead for pointers per element.
  • Lists can become long if hashing is poor or the load factor is high.

Open Addressing

All elements are stored directly in the table array. When a collision occurs, a probing sequence (deterministic function of the key and table size) is used to find an empty slot. Common variants:

  • Linear probing: Try indices (h(k) + 0) mod m, (h(k) + 1) mod m, (h(k) + 2) mod m, ...
  • Quadratic probing: Use quadratic offsets: (h(k) + c1*i + c2*i^2) mod m; reduces primary clustering but may have secondary clustering and requires careful choice of constants and table size.
  • Double hashing: Use two hash functions: h(k,i) = (h1(k) + i * h2(k)) mod m. It tends to spread probes well when h2(k) is relatively prime to m.

Advantages:

  • Memory efficient: no extra pointers, good cache locality (adjacent elements are near in memory).

Disadvantages:

  • Deletion is trickier; one must use tombstones or rehash clusters.
  • Performance degrades quickly as the table becomes full (α close to 1).

Note the cluster of filled slots around indices 2–6. Linear probing causes "primary clustering"—long contiguous runs of occupied slots—and that affects future probe lengths.

Basic Pseudocode

Separate chaining

function insert_chaining(table, key, value):
    i = hash(key) mod table.size
    for (k,v) in table.buckets[i]:
        if k == key:
            v = value         // update existing key
            return
    table.buckets[i].push_front((key,value))  // or push_back

function search_chaining(table, key):
    i = hash(key) mod table.size
    for (k,v) in table.buckets[i]:
        if k == key:
            return v
    return NOT_FOUND

function delete_chaining(table, key):
    i = hash(key) mod table.size
    remove key from table.buckets[i] if present

Open addressing (linear probing)

function insert_linear(table, key, value):
    if table.is_full(): resize_or_error()
    i = hash(key) mod table.size
    j = i
    while table.slot[j] is occupied and table.slot[j].key != key:
        j = (j + 1) mod table.size
    table.slot[j] = (key,value)

function search_linear(table, key):
    i = hash(key) mod table.size
    j = i
    while table.slot[j] not empty:
        if table.slot[j].key == key:
            return table.slot[j].value
        j = (j + 1) mod table.size
        if j == i:
            break  // full cycle
    return NOT_FOUND

function delete_linear(table, key):
    find slot j containing key as in search_linear
    if not found: return
    remove key at j, mark as "deleted" tombstone (not plain empty)

Deletion and Tombstones

In open addressing, a simple removal that makes the slot empty can break searches for keys that were displaced past that slot. Two common approaches exist:

  • Use tombstones (a special marker meaning "previously occupied"): treat tombstones as occupied for probing but available for insertion. Over time tombstones accumulate; periodic rehashing or cluster rebuilding is needed.
  • Perform "back-shift" or re-insert elements from the cluster following the removed slot until an empty slot is encountered; this preserves probe chains but is more complex to implement.

Performance and Complexity

Key observations, assuming a good (randomizing) hash function:

  • Expected average-time complexity for successful lookup, insertion, and deletion is O(1) (constant time) when the load factor α is maintained below a suitable threshold.
  • Worst-case time is O(n) (e.g., all keys collide into a single bucket or adversarial inputs for a poor hash). Universal hashing or randomized hashing mitigates adversarial cases.
  • For chaining, average cost of a search is O(1 + α) because you may traverse a short list of expected length α in the bucket.
  • For open addressing, the expected probe count depends strongly on α and the probing strategy; as α → 1, operations can degrade dramatically. Linear probing exhibits primary clustering, which can make expected probes ≈ O(1/(1−α)^2) under some analyses; double hashing typically gives better behavior, often with expected probes ≈ O(1/(1−α)).

Resizing and Amortized Cost

To keep average operation time O(1), tables are typically resized (grown) when α exceeds a chosen threshold (commonly 0.5–0.75). Resizing involves allocating a new table (typically around double the size) and rehashing all n elements into it. Although a single resize is O(n) work, resizing occurs rarely: amortized cost per insertion remains O(1).

Practical Considerations and Trade-offs

  • Chaining: Better when the table needs to handle many elements or heavy deletes; simple to code and robust.
  • Open addressing: Better cache performance and memory compactness; choose when memory layout and locality matter.
  • Hash choice: For strings and compound keys, combine components carefully (e.g., polynomial mixing, bit-mixing). Avoid naïve hashing that only uses a subset of bits.
  • Security: If keys come from untrusted users, use randomized hashing or defenses against hash-flooding (e.g., switch to different hash function) to avoid denial-of-service by causing many collisions.
  • Libraries: Most modern languages provide well-tested hash table implementations; use them unless you have special needs.

Common Applications

  • Caches and memoization.
  • Implementing sets and maps (key-value stores).
  • Indexing by arbitrary keys (strings, composite objects).
  • Counting occurrences (histograms), symbol tables in compilers, and many algorithms that require quick membership tests.

Quick Reference: When to Use Which Strategy

  • Need maximum throughput, compact memory layout, and mostly insert/lookups: open addressing (with good hash and load factor control).
  • Need many deletions or expect load factor > 1 or variable sized values frequently: chaining.
  • Adversarial inputs possible: use randomized or universal hashing, or switch to alternatives with guaranteed bounds (balanced trees).
Note: In many educational settings, hash table behavior is studied under the uniform hashing assumption—that each key is equally likely to land in any bucket independently. Real-world hash functions aim to approximate this behavior for expected performance guarantees.

Concise Pseudocode: Double Hashing Probe Function

function double_hash_index(h1, h2, i, m):
    return (h1 + i * h2) mod m

// h2 should be non-zero and relatively prime to m to probe all slots.
