Merge Sort — A Comprehensive Explanation

A careful walkthrough of the algorithm, theory, proofs, examples, variations, and practical considerations for Merge Sort.

Overview — What is Merge Sort?

Merge Sort is a classic, comparison-based, divide-and-conquer sorting algorithm. Its essential strategy is:

  • Divide the array into two roughly equal halves.
  • Recursively sort each half.
  • Merge the two sorted halves into a single sorted result.

Because it divides the input and merges sorted subproblems, Merge Sort guarantees a time complexity of O(n log n) in the worst, average, and best cases (assuming a standard merge implementation).

Intuition and Why It Works

The power of Merge Sort comes from two simple ideas:

  • Divide-and-conquer: Break a big problem into smaller similar problems that are easier to solve.
  • Efficient merging: Two sorted sequences can be combined into one sorted sequence in linear time by repeatedly taking the smaller head element.

When you sort smaller pieces and reliably merge them, correctness follows by induction: if the recursion sorts halves correctly, merging yields a correctly sorted whole.

Algorithm (High-level and Pseudocode)

Here is a concise description and pseudocode for the classical top-down Merge Sort.

Top-down Merge Sort (recursive)

// mergesort(A, lo, hi) sorts A[lo..hi) (half-open interval)
function mergesort(A, lo, hi):
    if hi - lo <= 1:
        return               // zero or one element is already sorted
    mid = floor((lo + hi) / 2)
    mergesort(A, lo, mid)   // sort left half
    mergesort(A, mid, hi)   // sort right half
    merge(A, lo, mid, hi)   // merge the two sorted halves back into A

// merge A[lo..mid) and A[mid..hi) into one sorted A[lo..hi)
function merge(A, lo, mid, hi):
    left = copy of A[lo..mid)
    right = copy of A[mid..hi)
    i = 0    // index into left
    j = 0    // index into right
    k = lo   // index into A
    while i < length(left) and j < length(right):
        if left[i] <= right[j]:
            A[k] = left[i]; i = i + 1
        else:
            A[k] = right[j]; j = j + 1
        k = k + 1
    // copy any remaining elements
    while i < length(left):
        A[k] = left[i]; i = i + 1; k = k + 1
    while j < length(right):
        A[k] = right[j]; j = j + 1; k = k + 1

Note: The pseudocode uses <= in the comparison so that the algorithm is stable (equal elements are taken from the left subarray first).

The Merge Operation — Step-by-step

The merge operation is the heart of Merge Sort. Given two sorted lists, merging them into a single sorted list takes linear time in the sum of their lengths.

Why merging is linear-time

At each step you inspect the heads of the two lists and move the smaller of the two into the output. Each move places one element into the final array and increases a pointer. No element is moved more than once during the merge, so the total work is O(n).

Detailed example of merging

Merge [2, 5, 7] and [1, 3, 6, 8]:

  • Compare 2 and 1 → pick 1 → output [1]
  • Compare 2 and 3 → pick 2 → output [1,2]
  • Compare 5 and 3 → pick 3 → output [1,2,3]
  • Compare 5 and 6 → pick 5 → output [1,2,3,5]
  • Compare 7 and 6 → pick 6 → output [1,2,3,5,6]
  • Right list exhausted? not yet → append remaining left→7 → final [1,2,3,5,6,7,8] (then append remaining 8)

A Full Example (Detailed Walkthrough)

We'll sort the array [38, 27, 43, 3, 9, 82, 10] using top-down merge sort and explain each step in words.

Splitting phase (divide)

We repeatedly split until we reach subarrays of size 1.

  1. Original: [38, 27, 43, 3, 9, 82, 10]
  2. Split into left [38,27,43] and right [3,9,82,10]
  3. Left [38,27,43] splits into [38] and [27,43]. Then [27,43] splits into [27] and [43].
  4. Right [3,9,82,10] splits into [3,9] and [82,10]; [3,9] → [3] and [9]; [82,10] → [82] and [10].
Merging phase (conquer + combine)

Now we merge back up, sorting as we go.

  1. Merge [27] and [43] → [27,43]. Now left becomes [38] and [27,43].
  2. Merge [38] and [27,43]: compare 38 and 27 → output 27; compare 38 and 43 → output 38; then append 43 → [27,38,43].
  3. Right side: merge [3] and [9] → [3,9]; merge [82] and [10] → [10,82].
  4. Merge [3,9] and [10,82]: compare 3 and 10 → 3; 9 and 10 → 9; then append remaining [10,82] → [3,9,10,82].
  5. Finally merge left [27,38,43] and right [3,9,10,82]: take 3 (from right), then 9, then 10, then 27 (left), then 38, then 43, then remaining 82 → final sorted [3,9,10,27,38,43,82].

At each merge step, elements move at most once. The merge process is where the linear cost per level of recursion arises.

Initial array: [38, 27, 43, 3, 9, 82, 10]
Split 1: left=[38,27,43], right=[3,9,82,10]
Split left: [38] and [27,43] → [27] and [43]
Merge [27] & [43]: → [27,43]
Merge [38] & [27,43]: compare 38 vs 27 → 27; 38 vs 43 → 38; append 43 → [27,38,43]
Split right: [3,9] and [82,10] → [3] & [9]; [82] & [10]
Merge [3] & [9]: → [3,9]
Merge [82] & [10]: → [10,82]
Merge [3,9] & [10,82]: → [3,9,10,82]
Final merge [27,38,43] & [3,9,10,82]: → compare & assemble → [3,9,10,27,38,43,82]

Time and Space Complexity — Formal Analysis

Time complexity

Let T(n) be the time to sort n elements using Merge Sort. Merge Sort performs two recursive sorts on inputs of size n/2 and one merge step which takes Θ(n) time. So the recurrence is:

T(n) = 2 T(n/2) + Θ(n)

Applying the Master Theorem (or a recursion-tree argument) yields T(n) = Θ(n log n). Therefore:

  • Worst-case time: Θ(n log n)
  • Average-case time: Θ(n log n)
  • Best-case time: Θ(n log n) with the standard algorithm (if you implement a check that both halves are already in order before merging, you can improve some best-case inputs)

Space complexity

Standard Merge Sort requires O(n) additional space because typical implementations copy halves into temporary arrays during merge. More precisely:

  • Top-down recursive Merge Sort: O(n) auxiliary space (plus O(log n) recursion stack in languages where recursion depth counts separately).
  • Bottom-up iterative Merge Sort: still O(n) auxiliary space but avoids recursion stack overhead (except in languages that allocate temporaries differently).
  • There exist in-place merge algorithms (that use O(1) additional memory) but they are complex and usually slower due to additional element moves — uncommon in practice.

Why Merge Sort Is Correct (Sketch of Proof)

We prove correctness by induction:

  • Base case: arrays of size 0 or 1 are trivially sorted.
  • Inductive step: assume mergesort correctly sorts arrays smaller than n. For array of size n, mergesort splits into two halves of sizes < n, recursively sorts them (by induction hypothesis), then merge combines two sorted sequences into one sorted sequence. Since merge preserves ordering and leaves no elements behind, the result is a fully sorted array of size n.

Thus, by induction, Merge Sort sorts every finite array correctly.

Stability, Comparisons, and Practical Notes

Stability

Merge Sort can be implemented as a stable sort: when two items compare equal, the implementation can consistently prefer the left element (or the earlier element) so that original relative order of equal keys is preserved. Stability is useful when sorting complex records by one key while preserving previous orderings.

Number of comparisons

Merge Sort does roughly n log2 n comparisons in the average case. The exact number varies but is bounded between n log2 n and n log2 n + O(n).

When to use Merge Sort?

  • When stable sort is required and n is large.
  • When worst-case O(n log n) performance is necessary (unlike Quick Sort's worst-case O(n^2)).
  • External sorting: Merge Sort variants (external merge sort) are the method of choice when data does not fit in memory because they can merge sorted runs efficiently from external storage.
  • Parallelizable: merging phases can be parallelized; divide-and-conquer nature maps well to parallel hardware.

When not to use Merge Sort?

  • When in-place sorting with minimal memory is required and stability is not important, Quick Sort or Heap Sort might be preferable.
  • For tiny arrays (< 30 elements), simple algorithms like insertion sort are often faster due to lower constant overhead.

Variations and Practical Optimizations

Bottom-up (iterative) Merge Sort

Instead of recursion, repeatedly merge runs of length 1, then 2, then 4, etc. This eliminates recursion overhead and is often used for iterative implementations.

Hybrid approaches

Many practical implementations use Merge Sort but switch to Insertion Sort for small subarrays (e.g., size < 16). This reduces overhead and takes advantage of insertion sort's efficiency on small inputs.

External Merge Sort

When data doesn't fit in RAM, we first create sorted "runs" (e.g., by reading chunks into memory, sorting them, and writing them out). Then we perform k-way merges of these runs, possibly in multiple passes. External merge sort is I/O efficient and widely used in databases and file sorting.

In-place merging (advanced)

There are algorithms to merge in-place with O(1) extra space, but they usually involve complex rotations or block-swapping and have worse constants; so in many practical contexts, the O(n) auxiliary array is preferred for simplicity and speed.

Reference Implementations and Tips

JavaScript (concise, functional style)

function mergeSort(arr) {
  if (arr.length <= 1) return arr;
  const mid = Math.floor(arr.length / 2);
  const left = mergeSort(arr.slice(0, mid));
  const right = mergeSort(arr.slice(mid));
  return merge(left, right);
}

function merge(left, right) {
  const out = [];
  let i = 0, j = 0;
  while (i < left.length && j < right.length) {
    if (left[i] <= right[j]) out.push(left[i++]);
    else out.push(right[j++]);
  }
  return out.concat(left.slice(i)).concat(right.slice(j));
}

Python (classic)

def merge_sort(a):
    if len(a) <= 1:
        return a
    mid = len(a) // 2
    left = merge_sort(a[:mid])
    right = merge_sort(a[mid:])
    return merge(left, right)

def merge(left, right):
    i = j = 0
    out = []
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            out.append(left[i]); i += 1
        else:
            out.append(right[j]); j += 1
    out.extend(left[i:])
    out.extend(right[j:])
    return out

Practical tips

  • Avoid excessive array copying by using a single temporary buffer and alternating roles or by implementing merges that write into a work array and copy back.
  • Use insertion sort for very small partitions inside recursion to reduce overhead.
  • Be mindful of recursion depth for very large arrays — bottom-up avoids deep recursion.

Comparison with Other Sorting Algorithms

  • Merge Sort vs Quick Sort: Quick Sort typically faster on average and uses less additional memory (in-place), but has worst-case O(n^2) without precautions. Merge Sort is stable and guarantees O(n log n).
  • Merge Sort vs Heap Sort: Heap Sort has O(n log n) worst-case and uses O(1) extra space, but is not stable and often slower in practice due to cache inefficiency. Merge Sort often has better constant factors for real-world data and is stable.
