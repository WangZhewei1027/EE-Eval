Transactions — A Comprehensive Educational Guide

Definitions, theory, algorithms, examples, and recovery strategies for transactional systems (database and distributed contexts). Designed as a textual, pedagogical resource.

What is a Transaction?

A transaction is a logical unit of work comprising one or more operations on data that must be executed as a single, indivisible action from the perspective of the system. Transactions appear to users/applications as atomic: they either complete fully (commit) or have no effect at all (abort/rollback). Transactions enforce correctness, consistency, isolation, and durability properties that are essential for reliable data management.

Common contexts

  • Relational databases (SQL transactions).
  • NoSQL systems that support transactions (key-value, document stores).
  • Distributed systems coordinating across multiple services/datastores.
  • Application-level logical transactions (sagas or compensating actions).

ACID Properties (Core Guarantees)

ACID summarizes the key database transaction properties:

  • Atomicity: The whole transaction succeeds or fails. Partial effects are not visible.
  • Consistency: A transaction moves the database from one valid state to another, preserving integrity constraints (schema rules, foreign keys, invariants).
  • Isolation: Concurrent transactions do not interfere; each transaction behaves as if it executed alone, up to a chosen isolation level.
  • Durability: Once a transaction commits, its effects persist despite subsequent crashes (typically by writing to stable storage or using replication).

ACID is a conceptual model. Practical systems implement these properties with trade-offs (performance, availability, and concurrency).

Transaction Lifecycle and States

A typical transaction progresses through states:

  • Active / Started: The transaction has begun and is performing operations.
  • Partially Committed: The transaction has finished its final operation, but changes may not be durable yet.
  • Committed: The transaction's effects are made durable and visible.
  • Aborted / Rolled Back: The transaction's effects have been undone; the database returns to the pre-transaction state.

Typical operations in a transaction:

  1. BEGIN TRANSACTION
  2. READ/WRITE/UPDATE statements
  3. COMMIT or ROLLBACK

Example: A Simple Bank Transfer (Step-by-step)

Consider transferring $100 from account A to B. A safe transaction ensures neither account ends up inconsistent even if failures occur.

Naive steps (conceptual)

  1. BEGIN TRANSACTION
  2. Read balance(A)
  3. If balance(A) < 100 then ROLLBACK (insufficient funds)
  4. Write balance(A) := balance(A) - 100
  5. Read balance(B)
  6. Write balance(B) := balance(B) + 100
  7. COMMIT

Crucial guarantees: the deduction and addition must both happen or neither. If the system crashes after step 4 but before step 6, recovery logic must either finish the transfer or undo the partial change.

What can go wrong without transactions?

  • Partial updates: money disappears (deduction without credit) or is duplicated.
  • Concurrent transfers causing lost updates if reads/writes overlap.

Isolation Levels and Anomalies

Isolation controls the visibility of concurrent transactions. Stronger isolation eliminates more anomalies but reduces concurrency.

Common isolation levels

  • Read Uncommitted: Allows dirty reads—transactions see uncommitted changes of others.
  • Read Committed: Prevents dirty reads; a transaction only sees committed changes. Non-repeatable reads and phantoms may occur.
  • Repeatable Read: Guarantees repeated reads of the same data within a transaction return the same results (prevents non-repeatable reads). Phantoms may still occur in some implementations.
  • Serializable: Strongest level; concurrent transactions appear to run in some serial order. Prevents all classical anomalies.

Classic anomalies (short descriptions)

  • Dirty Read: T1 reads data written by T2 that later rolls back.
  • Non-Repeatable Read: T1 reads the same row twice and sees different values because T2 committed an update between reads.
  • Phantom Read: T1 re-executes a range query and sees additional rows inserted by T2.

Concurrency Control Algorithms

Concurrency control (CC) enables safe concurrent execution. Major categories:

Two-Phase Locking (2PL)

2PL is a locking protocol with two phases:

  1. Growing phase: transaction acquires all locks (shared for reads, exclusive for writes).
  2. Shrinking phase: transaction releases locks and cannot acquire new ones afterwards.

Strict 2PL: holds exclusive locks until commit—this ensures recoverability and serializability.

// Simplified pseudocode: Strict 2PL (participant perspective)
begin_transaction(T):
  T.lockSet = {}
  // operations...
read(T, x):
  acquire_shared_lock(T, x)
  T.lockSet.add(shared(x))
  return x.value
write(T, x, v):
  acquire_exclusive_lock(T, x)
  T.lockSet.add(exclusive(x))
  x.temp = v  // write to private workspace or buffer
commit(T):
  flush_writes_from_private_workspace(T)
  release_all_locks(T)  // in strict 2PL, exclusive locks released here
abort(T):
  discard_private_writes(T)
  release_all_locks(T)

Pros: straightforward, ensures serializability. Cons: can block and lead to deadlocks; reduced parallelism.

Timestamp Ordering

Assign a unique increasing timestamp to each transaction. Ordering rules ensure transactions appear according to timestamps. If a transaction tries to perform an action that violates timestamp order, it aborts and restarts. 

// Simplified view of timestamp ordering rules
each data item x stores:
  read_ts(x) = max timestamp of transaction that read x
  write_ts(x) = max timestamp of transaction that wrote x

on read(T, x):
  if write_ts(x) > T.ts:
    abort T (T is too late - would see future write)
  else:
    read_ts(x) = max(read_ts(x), T.ts)

on write(T, x):
  if read_ts(x) > T.ts or write_ts(x) > T.ts:
    abort T (write would violate serial order)
  else:
    write_ts(x) = T.ts
    write x

Pros: non-blocking. Cons: can cause many aborts under contention and requires restarts.

Multi-Version Concurrency Control (MVCC)

MVCC stores multiple versions of data items. Reads can access a snapshot consistent with a transaction's timestamp without blocking writers. Writers create new versions. This enables high read concurrency and reduces blocking. 

// High-level MVCC idea
on begin(T):
  T.read_ts = current_global_ts  // snapshot

on read(T, x):
  return version of x with largest commit_ts <= T.read_ts

on write(T, x):
  create private new version (uncommitted)
commit(T):
  assign commit_ts
  make private versions visible with commit_ts

Pros: excellent read throughput, less blocking. Cons: storage overhead (garbage collection of versions) and complexity for write-write conflicts.

Recovery Techniques

Recovery ensures durability and atomicity across crashes by using logs, checkpoints, undo/redo strategies, and structured algorithms like ARIES.

Write-Ahead Logging (WAL)

WAL requires writing log records describing changes to durable storage before applying the corresponding changes to the database files. Logs contain enough information to undo or redo operations. 

// WAL rule (conceptual)
for each update U by transaction T:
  write_log_record(  )  // durable
  apply_update_to_database_buffers(U)
  // checkpoint later flushes buffers to disk

Undo and Redo

  • Undo log: log old values so we can revert uncommitted changes on crash.
  • Redo log: log new values so we can reapply committed changes that haven't been persisted yet.

ARIES (Advanced Recovery)

Key ideas at a high level:

  • Write-ahead logging with physiological logging (logical + physical mix).
  • Repeating history during analysis and redo phases to bring database to a state where undo can be applied consistently.
  • Selective undo of transactions that were active at crash time.

ARIES is complex and beyond a short snippet but conceptually provides efficient, fine-grained recovery for modern DBMS.

Distributed Transactions

Distributed transactions coordinate multiple participants (databases, services) so that a global operation commits or aborts atomically. They are necessary when a single logical operation spans multiple systems.

Two-Phase Commit (2PC)

2PC is the classic protocol with two roles: coordinator and participants.

  1. Phase 1 — Prepare: Coordinator asks all participants to prepare (vote). Each participant attempts to get necessary local locks and writes prepare log; it replies VOTE_COMMIT or VOTE_ABORT.
  2. Phase 2 — Commit/Rollback: If all vote commit, coordinator sends COMMIT; otherwise sends ABORT. Participants commit or roll back and acknowledge.
// Coordinator pseudocode
coordinator_begin_global_transaction(G):
  send PREPARE to all participants
  if all reply VOTE_COMMIT:
    send GLOBAL_COMMIT
  else:
    send GLOBAL_ABORT

Problems with 2PC: it can block if the coordinator fails after participants vote commit but before sending the global decision. It is blocking but ensures atomicity across systems.

Three-Phase Commit (3PC)

3PC tries to be non-blocking by introducing an additional phase to avoid the blocking window; it is more complex and still has limitations under network partitions and asynchrony.

Saga Pattern (Long-Running Business Transactions)

Sagas break a long-running transaction into a sequence of local transactions. Each local transaction has a compensating action to undo it. If one step fails, compensating transactions are executed to revert prior steps. 

// Example saga for two-step distributed transfer
step1: debit service A (local commit)
step2: credit service B (local commit)
if step2 fails:
  run compensate_step1: credit A (compensating action)

Sagas trade strong atomicity for availability and practicality in distributed environments. Compensating actions must be carefully designed and may not always perfectly invert the original action (idempotency and retries matter).

Algorithms Summary & Decision Guide

Choose a concurrency and transaction approach based on workload characteristics:

  • Heavy reads, few writes: MVCC for high read throughput.
  • High contention on same keys: careful application design to avoid hotspots; optimistic approaches may cause many aborts.
  • Distributed cross-service changes: avoid 2PC unless necessary; prefer eventual consistency, sagas, or compensating protocols when possible.
  • Need strict serializability: use strong CC (strict 2PL or serializable snapshot isolation implementations).

Developer Best Practices

  • Keep transactions short: minimize time holding locks or resources.
  • Avoid user interactions inside transactions (no long waits).
  • Retry idempotently: design operations to be safe under retries (use unique request IDs or idempotence keys).
  • Handle transient failures gracefully: exponential backoff, limit retry attempts, make compensations explicit.
  • Explicitly set isolation levels when the default is not appropriate for correctness or performance.
  • Test for anomalies: create tests simulating concurrency to uncover race conditions and anomalies (dirty reads, lost updates).

Common anti-patterns

  • Long-lived transactions that lock many rows (causes contention and blocking).
  • Mixing transactions across many distributed services without fallbacks (2PC complexity).
  • Assuming reads happen at a single instant in time when using weak isolation.

Glossary (Quick Reference)

Commit: Make changes permanent.
Rollback/Abort: Cancel changes.
Coordinator: Entity managing a distributed commit.
Participant: A local system inside a distributed transaction.
Lock: Mechanism to control access to data (shared/exclusive).
Snapshot: A consistent view of the database at a point in time.

Pseudocode: Two-Phase Commit and Participant Behavior

Coordinator and participant roles condensed into pseudocode to help study interactions and failure points.

// Participant pseudocode (brief)
on PREPARE:
  if local_preconditions_hold and can_acquire_resources:
    write_log("VOTE_COMMIT")
    reply VOTE_COMMIT
  else:
    write_log("VOTE_ABORT")
    reply VOTE_ABORT

on GLOBAL_COMMIT:
  write_log("COMMIT")
  apply_prepared_changes
  release_resources

on GLOBAL_ABORT:
  write_log("ABORT")
  undo_prepared_changes
  release_resources

// Coordinator pseudocode (brief)
start_2pc(global_tx):
  for p in participants:
    send PREPARE to p
  gather votes
  if all VOTE_COMMIT:
    send GLOBAL_COMMIT to all
  else:
    send GLOBAL_ABORT to all

Study the window between participants voting commit and receiving GLOBAL_COMMIT — coordinator failure there leads to blocked participants waiting for a decision.
