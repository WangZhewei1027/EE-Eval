Hash Map — Comprehensive Explanation

This page provides a thorough, text-first explanation of hash maps (also called hash tables or associative arrays). It covers definitions, core algorithms, implementation details, performance, common variants, practical pitfalls, and small illustrative examples. What is a Hash Map?

A hash map is a data structure that implements an associative array: it stores (key, value) pairs and allows efficient operations to insert a pair, retrieve a value given a key, and remove a key. The hallmark of a hash map is that it uses a hash function to map arbitrary keys to indexes in an internal array (often called buckets). When designed well, common operations — lookup, insertion, and deletion — run in expected constant time, O(1), independent of the number of stored items.

Core Concepts and Definitions

  • Key: an object or value used to look up data.
  • Value: the stored data associated with a key.
  • Hash function: a deterministic function that maps a key to an integer (the hash code). This is then reduced (usually via modulo) to an index in the bucket array.
  • Bucket / slot: an index or location in the internal array where items with the same target index are placed.
  • Collision: when two distinct keys map to the same bucket index.
  • Load factor (α): ratio of number of stored entries n to number of buckets m (α = n/m). It measures fullness and guides resizing.

How a Hash Map Works (High-Level)

On insertion: compute hash = hash_function(key), reduce to index = hash modulo m (or another reduction technique), store the (key,value) pair in bucket[index]. If bucket already contains other entries, use a collision-resolution strategy.

On lookup: compute index the same way, inspect bucket[index], find the entry whose key is equal to the searched key (equality comparison is required), and return its value if found.

Collision Resolution Strategies

Collisions are inevitable; how you handle them defines the hash map variant.

Separate chaining

Each bucket holds a linked list (or another container) of entries. When collisions occur, new entries are appended to the list at their bucket. Search walks the list and compares keys.

  • Pros: simple, handles high load factors gracefully.
  • Cons: requires extra memory for node pointers; poor cache locality if chains are long.
  • Average lookup: O(1 + α) where α = n/m.

Open addressing

All entries live in the bucket array itself. On collision, probe a sequence of other indices until an empty slot is found. Popular probing strategies:

  • Linear probing: probe i+1, i+2, ... (wrapping). Simple, good cache locality, but clustering can form (primary clustering).
  • Quadratic probing: probe i+1^2, i+2^2, ... reduces primary clustering but can produce secondary clustering and requires care with table size.
  • Double hashing: use a second hash to compute the probe step; typically offers better distribution and avoids clustering.

Open addressing requires careful handling of deletions (use tombstones or rehash) and generally needs α to remain below a threshold (e.g., 0.5–0.75) for good performance.

Advanced variants

  • Cuckoo hashing: keys can be in one of several candidate positions; insertion may evict existing keys, causing a chain of moves; guarantees constant worst-case lookup with small probe count.
  • Hopscotch hashing, Robin Hood hashing: improve probe distributions and cache behavior for open addressing.
  • Consistent hashing: used across distributed nodes to map keys to machines with minimal reassignments when machines change.

Operations: Step-by-Step (Pseudocode and Narrative)

Separate chaining — Insert (step-by-step)

function insert(hashmap, key, value):
  h = hash_function(key)
  index = reduce(h, hashmap.bucket_count)   // e.g., index = h mod m
  bucket = hashmap.buckets[index]
  for entry in bucket:
    if entry.key == key:
      entry.value = value
      return
  // key not found; append new entry
  bucket.append( (key, value) )
  hashmap.size += 1
  if hashmap.size / hashmap.bucket_count > hashmap.load_factor_threshold:
    resize_and_rehash(hashmap)

Notes: The equality test (entry.key == key) must match the same equality semantics used for hashing (i.e., keys that compare equal must produce equal hashes).

Separate chaining — Search

function get(hashmap, key):
  h = hash_function(key)
  index = reduce(h, hashmap.bucket_count)
  for entry in hashmap.buckets[index]:
    if entry.key == key:
      return entry.value
  return NOT_FOUND

Open addressing — Insert (linear probing, simple)

function insert_open(hashmap, key, value):
  h = hash_function(key)
  index = reduce(h, hashmap.bucket_count)
  i = index
  while true:
    if hashmap.slots[i] is EMPTY or hashmap.slots[i] is TOMBSTONE:
      hashmap.slots[i] = (key, value)
      hashmap.size += 1
      return
    if hashmap.slots[i].key == key:
      hashmap.slots[i].value = value
      return
    i = (i + 1) % hashmap.bucket_count
    if i == index: error "table full"

Delete in open addressing

Mark slot as a tombstone (deleted-but-occupied) to avoid breaking probe chains. Periodic rehashing cleans up tombstones.

Complexity and Amortized Analysis

- Average-case time for lookup/insert/delete is O(1) assuming a good hash function and load factor control. Separate chaining gives O(1 + α) where α is load factor. Open addressing costs grow as α approaches 1 (probe length increases).

- Worst-case time is O(n) when many keys collide to the same bucket or when a deliberately bad hash function (or adversarial inputs) causes chains or long probe sequences. This motivates good hashing and precautions against adversarial inputs (see security).

- Resizing (growing) an array-based hash map costs O(n) to reinsert all items, but if done by doubling the table size when a threshold is reached, the amortized cost per insertion is O(1). Briefly: doubling occurs infrequently; the total work over many inserts sums to O(n) extra work spread over n inserts.

Choosing a Hash Function

The hash function must be deterministic, fast, and distribute inputs uniformly across the output space. It is common to:

  • For integers: use the integer value mixed with bit operations to spread bits.
  • For strings: combine characters using multipliers and arithmetic (e.g., polynomial rolling), or use well-known non-cryptographic algorithms like MurmurHash or CityHash for speed.
  • For compound objects: combine the fields' hashes (e.g., pair hashing: h = mix(h1, h2)).

Important: A cryptographic hash (like SHA) is usually unnecessary and much slower. Use non-cryptographic hashes optimized for speed and quality unless you require cryptographic properties.

Equality and Hash Contracts

For a hash map to work correctly, the following must hold:

  • If key1.equals(key2) is true, then hash(key1) must equal hash(key2).
  • If two keys are not equal, their hashes may still collide; the data structure must handle collisions by comparing keys on equality when checking bucket entries.

In languages like Java, this is the contract between equals() and hashCode(). In Python, objects define __hash__() and __eq__().

Practical Considerations and Pitfalls

  • Mutable keys: if keys can change in a way that affects their hash or equality after insertion, the map will misbehave. Typically keys should be immutable (strings, numbers, tuples of immutables, etc.).
  • Poor hash functions: if many keys map to the same bucket, performance degrades. Use a well-tested hash implementation.
  • Equality mismatch: if your equality and hashing logic disagree, lookups fail.
  • Memory overhead: buckets with chaining require extra memory per node; open addressing reduces overhead but requires a lower maximum load factor.
  • Iteration order: most hash maps do not guarantee stable ordering. Some variants or implementations (e.g., Java LinkedHashMap, Python 3.7+ dict) provide deterministic insertion order as an added feature.

Security: Hash Collision Attacks

Hash-flooding attacks supply many keys that intentionally collide under a hash function, forcing the map into worst-case behavior and causing O(n) per operation. Mitigations include:

  • Randomized hashing: include a per-process or per-map random seed so that attackers cannot predict collisions.
  • Use of robust hash algorithms for high-risk inputs.
  • Hybrid or guaranteed-bounded strategies: e.g., switch to balanced tree buckets (as Java 8 does for long chains) or other mechanisms to guarantee O(log n) worst-case operations.

Language-Specific Notes (Short)

  • Java: HashMap uses separate chaining with an array of bins; since Java 8, bins are converted to balanced trees (red-black trees) when chains become long, giving O(log n) worst-case lookup for pathological cases.
  • Python: Python's dict is an open-addressed hash table optimized for speed and memory and preserves insertion order (as of Python 3.7). It uses randomized hashing for strings to mitigate hash-flooding.
  • C++: unordered_map uses buckets with separate chaining (implementation-dependent), with custom hash and equality functors available.

When to Use a Hash Map

Use a hash map when you need fast average-case key-based lookup, insertion, and deletion without an ordering constraint. Alternatives:

  • Use balanced binary search trees (e.g., TreeMap) when you need ordered keys or guaranteed logarithmic worst-case behavior.
  • Use arrays or vectors for dense integer keys within a small range.

Complete Example — Step-by-Step Narrative

Suppose we maintain a hash map with m = 7 buckets and use index = (hash(key) mod 7). We will insert the keys: "cat", "dog", "bird", "cow".

  1. Compute hash("cat") = H1 (assume H1 mod 7 = 3). Place ("cat", value) in bucket 3.
  2. Compute hash("dog") = H2 (assume H2 mod 7 = 5). Place ("dog", value) in bucket 5.
  3. Compute hash("bird") = H3 (assume H3 mod 7 = 3). Bucket 3 already has ("cat", ...). If using chaining, append ("bird", ...). If using open addressing, probe to next free slot.
  4. Lookup("bird"): compute index 3, inspect bucket 3 and find ("bird", ...).
  5. Delete("cat"): remove or mark as deleted; ensure future lookups for "bird" still succeed.
The above narrative demonstrates why equality checks are required even when two keys share the same bucket index.

Pseudocode: Simple Resizing Strategy

function resize_and_rehash(hashmap):
  old_buckets = hashmap.buckets
  new_size = next_prime_or_double(old_buckets.length * 2)
  hashmap.buckets = allocate_array(new_size)
  hashmap.size = 0
  for bucket in old_buckets:
    for entry in bucket:
      insert(hashmap, entry.key, entry.value)

Advanced Topics (Brief)

Cache behavior

Open addressing usually has better cache performance because entries are stored in a contiguous array; chaining incurs pointer chasing. However, open addressing becomes slow when the table is very full.

Memory layout

Decisions about storing key and value in the bucket, or storing pointers to heap-allocated nodes, impact memory usage and allocation overhead.
