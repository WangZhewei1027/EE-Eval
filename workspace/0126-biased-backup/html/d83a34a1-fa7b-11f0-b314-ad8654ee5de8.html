Introduction & Key Ideas

Garbage collection (GC) is the automatic reclaiming of memory that a program no longer needs. It frees programmers from manual memory deallocation (free/delete) and helps reduce memory-related bugs such as use-after-free and many kinds of leaks, but it introduces its own costs and complexities.

Core concepts

  • Object lifetime: the period from allocation to the point when no program-accessible reference remains.
  • Reachability: if an object can be accessed by the program through some chain of references starting from a root, it is considered live.
  • Root set: places where references can originate (stacks, registers, global/static variables, thread-local handles).
  • Collector: the component that identifies unreachable objects and reclaims their memory.

Note: "Garbage" here means memory that the program can no longer reach; it is not necessarily "trash data" in the colloquial sense.

Roots, Reachability, and the Object Graph

Visualize the heap as a directed graph: nodes are objects, edges are pointers/references. The GC's primary job is to determine which nodes are reachable from the root set.

Root set examples

  • Stack frames: local variables and temporary values.
  • CPU registers: values currently in use.
  • Global and static variables.
  • Handles the runtime exposes: JNI handles (Java), GC handles, or other runtime-managed references.

Reachability definition (informal)

An object is live if there exists a path of references from any root to that object. Any object not live is garbage and may be reclaimed.

Core Algorithms

There are multiple families of GC algorithms. Each has different performance characteristics, strengths, and weaknesses. We'll examine the most important ones:

Reference Counting

Every object maintains a counter of how many references point to it. When a reference is created, the count increments; when a reference is overwritten or removed, the count decrements. When the count reaches zero, the object can be reclaimed immediately.

Advantages

  • Deterministic reclamation: objects are collected immediately when the last reference is removed.
  • Simple to implement and incremental by nature.

Disadvantages

  • Overhead: additional memory for counters and time to update counts on every reference write.
  • Cyclic garbage: objects that reference one another can form a cycle and never reach zero even if unreachable from roots.

Cycle handling

To handle cycles, systems augment reference counting with a cycle detector (e.g., Python's cyclic GC), or use an alternative counting scheme like trial deletion or deferred counting.

// Simplified reference counting pseudocode
refcount_increment(obj)
  obj.count += 1

refcount_decrement(obj)
  obj.count -= 1
  if obj.count == 0:
    for child in obj.references:
      refcount_decrement(child)
    free(obj)

Mark-and-Sweep (Tracing Collectors)

Mark-and-sweep is a classic tracing collector. It works in two phases:

  1. Mark phase: starting from roots, recursively mark every reachable object.
  2. Sweep phase: walk the heap and reclaim objects that aren't marked.

Properties

  • Correctly handles cycles (tracing collectors don't rely on counters).
  • Can require pausing the application (stop-the-world) during the collection, unless implemented incrementally or concurrently.
  • May cause heap fragmentation unless compaction is performed.

Simplified pseudocode

function mark_and_sweep()
  // 1. Unmark all objects
  for obj in heap: obj.marked = false

  // 2. Mark reachable objects
  worklist = list(roots)
  while worklist not empty:
    obj = worklist.pop()
    if not obj.marked:
      obj.marked = true
      for child in obj.references:
        worklist.push(child)

  // 3. Sweep unmarked objects
  for obj in heap:
    if not obj.marked:
      free(obj)

Copying / Semispace Collectors

Divide the heap into two semispaces: "from" and "to". Allocation occurs only in "from". When "from" is full, the collector copies live objects from "from" to "to" and leaves behind garbage. Afterward, the roles swap.

Advantages

  • Compaction is implicit: live objects are copied close together, eliminating fragmentation.
  • Allocation can be extremely fast (bump pointer).

Disadvantages

  • Requires roughly double the memory for semispaces.
  • Overhead of copying live objects.
// Copying collector (simplified)
function collect_semispace():
  tospace.clear()
  scan = tospace.start
  for root in roots:
    root.ptr = copy_if_needed(root.ptr)
  while scan < tospace.end:
    for child in scan.object.references:
      child.ptr = copy_if_needed(child.ptr)
    scan = scan.next
  swap(fromspace, tospace)

Generational Garbage Collection

Based on the empirical observation known as the generational hypothesis: most objects die young (short-lived), while older objects tend to live longer. The heap is partitioned into generations (young/eden, survivor spaces, old/tenured). Collections focus on the young generation more frequently and occasionally promote surviving objects to older generations.

Key mechanisms

  • Minor collection: collects the young generation frequently (fast, short pause).
  • Major/Full collection: collects the entire heap less frequently.
  • Remembered sets / card tables: track references from old to young to allow correct scanning without scanning the whole old generation during a minor collection.

Example flow

  1. Allocate in Eden with a bump pointer.
  2. When Eden fills, copy live objects to survivor spaces.
  3. After enough survivor passes, promote to tenured generation.

Incremental & Concurrent Collectors

To reduce pause times, collectors can be incremental (interleave collector work with mutator work) or concurrent (collector runs in parallel with mutator threads). These techniques require careful synchronization and usually write/read barriers to maintain correctness.

Write and read barriers

Barriers are small pieces of code inserted where the program writes or reads pointers. For example:

  • A write barrier might record the fact that an old object now points to a young object (needed for generational GC).
  • Barriers are essential in concurrent collectors to handle "mutator" modifications that occur while collector threads are scanning.

Examples of advanced collectors

  • Concurrent Mark-Sweep (CMS) — reduces pauses but can fragment memory.
  • G1 (Garbage-First) — region-based, concurrent marking and evacuation to meet pause-time goals.
  • Shenandoah/ZGC — low-pause concurrent collectors that use regional compaction and sophisticated barriers.

Finalization, Weak References, and Determinism

Finalizers or destructors are language features invoked when an object is collected (e.g., Java finalizers, Python __del__). They complicate GC semantics and often are discouraged because they:

  • Make object reclamation non-deterministic — you don't know when the finalizer will run.
  • Can resurrect objects by re-establishing references, making collection more complex.
  • Often require special handling (finalizer queues, separate threads).

Reference types

Languages often offer different reference strengths:

  • Strong references: prevent collection while reachable.
  • Weak references: do not prevent collection; useful for caches.
  • Soft references: (in some runtimes) collected when memory is low — useful for memory-sensitive caches.
  • Phantom references: appear enqueued after finalization but before memory is reclaimed — advanced use cases.

Allocation Strategies & Fragmentation

Popular allocation techniques

  • Bump pointer allocation: very fast, move pointer forward; used with copying or contiguous arenas.
  • Free lists: keep linked lists of free blocks for different sizes; used in more general-purpose allocators.
  • Slab/page allocators: allocate fixed-size objects from pre-allocated pages for low fragmentation.

Fragmentation

Fragmentation occurs when free memory is split into many small blocks that are hard to use for large allocations. Copying collectors compact memory by copying live objects to contiguous areas; mark-and-compact collectors perform compaction after marking. Compaction can be expensive and complicated with pinned objects (objects that must not be moved, e.g., by native code pointers).

Performance, Tuning & Metrics

Key metrics

  • Throughput: fraction of time spent doing application work vs GC work.
  • Pause time: how long application threads are paused for collection (important for latency-sensitive apps).
  • Memory footprint: total memory used by heap and collector metadata.

Trade-offs

Many GC design decisions trade throughput for latency or space for speed. For example:

  • More frequent minor collections can reduce pause times but increase overhead.
  • Concurrent collectors lower pause times but need barriers and extra CPU resources.
  • Copying collectors reduce fragmentation but require extra space.

Tuning tips

  • Choose appropriate heap size; too small increases collections, too large increases pause time for full collections.
  • Profile allocation patterns: excessive allocation of short-lived objects favors generational collectors.
  • Use appropriate collector for workload: latency-sensitive (use low-pause or concurrent collectors), throughput-sensitive (use collectors that maximize mutator time).

Real-world Implementations and Notes

Here are some representative systems and what they use:

  • JVM (HotSpot): multiple collectors (Serial, Parallel, CMS legacy, G1, ZGC, Shenandoah) — generational, region-based, concurrent, and low-pause options.
  • .NET CLR: generational (0,1,2) with a large-object heap (LOH) for big allocations; supports server/workstation modes.
  • V8 (Chrome): generational with compacting / concurrent features and incremental marking, uses precise pointer maps for stacks/frames.
  • Go: concurrent, non-moving (historically), mark-and-sweep with concurrent mark and sweep and write barriers — designed for low latency.
  • Python (CPython): reference counting + periodic generational cycle detector.
  • PyPy: tracing/generational collector (JIT-friendly).

Important: Different languages and runtimes have different guarantees. For example, Go's collector is designed to minimize pause times but historically does not move objects (until newer versions added experimental options). Java's G1 and ZGC are regional and aim for short pauses even for large heaps.

Pseudocode, Examples, and Step-by-step Narratives

Example: Reference counting vs garbage cycle

Imagine two objects that reference each other but are no longer reachable:

// Objects A and B refer to each other; no roots refer to them
A.references = [B]
B.references = [A]

// Reference counts remain nonzero: both point to each other
// Reference counting alone will not reclaim them

A tracing collector (mark-and-sweep) will correctly identify that neither A nor B are reachable from roots and reclaim both.

Step-by-step: Mark-and-sweep narrative

  1. Pause application threads (in a stop-the-world collector).
  2. Walk roots: push all objects referenced directly by roots onto a worklist.
  3. Pop objects from the worklist; if not already marked, mark them and push their children.
  4. Continue until worklist empty — all reachable objects are marked.
  5. Sweep heap: for every object, if not marked, reclaim memory and possibly run finalizers; if marked, unmark for next collection or update metadata.
  6. Resume application threads.

Example pseudocode: Generational minor collection

// Allocator puts objects in Eden
when eden_full():
  // copy surviving objects to survivor space
  for obj in eden.live_objects():
    obj.survivor_age += 1
    if obj.survivor_age >= threshold:
      promote_to_old(obj)
    else:
      copy_to_survivor(obj)

  // clear eden and continue allocating

Write barrier (example concept)

When mutator writes a pointer from object P to object Q, a write barrier might run:

// generational write barrier (example)
on_pointer_write(P, old, new):
  if P.generation == old_gen and new.generation == young_gen:
    card_table.mark(P) // record that P contains reference to a young object
  store_pointer(P, new)

Complexity discussion (informal)

  • Mark-and-sweep: work proportional to live objects + (in a naïve sweep) all heap objects during sweep. Average cost dominated by live set size.
  • Copying: cost proportional to live objects copied (good if live set small relative to heap).
  • Reference counting: O(1) per write/update, but can incur cascading updates on deallocation (when decrementing triggers recursive decrements).
