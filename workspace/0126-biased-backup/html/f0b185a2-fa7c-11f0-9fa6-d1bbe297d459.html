AVL Tree: A Self-Balancing Binary Search Tree

Definition: An AVL tree is a self-balancing binary search tree where the difference between heights of left and right subtrees (the balance factor) for every node is at most 1. This property ensures that the tree remains balanced, maintaining O(log n) time complexity for search, insert, and delete operations.

1. Introduction to AVL Trees

AVL trees were invented in 1962 by Georgy Adelson-Velsky and Evgenii Landis, hence the name AVL. They were the first self-balancing binary search trees to be invented.

Binary search trees (BSTs) provide efficient search operations when they're balanced. However, in the worst case (when nodes are inserted in sorted order), BSTs can become linear structures with O(n) time complexity for operations. AVL trees solve this problem by automatically maintaining balance through rotations.

Key Properties of AVL Trees

  • For every node, the heights of its left and right subtrees differ by at most 1
  • Maintains all properties of a binary search tree (left child ≤ parent ≤ right child)
  • Height is always O(log n) where n is the number of nodes
  • All operations (search, insert, delete) have O(log n) time complexity

2. Balance Factor

The balance factor of a node is a crucial concept in AVL trees that determines whether the tree needs rebalancing.

Balance Factor: For any node N in an AVL tree, the balance factor BF(N) is defined as: BF(N) = height(left_subtree(N)) - height(right_subtree(N))

The balance factor can be one of three values for a balanced AVL tree:

  • -1: Right subtree is one level higher than left subtree
  • 0: Both subtrees are of equal height
  • 1: Left subtree is one level higher than right subtree

If the balance factor of any node becomes less than -1 or greater than 1, the tree is unbalanced and needs to be performed to restore the AVL property.

Example: Consider a node with left subtree height 3 and right subtree height 2. BF = 3 - 2 = 1 (valid, tree is balanced) If we insert a node making left height 4: BF = 4 - 2 = 2 (invalid, tree is unbalanced)

3. AVL Tree Rotations

When an insertion or deletion causes the AVL property to be violated, rotations are performed to rebalance the tree. There are four types of rotations:

3.1 Left Rotation (LL Rotation)

Performed when a node becomes right-heavy (balance factor < -1) and its right child is right-heavy or balanced.

Left-Rotation(x): 1. y = x.right 2. x.right = y.left 3. y.left = x 4. Update heights of x and y 5. Return y (new root of subtree)

3.2 Right Rotation (RR Rotation)

Performed when a node becomes left-heavy (balance factor > 1) and its left child is left-heavy or balanced.

Right-Rotation(y): 1. x = y.left 2. y.left = x.right 3. x.right = y 4. Update heights of y and x 5. Return x (new root of subtree)

3.3 Left-Right Rotation (LR Rotation)

Performed when a node becomes left-heavy (balance factor > 1) and its left child is right-heavy.

Left-Right-Rotation(z): 1. z.left = Left-Rotation(z.left) 2. return Right-Rotation(z)

3.4 Right-Left Rotation (RL Rotation)

Performed when a node becomes right-heavy (balance factor < -1) and its right child is left-heavy.

Right-Left-Rotation(z): 1. z.right = Right-Rotation(z.right) 2. return Left-Rotation(z)

4. Insertion in AVL Tree

Insertion in an AVL tree follows the standard BST insertion procedure, followed by checking and restoring the AVL property through rotations.

AVL-Insert(root, key): 1. Perform standard BST insertion 2. Update height of the current node 3. Get the balance factor of this node 4. If node is unbalanced, perform appropriate rotation: - Left Left Case: Right rotation - Right Right Case: Left rotation - Left Right Case: Left-Right rotation - Right Left Case: Right-Left rotation 5. Return the (possibly new) root of the subtree
Insertion Example: Let's insert the sequence [10, 20, 30, 40, 50, 25] into an empty AVL tree. 1. Insert 10: Tree is balanced (BF=0) 2. Insert 20: Tree is balanced (BF=-1 for 10) 3. Insert 30: Node 10 becomes unbalanced (BF=-2) - Left rotation After rotation: 20 is root with 10 and 30 as children 4. Insert 40: Tree is balanced (BF=-1 for 20, BF=-1 for 30) 5. Insert 50: Node 30 becomes unbalanced (BF=-2) - Left rotation After rotation: 40 becomes right child of 20 6. Insert 25: Node 20 becomes unbalanced (BF=2) - Right-Left rotation Final tree structure: 20 / \ 10 30 / \ 25 40 \ 50

5. Deletion in AVL Tree

Deletion in AVL trees is similar to BST deletion but requires additional steps to maintain balance.

AVL-Delete(root, key): 1. Perform standard BST deletion 2. If the tree had only one node, return null 3. Update height of the current node 4. Get the balance factor of this node 5. If node is unbalanced, perform appropriate rotation (same cases as insertion) 6. Return the (possibly new) root of the subtree

Deletion can sometimes require multiple rotations up the tree to the root, unlike insertion which requires at most one rotation.

6. Time and Space Complexity

AVL trees provide guaranteed logarithmic time complexity for all major operations:

  • Search: O(log n) - same as BST, but guaranteed due to balancing
  • Insertion: O(log n) - requires O(1) rotations after BST insertion
  • Deletion: O(log n) - may require O(log n) rotations in worst case
  • Space: O(n) - each node stores value, pointers, and height/balance info

7. Applications of AVL Trees

AVL trees are particularly useful in scenarios where:

  • Frequent insertions and deletions are needed while maintaining fast search
  • Predictable performance is required (no worst-case O(n) scenarios)
  • Memory is not extremely constrained (compared to simpler structures)

Common applications include:

  • Database indices
  • Language runtimes (implementing ordered maps/dictionaries)
  • File systems
  • Networking algorithms (e.g., routing tables)

8. Comparison with Other Data Structures

AVL Trees vs. Red-Black Trees

  • AVL trees are more strictly balanced (height difference ≤ 1 vs. ≤ 2 for Red-Black)
  • AVL trees provide faster lookups due to better balance
  • Red-Black trees have faster insertions/deletions (fewer rotations)
  • Red-Black trees are more commonly used in language libraries

AVL Trees vs. B-Trees

  • B-trees are better for disk-based storage (optimized for block access)
  • AVL trees are typically better for in-memory data
  • B-trees can have higher branching factors (more children per node)

9. Implementation Considerations

When implementing an AVL tree, several practical considerations arise:

Node Structure

A typical AVL tree node contains:

  • Value/key
  • Pointers to left and right children
  • Height information (can be stored explicitly or computed as needed)
  • Some implementations store balance factor directly

Height Management

Height can be stored explicitly in each node or computed recursively when needed. Storing it improves performance but uses more memory.

// Example height calculation height(node): if node is null: return 0 return 1 + max(height(node.left), height(node.right))

Recursive vs. Iterative Implementation

AVL trees can be implemented recursively (simpler code) or iteratively (better performance, avoids stack overflow for large trees).

10. Conclusion

AVL trees provide an excellent solution for maintaining sorted data with guaranteed logarithmic time operations. While they require more complex implementation than standard BSTs and may perform more rotations than some other balanced trees, their predictable performance makes them valuable in many applications.

The key advantages of AVL trees are:

  • Guaranteed O(log n) time complexity for all operations
  • Relatively simple rotation operations
  • Good for scenarios with frequent searches and moderate modifications

Understanding AVL trees provides a strong foundation for studying more complex data structures and algorithms in computer science.
