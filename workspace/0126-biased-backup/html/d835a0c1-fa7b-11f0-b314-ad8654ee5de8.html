Overview and Problem Statement

Dijkstra's algorithm is a classic algorithm for finding shortest paths from a single source node to all other nodes in a weighted graph where every edge weight is non-negative. It is widely used in routing, navigation, network optimization, and many other domains.

Formal problem

Given a directed or undirected graph G = (V, E) with non-negative edge weights w(u, v) ≥ 0 for each edge (u, v) ∈ E, and a source node s ∈ V, compute for every vertex v ∈ V the shortest path distance d(s, v). Optionally, reconstruct a shortest path tree showing the predecessor of each node along a shortest path from s.

Assumptions and important constraints

  • Edge weights must be non-negative. Dijkstra's algorithm does not produce correct results if negative weights are present.
  • Graphs can be directed or undirected; the algorithm works in both cases if weights are non-negative.
  • This is a single-source algorithm: it finds shortest paths from one source to all nodes.

Intuition and High-Level Idea

Think of the algorithm as a flood spreading outward from the source node: at each step, it permanently confirms the node that currently has the smallest known distance from the source. Once a node is confirmed, its shortest distance will never change — because any other path to that node would pass through an unconfirmed node that already had an equal or larger distance.

Key idea: maintain a set of nodes whose shortest distance is known (finalized) and a frontier of tentative distances to other nodes. Repeatedly pick the non-finalized node with the smallest tentative distance, finalize it, and relax (update) distances to its neighbors.

The greedy selection of the nearest tentative node is safe precisely because there are no negative edge weights. That ensures that once a node is chosen as the current minimum, any alternative path using unfinalized nodes would be at least as long.

Pseudocode and Step-by-Step Algorithm

Below is standard pseudocode for Dijkstra's algorithm in an easy-to-follow form. This version uses a priority queue keyed by tentative distances.

function Dijkstra(Graph, source):
    for each vertex v in Graph:
        dist[v] := +∞
        prev[v] := undefined
    dist[source] := 0

    // Q is a priority queue ordered by dist[]
    Q := all vertices in Graph, keyed by dist[]

    while Q is not empty:
        u := extract-min(Q)      // vertex with smallest dist[]
        // u now has final shortest distance dist[u]
        for each neighbor v of u:
            alt := dist[u] + weight(u, v)
            if alt < dist[v]:
                dist[v] := alt
                prev[v] := u
                decrease-key(Q, v, alt)

    return dist, prev

Explanation of steps

  1. Initialize distances: set every node's tentative distance to infinity, except the source which is zero.
  2. Maintain a priority queue (min-heap) of nodes keyed by their tentative distances.
  3. Repeatedly extract the node u with the smallest tentative distance. That distance is final for u.
  4. For each outgoing edge (u, v), compute alt = dist[u] + weight(u, v). If alt is smaller than the current dist[v], update dist[v] and set prev[v] = u. Also update the position of v in the priority queue (decrease-key).
  5. Continue until the priority queue is empty or until the target has been finalized if you're only interested in a single target.

Proof Sketch / Correctness

Correctness is commonly proved with a loop invariant: when a vertex u is extracted from the priority queue (i.e., chosen as the node with minimum tentative distance), dist[u] is the actual shortest-path distance from the source to u.

Sketch:

  1. Base: The source s is initialized with dist[s] = 0 which is correct.
  2. Inductive step: Assume every vertex extracted before the current step has correct shortest distances. Consider the next vertex u extracted, with tentative distance d = dist[u]. For contradiction, suppose there is a path from s to u with length < d. Let the first vertex on that shorter path that is not yet finalized be x; its predecessor on that path (call it y) must have been finalized already. Then dist[y] is the true shortest distance to y, and relaxing (y, x) would have set dist[x] ≤ dist[y] + weight(y, x). By the assumption of a shorter path to u, dist[y] + weight(y, x) ≤ true distance to u < d, so dist[x] would be < d. That contradicts the fact that u (distance d) was the smallest tentative vertex chosen. Therefore dist[u] is optimal.

This informal invariant-based argument shows the greedy step is safe because edge weights are non-negative; negative weights break the guarantee.

Complexity Analysis

Time complexity depends on how the priority queue is implemented:

  • Using a simple array (or linear scan to find minimum): O(V^2 + E) = O(V^2) for dense graphs. Practical for small graphs or when V is small.
  • Using a binary heap (or language-built priority queue): O((V + E) log V). For sparse graphs (E ≈ V), this is roughly O(V log V).
  • Using a Fibonacci heap (theoretically optimal for decrease-key): O(E + V log V). Rarely used in practice due to complexity of implementation.

Space complexity: O(V + E) to store the graph (adjacency lists) plus O(V) for dist[] and prev[] arrays and the priority queue overhead.

Implementation Tips and Variants

Adjacency representation

Prefer adjacency lists for general graphs; adjacency matrices are convenient for dense graphs or when V is small.

Priority queue details

  • If your priority queue doesn't support a decrease-key operation, a common trick is to insert the new (vertex, newDist) pair. When pulling items from the queue, ignore entries whose distance does not match the current dist[vertex]. This may increase queue size to O(E) but keeps correctness.
  • In many languages, pairing (distance, vertex) in the heap is sufficient.

Early exit

If you only need the shortest path to a single target node t, you can stop as soon as t is extracted from the priority queue (finalized). This can save work on large graphs.

Path reconstruction

Store a predecessor array prev[v] updated when dist[v] changes. After the algorithm, reconstruct a path from source s to target t by following prev pointers backwards from t to s.

Handling ties

Ties in distances can be broken arbitrarily. However, if you want deterministic behaviour, apply a consistent tie-breaker (e.g., vertex id) when retrieving the next vertex.

Edge cases

Graphs with multiple components: nodes unreachable from the source will retain dist = ∞.

Limitations and When Not to Use Dijkstra

  • Negative edge weights: Dijkstra's algorithm is not safe when edges can have negative weights. Use Bellman–Ford when negative weights are present (and you need to detect negative cycles).
  • All-pairs shortest paths: For dense graphs or when every pair distances are needed, consider Floyd–Warshall (O(V^3)) or repeated Dijkstra with a good priority queue, or Johnson's algorithm (reweights edges then runs Dijkstra from each source).
  • Dynamic graphs: For graphs with frequent updates, recomputing all shortest paths from scratch may be inefficient; consider dynamic shortest path techniques or incremental updates if applicable.

Related Algorithms and Extensions

  • Bellman–Ford: Single-source shortest paths with support for negative weights, O(V E).
  • A* Search: Dijkstra's algorithm with a heuristic to guide search towards a target. If heuristic is zero, A* reduces to Dijkstra.
  • Johnson's algorithm: Reweights edges (using Bellman–Ford) and runs Dijkstra from each vertex to compute all-pairs shortest paths efficiently for sparse graphs.
  • Floyd–Warshall: All-pairs shortest paths in O(V^3), supports negative weights but not negative cycles.

Practical Applications

Real-world uses of Dijkstra's algorithm include:

  • Network routing protocols (e.g., OSPF) for computing shortest paths between routers.
  • GPS and mapping software to find shortest driving or walking routes (often combined with heuristics like A*).
  • Robotics and game AI for path planning on discrete graphs.
  • Logistics and transportation planning, resource distribution, and many optimization problems which can be modeled as shortest-path problems.

Exercises and Practice Problems

  1. Hand-simulate Dijkstra on a graph that contains equal-weight edges. Observe tie-breaking behaviour and resulting shortest-path tree.
  2. Modify an implementation so decrease-key is not available: implement the "lazy" approach of pushing duplicates onto the heap and skipping outdated entries when popped.
  3. Compare running time of Dijkstra on a dense graph using array-based selection vs. binary-heap selection to see practical difference.
  4. Prove formally the loop-invariant correctness using induction (write a full proof).
