A* Search — Comprehensive Explanation

A thorough, text-centered educational resource describing the A* (A-star) search algorithm: principles, theory, step-by-step narrative, proofs of correctness, practical tips, variations, and a minimal demonstration.

1. What is A* Search?

A* (pronounced "A star") is a best-first search algorithm that finds a least-cost path from a start state to a goal state, given:

  • a model of the state space (nodes and transitions),
  • a cost function for transitions (often non-negative), and
  • a heuristic function h(n) that estimates the cost from a node n to the goal.

Intuitively, A* explores paths that appear promising according to an estimate that combines the cost so far and the estimated remaining cost. It uses the evaluation function: 

f(n) = g(n) + h(n)

where g(n) is the exact cost to reach node n from the start, and h(n) is the heuristic estimate of how much it will cost from n to the goal.

2. Core components and definitions

State space

A state space is a graph (V, E) where vertices V are states and edges E are transitions between states. Each edge (u, v) has an associated non-negative cost c(u, v) ≥ 0.

g(n) — exact cost-to-come

g(n) is the exact cost of the cheapest path found so far from the start node to node n. During A* execution, this value may be updated as better paths are discovered.

h(n) — heuristic estimate

h(n) is a problem-specific estimate of the cheapest cost from node n to the goal. The heuristic guides A*'s search order by estimating remaining effort.

f(n) = g(n) + h(n)

f(n) represents the algorithm's current estimate of the cheapest solution that goes through node n. A* always expands the node with the smallest f(n) value (breaking ties according to some rule).

Open and Closed sets

  • Open set: frontier nodes discovered, but not yet expanded (commonly implemented as a priority queue keyed by f).
  • Closed set: nodes already expanded (their best g-values are finalized if heuristics are consistent).

3. Pseudocode (standard, explicit)

Here is a standard pseudocode description of A* (textual):

function A_Star(start, goal, h):
  open = priority queue ordered by f = g + h
  open.insert(start, g=0, f = h(start))
  cameFrom = empty map
  gScore = map with default +∞
  gScore[start] = 0

  while open is not empty:
    current = open.pop()  // node with lowest f(n)
    if current == goal:
      return ReconstructPath(cameFrom, current)

    add current to closed set
    for each neighbor of current:
      tentative_g = gScore[current] + cost(current, neighbor)
      if tentative_g < gScore[neighbor]:
        cameFrom[neighbor] = current
        gScore[neighbor] = tentative_g
        f = tentative_g + h(neighbor)
        if neighbor not in open:
          open.insert(neighbor, f)
        else:
          open.decrease_key(neighbor, f)

  return FAILURE

In words: extract the frontier node with least estimated total cost, expand it (examine neighbors), and update each neighbor's best-known g value and f estimate. Continue until the goal is popped from the open set.

4. Heuristic properties and guarantees

Admissibility

A heuristic h is admissible if for every node n, h(n) ≤ h*(n), where h*(n) is the true minimal cost from n to the nearest goal. In other words, an admissible heuristic never overestimates the true remaining cost.

If h is admissible and A* terminates when the goal is first removed from the open set (popped), the path returned is optimal (a least-cost path).

Consistency (Monotonicity)

A heuristic is consistent (also called monotone) if for every node n and successor n' generated by an action with cost c(n, n'), 

h(n) ≤ c(n, n') + h(n')

Consistency implies admissibility. If the heuristic is consistent, then the f-values along any path are non-decreasing: f(n') ≥ f(n). One important practical consequence is that once a node is expanded, its best g value is final and the node never needs to be re-expanded.

Optimality proof sketch

Consider the first time the goal node is removed from the open set. Suppose there is a cheaper path to the goal that A* missed. Using the admissibility of the heuristic, you can show that at least one node along that optimal path must have been in the open set with f less than or equal to the cost of that optimal solution; A* would therefore expand nodes in order of increasing f until it finds that cheaper path and cannot prematurely return a suboptimal goal.

5. Complexity and practical behavior

Time complexity

The theoretical worst-case time complexity of A* depends on the heuristic. With a weak heuristic (e.g., h(n) = 0), A* reduces to Dijkstra's algorithm and may expand many nodes. In general, worst-case time complexity is exponential in the depth of the solution for uninformed or poorly informed heuristics. A good heuristic greatly reduces the number of expanded states.

Space complexity

A* stores all generated nodes in memory (open and closed sets). This can make A* memory intensive and a practical limitation for large problems. Variants such as IDA* (Iterative Deepening A*) seek to reduce memory usage at the cost of repeated work.

Effect of heuristic quality

The closer h(n) is to the true cost h*(n) (without exceeding it), the fewer nodes A* typically expands. The ideal heuristic h(n) = h*(n) would guide A* directly along an optimal path without branching. In practice, admissible heuristics are often derived from relaxed versions of the problem.

6. Heuristic design and examples

Grid pathfinding

For movement on a grid with unit-cost orthogonal moves:

  • Manhattan distance: h(n) = |x1 - x2| + |y1 - y2| (admissible for 4-neighbor grids)
  • Euclidean distance: h(n) = sqrt((x1-x2)^2 + (y1-y2)^2) (admissible for continuous or diagonal movement with Euclidean costs)
  • Chebyshev distance: h(n) = max(|dx|,|dy|) (admissible when 8-directional moves cost 1)

Graph problems

Common ways to derive admissible heuristics include:

  • Relax the problem constraints (e.g., ignore obstacles or turn costs) and compute exact shortest distances in the simpler problem.
  • Precompute lower bounds (pattern databases) and look up lower-bound values during search.

Weighted heuristics

Weighted A* uses f(n) = g(n) + w * h(n) with weight w ≥ 1. If h is admissible and w = 1, A* is optimal. With w > 1, search is more greedy (favors heuristics) and may find suboptimal solutions faster. Some variants provide provable bounded suboptimality: weight w gives an upper bound on solution cost relative to optimal.

7. Step-by-step narrative (small graph example)

Consider a simple graph: A start node S and a goal G with intermediate nodes B and C. 

Nodes: S, B, C, G
Edges and costs:
  S —(1)— B
  S —(4)— C
  B —(4)— G
  C —(2)— G

Heuristic h (admissible estimates to G):
  h(S) = 3, h(B) = 3, h(C) = 2, h(G) = 0

Step-by-step:

  1. Initialize: open = {S(f= g(=0)+h(=3) = 3)}. g(S)=0.
  2. Pop S (f=3); expand neighbors B and C:
    • B: tentative g = 0 + 1 = 1 → f = 1 + h(B)=1+3=4 → add B to open
    • C: tentative g = 0 + 4 = 4 → f = 4 + h(C)=4+2=6 → add C to open
  3. Open contains B(f=4), C(f=6). Pop B (f=4); expand B's neighbor G:
    • G via B: g = 1 + 4 = 5 → f = 5 + h(G)=5+0=5 → add G to open
  4. Open contains G(f=5), C(f=6). Pop G (f=5). Since G is the goal, stop and reconstruct path S→B→G with total cost 5.

Note: The direct path S→C→G would have cost 4 + 2 = 6. A* correctly picks the cheaper path because f-values led it to explore B first.

8. Formal correctness sketch (optimality)

Key idea (sketch): Let optimal cost from start to goal be C*. Suppose A* returns a path with cost C > C*. Let node n be the first node on some optimal path that is not expanded by the time A* returns. Because the heuristic is admissible, f(n) = g(n) + h(n) ≤ g(n) + h*(n) = cost of optimal solution through n, hence f(n) ≤ C*. But if A* returned a path with cost C > C*, all nodes with f ≤ C* must have been expanded before returning. Contradiction. Hence A* cannot return suboptimal path.

The full formal proof manipulates inequalities and relies on the definition of admissibility to compare f-values of frontier nodes to optimal solution cost.

9. Practical considerations, tips, and variants

Tie-breaking

When multiple nodes share the same f-value, tie-breaking affects performance. Common heuristics:

  • Prefer larger g (deeper) or smaller h to favor paths that are closer to the start or goal respectively.
  • Tie-breaking toward larger g tends to find paths that are slightly more direct in some grid search settings (breaking ties by larger g can reduce expansions).

Dealing with memory limits

  • Use IDA* (Iterative Deepening A*): repeated depth-first searches using f-cost thresholds; much lower memory but possibly repeated work.
  • Use external-memory or memory-bounded variants like SMA* (Simplified Memory-bounded A*).

Dynamic and incremental planning

In settings where the map or costs change, incremental planners such as D* and D*-Lite reuse previous search results to perform efficient replanning.

Graph search versus tree search

- Tree search does not record visited states (closed set). It may revisit states many times. Useful if the state space has unique states or if memory is extremely limited.
- Graph search uses a closed set to avoid reprocessing identical states; required for efficiency in graphs with cycles or multiple paths to same node.

Heuristic combinations

You may combine admissible heuristics by taking the maximum: h(n) = max(h1(n), h2(n), ...). The result is admissible and often stronger (closer to true cost) than each individually.

10. Variants and related algorithms

  • Weighted A*: trades optimality for speed by multiplying the heuristic.
  • IDA* (Iterative Deepening A*): depth-first iterative deepening using f-limits.
  • SMA* (Simplified Memory-bounded A*): works with memory bounds by discarding less-promising nodes.
  • D* and D*-Lite: incremental path planning for dynamic / changing graphs.
  • Theta*: allows any-angle pathfinding by connecting non-adjacent nodes directly when line-of-sight exists.
