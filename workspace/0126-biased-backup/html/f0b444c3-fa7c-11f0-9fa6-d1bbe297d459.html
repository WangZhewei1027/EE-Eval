Comprehensive Guide to Git

Introduction to Version Control

Version control is a system that records changes to files over time, allowing you to recall specific versions later. It's essential for:

  • Tracking changes in code or documents
  • Collaborating with multiple developers
  • Reverting to previous states
  • Maintaining different versions of a project
Git is a distributed version control system (DVCS) created by Linus Torvalds in 2005 for Linux kernel development. It's designed to be:
  • Fast and efficient
  • Distributed (every user has a complete repository)
  • Scalable for large projects
  • Secure (uses cryptographic hashing)

Core Concepts of Git

1. Repository

A Git repository (or "repo") is a collection of files and their complete history. There are two main types:

  • Local repository - On your computer
  • Remote repository - On a server (e.g., GitHub, GitLab)

2. Snapshots, Not Differences

Unlike other VCS, Git doesn't store differences between versions. Instead, it takes a "snapshot" of all files at each commit and stores references to these snapshots.

Traditional VCS: File1 (v1) -> diff1 -> File1 (v2) -> diff2 -> File1 (v3) Git: Snapshot1 -> Snapshot2 -> Snapshot3 | | | Files Files Files

3. Three Main States

Files in Git can be in one of three states:

  1. Modified - Changed but not committed
  2. Staged - Marked to go into the next commit
  3. Committed - Safely stored in the database
Working Directory -> Staging Area -> Git Directory (Modified) (Staged) (Committed)

4. Branches

A branch represents an independent line of development. The default branch is usually called main or master.

Example Branch Workflow:

  1. Create a new branch for a feature: git checkout -b new-feature
  2. Make changes and commit them
  3. Switch back to main: git checkout main
  4. Merge the feature: git merge new-feature

How Git Works Internally

1. The .git Directory

Every Git repository has a .git directory containing:

  • HEAD - Points to current branch
  • objects/ - Database of all content
  • refs/ - References to commit objects (branches, tags)
  • config - Repository configuration

2. Objects in Git

Git uses four fundamental object types:

  1. Blob - Stores file data (content only, no filename)
  2. Tree - Directory listing (references to blobs and other trees)
  3. Commit - Points to a tree, parent commit(s), and metadata
  4. Tag - Marks a specific commit (usually for versions)
Commit | Tree / \ Tree Blob / \ | Blob Blob File Content

3. SHA-1 Hashes

Git uses SHA-1 hashes to identify all objects. Each object is uniquely identified by a 40-character checksum:

5d2a3a5f5b5b5b5b5b5b5b5b5b5b5b5b5b5b5b5b

This is calculated based on the object's content, ensuring data integrity.

Basic Git Commands

1. Repository Setup

git init

Initializes a new Git repository in the current directory.

git clone <repository-url>

Creates a copy of an existing repository.

2. Making Changes

git status

Shows the status of files in the working directory.

git add <file>

Stages changes for the next commit.

git commit -m "Commit message"

Records changes to the repository with a descriptive message.

3. Viewing History

git log

Shows the commit history.

git diff

Shows changes between working directory and staging area.

4. Branching and Merging

git branch

Lists all branches.

git checkout <branch>

Switches to another branch.

git merge <branch>

Merges changes from another branch into the current branch.

Working with Remote Repositories

1. Adding Remotes

git remote add <name> <url>

Adds a remote repository reference.

2. Fetching and Pulling

git fetch <remote>

Downloads objects and refs from a remote repository without merging.

git pull <remote> <branch>

Fetches and merges changes from a remote branch.

3. Pushing Changes

git push <remote> <branch>

Uploads local branch commits to a remote repository.

Note: Before pushing, always pull the latest changes to avoid conflicts.

Advanced Git Concepts

1. Rebasing

Rebasing rewrites commit history by moving a branch to a new base commit. It creates linear history but changes commit hashes.

Rebase vs Merge:

Before:

A---B---C main \ D---E feature

After git rebase main on feature branch:

A---B---C---D'---E' main

After git merge feature on main branch:

A---B---C---F main \ / D---E feature

Warning: Don't rebase commits that have been pushed to a shared repository as it rewrites history.

2. Stashing

Temporarily shelves changes so you can work on something else:

git stash

Save changes to stash

git stash pop

Reapply stashed changes

3. Cherry-picking

Applies a specific commit from one branch to another:

git cherry-pick <commit-hash>

4. Hooks

Scripts that run automatically when certain Git events occur (pre-commit, post-commit, etc.). Stored in .git/hooks/.

Git Best Practices

1. Commit Guidelines

  • Commit often, in logical chunks
  • Write clear, descriptive commit messages
  • Follow the project's commit message conventions

2. Branching Strategy

Common strategies include:

  • Git Flow - Feature branches, develop branch, release branches
  • GitHub Flow - Simpler model with feature branches and main
  • Trunk-Based Development - Short-lived branches or direct commits

3. Handling Merge Conflicts

  1. Understand what changed with git diff
  2. Edit conflicted files (marked with <<<< and >>>>)
  3. Mark as resolved with git add
  4. Complete the merge with git commit

Conclusion

Git is a powerful version control system that enables efficient collaboration and version management. Key takeaways:

  • Git tracks content through snapshots, not file differences
  • The three-state architecture (working directory, staging area, repository) provides flexibility
  • Branching and merging enable parallel development
  • Distributed nature allows for robust collaboration

To learn more, consult the official Git documentation.
