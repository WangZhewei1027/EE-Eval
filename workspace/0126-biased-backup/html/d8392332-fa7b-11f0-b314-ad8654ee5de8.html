Congestion Control — A Comprehensive Textual Guide

This page explains the concept, theory, and common algorithms for congestion control in packet networks. It focuses on textual, step-by-step explanations and pedagogy. A single simple demonstration (AIMD behavior) is provided at the end to illustrate dynamics; the page is otherwise static reading material.

1. What is Congestion Control?

Congestion control is the set of techniques used by networked systems (endpoints, routers, or both) to avoid, react to, and mitigate congestion in packet-switched networks. Congestion occurs when the demand for packet transmission at a network resource (e.g., a link or router) surpasses its capacity, causing queues to grow, packets to be delayed, or packets to be dropped.

Congestion control aims to achieve several practical objectives:

  • Avoid sustained queue growth (keep latency bounded).
  • Prevent excessive packet loss.
  • Provide fair allocation of capacity across competing flows.
  • Utilize available capacity efficiently (throughput).
  • React quickly and stably to changes in available capacity and number of flows.

Important distinction: Congestion control is different from flow control. Flow control (e.g., TCP receive window) prevents a sender from overwhelming a particular receiver's buffer; congestion control prevents the network from being overwhelmed.

2. Ingredients — Where congestion information comes from

Congestion control systems must infer the state of the network. Common signals:

  • Packet loss: when a packet is not acknowledged in time it is assumed lost. Historically a primary signal for congestion.
  • ACK timing / RTT: increases in round-trip time (RTT) often indicate queue buildup and incipient congestion.
  • Explicit Congestion Notification (ECN): network marks packets with a bit to indicate impending congestion without dropping packets.
  • Throughput measurements: observing delivered rate over windows of time.
  • Active Queue Management (AQM) feedback: routers may explicitly drop or mark packets based on queue dynamics (e.g., RED, CoDel).

Design choice: signals may be implicit (loss, RTT) or explicit (ECN/marks). Each has tradeoffs: implicit signals are simple but can be ambiguous (loss can be due to corruption), explicit signals provide clarity but require network support.

3. Two general control styles

There are two broad ways to control the sending rate:

  1. Window-based control: The sender limits the number of unacknowledged bytes on the network (congestion window, cwnd). TCP is window-based.
  2. Rate-based control: The sender controls the pace directly, e.g., tokens per second. Common in RTP/QUIC media and some congestion control variants.

Both styles can implement the same high-level policies; mathematical analysis often uses fluid approximations for rates, or discrete-time models for windows.

4. Core principle: AIMD (Additive Increase, Multiplicative Decrease)

AIMD is the central control law for many congestion control algorithms. It balances efficiency and fairness and provides provable convergence properties in idealized settings.

How AIMD works (conceptually):

  • Additive Increase: In the absence of congestion signals, increase the sending rate slowly (add a fixed amount per RTT or per time). This probes for available capacity conservatively.
  • Multiplicative Decrease: Upon detecting congestion, decrease the sending rate multiplicatively (e.g., halve the window). This quickly relieves congestion.

Why AIMD? Heuristically, additive increase slowly explores spare capacity (leading to stability) while multiplicative decrease reduces load quickly. In multi-flow environments, AIMD on windows converges toward fairness: flows with similar RTTs share capacity roughly equally.

AIMD can be expressed in simple rules:

At each RTT, if no congestion:
  cwnd := cwnd + alpha      (additive increase)
On congestion event:
  cwnd := cwnd * beta       (0 < beta < 1, multiplicative decrease)

Typical TCP: alpha ≈ 1 MSS per RTT (in congestion avoidance); beta = 1/2 (halve cwnd on loss).

5. Classic TCP algorithms — step-by-step descriptions

TCP Tahoe (timeline)

  1. Start in Slow Start: cwnd = 1 MSS, SS threshold (ssthresh) large.
  2. Slow Start: cwnd doubles every RTT (increase by 1 MSS per ACK), until cwnd >= ssthresh or a loss is detected.
  3. On loss (timeout or triple duplicate ACKs): set ssthresh = cwnd/2, cwnd = 1 MSS, and enter Slow Start again.

Tahoe's behavior: aggressive probing (exponential growth) but when loss is detected it goes back to the slowest sending rate.

TCP Reno (improvement over Tahoe)

  1. Slow Start and Congestion Avoidance as before.
  2. On triple duplicate ACKs (indicating likely isolated loss): set ssthresh = cwnd/2, set cwnd = ssthresh, and enter Congestion Avoidance (do not revert to slow start).
  3. On timeout: behave like Tahoe (cwnd = 1 MSS).

Reno introduced Fast Retransmit + Fast Recovery: it recovers from a single packet loss without going back to slow start, improving throughput for flows with occasional single losses.

TCP NewReno

NewReno refines Reno's fast recovery to handle multiple losses in a single window more effectively by remaining in fast recovery until all the packets outstanding at the point of loss are acknowledged.

TCP Vegas (delay-based)

Vegas uses RTT changes as a congestion signal rather than loss. It measures difference between expected and actual throughput and adjusts cwnd up or down to keep a target number of packets queued.

Vegas can achieve lower loss and latency but may be at a disadvantage when competing with loss-based flows that push queue occupancy higher.

TCP Cubic

Cubic (widely used on Linux) uses a cubic function of time since the last loss to govern cwnd growth. The cubic shape is chosen to quickly utilize large bandwidth-delay product paths while maintaining stability near the previous loss point.

Key takeaways about these algorithms

  • Slow Start: exponential probe to quickly find capacity; risk: overshoot causing packet loss.
  • Congestion Avoidance: AIMD-style steady-state behavior.
  • Fast Retransmit/Recovery: mechanisms to repair isolated losses without timeout penalties.
  • Delay-based vs loss-based: tradeoff between low-latency (delay-based) and robustness when delay signals are noisy or cross-traffic is loss-based.

6. Active Queue Management (AQM) and router-side controls

Routers can help control congestion by managing queue lengths and by providing early signals to senders. Common AQM techniques:

  • Drop-tail: simple queue: drop packets when full. Leads to global synchronization and high latency under load.
  • RED (Random Early Detection): probabilistically drop packets before the queue is full, with drop probability increasing as queue length grows. Objective: avoid global synchronization and keep queue moderate.
  • CoDel (Controlled Delay): targets queueing delay rather than queue length. It drops packets if minimum delay exceeds a target for a certain interval.
  • PIE, fq_codel: modern variants combining fairness and delay targets.

AQM, combined with ECN marks, enables routers to signal congestion before drops occur, improving latency and throughput.

7. Fairness and resource allocation

A fundamental question: how should capacity be shared between competing flows? Several notions:

  • Equal share fairness: each flow gets the same bandwidth.
  • Weighted fairness: allocate relative shares according to weights.
  • Proportional fairness: maximize the sum of logarithms of rates; arises naturally from AIMD under certain conditions.
  • Max-min fairness: maximize the minimum allocation iteratively.

TCP's AIMD tends to converge to a proportional-fair point or equal-share depending on path symmetry. Flows with different RTTs can get different shares — shorter RTTs often get higher throughput with classic TCP.

8. Theoretical models (brief, conceptual)

Two common abstractions:

  • Fluid models: treat sending rates as continuous variables and model dynamics with differential equations. Useful for stability and equilibrium analysis.
  • Discrete round models: model window updates in terms of RTTs and packet events; closer to real TCP behavior.

One illustrative result: with many AIMD flows sharing a bottleneck and with synchronized RTTs, the long-run behavior tends to a fair division of capacity and a sawtooth pattern of window sizes. Convergence speed and stability depend on increase/decrease parameters (alpha, beta), RTTs, and AQM behavior.

9. Metrics to evaluate congestion control

  • Throughput / Goodput: amount of useful data delivered per unit time.
  • Latency / RTT: time for a packet to travel roundtrip, including queueing delay.
  • Packet loss rate: fraction of sent packets lost or dropped.
  • Fairness index: e.g., Jain's fairness index: (sum xi)^2 / (n * sum xi^2) ranges from 1/n to 1.
  • Responsiveness: how quickly a control law reacts to changes in capacity or competing flows.
  • Stability: absence of large oscillations or pathological behavior.

10. Practical considerations and trade-offs

  • Latency vs throughput: aggressive probing increases throughput but raises queueing delay. Applications (bulk transfer vs interactive) choose different points on this trade-off.
  • Competing flows: new algorithms must coexist fairly with legacy algorithms — a major deployment constraint.
  • Heterogeneous RTTs: fairness across diverse RTTs is hard; some approaches weight increases by RTT.
  • Wireless and non-congestion losses: loss-based control may wrongly interpret wireless corruption as congestion, reducing throughput unnecessarily.
  • Encrypted transports: protocols like QUIC move congestion control into user space, enabling faster deployment of innovations.

11. Step-by-step textual example — a TCP transfer

Consider a single TCP flow transferring a large file over a path with the following parameters:

  • Maximum Segment Size (MSS) = 1500 bytes.
  • Round-trip time (RTT) = 100 ms (fixed for the example).
  • Bottleneck capacity = 20 Mbps (≈ 20,000,000 bits/s ≈ 2.5 MB/s ≈ 1700 MSS per second).

We will illustrate the sender's cwnd evolution:

  1. Start: cwnd = 1 MSS. After 1 RTT and successful ACKs, cwnd = 2 MSS (slow start doubling).
  2. Next RTT: cwnd = 4 MSS; then 8, 16, 32, ... The window grows exponentially while ACKs arrive quickly.
  3. At some RTT the cwnd becomes large enough that the sending rate exceeds bottleneck capacity, causing queue buildup and eventual packet loss.
  4. On loss: typical TCP halves cwnd (multiplicative decrease). If cwnd had been 256 MSS, ssthresh becomes 128 MSS and cwnd becomes 128 MSS (Reno) or 1 MSS (Tahoe) depending on variant.
  5. After the loss, TCP increases cwnd additively (congestion avoidance) — roughly +1 MSS per RTT — until another loss or steady state is reached.

The sawtooth pattern (grow, drop, grow) is characteristic of AIMD systems.

12. Pseudocode: Simplified TCP congestion control (conceptual)

Initialize:
  cwnd = 1 MSS
  ssthresh = large value

On ACK received:
  if cwnd < ssthresh:                      // Slow Start
    cwnd := cwnd + 1 MSS per ACK
  else:                                     // Congestion Avoidance (AIMD)
    increase accumulator by (MSS*MSS / cwnd) per ACK
    if accumulator >= 1 MSS:
      cwnd := cwnd + 1 MSS
      accumulator := accumulator - 1 MSS

On triple duplicate ACK:
  ssthresh := max(cwnd/2, 2 MSS)
  cwnd := ssthresh
  enter fast-recovery (varies by TCP variant)

On retransmission timeout:
  ssthresh := max(cwnd/2, 2 MSS)
  cwnd := 1 MSS
  enter slow-start

This pseudocode highlights the transition between slow start and AIMD congestion avoidance, and the two kinds of reactions to loss (fast vs timeout).

13. Modern developments and alternatives

  • BBR (Bottleneck Bandwidth and Round-trip propagation time): rate-based estimator-driven approach that models bandwidth and RTT to set pacing rate, rather than relying on loss or delay signals. BBR seeks to operate at the bandwidth-delay product for low latency and high throughput.
  • QUIC and user-space congestion control: QUIC enables faster algorithm iterations and deployment since congestion control runs in user space rather than kernel space.
  • Machine learning approaches: research into RL-based congestion control exists, but deployment requires safety and fairness guarantees.

Each new approach must balance performance improvements with fairness to existing flows, deployability, and stability.

14. Common misconceptions

  • Loss always means congestion: Not in wireless networks—loss may be corruption. Modern protocols often combine signals to decide.
  • More aggressive always better: Aggressive increases can harm latency and cause unfairness or collapse of throughput due to repeated losses.
  • Routers alone solve congestion: Routers can help (AQM, ECN), but end-to-end congestion control is necessary for cooperative behavior across the network.

15. Practical tips for network engineers and developers

  • Choose a congestion control algorithm appropriate to your application's needs: bulk data (throughput) vs interactive (low latency) vs video (rate stability).
  • Use pacing (pace packets smoothly) to avoid bursts that create queueing spikes.
  • Consider ECN support across the network to signal congestion without drops.
  • Test under realistic conditions: variable RTTs, mixed flows, wireless losses, and transient overloads.
