Doubly Linked List — Comprehensive Guide

A thorough textual explanation: theory, algorithms, step-by-step descriptions, pseudocode, complexity, and practical guidance.

What is a Doubly Linked List?

A doubly linked list is a linear data structure made of elements called nodes, where each node stores:

  • a value (payload),
  • a pointer/reference to the next node in the sequence, and
  • a pointer/reference to the previous node in the sequence.

Because each node knows both its successor and predecessor, traversal and many updates can be performed in both directions.

Basic Node Structure and Invariants

Conceptual node fields (language-agnostic): 

Node {
  value
  next  // pointer/reference to Node or null
  prev  // pointer/reference to Node or null
}

Invariants to maintain for a non-circular doubly linked list:

  1. If node A's next points to B, then B's prev must point back to A.
  2. The head node's prev is null (or a sentinel), and the tail node's next is null (or a sentinel) for a standard linear list.
  3. There is no cycle unless explicitly using a circular variant.

Maintaining these invariants during insertions and deletions is essential for correctness. We'll show step-by-step procedures below.

Why use a Doubly Linked List?

Core advantages:

  • Bidirectional traversal: you can iterate forward and backward efficiently.
  • Constant-time O(1) insertion and deletion at known nodes (given a pointer to the node), because you can update neighboring pointers without scanning from the head.
  • Convenient to implement features like undo/redo, browsing history, or LRU caches where you frequently move elements between ends.

Trade-offs:

  • Each node stores an extra pointer (prev) compared with a singly linked list — higher memory cost.
  • Extra pointer updates increase the chance for bugs (e.g., forgetting to update both sides).

Core Operations — Step-by-step with Pseudocode

Traversal (forward and backward)

Forward traversal: start at head; while node != null, process node.value; node = node.next.

// Forward traversal
node = head
while node != null:
    process(node.value)
    node = node.next

Backward traversal: start at tail (or traverse to tail first); while node != null, process node.value; node = node.prev.

// Backward traversal
node = tail
while node != null:
    process(node.value)
    node = node.prev

Time: O(n) to visit all elements.

Insertion

We describe three common insertion points: at the head, at the tail, and after a given node.

Insert at head (step-by-step)

  1. Create new node N with N.value = newValue.
  2. Set N.next = head.
  3. Set N.prev = null.
  4. If head is not null, set head.prev = N.
  5. Set head = N.
  6. If tail was null (list was empty), also set tail = N.
// Insert at head
N = new Node(value=newValue)
N.next = head
N.prev = null
if head != null:
    head.prev = N
head = N
if tail == null:
    tail = N

Time: O(1).

Insert at tail (step-by-step)

  1. Create new node N.
  2. Set N.prev = tail.
  3. Set N.next = null.
  4. If tail is not null, set tail.next = N.
  5. Set tail = N.
  6. If head was null (empty list), set head = N.
// Insert at tail
N = new Node(value=newValue)
N.prev = tail
N.next = null
if tail != null:
    tail.next = N
tail = N
if head == null:
    head = N

Time: O(1).

Insert after a given node (given pointer to node X)

  1. Create N.
  2. Set N.prev = X.
  3. Set N.next = X.next.
  4. If X.next != null then set X.next.prev = N.
  5. Set X.next = N.
  6. If X was tail, update tail = N.
// Insert after node X
N = new Node(value=newValue)
N.prev = X
N.next = X.next
if X.next != null:
    X.next.prev = N
X.next = N
if tail == X:
    tail = N

Time: O(1) when X is known. Finding X may be O(n).

Deletion

To delete a node D when you have a direct pointer/reference to it:

  1. If D.prev != null, set D.prev.next = D.next; otherwise (D is head), set head = D.next.
  2. If D.next != null, set D.next.prev = D.prev; otherwise (D is tail), set tail = D.prev.
  3. Optionally clear D.next and D.prev for safety and free memory.
// Delete node D (given pointer to D)
if D.prev != null:
    D.prev.next = D.next
else:
    head = D.next

if D.next != null:
    D.next.prev = D.prev
else:
    tail = D.prev

// optional
D.next = null
D.prev = null

Time: O(1) when node pointer is known. Searching for the node first costs O(n).

Search

Linear scan: start at head and compare values until found or reach null.

// Search for value V
node = head
while node != null:
    if node.value == V:
        return node
    node = node.next
return null

Time: O(n) in the worst case.

Edge Cases and Correctness (common pitfalls)

  • For empty lists, ensure both head and tail are null. Many bugs occur when only one is updated.
  • When inserting into or deleting from a one-element list, ensure both head and tail are updated appropriately.
  • Always update both neighbors' pointers when inserting/deleting in the middle. Forgetting one leads to dangling references or broken links.
  • When using languages with manual memory management (C/C++), remember to free/delete removed nodes; in GC languages, clear references to help GC reclaim memory.
  • If you implement a circular doubly linked list, the invariants change: head.prev points to tail and tail.next points to head. Handle base cases accordingly.
Tip: When in doubt, write the operations as small atomic steps and reason about how the invariants change after each step. Unit tests covering empty, one-element, and multi-element cases will catch most bugs.

Variations and Patterns

Circular doubly linked list

In this variant, the list connects tail.next to head and head.prev to tail. This is useful when you need wrap-around iteration. Be careful to break loops by keeping a visited count or checking node identity.

Dummy/sentinel head and/or tail

Using sentinel nodes (dummy head/tail) simplifies code by removing special-case checks for null. All actual elements sit between sentinels. Sentinels never contain user data.

Indexed operations

Doubly linked lists do not support efficient random access by index (O(n) to reach index i). For frequent index-based access arrays or balanced trees are preferable.

Concurrent lists

Making a doubly linked list thread-safe requires synchronization for updates that modify pointers. Fine-grained locking or lock-free algorithms exist but are complex because updates touch multiple adjacent nodes and require careful ordering to maintain invariants.

Example Implementations (concise)

Python-like pseudocode

class Node:
    def __init__(self, value):
        self.value = value
        self.next = None
        self.prev = None

class DoublyLinkedList:
    def __init__(self):
        self.head = None
        self.tail = None

    def push_front(self, v):
        n = Node(v)
        n.next = self.head
        if self.head: self.head.prev = n
        self.head = n
        if self.tail is None: self.tail = n

    def push_back(self, v):
        n = Node(v)
        n.prev = self.tail
        if self.tail: self.tail.next = n
        self.tail = n
        if self.head is None: self.head = n

    def remove(self, node):
        if node.prev: node.prev.next = node.next
        else: self.head = node.next
        if node.next: node.next.prev = node.prev
        else: self.tail = node.prev
        node.next = node.prev = None
        return node.value

C-like pseudocode

struct Node {
    void* value;
    Node* next;
    Node* prev;
};

void insert_after(Node* X, Node* N, DoublyList* list) {
    N->prev = X;
    N->next = X->next;
    if (X->next) X->next->prev = N;
    X->next = N;
    if (list->tail == X) list->tail = N;
}

Use Cases and When Not to Use

Good use cases:

  • Implementing ordered lists where frequent insertions and removals occur at both ends.
  • LRU caches: moving nodes to front on access is O(1) given pointers.
  • Undo/redo structures and browser history stacks.

Not ideal when:

  • You need fast random access by index (arrays, vectors, or trees are better).
  • Memory overhead per element must be minimal (linked lists use more memory than arrays).

Troubleshooting Checklist

  1. Do head and tail pointers reflect the list correctly after every operation?
  2. For each node, if node.next != null then node.next.prev == node (and vice versa).
  3. Test boundary cases: empty list, one-element list, insert/delete at both ends.
  4. Clear node pointers on deletion to avoid accidental aliasing in manual memory-managed languages.
