1. What is version control?

A concise definition followed by an explanation.

Definition: Version control is a set of tools and practices that record, organize, and allow retrieval of historical states of a collection of files (often source code). It makes it possible to track changes over time, collaborate with others, compare versions, and reconstruct previous states.

Why version control matters

  • Collaboration: multiple people can work concurrently and integrate their changes.
  • History: you can inspect when and why changes happened, who made them, and revert if necessary.
  • Experimentation: branching lets you explore alternate designs in isolation.
  • Accountability and auditability: essential for legal, security, and quality purposes.
  • Automation: enables CI/CD pipelines to build and test changes reliably.

2. Types of version control systems

Three broad categories and their tradeoffs.

Local version control

Local VCS are simple systems that keep history on the same machine (e.g., early RCS). Useful for single-developer workflows but lack collaboration primitives.

Centralized version control (CVCS)

Systems like Subversion (SVN), Perforce (centralized mode) use a central server that stores the canonical repository. Clients check out working copies and commit to the server.

  • Advantages: single source of truth, centralized access control, simpler conceptual model.
  • Disadvantages: single point of failure, network dependence, less flexible offline operations.

Distributed version control (DVCS)

Systems like Git and Mercurial allow each clone to be a full repository copy, including history. Collaboration happens by exchanging changes between repositories (push/pull).

  • Advantages: offline operations, performance, rich local branching, multiple remote workflows.
  • Disadvantages: conceptual complexity (commit graphs), potential for divergent histories if workflows aren't coordinated.

3. Core operations and concepts (user-level)

Every VCS exposes a small set of powerful primitives. Here they are in plain language.

Repository

A repository (repo) is the data store that contains files, history, and metadata.

Working copy (working tree)

The set of files a developer edits. Changes can be staged and then committed to the repository.

Commit

A commit records a snapshot (or a delta, depending on system) of the tracked files, plus metadata: author, timestamp, message, and parent(s). Commits form a history.

Branch (or pointer)

A symbolic name for a commit (or a moving pointer that indicates the latest commit on a line of development).

Merge

Combining changes from two branches. Simple merges fast-forward; non-linear merges require conflict resolution.

Checkout

Switching the working copy to reflect a specified commit or branch.

Revert / Reset / Cherry-pick

Different ways to undo or reapply changes:

  • Revert: make a new commit that undoes a previous commit.
  • Reset: move a branch pointer (can change history—use with care in shared repos).
  • Cherry-pick: apply a single commit from elsewhere onto the current branch.

4. Core algorithms and data structures

This section explains the important algorithms used by VCS for diffing, merging, object storage, and graph handling.

4.1 Diff algorithms (textual difference)

Diff algorithms compute how to transform one sequence (typically lines of text) into another. The most common theoretical basis is the Longest Common Subsequence (LCS) problem. Practical diff tools use optimized algorithms for speed and human-friendly output.

Myers's diff algorithm (short description)

Myers's algorithm is widely used for generating minimal edit scripts. It runs in O(ND) time where N = lengthA + lengthB and D is the number of differences (edit distance). It is practical and efficient for typical file sizes.

High-level pseudocode for Myers's algorithm:
function MyersDiff(A[0..N-1], B[0..M-1]):
  for D from 0 to N+M:
    for k from -D to D step 2:
      if k == -D or (k != D and V[k-1] < V[k+1]):
        x = V[k+1]    // down (insertion)
      else:
        x = V[k-1] + 1 // right (deletion)
      y = x - k
      // "snake": advance while equal
      while x < N and y < M and A[x] == B[y]:
        x++; y++
      V[k] = x
      if x >= N and y >= M:
        return build_edit_script_from_trace()

Many practical diff tools add heuristics: patience diff for more readable hunks, or converting to line hashes before LCS to speed up comparisons.

4.2 Three-way merge

A three-way merge uses a common ancestor (base) and two diverging versions (ours, theirs). The algorithm aligns changes relative to the base and tries to combine non-overlapping edits automatically; overlapping edits become conflicts that require manual resolution.

Three-way merge (conceptual steps):
1. Compute diffBaseO = diff(base, ours)
2. Compute diffBaseT = diff(base, theirs)
3. Apply non-conflicting edits from both diffs to base.
4. If both diffs modify the same region differently, mark a conflict.
5. Produce a merged file with conflict markers or resolved content.

Three-way merges are robust because they reason about how each side changed the base rather than directly diffing the two sides against each other.

4.3 Directed Acyclic Graph (DAG) of commits

Most modern VCS represent commit history as a DAG: commits are nodes; edges point from a child commit to its parent(s). A merge commit has multiple parents. Key graph algorithms include:

  • Topological sort (for linearizing history for traversal)
  • Lowest common ancestor (LCA) or nearest common ancestor (to find merge base)
  • Reachability (what commits are in a branch vs another)
Simple algorithm to find a merge-base (nearest common ancestor):
- Walk parents from both commits, marking visited nodes with distance.
- The common ancestor with minimal max(distanceA, distanceB) is a good base.
- Optimizations: bidirectional BFS, heuristics based on commit timestamps.

4.4 Delta compression and storage

To save space, VCS use delta compression: storing differences between objects rather than full copies. Packfile formats (e.g., Git packs) choose reference objects and encode deltas against them. Algorithms aim to minimize total storage size and access cost; heuristics choose which objects to delta against based on similarity and depth limits.

4.5 Hashing and content-addressed storage

Systems like Git use cryptographic hashes (SHA-1 historically, moving to SHA-256) to name objects (blobs, trees, commits). This provides:

  • Integrity: content is verifiable by recomputing the hash
  • De-duplication: identical content maps to the same hash
  • Efficient references: commits refer to trees and parents by hash

Important: Cryptographic hashes are not a substitute for signatures: they verify consistency, not authorship. Signed tags or commits provide cryptographic attribution.

5. How Git works internally (a deep dive)

Git is a representative DVCS; understanding its model clarifies many typical VCS concepts.

Core object types

  • Blob: raw file content (no filename stored in blob); identified by hash of header + content.
  • Tree: a directory listing that maps names to blobs or other trees (stores mode and name entries).
  • Commit: metadata that points to a tree and parent commit(s). Contains author, committer, message.
  • Tag: optional annotated tag object pointing to another object (often a commit).

Refs and HEAD

Refs are pointers to commits: branches (refs/heads/*), remotes (refs/remotes/*), tags (refs/tags/*). HEAD indicates the currently checked-out commit or branch.

Index (staging area)

The index is an intermediate structure that records the exact set of files to include in the next commit. It allows composing a commit from changes in the working tree.

Loose objects vs packfiles

Initially Git stores objects as files (loose). For efficiency, Git packs many objects into a packfile using delta compression. Packfiles are accompanied by an index for fast lookup by hash.

Basic Git command map

git init               # create repository
git clone URL          # copy a remote repository
git status             # see changes
git add          # stage change(s)
git commit -m "msg"    # create a commit from staged content
git branch             # list/create branches
git checkout   # switch branches (or use git switch)
git merge      # merge another branch into current
git rebase       # replay commits onto base (rewrite history)
git fetch              # download remote refs and objects
git pull               # fetch + merge (or fetch + rebase)
git push               # send commits to remote
git log --graph --oneline   # view commit DAG

6. Branching and merging — strategies and narratives

Branching is a key productivity tool. This section gives clear workflows and step-by-step narratives.

Common branching strategies

  • Feature branches: create a branch per feature/bugfix; merge into main when ready.
  • Gitflow: a strict model with develop, feature, release, and hotfix branches. Useful for teams that need formal release staging.
  • Trunk-based development: short-lived branches or direct commits to trunk/main with feature flags. Favored for continuous delivery.

Step-by-step: a safe feature branch workflow

1. git checkout -b feature/awesome
2. Make changes locally.
3. git add file; git commit -m "Implement part"
4. Push the branch: git push -u origin feature/awesome
5. Open a pull request / merge request for review.
6. Address review comments; add commits.
7. When CI passes and review is approved: git merge --no-ff feature/awesome into main (or rebase onto main then merge).
8. Deploy from main.

Merging vs Rebasing (educational narrative)

Both produce a branch that contains other changes; they differ in how history is represented:

  • Merging preserves the exact history and records a merge commit that shows both parents.
  • Rebasing rewrites commits to appear as if they were made on top of another base—this produces a linear history but changes commit hashes (not safe on shared branches unless coordinated).

7. Conflict resolution patterns

Conflicts arise when concurrent changes overlap. Here are patterns to resolve them.

  • Manual edit and test: open conflict markers, edit, run tests, commit.
  • Choose one side: "ours" or "theirs" through VCS options when you intentionally prefer one change.
  • Semantic merge tools: use language-aware merges for structured files (e.g., JSON, XML) to reduce false positives.
  • Divide and reapply: break large changes into smaller commits to ease later merges.

8. Advanced topics and system-level concerns

History rewriting and risks

Operations like rebase, filter-branch, and git commit --amend rewrite history by creating new commits. Avoid rewriting public/shared commits unless everyone agrees because it changes hashes and forces collaborators to reconcile divergent histories.

Garbage collection and pruning

Git runs garbage collection (git gc) to repack objects and remove unreachable objects after a grace period. Concepts include reachability, reflog (a safety mechanism), and expiration policies for unreachable objects.

Large files and Git LFS

Standard Git performance degrades with very large binary files. Git LFS (Large File Storage) stores pointers in Git and keeps large content on a separate server. Alternatives include external artifact stores.

Submodules and subtrees

Submodules embed external repositories at a path by storing a reference to a specific commit. Subtrees allow history from another repo to be merged into a subdirectory. Both have tradeoffs in complexity and independence.

Monorepos vs Polyrepos

Monorepos store multiple projects in a single repo; polyrepos keep separate repos. Monorepos ease cross-cutting changes and tool integration but require careful tooling to scale (partial clones, sparse checkouts, fine-grained CI). Polyrepos reduce repository size and may simplify permissions per project.

Security, signing, and permissions

  • Use signed commits/tags (GPG, SSH signatures) for provenance where required.
  • Enforce access controls at the server or platform level (branch protection rules).
  • Scan history for secrets (API keys) and rotate/rewrite history if leaks occur (with care).

9. Key algorithms — more formal pseudocode and complexity

Patience diff (idea)

Patience diff finds unique matching lines and uses them as anchors; it often yields more comprehensible human diffs. It's a heuristic with good practical results.

Myers complexity recap

Myers runs in O(ND) time and O(N) memory with optimizations. For files with small D compared to length, it is very fast. Worst-case D = N + M (completely different), but typical source code diffs have small D.

Three-way merge pseudocode (simplified)

function ThreeWayMerge(base, left, right):
  editsLeft = Diff(base, left)
  editsRight = Diff(base, right)
  merged = base.clone()
  conflicts = []
  for each region in mergeRegions(base, editsLeft, editsRight):
    if region modified only in one side:
      apply modification to merged
    else if modifications are identical:
      apply modification
    else:
      record conflict for region; insert conflict markers
  return {merged, conflicts}

11. Best practices and team guidance

  • Write clear commit messages: short summary + optional body explaining why.
  • Make small, focused commits that represent a single logical change.
  • Use branch protection to require CI and reviews before merging to main.
  • Automate tests and linting in CI so code quality gates are enforced on every change.
  • Document branching strategy and when to rebase vs merge for your team.
  • Keep secrets out of repository history; scan and rotate if leaked.

12. Glossary (concise)

Blob
Content of a file (in Git terminology).
Tree
Directory listing mapping names to blobs/trees.
Commit
Snapshot metadata referencing a tree (and parent commits).
Merge commit
A commit with two or more parents representing a merge operation.
Index / Staging area
Intermediate area to build the next commit.
Packfile
Compressed storage of many objects with deltas.
Rebase
Rewrite commits to base them onto a different commit.
Fast-forward
When branch pointer can be moved forward without a merge commit.

13. Troubleshooting & recovery patterns

  • Forgot to stage a file before commit: use git add and git commit --amend to update the last commit (if safe).
  • Accidentally deleted a branch: if the commits exist in reflog or another ref, restore with git reflog and git branch or use git fsck to find dangling commits.
  • Merge created conflicts: open conflicted files, resolve, git add them, git commit to finish merge.
  • Corrupt repository: check with git fsck; pack repairs and restore from backups if needed.
