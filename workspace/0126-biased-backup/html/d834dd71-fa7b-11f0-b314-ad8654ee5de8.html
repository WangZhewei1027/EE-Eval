Radix Sort — A Comprehensive Explanation

An in-depth, pedagogical guide to the radix sort family of algorithms: theory, step-by-step descriptions, pseudocode, examples, variations, and practical notes.

Overview and Basic Idea

Radix sort is a non-comparative integer (or fixed-length key) sorting algorithm that sorts items by processing individual digits (or "radices") of the keys. Instead of comparing elements pairwise (as comparison sorts like quicksort or mergesort do), radix sort groups keys by their digits, starting either from the least significant digit (LSD) or the most significant digit (MSD), and uses a stable intermediate sort to maintain the relative order of keys with the same digit.

Key properties (summary):

  • Type: Non-comparative sort (uses digit extraction + stable distribution)
  • Typical time complexity: O(d × (n + k)) where n = number of items, d = number of digits (or key length), k = radix (number of possible digit values, e.g., 10 for decimal)
  • Space: O(n + k) typically, because of buckets or temporary arrays used by the stable subroutine (often counting sort)
  • Stability: Radix sort relies on a stable intermediate sort—stability is essential to preserve order across digit passes

Why radix sort works (intuition)

Think of sorting integers by looking at their digits like sorting words by their letters. If you first group values by least significant digits and place them back keeping their previous order, then group by the next digit, and so on, the final order will be globally sorted. The crucial ingredient is that the intermediate grouping is stable: when two keys have the same digit value on a pass, they must remain in the same relative order they were before the pass. Repeating stable grouping for each digit produces a fully sorted sequence.

There are two main strategies:

  • LSD (Least Significant Digit) radix sort — process digits from least significant to most significant (common for fixed-length numeric keys). Typically uses counting sort as the stable subroutine for each digit.
  • MSD (Most Significant Digit) radix sort — process digits from most significant to least significant, often recursively partitioning the array by digit buckets (useful for strings or variable-length keys).

Pseudocode and step-by-step descriptions

LSD Radix Sort (using Counting Sort per digit)

This is probably the most common presentation used for sorting integers (non-negative) with a fixed base, for example base 10 digits. The algorithm processes digits from the 1s place upward.

function LSDRadixSort(array, base = 10):
    // array: list of non-negative integers
    // base: radix used to extract digits (10 for decimal, 2 for binary, etc.)
    maxValue = max(array) if array not empty else 0
    d = number of digits needed to represent maxValue in given base
    for digitPosition from 0 to d-1:
        // stable sort the array by the digit at digitPosition
        array = stableCountingSortByDigit(array, digitPosition, base)
    return array

function stableCountingSortByDigit(array, position, base):
    // position: 0 for least significant digit, 1 for next, ...
    count = array of zeros length base
    output = array of same length as input
    // 1) Count occurrences of each digit
    for each value in array:
        digit = (value / base^position) % base   // integer division
        count[digit] += 1
    // 2) Accumulate positions (prefix sums) to make it stable
    for i from 1 to base-1:
        count[i] += count[i-1]
    // 3) Place elements into output in reverse order to ensure stability
    for i from length(array)-1 downto 0:
        digit = (array[i] / base^position) % base
        count[digit] -= 1
        output[count[digit]] = array[i]
    return output

Step-by-step narrative for LSD:

  1. Find the maximum key to determine how many digit passes (d) are required.
  2. For each digit position from least significant to most significant:
    • Use a stable distribution (commonly counting sort) keyed by that digit.
    • Copy the result back into the array (or use the output as the new array for the next pass).
  3. After processing all digit positions, the array is sorted.

MSD Radix Sort (recursive bucket partitioning)

MSD sorts by examining the most significant digit first and distributing keys into buckets, then recursively sorts each non-empty bucket by the next digit. This is similar to a radix-based quicksort but without comparisons; stable partitioning per recursion level is not required in the same way as LSD because recursion handles ordering from most significant to least.

function MSDRadixSort(array, base = 10, position = mostSignificantPosition):
    if array size ≤ 1 or position < 0: return array
    // Bucketize by digit at current position
    buckets = array of lists length base
    for value in array:
        digit = (value / base^position) % base
        buckets[digit].append(value)
    // Recursively sort non-empty buckets on the next less significant position
    result = empty list
    for i in 0..base-1:
        if buckets[i] not empty:
            sortedBucket = MSDRadixSort(buckets[i], base, position-1)
            append sortedBucket to result
    return result

MSD is often used for strings or variable-length keys because you can stop recursion on buckets that contain single keys or when you run out of characters.

Complexity and performance considerations

Let n be the number of elements, d the number of digit positions (key length), and k the radix (number of bucket values per digit, e.g., 10 for decimal). Using counting sort (which runs in O(n + k) and is stable) for each digit pass, the time complexity becomes O(d × (n + k)).

Special cases and simplifications:

  • If d is constant or small (for example, sorting 32-bit integers with base 2^8 → d = 4), complexity is effectively O(n).
  • If k is O(n) (e.g., very large base), then the cost for counting becomes O(n) per pass; careful choice of base balances the number of passes d and cost per pass.
  • Space: typically requires O(n + k) auxiliary space for counting arrays and temporary output arrays.

When is radix sort faster in practice?

  • When keys are fixed-length (or have small maximum length) and radix can be chosen to minimize passes.
  • When keys are integers and d is small relative to n (e.g., 32-bit or 64-bit integers using radix 2^8 or 2^16).
  • When avoiding comparisons is beneficial, such as on architectures where branch mispredicts are expensive.

When might you avoid radix sort?

  • If keys are arbitrary precision or variable-length with large d, radix sort may be expensive.
  • If memory is extremely constrained and you cannot afford O(n) temporary arrays.
  • If you need an in-place comparison sort, classical algorithms like quicksort or heapsort might be preferred.

Stability and the role of the stable subroutine

Stability means equal keys preserve their relative order from before to after a sort. Radix sort fundamentally depends on stability when using LSD. If the per-digit sort is not stable, earlier passes' ordering is lost and the algorithm will produce incorrect results.

Counting sort is often used as the stable per-digit sort because it runs in linear time with small alphabet sizes (k small) and is straightforward to make stable (using prefix sums and placing elements in output in reverse order).

Handling negative numbers and non-integer keys

Negative integers

Radix sort is naturally described for non-negative integers. Common ways to handle negatives:

  • Offset technique: add a constant offset to all numbers so the minimum becomes 0, sort, then subtract offset. This only works if offsets don't overflow and keys are bounded.
  • Partition into negatives and non-negatives: move negative numbers to one array, make them positive by absolute value (or invert mapping), sort both halves (negatives and non-negatives) with radix sort, then combine in proper order (negatives need to appear before positives; if you invert absolute values handle ordering and reverse negatives if necessary).
  • Use sign-aware bucketization in MSD approach, or use two passes where one handles sign bit separately if using binary radix.

Strings and lexicographic keys

Strings can be sorted with radix techniques. Two approaches:

  • LSD string sort — works when all strings have fixed length; process characters from rightmost to leftmost using a stable sort by character.
  • MSD string sort — better for variable-length strings; partition strings by their first character into buckets, then recursively sort each bucket by the next character. MSD can be faster when strings have shared prefixes because buckets shrink quickly.

Careful treatment is needed for variable-length strings (for example, using a special end-of-string marker that sorts before any real character, or by tracking lengths in recursion).

Variations, optimizations, and practical notes

  • Choose base (radix) carefully: larger radices reduce d (passes) but increase k (count array size). Typical choices for 32-bit integers are base 2^8 (256) with 4 passes, or base 2^16 (65536) with 2 passes (but counting array becomes bigger).
  • Byte-wise radix on 32/64-bit integers is popular (radix 256) because extracting a byte is cheap and counting array small and cache-friendly.
  • Hybrid approaches: use radix for coarse sorting then switch to insertion/quick/merge sort for small buckets (to reduce overhead).
  • In-place variants exist but are complex and often sacrifice stability or add significant overhead. Many practical implementations use auxiliary arrays for simplicity and performance.
  • Parallelization: radix passes or independent bucket sorts are amenable to parallel execution. This makes radix attractive for GPUs and parallel CPUs.

Implementation notes and pitfalls

  • Always ensure the per-digit sort is stable when using LSD radix sort.
  • Watch out for integer overflow when choosing base and calculating digit extraction. Use unsigned arithmetic or careful shifting when working in binary radices.
  • When sorting pairs or records keyed by integers, move the entire record during bucket placement, not just keys.
  • For strings, variable-length keys require careful handling of end-of-key conditions during MSD or padding for LSD.

Summary and key takeaways

Radix sort is a powerful technique for sorting keys when keys can be broken into fixed-width digits or characters and where a stable per-digit distribution is available. It's especially practical for integer keys (e.g., 32-bit) and for string sorting where MSD/LSD variants can be applied depending on whether keys are variable-length or fixed-length.

Remember:

  • Radix sort is non-comparative; it avoids pairwise comparisons by distributing by digits.
  • Stability of the per-digit sort (for LSD) is essential.
  • Complexity is O(d(n + k)) — choose d (number of digits) and k (radix) to balance performance and memory.
  • Consider memory usage, key ranges, and practicality (e.g., hybridizing with comparison sorts for small buckets).
