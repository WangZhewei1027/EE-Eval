Load Balancing — A Comprehensive Educational Guide

A detailed, text-first explanation of what load balancing is, why it matters, popular algorithms and trade-offs, step-by-step request flows, pseudo-code, examples, operational considerations, and a tiny demo you can run.

What is Load Balancing?

Load balancing is the practice of distributing work (requests, sessions, tasks) across a set of resources (servers, CPUs, network links) so that overall system goals are achieved: maximize throughput, minimize latency, avoid overload of any single resource, and provide fault tolerance. In modern systems, "load balancer" refers both to the conceptual function (deciding where to send work) and to the concrete component that performs that function (software reverse proxies, hardware appliances, DNS layers, client libraries).

Key goals of load balancing

  • Reliability: route around failed or unhealthy nodes.
  • Scalability: allow capacity to grow or shrink by adding/removing nodes.
  • Performance: reduce latency and increase throughput.
  • Fairness: distribute requests to avoid hotspots and resource exhaustion.
  • Predictability: ensure reasonable response times and tail latency behavior.

Core Concepts and Metrics

Before diving into algorithms, it helps to agree on common metrics and concepts used to evaluate load balancing strategies:

  • Throughput — requests processed per second across the pool.
  • Latency — response time per request (mean and tail percentiles).
  • Utilization — fraction of capacity used on each node (CPU, memory, network).
  • Connection count — active concurrent connections or in-flight requests.
  • Fairness — difference between the most and least loaded nodes.
  • Stability — how quickly the balancer reacts to load and how much it oscillates.

Categories of Load Balancing

Algorithms can be grouped by where they make decisions and how much information they use:

  • Client-side — clients pick a node (e.g., service discovery + client library). Good for scaling, less central bottleneck.
  • Network/DNS-based — DNS round-robin or network hardware. Simple, but can be coarse and have caching issues.
  • Proxy/load balancer (server-side) — a reverse proxy (HAProxy, NGINX) routes traffic. Centralized control, rich feature set.
  • Distributed — many peers performing balancing using consistent hashing or rendezvous hashing (e.g., caches, DHTs).
  • Static vs Dynamic — static algorithms ignore current load (e.g., simple round-robin); dynamic algorithms use runtime metrics (e.g., least-connections).

Common Algorithms: Explanations and Pseudocode

Here are core algorithms you will encounter. Each includes a short textual explanation and simple pseudocode.

1) Round-Robin

Definition: Cycle through the backend list in order, sending each incoming request to the next server.

When to use: Stateless requests with similar work per request; very simple.

state: index = 0
onRequest(request):
  server = servers[index]
  index = (index + 1) mod servers.length
  route request to server

Pros: Simple, even distribution if requests are uniform. Cons: doesn't account for server capacity or differing request sizes; fails with long-lived connections unless adapted.

2) Weighted Round-Robin

Definition: Each server has a weight proportioning how many requests it should receive. Useful when servers have different capacities.

state: index = 0
onRequest(request):
  find next server using weights (e.g., expand list by weights or use smooth weighted algorithm)
  route request to chosen server

Note: Many implementations use the "smooth weighted round-robin" algorithm to avoid bursts and better distribute weights over time.

3) Least-Connections

Definition: Send the request to the server with the fewest active connections (in-flight requests). This is a simple dynamic algorithm that assumes fewer active connections => more free capacity.

onRequest(request):
  server = argmin(servers, server.activeConnections)
  server.activeConnections += 1
  route request to server
onResponse(server):
  server.activeConnections -= 1

Pros: Adapts to uneven request lengths. Cons: Requires accurate tracking of active connections; can oscillate if many decisions occur quickly.

4) Least-Response-Time

Definition: Choose the server with the smallest average response time (adjusted by active connections). Typically needs metrics collection.

This is more precise but requires telemetry and can be noisy; smoothing and exponential averaging are commonly used.

5) Randomized

Definition: Pick a server at random. Surprisingly effective in large pools and is simple and stateless.

Variant: Weighted random (probability proportional to capacity).

6) IP Hash / Source Hashing

Definition: Hash client IP (or another attribute) to pick server deterministically. Useful for session affinity without shared session storage.

Limitations: Uneven distribution if client IPs are skewed; sensitive to topology changes (node addition/removal causes wide remapping).

7) Consistent Hashing

Concept: Map keys (e.g., session ID, object key) and servers onto the same hash ring; a key maps to the next server clockwise. When nodes change, only a small subset of keys remap.

# Simplified conceptual flow:
for each server:
  place server on hash ring (often with multiple virtual nodes)
onRequest(key):
  h = hash(key)
  server = ring.findFirstNodeClockwise(h)
  route to server

Pros: Minimal remapping on membership changes—excellent for caches and partitioning. Requires careful choice of virtual nodes to smooth distribution.

8) Rendezvous (Highest Random Weight) Hashing

Definition: For each request key, compute a score per server (hash(server_id, key)); choose server with highest score. Simple, deterministic, and also minimizes remapping.

onRequest(key):
  best = null; bestScore = -inf
  for each server:
    score = hash(server.id, key)
    if score > bestScore:
      bestScore = score
      best = server
  route to best

Rendezvous hashing is often simpler to implement than ring-based consistent hashing and has good distribution with fewer virtual nodes.

Step-by-Step: How a Request Flows Through a Typical HTTP Load Balancer

  1. Client opens TCP connection and sends HTTP request to LB address.
  2. LB performs health check evaluation and picks a backend according to its algorithm and current metrics.
  3. If the protocol requires session affinity, LB determines whether to honor sticky session and may consult cookies or source IP hashing.
  4. LB optionally terminates TLS (SSL offload) and can inspect HTTP headers for routing decisions (L7 routing).
  5. LB forwards the request via a new or pooled connection to the chosen backend (preserving client IP via PROXY protocol or X-Forwarded-For if needed).
  6. Backend processes request and returns response; LB may stream response back to client while monitoring connection state.
  7. Connection draining: when removing a backend, LB stops sending new requests to it and waits for in-flight requests to complete or forcibly terminates after timeout.

Operational Considerations

Health checks

Active health checks probe backends (HTTP GET, TCP connect, application-level checks) and mark servers IN/OUT. Only healthy servers should receive traffic.

Session affinity (sticky sessions)

Sticky sessions keep a client's successive requests on the same backend (via cookies or source IP). This simplifies stateful apps but reduces load distribution flexibility and complicates maintenance.

SSL/TLS termination and re-encryption

Load balancers often terminate TLS to inspect and route traffic, possibly re-encrypting to backends. Consider certificate management, key handling, and performance impact.

Connection pooling and keep-alives

Reusing backend connections reduces latency and CPU overhead on backends. Balancers may multiplex many client connections onto fewer backend connections.

Sticky state vs externalizing state

Prefer stateless services with external state (databases, caches) to allow flexible balancing and autoscaling. Stateful services complicate node replacement and rolling updates.

Autoscaling and coordination

Load balancers should integrate with auto-scaling: add instances and register them with health checks; on removal, drain traffic before termination.

Performance and Complexity Trade-offs

  • Stateless algorithms (round-robin) are O(1) per decision and cheap but may be suboptimal when work is heterogeneous.
  • Dynamic algorithms (least-connections, response-time) require telemetry and additional state, raising operational complexity.
  • Consistent hashing adds complexity for membership changes but improves data locality for caches.
  • Centralized balancers provide powerful features but can be a single point of failure; typically you deploy them in high-availability pairs or use DNS failover.

Example: Smooth Weighted Round-Robin (concept)

Used by NGINX and others. Smoothly distributes requests according to weights over time.

# servers each have: weight, current
onStart:
  for each server s:
    s.current = 0
onRequest:
  for each server s:
    s.current += s.weight
  choose server with max(s.current)

  route to server

Monitoring, Testing and Hardening

Measure and monitor:

  • Per-backend request rate, latency percentiles, error rates.
  • Active connections, CPU/memory utilization, network bandwidth.
  • Health check status, number of removed/added hosts over time.

Testing strategies:

  • Load test with realistic traffic patterns and tail-latency focus.
  • Chaos testing: kill nodes and observe how quickly LB routes around failures.
  • Capacity planning: simulate peak traffic and autoscaling latency.

Common Pitfalls

  • Sticky sessions preventing even distribution and hindering scaling.
  • Large variance in request workload makes simple round-robin inefficient.
  • DNS-based balancing suffers from caching and client-side TTLs.
  • Backend slow-start/ warm-up: adding many servers instantly may cause overload if they receive traffic faster than they can initialize caches.
  • Health check flapping causing oscillation—use hysteresis and cooldowns.

When to Use Specific Algorithms (Guidance)

  • Round-robin: stateless, homogeneous servers, simple setup.
  • Weighted RR: servers with differing capacities.
  • Least-connections: varied request durations, state tracked by active connections.
  • Hashing (consistent / rendezvous): you need stable key -> server mappings (caches, sharding).
  • Client-side balancing: when you want to avoid central bottlenecks or prefer library-based routing with service discovery.

Short Case Study: Cache Tier with Consistent Hashing

Problem: You have a distributed in-memory cache (e.g., memcached) and want minimal key remapping when scaling up/down.

Solution: Use consistent hashing with virtual nodes. When you add a server, only the range of keys between the previous node and the new node are remapped. To smooth distribution, configure ~100 virtual nodes per physical server (tune based on pool size).

Example: With 4 real servers and 100 virtual nodes each, only ~1/5 of keys move when removing one server from a pool of five (roughly proportional).
