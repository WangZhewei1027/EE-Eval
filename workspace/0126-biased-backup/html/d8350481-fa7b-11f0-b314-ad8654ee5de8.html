TimSort — A Comprehensive Educational Explanation

TimSort is a hybrid, stable sorting algorithm derived from merge sort and insertion sort. It is the standard sorting algorithm for Python's lists and for Java's Arrays.sort for certain object types. This document explains the theory, algorithmic steps, implementation notes, invariants, and examples in a thorough, educational manner.

1. High-level summary and motivation

TimSort was designed by Tim Peters in 2002. Its goal is to take advantage of the existing order in real-world data (common partially-sorted patterns) to achieve better-than-generic mergesort performance in practice while keeping worst-case guarantees of O(n log n). It is a "natural" mergesort variant augmented with insertion sort for short runs, and it uses a carefully managed run stack and an adaptive merging strategy (including "galloping") to be both efficient and robust.

  • Stable: preserves relative order of equal elements.
  • Adaptive: runs (already-sorted subsequences) are exploited.
  • Hybrid: uses insertion sort for small runs and merge-based merging for larger runs.
  • Used in practical systems: Python lists, Java (for non-primitive objects), Android.

2. Core ideas and components

  1. Run detection: Scan the array to find runs (monotonic runs: ascending or descending). Reverse descending runs to make them ascending.
  2. Minrun: Ensure every run has at least a minimum length ("minrun") by extending short runs using insertion sort.
  3. Run stack and invariants: Push runs onto a stack and merge runs based on size invariants to keep merges balanced.
  4. Merge and galloping: Merge runs using a stable merge. If one side starts winning repeatedly, switch to "galloping" (exponential search + binary search) to reduce comparisons.

Why these pieces? Natural runs give a linear-time best case (already sorted). Small-run insertion sorts are fast for short data. Carefully merging with stack invariants keeps the overall merge tree balanced, yielding O(n log n) worst case while being fast in practice.

3. Detailed algorithmic steps (narrative)

We will narrate the algorithm as if walking through the code:

  1. Compute minrun for the array length n. This controls the size threshold of runs to produce; minrun is typically between 32 and 64 for practical n.
  2. Linearly scan the array to identify runs:
    • Start at index i. If a[i] <= a[i+1], it's an ascending run. Continue until order breaks.
    • If a[i] > a[i+1], it's a descending run. Continue until it stops decreasing, then reverse the whole run to make it ascending.
  3. If the run length is less than minrun, extend the run to minrun (or to array end) using binary/insertion sort to place elements into the run.
  4. Push the run length and start index onto a run-stack structure.
  5. After pushing, check the run-stack invariants. While invariants are violated, merge runs on the stack. The merge chooses adjacent runs to merge to restore invariants; this keeps merges balanced.
  6. After processing all runs from the input, merge the remaining runs on the stack until only a single run remains — the fully sorted array.

4. Computing minrun (detailed)

Minrun is computed from n using a small routine. The idea is to choose a minrun between 32 and 64 that evenly balances the number of runs vs run sizes.

function computeMinRun(n):
    r = 0
    while n >= 64:
        r |= (n & 1)
        n >>= 1
    return n + r

Explanation: repeatedly shift n right until it is less than 64. Keep track if any low-order 1 bits were lost (using r). The final minrun is n + r, producing a value in [32, 64] for typical n and giving desirable properties for run balancing.

5. Run detection and extension

Run detection is one linear pass. For each run:

  • If the run is descending, reverse it in-place to become ascending. Reversing is O(length).
  • If run length < minrun, perform insertion sort on the next (minrun - length) items to extend the run to minrun (or to the end of the array).

Why insertion sort? Because insertion sort is excellent on small arrays and on arrays that are already nearly sorted; using it to grow short runs is efficient and keeps the algorithm stable.

6. Run stack invariants and merge policy

When runs are discovered, TimSort pushes their start index and length onto a stack. To avoid creating very unbalanced merges (which would harm performance), TimSort maintains invariants that force some merges early. Let the runs on the stack (from bottom to top) be R0, R1, R2, ..., Rk with lengths L0, L1, L2, ..., Lk. Consider the top three run lengths: X=L[k-2], Y=L[k-1], Z=L[k] (if they exist).

The invariants (expressed informally) are:

  • While there are at least three runs on the stack and X ≤ Y + Z, merge either X and Y or Y and Z (choose the smaller merge to keep balance).
  • Then, while there are at least two runs and Y ≤ Z, merge Y and Z.

In implementation form, after each push, TimSort repeatedly performs: 

while stack has at least 2 runs:
    if (len[-3] <= len[-2] + len[-1]) then
        if len[-3] < len[-1]:
            merge runs -3 and -2
        else:
            merge runs -2 and -1
    else if (len[-2] <= len[-1]):
        merge runs -2 and -1
    else:
        break

This ensures that no run is overwhelmingly larger than the sum of the next two runs, which provides a good bound on the number of merges and keeps merge trees shallow.

7. Merging runs

Merging two sorted runs is the familiar stable merge. TimSort optimizes this in two ways:

  1. Copy the smaller of the two runs into a temporary buffer, then merge from the buffer into the destination. This minimizes temporary storage and data moves.
  2. Use galloping mode when one side repeatedly "wins": instead of taking one element at a time, do an exponential search (gallop) to find a block of elements that can be copied in bulk using binary searches. This reduces comparisons dramatically on certain inputs (e.g., when runs are highly unbalanced or when data has long stretches where one run's elements are all smaller).

Galloping mode is adaptive: there is a threshold (initially often 7) that determines when to switch to galloping. The threshold can change during runtime depending on success of galloping.

Important: even with galloping, TimSort remains stable because elements taken from the left run are always placed before equal elements from the right run.

8. Complexity and resource usage

  • Best case: O(n) — when the array is already composed of long runs (e.g., fully sorted).
  • Average: O(n log n) — similar to mergesort on random inputs, but with smaller constants in practice.
  • Worst case: O(n log n) — the stack invariants and merge policy guarantee the bound.
  • Extra space: O(n) worst-case for temporary storage during merges, but typical temporary usage is much smaller (often proportional to the smaller run being merged).

TimSort design focuses on practical speed on real-world inputs rather than minimizing theoretical space. The algorithm's constants and adaptive techniques make it faster on many inputs than a plain mergesort or quicksort (despite equivalent asymptotic complexity).

9. Stability and correctness

TimSort is stable by construction because when merging, equal elements from the left run are always copied before equal elements from the right run. The algorithm performs only comparisons and stable merges; reversing descending runs and using insertion sort preserves stability (in insertion sort, stable insertion is used).

Correctness outline: each run is always sorted; merges produce sorted runs covering the union of their inputs; stack invariants ensure a sequence of merges that eventually combines all runs into one. The sequence of merges preserves order and sorts by pairwise stable merging — hence the final array is sorted and stable.

10. Pseudocode (comprehensive)

TIMSORT(A):
    n = length(A)
    if n < 2: return A

    minrun = computeMinRun(n)
    runStack = empty stack

    i = 0
    while i < n:
        runLen = countRun(A, i)            # find ascending or descending run; reverse if descending
        if runLen < minrun:
            force = min(minrun, n - i)
            binaryInsertSort(A, i, i + force)
            runLen = force
        push (i, runLen) onto runStack
        collapseWhileNeeded(runStack, A)   # enforce invariants by merging top runs
        i += runLen

    # merge remaining runs until one remains
    while runStack.size > 1:
        mergeTopTwoRuns(runStack, A)

    return A

Helper functions described above: countRun detects monotonically ascending/descending sequences. collapseWhileNeeded follows the merge-invariant rules described in section 6. mergeTopTwoRuns performs a stable merge using temporary buffer + galloping.

11. Example walkthrough (step-by-step)

Take the small array: [5, 21, 7, 23, 19, 3, 2, 9, 10, 11]

  1. Assume computeMinRun(10) returns 10 because n is small (for the example we will extend to minrun=4 for demonstration). Start scanning:
  2. Index 0: 5 < 21 (ascending): run = [5,21] length 2. Since 2 < minrun (4), extend using insertion sort on next elements to make run length 4. After extension and insertion, the first run might become [5,7,21,23] (depending on values).
  3. Continue after the first run boundary, find next run(s), reverse descending runs if found, extend to minrun, push onto stack after each run.
  4. After several runs are pushed, the stack invariants are checked and small merges occur (e.g., merge adjacent runs of lengths 4 and 4 to produce an 8-length run).
  5. At the end, runs are merged until a single sorted array emerges.

This walkthrough abstracts details but highlights the flow: detect run, extend, push, and merge according to invariants — repeatedly until done.

12. Implementation notes & practical tips

  • Binary insertion sort is commonly used to extend runs because it minimizes comparisons when inserting into already-sorted prefixes.
  • The galloping threshold (often started at 7) is tuned adaptively: when galloping helps, decrease threshold; when it hurts, increase threshold. This makes the algorithm self-tuning for different inputs.
  • Carefully implement temporary buffers: copy the smaller run to temporary memory to reduce movement and memory usage.
  • Beware integer overflow in run-size calculations in low-level languages when dealing with huge arrays; use appropriate integer types.
  • Edge cases: arrays with many equal elements; stability is crucial and TimSort handles it. Binary searches used in galloping must also be stable (tie-breaking keeps left before right).

13. Comparisons and when to use TimSort

TimSort is excellent in contexts where:

  • Input often has partial order (sorted or nearly-sorted sections).
  • Stability is required.
  • Predictable worst-case O(n log n) behavior is desired and low-level tuning of comparisons matters.

Compared to quicksort: TimSort is stable and often faster on partially sorted or real-world data. Quicksort can be faster on random data in low-overhead implementations but requires measures to avoid worst-case quadratic behavior (randomization or pivot selection).

15. Common misconceptions

  • TimSort is not "just mergesort": it is a carefully tuned hybrid that leverages natural runs and adaptive merging for real-world performance.
  • TimSort is not unstable: it's explicitly stable by design.
  • TimSort does not always use more memory than mergesort — it copies only the smaller of the two runs during each merge, minimizing temporary usage in practice.
