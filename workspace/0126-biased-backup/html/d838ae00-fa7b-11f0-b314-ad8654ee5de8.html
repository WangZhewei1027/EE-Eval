Query Optimization — A Comprehensive Textual Guide

This document explains the theory, algorithms, and practical considerations behind query optimization. It is written as an educational reference with definitions, algorithm descriptions, step-by-step examples, and taxonomy of techniques. Interactivity is intentionally minimal: a single demonstration button reveals a short, static walkthrough of join enumeration and costing for a small example.

Overview and Motivation

Query optimization is the process of transforming a high-level declarative query (for example SQL) into an efficient execution strategy that returns the required results. The optimizer chooses how to access and combine data while minimizing a cost metric (typically execution time or resource consumption).

Goals of query optimization:

  • Produce a correct plan that returns the same result as the query.
  • Minimize a cost metric: wall-clock time, CPU, I/O, network, or a weighted combination.
  • Make robust choices even with imperfect statistics or changing runtime conditions.

High-level Architecture and Phases

Most relational DBMS optimizers follow a pipeline with distinct phases:

  • Parsing and validation: Convert SQL text into an abstract syntax tree (AST); check names, types, privileges.
  • Translation to logical algebra: Produce a relational algebra expression (operators like Scan, Join, Filter, Project, Aggregate).
  • Logical optimization / rewrite: Apply semantics-preserving rewrites (predicate pushdown, projection pruning, join associativity/commutativity, subquery unnesting).
  • Plan enumeration and cost-based selection: Explore physical plan alternatives and estimate costs to choose an optimal plan.
  • Code generation / plan finalization: Generate an execution plan (possibly compiled code) and submit to the runtime engine.
  • Execution: Execute the chosen plan; modern systems may adapt during execution.

Key Definitions

Important terms used throughout:

  • Logical plan: An operator tree expressing what to compute (e.g., a join of R and S) without specifying algorithms.
  • Physical plan: A plan that specifies concrete algorithms and access methods (e.g., Hash Join, Index Scan).
  • Plan space: The set of all physically distinct plans that produce the query result.
  • Cost model: A mathematical model that assigns a numeric cost to a physical plan, often based on estimated I/O, CPU, and memory.
  • Cardinality: The estimated number of rows produced by an operator.
  • Selectivity: The fraction of rows that satisfy a predicate.

Cost Models and Cardinality Estimation

Cost-based optimization depends critically on accurate estimates of cardinality and cost. Cost models vary:

  • Simple: counts of logical I/O, sequential page reads.
  • Refined: separate CPU vs I/O, consider cache effects.
  • Complex: include network transfer cost (distributed systems), disk seek models, multi-level storage tiers.

Cardinality estimates come from database statistics:

  • Table cardinality (row counts)
  • Histograms for column value distributions (equally-spaced, equi-height, V-Optimal)
  • Number of distinct values (NDV) or distinct count sketches
  • Multi-column statistics (correlation information), samples, sketches

Common selectivity assumptions (and their limitations): independence assumption — predicates on different columns are treated as independent (often wrong when columns are correlated). uniformity — values are assumed uniformly distributed across ranges (rarely true in practice).

Logical Transformations and Rewrites

Logical rewrites improve the plan space without changing semantics. Examples:

  • Predicate pushdown: Move filters as close as possible to data sources to reduce intermediate sizes.
  • Projection pushdown / column pruning: Keep only needed columns to reduce I/O and memory footprint.
  • Join reordering: Use associative and commutative properties to explore alternative join sequences.
  • Subquery unnesting: Convert correlated subqueries to joins where beneficial.
  • View and expression simplification: Replace expressions with precomputed or cheaper forms.

Plan Enumeration Strategies

The optimizer must search a combinatorial plan space. Common strategies:

Exhaustive dynamic programming (System R / Selinger)

Dynamic programming builds optimal subplans for every subset of relations, then combines them. It guarantees the best plan among considered physical operator choices under the cost model for left-deep (or more generally restricted) search spaces. 

// Simplified dynamic programming outline (textual pseudocode)
For each single relation R:
  bestPlan[{R}] = best physical access plan for R
For size = 2 to N:
  For each subset S of relations with |S| = size:
    For each partition (A,B) of S:
      candidate = combine(bestPlan[A], bestPlan[B]) using a join algorithm
      if cost(candidate) < cost(bestPlan[S]):
        bestPlan[S] = candidate
Return bestPlan[All relations]

Notes: Complexity is exponential in number of relations; pruning and memoization mitigate cost but cannot avoid worst-case exponential growth.

Greedy / Heuristic Methods

Greedy approaches choose the locally cheapest join at each step (e.g., pairwise join the best two relations), often using heuristics like choosing the join with smallest estimated intermediate size. They are fast but not guaranteed optimal.

Randomized and Iterative Improvement

Approaches such as simulated annealing, genetic algorithms, or hill-climbing explore the plan space by perturbing plans and accepting improvements (and sometimes worse plans to escape local minima). They are useful for large queries where exhaustive search is infeasible.

Top-down and Memoization (Cascades Framework)

The Cascades framework organizes exploration as rule-driven transformations using a memo structure: logically equivalent expressions are grouped and enumerated, and cost-based pruning is applied. It cleanly separates search control from rewrite rules.

Join Ordering and Complexity

Choosing the best join order is one of the core challenges: for N relations, the number of join orders is super-exponential. Formally, join-order optimization is NP-hard. Therefore, practical optimizers use a mix of exact algorithms for small N and heuristics or randomized search for larger N.

Important plan shapes:

  • Left-deep trees: Each join's right child is a base relation. Favored by many cost-based systems for pipeline-friendly execution and easier plan enumeration.
  • Right-deep trees: Useful for parallel hash-join builds.
  • Bushy trees: Fully general; allow joining arbitrary subtrees (often leads to better plans but larger search space).

Join Algorithms (Physical Operators)

Common join algorithms include:

  • Nested Loop Join (NLJ): For each row in outer, probe inner (inner can be an indexed lookup). Costly for large inputs unless one side is small or indexed.
  • Hash Join: Build hash table on smaller input; probe with other input. Good for equality joins and large unsorted data when memory permits.
  • Sort-Merge Join: Sort both inputs on join key and merge. Good for large inputs already sorted or for range joins.

Choosing a physical algorithm depends on input sizes, available memory, presence of indexes, and join predicates.

Access Methods and Indexes

How you access base tables matters: a full table scan, an index range scan, or an index seek produce different costs and cardinalities. The optimizer considers:

  • B-tree / clustered index scans for range queries.
  • Bitmap indexes for low-cardinality attributes or combined predicates.
  • Index intersection/covering indexes to avoid lookups.
  • Index selection based on selectivity and predicate shapes.

Example: Step-by-step Join Enumeration and Cost Comparison

Below is a compact, self-contained demonstration for a three-table join. Click the button to reveal a short walkthrough that explains how a small optimizer might enumerate plans and pick a best plan using simple cost estimates.

Example SQL

SELECT *
FROM R, S, T
WHERE R.a = S.a AND S.b = T.b AND R.filter = 'X';

Assume the following:

  • |R| = 10,000 rows; predicate R.filter='X' has selectivity 0.05 → R_filtered ≈ 500 rows.
  • |S| = 20,000 rows; join selectivity for equality is approximated via NDV and uniformity.
  • |T| = 50,000 rows.
  • Cost model (very simplified): cost ≈ number of rows processed (I/O/CPU proxy).
Step 1 — Pre-filtering: Apply R.filter early (predicate pushdown) → R' ≈ 500 rows. This reduces downstream costs.
Step 2 — Consider pairwise join costs: Compute estimated output sizes for joins (hypothetical calculations):
  • R' ⋈ S: assume selectivity 0.01 → output ≈ 500 * 20,000 * 0.01 = 100,000 rows.
  • S ⋈ T: assume selectivity 0.005 → output ≈ 20,000 * 50,000 * 0.005 = 5,000,000 (large).
  • R' ⋈ T (via S): not directly joinable on keys, so skipped for pairwise).
Using these crude numbers, R' ⋈ S seems much smaller than S ⋈ T.
Step 3 — Join ordering options (left-deep): Options:
  1. ((R' ⋈ S) ⋈ T): cost ≈ cost(R' scan + S scan + cost join producing 100k) + join with T (100k ⋈ 50k → maybe 100k*50k*sel_small). Rough simplified score: low-medium.
  2. ((S ⋈ T) ⋈ R'): cost likely very large because S ⋈ T is huge (5M rows), then joining R' to that big result.
Heuristically prefer plan (1).
Step 4 — Pick physical algorithms: If R' is small, build hash on R' and probe with S (or vice versa). For (R' ⋈ S) choose HashJoin with R' as build side; then join result with T using HashJoin with smaller side as build.
Conclusion: Simple optimizer selects plan: Filter R first → HashJoin(R',S) → HashJoin(result,T). The exact costs depend on more precise cardinality estimates; wrong estimates can reverse choices.
This small demo uses simplified arithmetic to illustrate reasoning. Real optimizers use richer statistics and more nuanced cost functions.

Pruning and Optimization Tricks

To handle large plan spaces, optimizers use pruning:

  • Cost-based pruning: Keep only k best plans for each expression (top-k pruning).
  • Dominance pruning: Discard plans that are strictly worse in cost and output properties than another.
  • Property-based pruning: Use required properties (e.g., sort order) to avoid planning unnecessary alternatives.
  • Bushy search limits: Limit bushy exploration to small subproblems or use heuristics to decide when to allow bushy trees.

Adaptive and Re-Optimization Techniques

When runtime observations differ from estimates, adaptive techniques can improve performance:

  • Runtime re-optimization: Detect major cardinality deviations and re-plan mid-query.
  • Eddies and adaptive join operators: Dynamically route tuples through join operators and reorder operators at runtime.
  • Runtime feedback-driven tuning: Use past execution statistics to refine future estimates (plan cache with feedback).

Distributed and Parallel Query Optimization

In distributed systems, optimizers must also consider data placement, network cost, data shipping, and parallel execution strategies:

  • Operator placement: where to run each operator (co-located with data or centralized).
  • Partition-wise joins: leverage data partitioning by join key to avoid shuffling.
  • Shuffle cost modeling: network I/O can dominate cost, so plan selection must minimize cross-node transfer.
  • Replica-aware optimization: choose replicas considering load and locality.

Practical Challenges and Pitfalls

Optimizers face multiple practical difficulties:

  • Cardinality estimation errors: Small estimation mistakes can cascade into drastically wrong join orders.
  • Data skew: Non-uniform distributions cause hash bucket imbalances and mispredicted costs.
  • Correlated predicates: Independence assumptions fail for correlated columns.
  • Parameter sniffing / prepared statements: Plans optimized for particular parameter values may perform poorly for others.
  • Cost model inaccuracies: Real-world systems have caches, multiple layers, asynchronous I/O; simple models may mispredict.

Optimization Algorithms — Summary and Comparative Notes

Which algorithm to use depends on workload and query size:

  • Exact dynamic programming: Good for small-medium queries; guarantees optimality within search space.
  • Greedy heuristics: Fast and predictable; may produce suboptimal plans for complex joins.
  • Randomized search: Scales to larger queries, can escape local minima; repeatability can be an issue.
  • Cascades-style frameworks: Flexible rule-based rewriting with cost-driven enumeration; supports sophisticated transformations.

Emerging and Research Directions

Recent work explores learning-driven optimization:

  • Learned cost models: Use ML to predict cardinalities and costs from features instead of hand-crafted formulas.
  • Reinforcement learning for join ordering: Train agents to select join orders from state features.
  • End-to-end learned optimizers: Replace parts of or whole optimizer logic with models trained on query workloads.

These approaches promise adaptability but must integrate well with correctness guarantees and runtime variability.

Practical Tips for Developers and DBAs

  • Collect and refresh statistics regularly (histograms, NDV, multi-column stats for known correlated columns).
  • Write predicates to enable index usage and predicate pushdown; avoid non-sargable expressions on indexed columns.
  • Be cautious with views and abstractions that hide cardinality effects from optimizer.
  • Test queries with realistic data distributions and parameter values.
  • Monitor actual runtime plans and consider hints sparingly for critical queries when optimizer repeatedly makes poor choices.

Concise Reference: Checklist of Optimizer Techniques

  • Logical rewrites: pushdown, prune, flatten, unnest.
  • Enumeration: dynamic programming, heuristics, randomized search, cascades.
  • Physical selection: join algorithms, scans, indexing, materialized view usage.
  • Pruning: top-k, dominance, property-driven, bushy-limits.
  • Cardinality & statistics: histograms, NDV, samples, multi-column stats.
  • Adaptive techniques: re-optimization, Eddies, runtime feedback.
  • Distributed concerns: partitioning, data shipping, operator placement.
