SQL — Structured Query Language

A thorough, text-centered introduction to the concepts, theory, algorithms, and practical considerations of SQL and relational database querying. This document emphasizes explanations and pedagogy. Minimal interactivity is provided only to illustrate a short example execution plan.

1. What is SQL?

SQL (Structured Query Language) is a domain-specific, declarative language used to interact with relational database management systems (RDBMS). SQL expresses what result you want (for example, rows that meet conditions) rather than how to compute it. The RDBMS interprets SQL statements and performs the necessary steps to retrieve or modify data.

Key categories of SQL statements

  • DDL (Data Definition Language): CREATE, ALTER, DROP — define and modify schema.
  • DML (Data Manipulation Language): SELECT, INSERT, UPDATE, DELETE — query and change data.
  • DCL (Data Control Language): GRANT, REVOKE — security and privileges.
  • TCL (Transaction Control Language): BEGIN/START TRANSACTION, COMMIT, ROLLBACK — manage transactions.

2. The Relational Model (brief)

The relational model organizes data into relations (tables). Each relation consists of tuples (rows) and attributes (columns). Core theoretical concepts include:

  • Schema: the structure of a table (column names and types).
  • Primary key: one or more columns that uniquely identify a row.
  • Foreign key: a column that references a primary key in another table to express relationships.
  • Relational algebra: a formal system (set, tuple, and predicate operations) underpinning SQL—includes selection, projection, join, union, difference, aggregation.

3. Basic SQL: syntax and examples

Below are canonical examples demonstrating structure and intent. They highlight typical usage and common clauses.

Defining a table (DDL)

CREATE TABLE employees (
  id INT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  department_id INT,
  salary DECIMAL(10,2),
  hired_at DATE
);

Inserting rows (DML)

INSERT INTO employees (id, name, department_id, salary, hired_at)
VALUES (1, 'Alice', 10, 75000.00, '2020-06-01');

Querying rows

SELECT id, name, salary
FROM employees
WHERE salary > 60000
ORDER BY salary DESC
LIMIT 10;

SELECT statement anatomy

The canonical SELECT structure (simplified) is:

SELECT [DISTINCT] select_list
FROM table_expression
[WHERE predicate]
[GROUP BY grouping_list]
[HAVING group_predicate]
[WINDOW ...]
[ORDER BY order_list]
[LIMIT ...];

Notes:

  • SELECT list can contain columns, expressions, aggregates, or window functions.
  • WHERE filters rows before grouping. HAVING filters groups after aggregation.
  • ORDER BY is applied after selection and aggregation; sorting can be expensive.

4. Joins and set operations

Joins combine rows from two or more tables based on related columns. Understanding joins is fundamental.

Join types and semantics

  • INNER JOIN: returns rows where join condition matches in both tables.
  • LEFT (OUTER) JOIN: keeps all left-table rows, adds matching right rows or NULLs if none.
  • RIGHT (OUTER) JOIN: symmetric to LEFT JOIN.
  • FULL (OUTER) JOIN: keeps rows present in either table, filling NULLs where missing.
  • CROSS JOIN: Cartesian product (every row of A with every row of B).
  • SELF JOIN: joining a table to itself (useful for hierarchical or comparative queries).

Example: inner join

SELECT e.name, d.name AS department
FROM employees e
JOIN departments d ON e.department_id = d.id
WHERE e.salary > 50000;

5. Aggregation, grouping, and window functions

Aggregates compute summary values across groups of rows. Window functions compute values across partitions without collapsing rows.

Aggregates

  • Common aggregate functions: COUNT(), SUM(), AVG(), MIN(), MAX().
  • GROUP BY forms groups; every non-aggregated column in SELECT typically must appear in GROUP BY.
  • HAVING filters groups (e.g., groups with COUNT() > 10).

Window functions

Window functions operate over a set of rows related to the current row, defined by OVER(...) with optional PARTITION BY and ORDER BY. They do not reduce rows.

SELECT id, name, department_id, salary,
  RANK() OVER (PARTITION BY department_id ORDER BY salary DESC) AS dept_rank
FROM employees;

6. Subqueries, correlated subqueries, and CTEs

Subqueries are nested queries used in SELECT, FROM, WHERE, or HAVING. A correlated subquery references outer query columns and is evaluated per-row (and thus can be expensive).

-- correlated subquery example:
SELECT e.id, e.name,
  (SELECT AVG(salary) FROM employees e2 WHERE e2.department_id = e.department_id) AS dept_avg
FROM employees e;

Common Table Expressions (CTEs) provide named temporary result sets that can improve readability and modularity:

WITH dept_salary AS (
  SELECT department_id, AVG(salary) AS avg_salary
  FROM employees
  GROUP BY department_id
)
SELECT e.name, d.avg_salary
FROM employees e
JOIN dept_salary d ON e.department_id = d.department_id;

7. Transactions, ACID, and isolation

Transactions group multiple operations into an atomic unit. ACID stands for Atomicity, Consistency, Isolation, Durability.

ACID explained

  • Atomicity: all operations in a transaction succeed or none do.
  • Consistency: transactions move the database from one valid state to another (application-defined integrity constraints hold).
  • Isolation: concurrently executing transactions appear as if executed serially to each other, to a degree determined by isolation level.
  • Durability: once committed, changes persist even after a crash.

Isolation levels (ANSI SQL)

Common isolation levels with typical phenomena prevented:

  • READ UNCOMMITTED: allows dirty reads (lowest isolation)
  • READ COMMITTED: prevents dirty reads, still allows non-repeatable reads and phantoms
  • REPEATABLE READ: prevents dirty and non-repeatable reads; some systems also prevent phantoms
  • SERIALIZABLE: strongest; transactions are serializable — no phantoms, provides a serial execution appearance

Example (conceptual): Dirty read occurs when transaction A writes a value, transaction B reads it before A commits, then A rolls back — B saw a value that never persisted.

8. Concurrency control and locking

Databases use locking or multi-version concurrency control (MVCC) to manage concurrent access. Key approaches:

  • Lock-based systems acquire locks (row, page, table) — can cause blocking and deadlocks.
  • MVCC keeps multiple versions of rows so readers don't block writers; readers see a snapshot of data as of a point-in-time.

9. Indexes and physical design

Indexes speed up lookups at the cost of storage and write overhead. Use them selectively based on query patterns.

Common index types and characteristics

  • B-tree (balanced tree): general-purpose ordered index for range queries, equality, ordering. Most commonly used.
  • Hash: very fast for equality lookups but not for range scans.
  • Bitmap: used in data warehouses for low-cardinality columns.
  • Covering index: an index that contains all columns required for a query, avoiding accessing the table rows.
  • Composite index: index on multiple columns — careful about column order; often used to support queries that filter by leading columns.

Index design tips

  • Index columns used in WHERE, JOIN, ORDER BY, and GROUP BY frequently.
  • Avoid indexing low-selectivity columns (e.g., boolean flag) unless combined with others.
  • Beware write amplification: every INSERT/UPDATE/DELETE may update indexes.
  • Keep statistics up to date; optimizers use statistics to choose plans.

10. Query processing and optimization (theory and algorithms)

When you submit SQL, the RDBMS typically executes these stages:

  1. Parsing: syntax check and parse tree creation.
  2. Binding/Resolution: resolve table and column names; check types and privileges.
  3. Rewriting/Normalization: semantic transformations (e.g., view expansion, subquery flattening).
  4. Logical optimization: apply relational-algebraic rewrites (predicate pushdown, projection trimming, join reordering heuristics).
  5. Physical optimization / costing: enumerate physical plans (access methods, join algorithms), estimate costs using statistics, choose cheapest plan.
  6. Plan execution: the executor performs the operations (scans, joins, sorts, aggregates).

Join algorithms (commonly implemented)

Different physical strategies for joining two sets A and B:

  • Nested-loop join: for each row in outer, scan inner (or index lookup). Good when outer is small or inner is indexed.
  • Hash join: build a hash table on the smaller relation using the join key, then probe with the larger relation. Efficient for equality joins and large unsorted inputs.
  • Sort-merge join: sort both inputs by join key and merge. Ideal when inputs are already sorted or when performing range joins.

Cost estimation basics

Cost-based optimizers estimate I/O and CPU costs using statistics: row counts, column cardinalities, histograms, and distribution metrics. Cardinality estimation (predicting how many rows satisfy predicates) is the single most important factor affecting plan choice.

11. Normalization and schema design

Normalization is the process of structuring a database to reduce redundancy and improve integrity. Common normal forms:

  • 1NF: atomic attribute values (no repeating groups).
  • 2NF: 1NF plus every non-key attribute is fully functionally dependent on the primary key (applies to composite keys).
  • 3NF: 2NF plus no transitive dependencies (non-key attributes depending on other non-key attributes).
  • Beyond 3NF: BCNF, 4NF, 5NF address more specialized anomalies.

In practice, designers balance normalization and performance. Denormalization (duplicating data) can improve read performance at the cost of more complex writes and potential inconsistencies.

12. Security, injection, and best practices

Key practices to keep SQL-based applications secure and robust:

  • Use parameterized queries or prepared statements to prevent SQL injection.
  • Grant least privilege — avoid using superuser credentials in application code.
  • Validate input at application level; don't rely solely on database constraints.
  • Use TLS to protect data in transit between app and database.
  • Sanitize administrative inputs and audit logs for changes to schema and permissions.

13. Performance tuning and common pitfalls

Performance tuning typically focuses on:

  • Monitoring and using EXPLAIN/EXPLAIN ANALYZE to inspect query plans and execution statistics.
  • Ensuring useful indexes exist for critical queries; avoid over-indexing.
  • Reducing volume of data scanned: push predicates early, avoid functions on indexed columns used in WHERE.
  • Avoid SELECT * in production queries — fetch only necessary columns to reduce I/O.
  • Consider materialized views for expensive repeated aggregations and summary tables for OLAP workloads.

14. Example scenarios and explanations

Example: retrieving top earners per department

Goal: for each department, find the top 3 earners.

SELECT department_id, id, name, salary
FROM (
  SELECT id, name, department_id, salary,
    RANK() OVER (PARTITION BY department_id ORDER BY salary DESC) AS rnk
  FROM employees
) t
WHERE rnk <= 3;

Why it works: the window function computes a ranking per department, preserving rows; the outer query filters the ranked rows to get the top N.

Example: avoiding correlated subquery when possible

Correlated subqueries can be expensive because they can be evaluated per outer row. Often a join or CTE yields a better plan:

-- correlated (potentially costly)
SELECT e.id, e.name
FROM employees e
WHERE e.salary > (SELECT AVG(salary) FROM employees WHERE department_id = e.department_id);

-- equivalent using CTE (often lets optimizer do grouping once)
WITH dept_avg AS (
  SELECT department_id, AVG(salary) AS avg_salary
  FROM employees
  GROUP BY department_id
)
SELECT e.id, e.name
FROM employees e
JOIN dept_avg d ON e.department_id = d.department_id
WHERE e.salary > d.avg_salary;

15. Common commands and syntax reference (cheat-sheet)

  • CREATE TABLE, ALTER TABLE, DROP TABLE — schema operations.
  • INSERT INTO ... VALUES / INSERT ... SELECT — append data.
  • UPDATE ... SET ... WHERE ... — modify rows.
  • DELETE FROM ... WHERE ... — remove rows.
  • TRUNCATE TABLE — quick removal of all rows (varies by engine).
  • CREATE INDEX / DROP INDEX — add and remove indexes.
  • EXPLAIN [ANALYZE] <query> — view optimizer/execution plan (DB-specific output).

16. EXPLAIN and interpreting plans (practical guidance)

Use EXPLAIN to understand how the database plans to execute a query. Plan components often include:

  • Access method: sequential scan / index scan / index seek
  • Join strategy: nested loop, hash join, merge join
  • Estimated vs actual rows processed (some systems show both)
  • Cost estimates: rough measures combining I/O and CPU
  • Where the work is concentrated (sorts, aggregates, nested loops)

When reading plans, pay particular attention to nodes with high estimated cost or high actual time/rows. Differences between estimated and actual cardinalities are a strong sign that statistics are missing or outdated.

17. Practical checklist for writing good SQL

  • Start by writing a clear, correct query before optimizing.
  • Profile with EXPLAIN ANALYZE, then fix the high-cost operations.
  • Add appropriate indexes; consider covering indexes for frequent queries.
  • Update statistics periodically, especially after large data changes.
  • Prefer set-based operations over row-by-row loops (e.g., avoid cursors if possible).
  • Use explicit column lists, not SELECT *.
  • Parameterize queries to prevent injection and allow plan reuse.

Pseudo-execution plan (explanation)

  1. Index scan or table scan to find rows where department_id = 10. If an index on department_id exists, the engine uses it (low cost); otherwise it scans the table.
  2. Filter rows: yields rows for id 1, 3, 5 (Alice, Carol, Eve).
  3. Sort the filtered rows by salary DESC. Sorting cost depends on number of rows; with LIMIT 2 some engines use a top-N algorithm to avoid full sort.
  4. Return top 2 rows to the client.

Observation: adding an index on department_id, salary DESC could let the engine find the top-2 rows directly without an additional sort (covering index), improving performance.

19. Commonly asked questions (FAQs)

Q: Is SQL procedural or declarative?

A: SQL is primarily declarative: you state the desired result and the DBMS decides how to compute it. However, some extensions (procedural languages like PL/pgSQL, T-SQL, stored procedures) enable procedural constructs for control flow.

Q: When should I denormalize?

A: Denormalize when read performance demands outweigh the costs of maintaining duplicate data. Typical cases: OLAP/analytics, caching precomputed aggregates, or reducing complex joins for low-latency queries. Always evaluate trade-offs and maintain safeguards for integrity.

Q: What are the most important metrics for tuning queries?

A: Look at execution time, I/O read counts, CPU time, row counts per operator, and differences between estimated versus actual cardinalities. Use these to guide index creation, rewriting queries, or changing schema/statistics.

References and further reading: relational algebra texts, the documentation of your chosen RDBMS (PostgreSQL, MySQL, SQL Server, Oracle), and canonical books like "Database System Concepts" (Silberschatz, Korth) and "Readings in Database Systems" (The Red Book) for deeper theory.
