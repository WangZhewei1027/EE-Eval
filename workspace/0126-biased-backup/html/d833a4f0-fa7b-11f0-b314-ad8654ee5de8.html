AVL Tree — Comprehensive Exposition

A compact, text-heavy guide explaining the concept, theory, algorithms, and practical aspects of AVL trees. This page focuses on careful, step-by-step written explanations and pedagogical content. A single small demonstration is provided at the end.

1. Overview and Motivation

An AVL tree is a type of self-balancing binary search tree (BST). It was the first such data structure invented (by Adelson‑Velsky and Landis in 1962), and it guarantees that the tree's height remains logarithmic in the number of nodes. The core idea is to maintain a balance condition after insertions and deletions so that search, insertion, and deletion operations run in O(log n) time.

2. Definitions and Key Concepts

  • Binary Search Tree (BST): a tree where each node has up to two children; all keys in the left subtree are less than the node key, and all keys in the right subtree are greater than the node key (for strict order).
  • Height of a node: the length of the longest path from that node down to a leaf. A leaf has height 0 (or 1, depending on convention). In this guide we will define a leaf node's height as 0 and an empty subtree (null) as height -1 when convenient.
  • Balance Factor (BF): for a node, BF = height(left subtree) − height(right subtree). In AVL trees, BF must always be one of {−1, 0, +1} for every node.
  • AVL condition: every node in the tree must have balance factor -1, 0, or +1.
  • Rotation: a local restructuring operation that changes parent-child relationships between a small number of nodes while preserving the in-order sequence of keys (thus preserving the BST property).

3. Why balancing matters (intuitive)

In a plain BST, insertion order can make the tree degenerate into a linear chain (height n), causing operations to degrade to O(n). AVL trees prevent long chains by performing rotations right after operations that cause imbalance, so the height remains O(log n).

4. Core rotational operations

There are two basic rotations and two compound rotations (combinations) used by AVL rebalancing. Below are textual diagrams (ASCII) showing the structure before and after each rotation. Letters like T1, T2, ... indicate subtrees that keep their internal structure.

Right Rotation (also called single right or "rotateRight")

Before rotation (unbalanced at z, left-heavy):
        z
       / \
      y   T4
     / \
    x   T3
   / \
  T1  T2

After rotateRight(z):
        y
       / \
      x   z
     / \ / \
    T1 T2 T3 T4

Effect: the left child y becomes the new root of this subtree, z becomes y's right child, and T3 moves between them appropriately. Heights and balance factors must be updated after the rotation.

    T2  x
       / \
      T3 T4

After rotateLeft(z):
        y
       / \
      z   x
     / \ / \
    T1 T2 T3 T4

5. Insertion algorithm — step-by-step textual description

We describe a recursive insertion procedure that maintains the BST property and then restores the AVL property by updating heights and performing rotations when necessary.

  1. Perform the usual BST insertion: recursively descend left or right depending on key until you reach a null and insert a new node (height = 0).
  2. On unwinding the recursion, update the height of each ancestor node: height(node) = 1 + max(height(left), height(right)).
  3. Compute the balance factor BF = height(left) − height(right).
  4. If BF is within {-1, 0, +1}, the node is balanced — continue unwinding.
  5. If BF becomes +2 or −2, the node is unbalanced — identify which of the four cases applies based on the newly inserted key's position:
    • LL case: insertion in left subtree of left child → perform single right rotation.
    • RR case: insertion in right subtree of right child → perform single left rotation.
    • LR case: insertion in right subtree of left child → perform left rotation on left child, then right rotation on node.
    • RL case: insertion in left subtree of right child → perform right rotation on right child, then left rotation on node.
  6. After rotation(s), update heights of affected nodes (the rotated nodes) and continue unwinding.

Important implementation detail: to achieve O(1) time per height update, store the height in each node and recompute it based on child heights when visiting a node.

Pseudocode for insertion (concise)

function insert(node, key):
  if node == null:
    return new Node(key)   // height = 0

  if key < node.key:
    node.left = insert(node.left, key)
  else if key > node.key:
    node.right = insert(node.right, key)
  else:
    handle duplicates (depends on policy) 

  update node.height = 1 + max(height(node.left), height(node.right))
  balance = height(node.left) - height(node.right)

  if balance > 1:
    if key < node.left.key:          // LL
      return rotateRight(node)
    else:                            // LR
      node.left = rotateLeft(node.left)
      return rotateRight(node)
  if balance < -1:
    if key > node.right.key:         // RR
      return rotateLeft(node)
    else:                            // RL
      node.right = rotateRight(node.right)
      return rotateLeft(node)

  return node

6. Deletion algorithm — overview and steps

Deletion in an AVL tree follows the same high-level approach as deletion in a BST, then it repairs any imbalance encountered while unwinding recursion. Steps:

  1. Find and remove the node like in a BST:
    • If the node has no children: remove it.
    • If it has one child: replace the node with its child.
    • If it has two children: find the in-order successor (smallest node in right subtree), copy its key to the node, and recursively delete the successor.
  2. On unwinding, update heights and compute the balance factor at each ancestor.
  3. If a node becomes unbalanced (BF = +2 or −2), identify one of the four cases. The determination of whether it's LL, LR, RL, or RR is done by looking at the balance factor of the offending child (or by comparing keys if easier in a particular implementation), and perform the corresponding rotation(s).
  4. Continue unwinding; deletion may require rebalancing at multiple nodes up to the root.

Note: Unlike insertion (which triggers at most one or two rotations near the insertion site), deletion can cause rebalancing at multiple levels while unwinding, potentially O(log n) rotations in the worst case.

7. Complexity

  • Search: O(h) = O(log n) where h is tree height.
  • Insertion: O(log n) for finding the insertion point plus O(1) rotations in most cases; overall O(log n).
  • Deletion: O(log n) for finding node and O(log n) worst-case rebalancing; overall O(log n).
  • Space: O(n) for storing nodes. Additional memory: O(h) recursion stack for recursive implementations.

8. Mathematical height bound (concise proof sketch)

Let N(h) be the minimum number of nodes in an AVL tree of height h (assuming height of empty tree is -1, height of leaf is 0). The worst-case (minimum nodes for a given height) occurs when each node's children heights differ as little as possible but still reach height h:

Recurrence: N(h) = 1 + N(h−1) + N(h−2) for h ≥ 1 with base values N(0) = 1, N(−1) = 0.

This recurrence is the Fibonacci recurrence shifted. Hence N(h) ≥ F(h+2) − 1, and Fibonacci numbers grow exponentially (approximately φ^h where φ ≈ 1.618...). Therefore h = O(log n). Explicit bound:

h ≤ log_φ(n + 1) − 2 = O(log n).

9. Example walkthrough (textual)

Insert the sequence: 30, 20, 10, 25, 40, 22 and observe what happens step-by-step.

  1. Insert 30 → tree has single node 30 (balanced).
  2. Insert 20 → 20 becomes left child of 30. Heights and BF are OK (BF of 30 = +1).
  3. Insert 10 → 10 goes to left of 20 → now 30 is left-heavy with BF = +2. This is LL case (left child 20 has BF +1). Do a single right rotation at 30. Resulting root becomes 20, with left child 10 and right child 30. Balanced.
  4. Insert 25 → goes to right of 20's left/right? Starting at root 20: 25 > 20 → go right to 30: 25 < 30 → goes left of 30. Heights are updated; tree remains balanced.
  5. Insert 40 → goes to right of 20 → right of 30 → becomes right child of 30. Balanced overall.
  6. Insert 22 → follows path: 20 → right to 30 → left to 25 → left to null → inserted as left child of 25. This insertion makes node 30 left-heavy (its left subtree height increases), which then may cause node 20 to become unbalanced. If at any point a BF of +2 or −2 arises, identify LL/LR/RR/RL and rotate accordingly. In this sequence a rotation (LR or similar) may be triggered depending on exact heights; the algorithm will perform the correct local rotations to restore BF in {-1,0,1} for all nodes.

10. Implementation tips and pitfalls

  • Store node height explicitly (update it as 1 + max(height(left), height(right))). This avoids costly subtree traversals.
  • Make rotation functions that update heights of involved nodes in the correct order — the child nodes first, then the parent (new root) — to keep BF calculations correct.
  • Watch out for off-by-one in height definition (leaf as 0 or 1). Pick one definition and use it consistently.
  • Decide on a policy for duplicate keys: ignore insertion, count occurrences, or store duplicates in a list or in the right subtree consistently. In AVL literature duplicates are usually disallowed or handled by a chosen convention.
  • Thoroughly unit test rotations: create isolated small subtrees and rotate them to verify structural correctness and heights.

11. Comparisons and trade-offs

AVL trees provide tighter balancing than Red‑Black trees: the height of an AVL tree is smaller (closer to optimal) compared to a Red‑Black tree for the same number of nodes. This often yields faster lookups. However, tighter balancing means insertions and deletions may perform more rotations on average than Red‑Black trees. Accordingly AVL trees are a good choice when reads/searches are much more frequent than inserts/deletes and strict balance (fast lookups) is desired.

12. Common test cases for verifying an AVL implementation

  • Insert an increasing sequence of keys (1..n) and verify tree remains balanced and height is O(log n).
  • Insert a decreasing sequence of keys (n..1).
  • Insert keys in a random order and compare search results to a trusted structure.
  • Delete nodes in different orders: leaves, nodes with one child, nodes with two children (in-order successor case).
  • Edge cases: single element, empty tree, duplicates, null child pointers.

13. Example pseudocode: rotate functions

function rotateLeft(z):
  y = z.right
  T2 = y.left

  y.left = z
  z.right = T2

  // update heights
  z.height = 1 + max(height(z.left), height(z.right))
  y.height = 1 + max(height(y.left), height(y.right))

  return y  // new root of this subtree

function rotateRight(z):
  y = z.left
  T3 = y.right

  y.right = z
  z.left = T3

  // update heights
  z.height = 1 + max(height(z.left), height(z.right))
  y.height = 1 + max(height(y.left), height(y.right))

  return y
Note: The above pseudocode assumes a non-null z where z.left or z.right exists appropriately. Always guard against null pointers in actual code.
