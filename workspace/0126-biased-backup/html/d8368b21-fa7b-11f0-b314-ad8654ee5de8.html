Dynamic Programming — A Comprehensive Textual Guide

Deep explanation of the concept, strategies, algorithm patterns, and many textual step-by-step walkthroughs.

1. What is Dynamic Programming (DP)?

Dynamic Programming is a method for solving complex problems by breaking them down into simpler subproblems, solving each subproblem once, and storing their solutions — typically using a table — so the solution to the same subproblem is available when needed. It is an algorithmic paradigm ideally suited for optimization problems and counting problems that exhibit two key properties:

  • Optimal substructure: An optimal solution to the problem contains optimal solutions to its subproblems.
  • Overlapping subproblems: The problem can be decomposed into subproblems which are reused multiple times.

Important: Dynamic Programming is not a specific algorithm; it is a way to structure computation to reuse work. It trades extra memory (to store subproblem results) for time efficiency.

2. Two main approaches

Top-down (Memoization)

Start with the original problem and write a recursive solution that directly expresses the recurrence. Use a table (often an associative map or an array) to cache results of subproblems. When a recursive call requests a value that is already computed, return it from cache instead of recomputing.

// Pseudocode (top-down)
function solve(params):
  if params in memo: return memo[params]
  if base case: return base-value
  result = combine( solve(smaller-subproblem1), solve(smaller-subproblem2), ... )
  memo[params] = result
  return result

Bottom-up (Tabulation)

Identify a suitable order of evaluation for subproblems (typically increasing size or complexity) and fill a table iteratively so that when computing any entry, all needed entries are already computed.

// Pseudocode (bottom-up)
initialize dp table with base values
for each state in chosen order:
  fill dp[state] using earlier dp values
return dp[final-state]

Both approaches compute the same results; choose top-down for clarity and quick implementation, and bottom-up for better control of iteration, memory layout, and sometimes slightly better performance.

3. How to design a Dynamic Programming solution — step by step

  1. Understand and characterize the structure of an optimal solution. Ask: can the problem be decomposed into smaller subproblems that combine to form the solution?
  2. Define the state. Formally specify what a subproblem is. A state is a tuple (i, j, ...) describing the portion of the original problem the subproblem solves.
  3. Write the recurrence (transition). Express the solution of a state in terms of solutions of smaller states.
  4. Choose base cases. Identify trivial states that have immediate answers.
  5. Choose evaluation order. For bottom-up: pick an order that guarantees dependencies are computed first. For top-down: memoize results as they are first needed.
  6. Implement and analyze complexity. Determine number of distinct states and the cost to compute each state; total cost = states × costPerState.
  7. Optimize. Reduce memory (e.g., compress dimensions), simplify transitions, or apply known optimizations (Convex hull trick, Divide-&-Conquer DP, Knuth, etc.) when applicable.

4. Classic example — Fibonacci (step-by-step)

Problem: compute the n-th Fibonacci number where F(0)=0, F(1)=1, F(n)=F(n-1)+F(n-2).

Naive recursion

The naive recursion directly follows the recurrence, but recomputes many subproblems: exponentially many calls.

Top-down memoization

Store computed F(k) in memo; each F(k) is computed once. Complexity becomes O(n) time and O(n) memory.

Bottom-up tabulation

Compute F(0), F(1), then iterate i from 2..n: F(i)=F(i-1)+F(i-2). Complexity O(n) time and O(n) memory; memory can be optimized to O(1) by keeping just two last values.

// Example: bottom-up iterative
function fib(n):
  if n==0: return 0
  a = 0
  b = 1
  for i in 2..n:
    c = a + b
    a = b
    b = c
  return b

This example demonstrates overlapping subproblems (F(k) used repeatedly) and optimal substructure (F(n) is combination of optimal subsolutions F(n-1) and F(n-2)).

5. Example walkthrough — Coin Change (minimum coins)

Problem: given coin denominations coins[] and target amount T, find minimum number of coins to make amount T (unlimited supply of each coin). If impossible, return -1.

State design

Let dp[x] = minimum coins to make amount x. We want dp[T].

Recurrence

dp[0] = 0. For x >= 1: dp[x] = min_{c in coins, c <= x} (dp[x - c] + 1).

Bottom-up iteration

initialize dp[0] = 0
for x from 1 to T:
  dp[x] = +infinity
  for each coin c:
    if c <= x and dp[x-c] is not infinity:
      dp[x] = min(dp[x], dp[x-c] + 1)
return dp[T] if not infinity else -1

Step-by-step textual example

Suppose coins = [1, 3, 4], T = 6. Table progress (x from 0..6):

  • dp[0] = 0
  • dp[1] = min(dp[0]+1) = 1 (use 1)
  • dp[2] = dp[1]+1 = 2 (1+1)
  • dp[3] = min(dp[2]+1, dp[0]+1) = min(3, 1) = 1 (use 3)
  • dp[4] = min(dp[3]+1, dp[1]+1, dp[0]+1) = min(2,2,1) = 1 (use 4)
  • dp[5] = min(dp[4]+1, dp[2]+1, dp[1]+1) = min(2,3,2) = 2 (e.g., 1+4)
  • dp[6] = min(dp[5]+1, dp[3]+1, dp[2]+1) = min(3,2,3) = 2 (e.g., 3+3)

Answer dp[6] = 2. This step-by-step filling demonstrates how earlier computed values are reused to compute later ones.

6. More canonical DP problems (brief descriptions)

  • Longest Common Subsequence (LCS): dp[i][j] = LCS length of prefixes A[0..i-1], B[0..j-1]. Transition: if A[i-1]==B[j-1] then dp[i][j]=dp[i-1][j-1]+1 else max(dp[i-1][j],dp[i][j-1]).
  • Edit Distance (Levenshtein): dp[i][j] = minimum edits to convert A[0..i-1] to B[0..j-1]; consider insert, delete, replace operations.
  • 0/1 Knapsack: dp[i][w] = best value using first i items with capacity w. Can be optimized to 1D from high-to-low.
  • Unbounded Knapsack / Coin Change (combinatorial): variations on order and counting distinct ways.
  • Matrix Chain Multiplication: dp[i][j] = min cost to multiply matrices i..j, try splitting at k and add multiply cost.
  • DP on Trees: rerooting DP, compute values for subtrees, combine child contributions.
  • Bitmask / Subset DP: States represent subsets (mask); used for TSP and some combinatorial problems. Complexity often O(2^n * n).
  • Digit DP: Count numbers satisfying constraints by processing digits with state (position, tight, leadingZeros, ...).

7. Advanced topics and optimizations (overview)

Some specialized optimizations reduce time from naive DP or reduce memory:

  • Space optimization: Many DPs only need a few recent rows/columns; compress memory to O(1) or O(k).
  • Divide & Conquer DP optimization: When transitions have monotone decision points, compute DP rows faster with divide-and-conquer technique.
  • Knuth optimization: For quadrangle inequality and monotone decisions, reduces matrix chain-like DP to O(n^2).
  • Convex hull trick / Li Chao tree: For DP with linear terms and monotone slopes, speed up min/max over lines to amortized O(log n) or O(1).
  • Meet-in-the-middle: When n is ~40, split into two halves and combine rather than 2^n brute force; not DP but often a competitor.

Applying these optimizations typically requires mathematical properties (monotonicity, convexity) of your recurrence.

8. Common pitfalls and debugging tips

  • Wrong state: If your state doesn't capture all necessary information, your recurrence will be incorrect. Make the state sufficiently expressive.
  • Off-by-one in indices: DP tables commonly use lengths or indices — be consistent about inclusive/exclusive ranges.
  • Uninitialized base cases: Ensure base cases are correctly set (e.g., dp[0] for coin change, dp[empty] for subsequence problems).
  • Iteration order: For bottom-up, compute states in an order that satisfies dependencies. If you need values from larger indices, iterate backward or restructure state.
  • Memory/time blow-up: Count distinct states; if it's exponential you may need a different approach (e.g., greedy, meet-in-the-middle, heuristic).
  • Precision and limits: For counting problems, results may overflow — use modulo arithmetic if required.

9. How to read a DP problem in contests or interviews

  1. Identify whether the problem asks for an optimal value, count, or feasibility.
  2. Look for overlap: are the same subproblems recomputed in naive recursion?
  3. Try to write a recursive formulation (even if inefficient). From recursion you can often extract state and transitions.
  4. Translate recursion to memoization first for correctness, then optimize to tabulation if needed.
  5. Check constraints: if n <= 2000, O(n^2) may be OK; if n <= 50, bitmask DP might be fine; if n <= 1e5, you need linear or n log n algorithms.

10. Concrete pseudo-code examples

0/1 Knapsack — bottom-up

// items indexed 1..N, weight limit W
for i = 0..N:
  for w = 0..W:
    if i == 0 or w == 0: dp[i][w] = 0
    else if weight[i] <= w:
      dp[i][w] = max(dp[i-1][w], dp[i-1][w-weight[i]] + value[i])
    else:
      dp[i][w] = dp[i-1][w]

Longest Common Subsequence (LCS)

// strings A (length n), B (length m)
for i=0..n:
  dp[i][0] = 0
for j=0..m:
  dp[0][j] = 0
for i = 1..n:
  for j = 1..m:
    if A[i-1] == B[j-1]:
      dp[i][j] = dp[i-1][j-1] + 1
    else:
      dp[i][j] = max(dp[i-1][j], dp[i][j-1])

11. Complexity analysis (how to compute)

To estimate DP complexity, do two things:

  1. Count the number of distinct states. This is the size of your dp table (e.g., dp[n][m] has O(n*m) states; bitmask with n items has 2^n states).
  2. Estimate the work needed per state to compute the transition (e.g., O(1), O(n), O(log n)).

Total time = (number of states) × (cost per state). Memory equals the number of states (unless optimized).

Example: LCS uses O(n*m) states and O(1) work per state ⇒ O(n*m) time and memory.

12. FAQ / Short answers

When should I use DP vs greedy?

Use greedy when a local choice that looks best can be proven to lead to a global optimum. If you cannot prove such property or you observe overlapping subproblems, DP is safer.

Does memoization always use recursion?

Typically yes — memoization is implemented with recursion. But you can simulate recursion with an explicit stack and memo table if desired.

What is state explosion and how to avoid it?

State explosion is when the number of distinct states is too large (exponential). Avoid it by tightening the state to only necessary information, using bitmasks for small n, or by using alternative algorithms.
