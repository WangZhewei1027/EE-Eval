Introduction & Motivation

Virtual memory (VM) is a fundamental abstraction provided by modern operating systems that gives each process the illusion of having a large, contiguous block of private memory starting at address zero. The OS and hardware collaborate to map these virtual addresses to physical memory (RAM), and to secondary storage (disk) when necessary.

Primary motivations for virtual memory include:

  • Isolation and protection: Processes cannot (normally) access the physical memory of others.
  • Simplified programming model: Programs can be written assuming large contiguous memory.
  • Efficient use of physical memory: Only needed pages are loaded (demand paging), and pages can be shared between processes (shared libraries).
  • Support for memory overcommit and swapping: The OS can run programs whose combined virtual memory exceeds physical RAM.

Note: Virtual memory is an abstraction implemented via both hardware (MMU and TLB) and software (page fault handler, page replacement policies).

Address Spaces and Address Translation

Conceptually, a process operates in a virtual address space: a range of virtual addresses (for example, 0 .. 2^32-1 on a 32-bit system). The Memory Management Unit (MMU) translates each virtual address into a physical address during memory accesses.

A virtual address is typically decomposed into:

  • Virtual Page Number (VPN): identifies the page in virtual address space.
  • Offset: the position inside the page (determined by page size).

Translation: physical_address = frame_base + offset, where the virtual page maps to a physical frame (if resident).

Example decomposition for 32-bit address with 4 KB page size:

  • Offset: lower 12 bits (2^12 = 4096)
  • VPN: upper 20 bits

Hardware accelerates translation through a Translation Lookaside Buffer (TLB): a small cache of recent VPN→PFN (physical frame number) mappings. A TLB hit avoids walking the page table; a TLB miss may require a page table lookup and possibly walk(s).

Paging: Concept and Mechanics

Paging breaks both virtual and physical memory into fixed-size blocks: pages (virtual) and frames (physical). Common page sizes: 4 KB, 2 MB (huge pages), or 1 GB (on x86-64 with extended modes).

Key ideas:

  • Each virtual page maps to either a physical frame (resident) or to disk (not resident).
  • Page table entries (PTEs) store metadata: present bit, dirty bit (modified), access bits (reference), protection bits (R/W/X), and frame number.
  • Only a subset of pages—those actually used—need to be resident in RAM at any time (demand paging).

Page size trade-offs

  • Small pages reduce internal fragmentation and allow finer-grained memory use, but increase page table size and TLB pressure.
  • Large pages (huge pages) reduce page table overhead and TLB pressure per byte, but increase internal fragmentation and may waste memory.

Fragmentation

Paging eliminates external fragmentation (since frames are uniform), but introduces internal fragmentation (unused space inside the last page of a region).

Page Tables: Designs and Trade-offs

Page tables map VPN -> PTE. Designs include:

  • Single-level (flat) page table: simple array indexed by VPN. Size can be large (memory cost proportional to virtual address space).
  • Multi-level (hierarchical) page tables: break the VPN into multiple indices; allocate levels on demand to avoid reserving space for unused parts of the address space. Commonly used in 32/64-bit OSes.
  • Inverted page table: one entry per physical frame; entries contain the owning process and VPN. Space proportional to physical memory, but lookup needs hashing or associative structures.

Multi-level example: On a 64-bit system with 4-level paging, the top levels are often sparse; only the parts in use are allocated, making large virtual spaces practical.

Page table entry fields (typical)

  • Present/valid bit
  • Frame number (physical frame / base address)
  • Accessed/reference bit (set by hardware when page is read/written)
  • Dirty bit (set when the page is written)
  • Protection bits: readable, writable, executable
  • Cache and global flags (architecture-specific)

Demand Paging & Page Fault Handling

Demand paging delays loading a page until the process first accesses it. When a process accesses a page that is not resident, the MMU triggers a page fault (trap to the kernel). The kernel handles the fault:

  1. Validate the access (is the virtual address valid for the process?). If not, send a segmentation fault to the process (terminate or signal).
  2. If valid but not resident, locate a physical frame: either find a free frame or select a victim using a page replacement algorithm.
  3. If the victim page is dirty, write it back to disk (swap area or file backing). If clean, it can be freed immediately.
  4. Load the requested page from its backing store (file, swap) into the chosen frame.
  5. Update the page table entry to mark the page present and set the frame number; invalidate/flush TLB entries as necessary.
  6. Resume the faulting instruction (often the CPU re-executes automatically after return from kernel). The instruction now succeeds.

Handling page faults requires cooperation between hardware (to trap and record faulting address) and the OS (to update tables and schedule I/O).

Disk I/O is slow compared to RAM (orders of magnitude). Therefore, page faults are expensive; the OS must minimize them through good replacement algorithms and locality-aware scheduling.

Page Replacement Algorithms

When physical memory is full, the OS must choose a resident page to evict to make space. The choice affects performance. Common algorithms:

Optimal (Belady's algorithm)

Evict the page that will not be used for the longest time in the future. This is optimal but requires future knowledge of references (unimplementable in general). Used as a theoretical baseline.

FIFO (First-In First-Out)

Evict the oldest loaded page. Easy to implement (queue). Can suffer from Belady anomalies where increasing frames increases faults.

LRU (Least Recently Used)

Evict the page that has not been used for the longest time in the past. An excellent practical approximation of optimal under locality assumptions. Precise LRU requires maintaining exact recency order (costly), so approximations are often used.

Clock (Second-Chance)

A practical approximation of LRU with low overhead. Pages are organized in a circular list with a pointer. Each page has a reference bit (set by hardware on accesses). On replacement, the OS scans pages: if reference bit is 0, evict; if 1, clear it and advance pointer ("second chance").

NRU, Aging, Working Set

Other algorithms use periodic sampling of reference bits (aging) or the working-set model (maintain set of pages used in last T time units) to make decisions. NRU (Not Recently Used) classifies pages by referenced/dirty bits and prefers certain classes to evict.

Algorithm properties

  • Complexity (time/space): LRU exact needs O(number of pages) updates per access unless approximated.
  • Hardware support (reference bits) enables practical approximations.
  • Locality assumptions: workloads with strong locality are well served by LRU-like policies.

Pseudo-code (textual)

FIFO (conceptual):

On page fault:
  if free frame exists:
    allocate it and load page
  else:
    victim = dequeue(oldest)
    evict(victim)
    allocate victim's frame and load page
    enqueue(new page)

LRU (conceptual with a recency list):

On memory access to page P:
  if P is in memory:
    move P to front of recency list (most-recent)
  else:
    if free frame exists:
      allocate frame and load P; insert P at front
    else:
      victim = tail of recency list (least-recent)
      evict(victim)
      load P into frame; insert P at front

Clock (conceptual):

Pointer = start of circular list
On page fault:
  while true:
    if reference_bit[Pointer] == 0:
      victim = Pointer; break
    else:
      reference_bit[Pointer] = 0
      advance Pointer
  evict(victim); load new page at victim; set reference bit = 1; advance Pointer

Step-by-step Example: Address Translation & Paging

Address translation example (simple)

Assume:

  • 32-bit virtual addresses
  • Page size = 4096 (4 KB) → offset = 12 bits
  • Virtual address: 0x12345678

Compute:

  1. Offset = lower 12 bits = 0x678 & 0xFFF = 0x678 (decimal 1656)
  2. VPN = upper bits = 0x12345 (since 0x12345678 >> 12 = 0x12345)

If the page table maps VPN 0x12345 → PFN 0x9AB (frame base physical address = PFN << 12), then physical address = (0x9AB << 12) | 0x678.

Paging example: page table & TLB interaction

Sequence:

  1. Process reads address 0x12345678.
  2. MMU looks up VPN 0x12345 in TLB — if present, it uses PFN and returns physical address quickly.
  3. If TLB miss, the MMU walks page table (possibly multi-level) to find PTE. If PTE present, the mapping is loaded into the TLB and access continues.
  4. If PTE indicates not-present, page fault occurs; OS loads page and updates PTE. Once present, the original memory access is retried.

Performance: Effective Access Time (EAT), Hits, Misses, Thrashing

Effective Access Time (EAT)

EAT measures the average time to access memory considering page faults:

Let p = page-fault rate (probability of a memory access causing a page fault). Let m = memory access time (e.g., 100 ns), and s = page fault service time (including disk I/O, often milliseconds).

EAT = (1 - p) * m + p * s

Because s is orders of magnitude larger than m, even a tiny p (like 10^-6) can significantly increase EAT. Thus reducing page-fault rate is critical.

Working set and thrashing

The working set of a process at time t is the set of pages it has referenced within a window of recent time (or references). If the total working sets of all processes exceed physical memory, the OS may experience thrashing: constant page faults, low CPU utilization, and low throughput.

Ways to avoid thrashing:

  • Admit control: restrict degree of multiprogramming
  • Use working-set based replacement and page-fault frequency (PFF) schemes
  • Use pre-paging or clustering to bring needed pages in groups

Advanced Topics

Copy-On-Write (COW)

COW is an optimization when processes are forked. The parent and child initially share the same physical pages marked read-only. On a write attempt, a page fault occurs; the OS allocates a new frame and copies the page, so each process gets its private copy only when it writes.

Shared pages

Shared libraries or shared memory segments map the same physical frames into multiple processes' virtual address spaces; page protection and consistency must be managed. Mapping files into memory (mmap) is a common mechanism for sharing code and data among processes.

Protection and Access Control

PTEs include permission bits (read/write/execute). The MMU enforces those bits and can raise protection faults for unauthorized access. Kernel mappings often live in upper part of address space; user processes cannot access them directly.

Memory-mapped files and demand paging

Files mapped via mmap are loaded into memory on demand. Modified pages may be written back to the file (if MAP_SHARED) or discarded (if MAP_PRIVATE without COW semantics).

Implementation Considerations

Hardware support

  • MMU: performs address translation.
  • TLB: hardware cache of translations.
  • Reference and dirty bits: hardware updates them on access/write, enabling approximations to LRU.
  • Page table base registers (CR3 on x86) and page levels implemented by hardware.

OS responsibilities

  • Maintain page tables and PTE metadata.
  • Handle page faults, perform I/O to load/store pages.
  • Choose and implement page replacement policy.
  • Manage shared mappings, copy-on-write, and memory allocation.

Multi-core and concurrency

On multi-core systems, TLB coherence and per-core caches complicate VM. The OS must flush or invalidate TLB entries when page table mappings change. Kernel data structures (free frame lists, page replacement queues) require synchronization.

Common Questions & Short Answers

Q: Why use virtual memory rather than letting programs manage addresses?

A: VM provides protection, isolation, simpler programming model, sharing, and better utilization. Letting programs manage addresses would be error-prone and insecure.

Q: How does TLB affect performance?

A: TLB hits are fast and avoid page-table walks; TLB misses require MMU to consult page tables (possibly multiple memory accesses), increasing latency. TLB design (associativity, size, page size mappings) critically affects memory performance.

Q: What is the difference between paging and segmentation?

A: Segmentation divides memory into variable-size logical segments with base+limit, reflecting program structures (code, stack, heap). Paging breaks memory into fixed-size pages. Many modern systems combine them (segmentation seldom used in mainstream x86-64 except for legacy features).

Q: What causes thrashing?

A: When the system has too little physical memory for the active working sets, frequent page faults force constant swapping, reducing actual productive computation.
