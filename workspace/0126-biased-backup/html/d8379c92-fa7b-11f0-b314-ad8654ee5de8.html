1. Definition and big picture

In operating systems and low-level concurrent systems, a context switch is the act of the CPU switching from executing one logical thread of execution (a process, kernel thread, or user thread) to another. It requires saving the state (the "context") of the currently running entity and restoring the state of the entity to run next. After the switch the CPU continues execution as if the new entity had been running all along.

Note: "Context switching" is also used in cognitive psychology to describe humans switching attention between tasks. This document focuses on the computing/operating-system meaning, but sometimes uses the human analogy to clarify ideas.

2. What "context" contains

The "context" is everything necessary to resume execution later without semantic change. Typical items:

  • CPU registers: general-purpose registers, stack pointer (SP), program counter (PC) / instruction pointer (IP), status or flags register.
  • Floating-point/vector unit registers and state (when applicable).
  • Memory mapping state: page table pointer or CR3 (x86), TLB entries conceptually.
  • Processor mode (user/kernel), CPU flags, interrupt masks.
  • Process control block metadata: PID, scheduling priority, runtime accounting (CPU time used), memory limits.
  • I/O and device context if the OS tracks it per process (file descriptors, open sockets) — these are often in process descriptors rather than CPU context.

The exact contents depend on the architecture and implementation choices. For example, x86 can use a Task State Segment (TSS) to hold some context; modern kernels usually manage context explicitly in memory rather than relying on hardware TSS switches.

3. A step-by-step textual description

Below is a canonical, step-by-step textual narrative of a preemptive context switch triggered by a timer interrupt (common for time-sliced multitasking).

Before the switch: a process A is running in user mode

  1. Timer interrupt fires. The CPU completes the current instruction (or reaches a safe point) and performs an interrupt entry to the kernel: the CPU switches to kernel mode and jumps to the interrupt vector/handler.
  2. The interrupt handler (partly in assembly) saves a minimal hardware-saved state (often PC/IP and some flags) automatically or explicitly saves registers to a kernel stack. At this point minimal state for A is preserved temporally.
  3. The kernel's interrupt handler calls into the scheduler: it updates accounting for process A (credit/delta of CPU time used), checks if A's quantum expired or a higher-priority process is runnable.
  4. If the scheduler decides to continue running A, it arranges to return from interrupt to user mode (no full switch). Otherwise, it selects the next runnable entity B.
  5. The kernel completes the save of A's full context: saves remaining general-purpose registers, stack pointer, program counter, and any non-saved state into A's process control block (PCB) or thread control block (TCB). If FPU registers were in use, kernel may lazily save them on demand or save them now.
  6. The kernel updates A's state to "runnable" or "waiting" as appropriate and updates B's state to "running".
  7. The kernel restores B's context: loads registers, stack pointer, program counter, CPU flags, and restores memory mapping (switch page table base register), and if needed restores FPU state.
  8. Return from interrupt to user mode causes the CPU to begin executing B at the restored instruction pointer; switching is complete.

Important: The kernel must switch memory context before executing B if the two processes have different address spaces; otherwise the restored stack pointer and code would be interpreted in the wrong page tables.

4. Pseudocode: a minimal context switch routine

Here is simplified pseudocode that illustrates the core actions. Real kernels are more complex and include race handling, locking, and hardware-specific details.

// Save current context (running on CPU)
function save_context(current) {
  current.regs = save_cpu_registers();   // general registers, flags
  current.sp   = read_stack_pointer();
  current.ip   = read_instruction_pointer();
  if (fpu_in_use_by(current)) {
    save_fpu_state(current.fpu);
  }
  current.page_table_base = read_page_table_base();
}

// Restore the next context and switch to it
function restore_context(next) {
  write_page_table_base(next.page_table_base); // switch address space first
  write_stack_pointer(next.sp);
  restore_cpu_registers(next.regs);
  if (next.fpu) restore_fpu_state(next.fpu);
  jump_to_instruction_pointer(next.ip); // return to user or thread code
}

// Top-level scheduler invoked from interrupt or syscall
function context_switch(current, next) {
  save_context(current);
  current.state = (current.should_sleep ? SLEEPING : READY);
  next.state = RUNNING;
  restore_context(next);
}

This pseudocode omits locking and the kernel stack management required to run save/restore code safely on modern SMP systems.

5. When context switches happen (scheduling events)

Common reasons for context switches include:

  • Preemption due to time slice expiration (timer tick).
  • Blocking system calls (I/O wait, sleep, waiting on locks/condition variables).
  • Voluntary yield by the running thread (cooperative multitasking).
  • Creation or termination of a thread/process (the scheduler may pick a different runnable task).
  • Interrupts: a hardware interrupt may cause a short switch into the kernel and possibly a full context switch if a higher-priority task becomes runnable.
  • Asynchronous events: signals, wakeups, inter-processor interrupts causing migration.

Interaction with scheduling algorithms

The scheduler's policy determines which runnable entity gets the CPU next and thus how often switches occur. Various strategies:

  • Round-Robin: fixed time quantum, switch on expiry to next runnable in queue.
  • Priority-based: prefer highest priority, possibly preempt lower-priority tasks immediately.
  • Multi-level feedback queues: dynamic priorities based on history (aging and penalties for long CPU bursts).
  • Completely Fair Scheduler (CFS): attempts to provide proportional CPU time; chooses the task with minimal virtual runtime.
  • Real-time schedulers: RMS, EDF — switches are driven by deadlines and rate constraints.

6. Hardware support, special cases, and optimizations

Hardware support

  • Some architectures provide hardware facilities (e.g., x86's Task State Segment or special instructions) to ease switching. Modern kernels often prefer explicit software-managed saves/restores for flexibility.
  • Hardware automatically saves minimal state on interrupts (PC, some flags). Full register save is typically done by kernel code.
  • Virtualization adds another layer: a VM exit/entry (vCPU switch) requires saving/restoring virtual CPU state and possibly nested virtualization state.

Performance optimizations used in kernels

  • Lazy FPU/SIMD save/restore: delay saving FPU registers until another process uses the FPU.
  • Avoid switching address space when scheduling threads from the same process (e.g., kernel threads, or when doing threads within one address space).
  • Reduce switch frequency: use longer time quanta when latency permits; use cooperative user-level threading for fine-grained concurrency.
  • Per-CPU run queues and avoiding global locks to reduce contention and unnecessary cross-CPU switches.
  • Tickless kernels to avoid periodic preemption if not needed.

Special cases

  • Context switching on interrupt vs. scheduler-initiated: interrupts may be nested; kernel must often handle return-from-interrupt semantics carefully.
  • Lazy propagation of user-space page tables and copy-on-write may reduce perceived cost of switching but can increase TLB shootdown costs when migrating pages.

7. Cost and performance implications

Context switches are not free. Costs include:

  • CPU cycles to save/restore registers and kernel bookkeeping.
  • Cache pressure: switching to a different process invalidates working set locality; instruction and data caches may suffer.
  • TLB / MMU effects: switching address spaces changes page table base register and flushes relevant TLB entries — costly for memory-heavy workloads.
  • Pipeline flushes and branch predictor state loss.
  • Overhead from scheduler decisions and locking when selecting the next task.

Rough example numbers (approximate and architecture-dependent):

  • Simple kernel context switch (save/restore registers, minimal bookkeeping) can cost a few hundred to a few thousand cycles on modern CPUs (tens of microseconds on older hardware).
  • If FPU state, TLB flush, and cache effects are included, the effective latency and lost work can be much higher.

When designing systems, prefer fewer context switches for high-throughput code paths and lower switch cost for latency-critical systems. Real-time systems may tune scheduling to minimize worst-case context-switch latency.

8. Example walkthroughs and narratives

Example A — A preemptive switch: Process A → Process B

  1. Process A is executing user code. Timer interrupt occurs after its quantum ends.
  2. Hardware vector to interrupt handler; CPU pushes return IP/flags to kernel stack and switches to kernel mode.
  3. Interrupt stub in assembly saves volatile registers and calls the kernel C handler.
  4. The kernel updates A's runtime accounting, then calls the scheduler.
  5. Scheduler finds B runnable and better choice (higher priority or next in round-robin).
  6. Kernel saves the rest of A's registers to A's PCB and sets A.state = RUNNABLE.
  7. Switch page tables to B's address space (if needed), restore B registers and stack pointer, and set B.state = RUNNING.
  8. Return-from-interrupt sequence restores CPU flags and jumps to B's instruction pointer — B runs.

Example B — A blocking system call: A sleeps, B runs

  1. A performs a blocking read (no data available). The syscall traps into kernel; kernel marks A as SLEEPING and enqueues it on the I/O wait queue.
  2. The scheduler immediately chooses next runnable B without a timer interrupt because A voluntarily gave up the CPU.
  3. No timer-induced preemption overhead occurred; the kernel still performs the same save/restore steps but may skip some preemptive accounting.

These scenarios show that switching can be voluntary (blocking syscalls) or involuntary (preemption).

9. Context switches for threads vs. processes

Threads within the same process share an address space but have separate stacks and register contexts. Switching between threads of the same process can avoid changing page tables and thus be faster than switching between different processes. This is why lightweight threads and kernel-level thread implementations aim to reduce costly address-space changes.

User-level (green) threads can switch contexts purely in user space with no kernel involvement, avoiding kernel context switch costs entirely — but they cannot utilize multiple CPUs without special arrangements (unless paired with multiple kernel threads).

10. Virtualization and context switching

Virtual machines require switching not just between guest processes but between host and guest contexts. A VM exit (to the hypervisor) and VM entry back add overhead beyond a normal context switch: virtual CPU registers, virtualization-specific control structures, and nested page tables must be saved/restored.

Hence hypervisors try to minimize VM exits and batch emulation to reduce effective switching costs.

11. Measuring and tuning

Tools and approaches:

  • perf (Linux) to measure context-switch counts and CPU time lost to scheduling.
  • vmstat and top for coarse-grained context switch counts per second (cs/s).
  • tracepoints and ftrace/eBPF for detailed tracing of scheduler events and timings.
  • Microbenchmarks: measure a task that yields frequently and measure latency/throughput impact of different quantum sizes.

When tuning, measure both system-wide and application-level performance: fewer context switches isn't always better if it harms responsiveness or fairness.

12. Glossary

PCB / TCB
Process/Thread Control Block — structure storing context and metadata.
Preemptive multitasking
Kernel forcibly interrupts a running thread to switch.
Voluntary context switch
When a thread yields or blocks, allowing scheduler to choose another thread.
FPU laziness
Deferring FPU state save/restore until necessary to reduce overhead.
TLB shootdown
Operation that invalidates TLB entries across CPUs; expensive during migrations.
Tickless kernel
Kernel mode that avoids periodic timer ticks when idle to reduce unnecessary interrupts and switches.
