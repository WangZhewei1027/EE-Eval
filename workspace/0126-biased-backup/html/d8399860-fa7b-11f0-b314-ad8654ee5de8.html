Git — A Comprehensive Conceptual and Technical Guide

This page explains Git's concepts, data model, algorithms, common workflows, and practical step-by-step examples. It focuses on clear textual explanations; interactivity is minimal and purely demonstrative.

High-level overview

Git is a distributed version control system (DVCS). Its purpose is to record the history of files (source code, documents, etc.) and coordinate collaboration across multiple contributors. "Distributed" means every clone of a repository contains the full history and can operate independently.

At its core Git models history as a content-addressable, immutable graph of objects: commits point to tree objects (snapshots of a directory), trees point to blobs (file contents) and other trees, and commits reference parent commits. This structure makes Git robust, efficient, and capable of representing branching and merging histories naturally.

Key terms and definitions

  • Repository (.git): The local database storing objects, refs (branches, tags), hooks, configuration, and logs.
  • Working directory: The checked-out files you edit.
  • Index (staging area): An intermediate state where you prepare what will go into the next commit.
  • Blob: An object containing file data (no metadata except size/type).
  • Tree: An object representing one directory — links names to blobs/trees and stores file modes.
  • Commit: An object that points to a tree object, contains metadata (author, message, timestamp), and points to parent commits.
  • Ref: A name that points to a commit (branch or tag).
  • HEAD: A special ref pointing to the current checked-out commit (usually the tip of a branch).

Git's data model (content-addressable storage)

Git stores four primary object types: blob, tree, commit, and tag. Each object is stored by the cryptographic hash of its contents (historically SHA-1; modern Git supports SHA-256). Because the identifier is computed from content, the same content always yields the same object id (OID). This makes the data model a Merkle DAG (a directed acyclic graph where nodes are identified by hashes).

How an object is stored (conceptually):

object = type + ' ' + size + NUL + content
OID = HASH(object)

Example: a commit object includes:

  • tree <tree-id> (root snapshot)
  • parent <parent-commit-id> (zero or more parents)
  • author & committer metadata
  • commit message

Because commits point to trees (snapshots), Git models history as snapshots, not as a sequence of file diffs. The snapshot approach simplifies reasoning about repository state.

Storage, packfiles, and delta compression

On disk, objects are stored compressed (zlib) as loose objects or combined into packfiles for efficiency. Packfiles contain many objects and use delta compression — instead of storing multiple full versions of similar files, Git can store a base object and a delta (patch) from that base to another object. Creating good packs is an NP-hard optimization, so Git uses heuristics to find efficient pack ordering and delta chains.

Garbage collection (git gc) consolidates loose objects into packs, prunes unreachable objects, and repacks to optimize space and performance.

Integrity and hashing

Every object is referenced by a cryptographic hash; this provides integrity: the content cannot be changed without changing its id. Git originally used SHA-1. Due to collision concerns, newer versions support SHA-256. Besides integrity, this design enables cheap content deduplication and efficient synchronization by exchanging object ids.

Branches, refs, and HEAD

A branch is simply a ref file that contains the id of a commit (the tip). HEAD points to the current ref (often refs/heads/main) or directly to a commit (detached HEAD). Moving a branch just updates the ref to point to another commit — this is cheap and constant-time.

Typical operations:

# create a branch and switch to it
git checkout -b feature

# move the branch tip (commit)
git commit -m "work"

# merge feature into main
git checkout main
git merge feature

Index (staging area) and workflows

The index stores a list of file paths and corresponding blob ids: it represents what will become the next commit's tree. The workflow is often:

  1. Edit files in the working directory.
  2. git add <path> — record the new blob and update the index (stage changes).
  3. git commit — create a commit object referencing the tree described by the index.

The staging area enables fine-grained control: you can stage partial changes (with git add -p), collect multiple logical changes into separate commits, and craft good commit messages.

Merge algorithms — three-way merge and strategies

When combining branches, Git must incorporate changes from different commits. The primary algorithm is the three-way merge:

  1. Find the merge base (the best common ancestor commit of the two tips).
  2. Compute the diff from the base to each tip (A and B): base → A, base → B.
  3. Apply the changes to the base, preferring non-conflicting edits. If both touched the same part of a file and changes overlap semantically, a conflict is produced for human resolution.

Merge strategies select how to combine more than two heads or handle special cases:

  • fast-forward: if the target branch is ancestor of the source, Git just moves the ref forward (no new commit).
  • recursive: default for two heads; performs a recursive merge and can handle criss-cross merges by creating a synthetic merge base.
  • octopus: used to merge multiple branches at once (no conflict resolution for overlapping edits).

Three-way merge is deterministic and relies heavily on the quality of the merge base. When merge conflicts occur, Git marks conflict regions in files using conflict markers for manual resolution.

Rebase and the "replay" algorithm

Rebase replays commits from one base onto another by:

  1. Finding the set of commits to replay (commits reachable from your branch but not from the new base).
  2. For each commit, compute the patch relative to its parent, then apply that patch onto the new base, creating new commits with potentially different ids (different parents and timestamps).

Rebase rewrites history (it creates new commits) and is useful to keep a linear history. It should be used carefully on public/shared branches because rewriting published history can confuse collaborators.

Diffs and the underlying algorithms

For user-visible diffs, Git typically uses line-based diff algorithms (variants of the Myers diff algorithm) to compute minimal edit scripts. For internal storage (pack deltas), Git uses binary delta algorithms and heuristics to find similar objects to delta against. These algorithms try to minimize packed size and optimize transfer.

Networking and transfer: clone, fetch, push

Git sends object ids (and refs) across the network and transfers only objects the other side does not have. Protocols supported include the native git protocol (high performance), SSH, HTTP(S) (smart and dumb variants). The "smart" HTTP/HTTPS endpoints understand Git packfile negotiation and serve efficient transfers; the "dumb" ones rely on static file downloads.

Typical remote operations:

# copy a repository locally
git clone url

# update local refs with remote commits (no merge)
git fetch origin

# fetch and merge (or rebase) into current branch
git pull

# upload local commits to remote
git push origin main

Safety nets: reflog, recoverability, and garbage

Although committing and rebase operations can change history, Git keeps a reflog — a local record of where refs (like HEAD) have pointed recently. This makes it possible to recover commits after accidental branch moves for some time. Unreachable objects remain in the object database until garbage collection prunes them (usually after a safety window).

Security, signing, and integrity

Git integrity is based on cryptographic hashes. For authenticity of user actions, you can sign commits and tags (PGP/GPG or other mechanisms). Signed commits let consumers verify who authored or approved a change. Server-side protections (branch protection, signed tags, forced pushes disabled) add additional safety for shared repositories.

Advanced topics (brief overview)

  • Submodules: Git references other repositories at particular commits — useful but complex; they embed a pointer and require explicit updates.
  • Worktrees: Multiple working copies attached to a single .git repository to work on several branches simultaneously.
  • Large File Storage (Git LFS): Stores large binary files outside core Git objects, replacing them with lightweight pointers to reduce repository bloat.
  • Custom hooks: Scripts that run at lifecycle events (pre-commit, pre-receive, post-receive) to enforce policies or integrate with CI.
  • Partial clones and shallow clones: Techniques to avoid fetching full history for large repos (e.g., depth parameter for shallow clone).

Step-by-step examples and narratives

This section gives textual walkthroughs for common tasks — no real changes happen here; these are examples you can follow in your own environment.

Creating a new repository and making your first commit

  1. Start a repository: 
    git init myproject && cd myproject
  2. Create a file: 
    echo "Hello Git" > README.md
  3. Stage it: 
    git add README.md
  4. Commit: 
    git commit -m "Initial commit: add README"
  5. What happened internally: Git created a blob for README.md, a tree for the root directory, and a commit that points to that tree. The branch ref (usually refs/heads/main) now points to the commit; HEAD points to the branch.

Making a feature branch and merging

  1. Create and switch: 
    git checkout -b feature-x
  2. Make changes and commit on feature-x.
  3. Switch back: 
    git checkout main
  4. Merge feature: 
    git merge feature-x
  5. If the merge is a fast-forward, main will simply point to the feature commit. Otherwise, Git will run a three-way merge and potentially create a merge commit.

Handling a merge conflict (high-level)

  1. Attempt merge: git merge other-branch
  2. Git reports conflicts and marks files with conflict markers. Open the file(s) and look for: 
    <<<<<< HEAD
your changes
=======
their changes
>>>>>> other-commit
  3. Edit to select or combine changes, remove markers.
  4. Stage the resolved files: 
    git add resolved-file
  5. Complete the merge: 
    git commit
    (Git will create a merge commit if necessary).

Common commands and what they do (concise reference)

Local workflow

  • git init — create repository
  • git add — stage changes
  • git commit — record staged changes
  • git status — show state of working dir/index
  • git diff — view differences (working vs index or index vs HEAD)
  • git log — inspect commit history

Collaboration & history

  • git clone — copy remote repo
  • git fetch — update remote refs locally
  • git pull — fetch + merge (or rebase)
  • git push — send commits to remote
  • git rebase — replay commits onto another base
  • git reset — move HEAD and optionally index/working tree

Best practices and pedagogical advice

  • Make small, focused commits with clear messages (what and why).
  • Use branches for features and bugfixes; keep main branch stable.
  • Prefer rebasing local private branches to keep history linear, but avoid rewriting public/shared history.
  • Use code reviews and continuous integration to maintain quality.
  • Learn to read git log, gitk, or graphical tools to understand history structure rather than only commands like blame.

Troubleshooting tips

  • If you accidentally deleted a branch: check reflog (
    git reflog
    ) and create a branch at the commit id you want (
    git branch recover <sha>
    ).
  • If push fails due to non-fast-forward: fetch first, inspect differences, and either merge or rebase as appropriate.
  • To undo a local commit but keep changes staged: 
    git reset --soft HEAD^
  • To undo and throw away changes (be careful): 
    git reset --hard <commit>
