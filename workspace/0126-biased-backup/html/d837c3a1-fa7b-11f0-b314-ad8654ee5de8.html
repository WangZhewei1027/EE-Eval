Definition and Intuition

At a high level, a deadlock is a situation in concurrent computing where a set of processes becomes permanently blocked because each process is waiting for a resource that another process in the set holds. No one can proceed—the system is stuck.

Intuition: imagine two people trying to pass in a narrow corridor. Each refuses to step back. Both wait indefinitely for the other to move. In computing, the "refusal" corresponds to holding a resource while waiting for another.

Formal View and Notation

Common notation: let P = {P1, P2, ..., Pn} be processes and R = {R1, R2, ..., Rm} be resource types (with possibly multiple identical instances per type).

State variables usually include:

  • Allocation A: how many instances of each resource are currently held by each process.
  • Request Rq: outstanding requests by processes for resource instances.
  • Available V: the count of free instances for each resource type.

A deadlock is a global state where every process in some non-empty subset S ⊆ P is blocked, waiting for resources held by processes in S.

Alternative formalization: a deadlock exists iff there is a circular wait (a cycle in the wait-for graph) and other necessary conditions (below) hold.

Coffman Conditions (Necessary Conditions)

In 1971, Coffman et al. identified four necessary conditions that must hold simultaneously for deadlock to occur. Understanding how to break these conditions is the basis for prevention techniques.

  1. Mutual Exclusion: At least one resource must be held in a non-sharable mode (only one process can use it at a time).
  2. Hold and Wait: A process is holding at least one resource and waiting to acquire additional resources that are currently held by other processes.
  3. No Preemption: Resources cannot be forcibly taken from a process; they must be released voluntarily.
  4. Circular Wait: There exists a set of processes {P1, P2, ..., Pk} such that P1 is waiting for a resource held by P2, P2 is waiting for a resource held by P3, ..., Pk is waiting for a resource held by P1.

Breaking any one of these conditions eliminates the possibility of deadlock in that system design.

Resource Allocation Graphs (RAG)

A resource allocation graph is a directed graph used to model resource assignments and requests:

  • Process nodes (usually circles)
  • Resource nodes (usually squares), each resource may have multiple instances
  • An edge from resource R to process P (R → P) indicates R is allocated to P
  • An edge from process P to resource R (P → R) indicates P is requesting R

Key properties:

  • For single-instance resources, the existence of a cycle in the RAG is necessary and sufficient for a deadlock.
  • For multi-instance resources (multiple identical units), a cycle is necessary but not sufficient; more sophisticated checks are required.

Concrete Examples (step-by-step)

Classic two-process, two-resource example

Consider processes P1 and P2 and resources R1 and R2 (one instance each). Sequence:

  1. P1 acquires R1 (holds it).
  2. P2 acquires R2 (holds it).
  3. P1 requests R2 and blocks because R2 is held by P2.
  4. P2 requests R1 and blocks because R1 is held by P1.

Neither can proceed and neither will release the resource they already hold — this is a deadlock. The RAG contains a cycle: P1 → R2 → P2 → R1 → P1.

Step-wise explanation of why no single process can break the deadlock

  1. To finish, P1 needs R2 (but R2 is held by P2).
  2. P2 needs R1 (but R1 is held by P1).
  3. Neither will relinquish their held resource without making forward progress — so they wait forever.

Example with multiple instances

Suppose there are 3 processes and a resource type with 2 identical instances. Even if a cycle exists, there might be enough spare instances to satisfy requests and avoid deadlock. Thus more careful analysis is needed (see detection and Banker's algorithm).

Deadlock Avoidance: Banker's Algorithm

Deadlock avoidance allows the system to make runtime decisions to ensure it never enters an unsafe (potentially deadlocked) state. The most famous avoidance algorithm is Dijkstra's Banker's Algorithm.

Concept

The system treats resource allocation like a bank loan: each process announces the maximum resources it may need. The system will only grant requests that leave the system in a safe state. A safe state is one where there exists a sequence of processes that can complete sequentially with current available resources.

Safety algorithm (informal step-by-step)

  1. Work ← Available (vector of free instances).
  2. Finish[i] ← false for all processes.
  3. Find a process i such that Finish[i] is false and Need[i] ≤ Work.
  4. If found, set Work ← Work + Allocation[i], Finish[i] ← true, and repeat step 3.
  5. If all Finish[i] become true, the state is safe; otherwise it's unsafe.

Pseudocode (request handling)

// When process P requests Request[P]:
if Request[P] <= Need[P] then
  if Request[P] <= Available then
    // Pretend to allocate
    Available = Available - Request[P]
    Allocation[P] = Allocation[P] + Request[P]
    Need[P] = Need[P] - Request[P]
    // Check safety
    if safe_state() then
      // grant request permanently
    else
      // rollback the pretend allocation, deny request (or delay)
    end
  else
    // not enough resources now: block P
  end
else
  // error: process asked more than declared maximum
end

Note: the algorithm requires knowledge of maximum needs and works best for systems with known bounds. It can be conservative and reduce concurrency.

Deadlock Prevention Techniques

Prevention aims to design resource policies that make at least one of the Coffman conditions impossible.

  • Eliminate Mutual Exclusion: If resources can be shared safely (e.g., read-only data), avoid exclusive locks.
  • Eliminate Hold-and-Wait: Require processes to request all needed resources at once and hold none while waiting. This limits concurrency and can result in low resource utilization or starvation.
  • Eliminate No Preemption: Allow preemption: if a process holding some resources is denied another, force it to release held resources and try again later (complex to implement if resources are not easily roll-backable).
  • Eliminate Circular Wait: Impose a global ordering on resource types and require that each process requests resources in ascending order of this ordering. This prevents cycles in the wait graph.

Each technique has trade-offs in performance and complexity.

Deadlock Detection and Recovery

Detection permits the system to enter deadlocked states but includes mechanisms to detect and recover. Detection algorithms differ for single-instance vs multiple-instance resources.

Wait-for Graph

A wait-for graph is derived from an RAG by collapsing resources: an edge Pi → Pj means Pi is waiting for a resource held by Pj. Deadlock detection reduces to cycle detection in this directed graph. Cycle detection uses depth-first search (linear time in edges+nodes).

Detection with multiple resource instances

An algorithm similar to the Banker's safety check can be used periodically: simulate allocations and releases to see if all processes can complete. Complexity is polynomial in number of processes and resource types.

Recovery strategies

  • Process termination: kill one or more processes in the deadlock cycle until the deadlock is broken. Choose victims based on priority, work done, rollback cost.
  • Resource preemption: forcibly take resources away from some processes and give them to others; may require rollback to a safe state.
  • Rollback: if processes are checkpointed, roll back one or more processes to earlier safe points and restart them.

Recovery requires careful policy design to minimize wasted work and maintain correctness, especially for non-transactional resources.

Distributed Deadlocks

Worse in distributed systems because no global view exists and communication delays complicate detection and recovery.

  • Global wait-for graph: edges can span machines; constructing it requires message exchanges.
  • Edge-chasing algorithms: processes send probes along requests to detect cycles (e.g., Chandy-Misra-Haas probe algorithm).
  • Timeouts and heuristics: often used in practice because exact detection is expensive; timeouts risk false positives.

Distributed deadlock detection must handle message loss, process crashes, and partial failures, increasing complexity.

Databases and Two-Phase Locking

Databases commonly use locking protocols to ensure serializability. Two-Phase Locking (2PL) acquires locks in a growing phase and releases them in a shrinking phase.

Strict 2PL (release all locks only after transaction commits or aborts) is common, but it can create deadlocks. Detection via wait-for graphs and periodic deadlock checkers is typical. Simple recovery is to abort one transaction from a detected cycle.

Related Concepts: Livelock, Starvation

It's important to distinguish deadlock from similar phenomena:

  • Livelock: processes are not blocked but constantly change state in response to others—yet make no progress (e.g., two processes repeatedly yielding to each other).
  • Starvation (indefinite postponement): a process never acquires needed resources because the scheduling policy always favors others; not everyone is blocked simultaneously, and the system may still be live.

Prevention and fairness mechanisms help mitigate starvation. Livelock often requires back-off strategies or randomized delays.

Modeling, Complexity, and Formal Analysis

Formal techniques include:

  • State-space exploration: enumerating global states to detect cycles and dead states (feasible for small systems only).
  • Model checking: temporal logic specifications can assert "eventually progress" properties and find counterexamples (deadlocks).
  • Petri nets: a natural model for concurrency and resource sharing; deadlocks correspond to markings with no enabled transitions for a subset of processes.

Complexity: cycle detection is linear in graph size. Banker's algorithm and safety checks are polynomial (O(n*m) roughly). Exhaustive detection by state-space exploration can be exponential in process and resource counts.

Practical Advice and Design Guidelines

  • Favor simpler prevention techniques when the cost of deadlock is low; use detection and recovery when prevention is too restrictive.
  • Implement monitoring: detect cycles early and log detailed state for postmortem analysis.
  • Use timeouts and exponential backoff in distributed systems to limit prolonged waits, but be careful about false positives.
  • When resources are not preemptable, design for ordered acquisition or batch allocation to avoid circular waits.
  • Write clear policies for selecting victims for termination or preemption, and consider transaction semantics to preserve correctness.
