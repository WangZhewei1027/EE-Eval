B+ Tree — Comprehensive Educational Guide

A thorough textual explanation of B+ trees: structure, theory, algorithms (search, insert, delete), design trade-offs, examples, and implementation guidance. This page is intentionally static and text-focused — a single simple demonstration is provided below.

What is a B+ Tree?

A B+ tree is a balanced tree data structure commonly used for indexing and database storage. It is a variant of the B-tree family where all actual record values (or pointers to records) are stored in the leaves, and internal nodes store only keys (one-step routing information). Leaf nodes are typically linked together in a doubly linked list to support efficient range queries.

High-level goals: minimize disk I/O, provide high fan-out (small height), support efficient point and range queries, and allow efficient insertions and deletions while keeping the tree balanced.

Structure and Invariants

A B+ tree of order m (also called degree m) has these common structural properties (definitions vary slightly between authors — here is the conventional variant used in databases):

  • Every node (internal or leaf) can have at most m children (for internal nodes) or hold at most m-1 keys (for some definitions, the order is number of keys per node; be clear in your code/spec).
  • Every internal node (except the root) has at least ceil(m/2) children (thus at least ceil(m/2)-1 keys).
  • All leaves are at the same depth (the tree is height-balanced).
  • Internal nodes contain keys to guide searches, but they do not contain actual record pointers (data) — leaves do.
  • Leaf nodes contain keys and pointers to records (or the records themselves), and leaves are linked in order via sibling pointers to support sequential scans.

Note: Different sources may define "order" differently (maximum keys vs maximum children). When implementing, pick a clear, consistent definition and document it.

Node Types: Internal vs Leaf

Conceptually there are two main node types:

  • Internal nodes (index nodes): store an ordered list of keys and child pointers. If internal node K has keys k1,k2,...,kn, it has n+1 child pointers: p0,p1,...,pn. Key i separates ranges: p0 → keys < k1, p1 → keys in [k1,k2), ...
  • Leaf nodes: store actual key-value entries (or pointers to records). They also include sibling pointers (next/previous) to enable fast scans across ranges.

Visual metaphor: internal nodes act as routing tables; leaves hold the data and are linked left-to-right in key order.

Core Operations — Overview

We present algorithmic descriptions and step-by-step reasoning for:

  • Search (point queries)
  • Range queries (scan between two keys)
  • Insertion (including splitting and promotions)
  • Deletion (including redistribution and merging)

Search (Point Query) — Step-by-step

  1. Start at the root node.
  2. In the current internal node, perform a binary search (or linear search if small) on the node's keys to choose the correct child pointer to follow. If searching key K, choose the child whose interval contains K.
  3. Follow the chosen child pointer and repeat until you reach a leaf.
  4. In the leaf node, search for the exact key K. If present, return the associated value/pointer; if not, the search fails (key not found).

Range Query (K1 to K2)

Perform a search for the lower bound K1 to locate the leaf containing K1 (or the successor if K1 does not exist). Then follow the leaf sibling pointers, scanning leaf entries sequentially until you pass K2. This is extremely efficient because sequential reads of adjacent leaf nodes are often colocated on disk.

Search Complexity

Time complexity: O(h) node accesses to reach a leaf (h = tree height). Within-node key search is O(log m) if using binary search on keys; typically m is large (fan-out), so h is small (often 2–4 for large datasets). Disk I/O cost is proportional to h for single lookups.

Insertion — Detailed Algorithm and Example Walkthrough

Insertion maintains order and balance. The high-level steps:

  1. Locate the leaf node L where the new key K should be inserted (use the search procedure).
  2. Insert K (and its record pointer) into L in sorted order.
  3. If L now has at most the maximum allowed keys, we are done.
  4. If L overflows (exceeds maximum keys), split L into two nodes L and L2:
    • Keep the lower half of keys in L, move the upper half to new leaf L2 (for even splits, conventions may vary about median placement).
    • Insert the separator key (the first key of L2) into the parent internal node as a key and pointer to L2.
  5. If the parent overflows due to the added key, recursively split the parent internal node and propagate a key upward. If the root splits, create a new root (height increases by 1).
  6. Update sibling pointers for leaves; update parent pointers as needed.

Important detail: in a B+ tree the promoted key for the parent during a leaf split is typically the smallest key in the new (right) sibling. Internal node splits promote a middle key (depending on your key-count convention).

Pseudocode (Insertion, simplified)

function BPlus_Insert(root, key, value):
  leaf = find_leaf(root, key)
  insert_in_leaf(leaf, key, value)

  if leaf.key_count <= MAX_KEYS:
    return root

  // Split leaf
  new_leaf = allocate_leaf()
  split_index = ceil((MAX_KEYS+1)/2)
  move keys >= split_index from leaf to new_leaf
  new_leaf.next = leaf.next
  leaf.next = new_leaf

  promoted_key = new_leaf.keys[0]
  return insert_into_parent(root, leaf, promoted_key, new_leaf)


function insert_into_parent(root, node, key, new_node):
  if node is root:
    new_root = allocate_internal()
    new_root.keys = [key]
    new_root.children = [node, new_node]
    return new_root

  parent = node.parent
  insert key and new_node into parent in order

  if parent.key_count <= MAX_KEYS_INTERNAL:
    return root

  // Split parent and propagate
  new_parent = allocate_internal()
  split_index = ceil((MAX_KEYS_INTERNAL+1)/2)
  promote_key = parent.keys[split_index]
  move keys and children right of promote_key to new_parent
  return insert_into_parent(root, parent, promote_key, new_parent)

Illustrative Example (textual)

Suppose a small B+ tree with leaf capacity 3 (max 3 keys per leaf). We'll insert keys in this order: 10, 20, 5, 6, 12, 30.

  1. Insert 10: root is a leaf (empty) — leaf now [10].
  2. Insert 20: leaf [10,20].
  3. Insert 5: leaf [5,10,20] — still at capacity.
  4. Insert 6: attempt to insert into leaf [5,6,10,20] → overflow (4 > 3). Split leaf into:
    • Left leaf: [5,6]
    • Right leaf: [10,20]
    • Promote key 10 to new root internal node
  5. Insert 12: locate leaf for 12 (right leaf [10,20]) → becomes [10,12,20] (fits).
  6. Insert 30: right leaf overflows to [10,12,20,30] → split:
    • Right-left: [10,12]
    • Right-right: [20,30]
    • Promote key 20 to the root internal node — root had key 10; after promoting 20, root becomes internal with keys [10,20] and three child pointers.

At the end, leaves contain the sorted keys linked left-to-right: [5,6] -> [10,12] -> [20,30]. Internal nodes provide routing keys [10,20].

Initial: Tree empty.
Insert 10: Leaf becomes [10].
Insert 20: Leaf becomes [10, 20].
Insert 5: Leaf becomes [5, 10, 20].
Insert 6 (overflow): Pre-split would be [5,6,10,20]. Split into [5,6] and [10,20]. Promote 10 to root.
Insert 12: Goes to right leaf → [10,12,20].

Deletion — Detailed Algorithm and Explanations

Deletion in a B+ tree carefully preserves balance and minimum occupancy. High-level process:

  1. Find the leaf that contains the key K. If not found, nothing to do.
  2. Remove K from the leaf node.
  3. If the leaf still has at least the minimum number of keys, stop.
  4. If it falls below minimum, try to redistribute (borrow) a key from a sibling leaf that has extra keys:
    • If a left sibling has extra keys, move its largest key into the deficient leaf, and adjust the parent separator key.
    • If a right sibling has extra keys, move its smallest key into the deficient leaf, and adjust the parent separator key.
  5. If redistribution is not possible (both siblings at minimum), merge the deficient leaf with a sibling (left or right). Remove the separator key from the parent that separated the merged nodes.
  6. If removing the separator from the parent causes it to go below minimum, repeat the redistribution/merge process up the tree. If the root becomes empty (no keys, single child), replace root with its child (tree height decreases).

Practical note: many implementations prefer borrowing from the closest sibling to minimize key movement. Merges reduce the number of keys in the parent, possibly cascading upwards.

Pseudocode (Deletion, simplified)

function BPlus_Delete(root, key):
  leaf = find_leaf(root, key)
  if key not in leaf: return root

  delete key from leaf

  if leaf is root:
    if leaf.key_count == 0:
      return empty tree
    else:
      return root

  if leaf.key_count >= MIN_KEYS:
    return root

  sibling, is_left = find_sibling(leaf)
  if sibling.key_count > MIN_KEYS:
    redistribute_between(leaf, sibling, is_left)
    return root
  else:
    merge_leaf_and_sibling(leaf, sibling, is_left)
    delete_separator_from_parent(leaf.parent, separator_key)
    if parent.key_count < MIN_KEYS:
      // handle parent underflow recursively
      fix_parent_underflow(root, parent)
    return root

Deletion Example (scenario)

Consider leaves with capacity 4 and minimum 2. Suppose you have two neighboring leaves:

  • Left leaf: [10, 12]
  • Right leaf: [20, 30]

If you delete 12, left leaf becomes [10] which is below minimum. If the right sibling has > min keys you can borrow 20 (smallest of right) and update parent separator key from 20 to 20 (or adjust according to your representation). If the sibling cannot lend, you merge leaves to [10,20,30] and remove the separator in the parent — potentially cascading.

Complexity, Height, and Disk Considerations

Key characteristics and performance analysis:

  • Height (h) is O(log_m N) where m is the average fan-out (children per internal node) and N is the number of keys. If nodes fit a whole block (e.g., 4KB) and keys/pointers are small, m can be large, so h is very small.
  • Search/insert/delete cost measured in disk I/Os is roughly proportional to h (number of node accesses). Internal node scans are memory-friendly; leaves hold record pointers or record ranges.
  • B+ trees are designed to maximize fan-out (m) to minimize height and number of disk accesses.

Example: If a node can hold 100 keys (m ≈ 100), then h for a million keys is roughly log_100(1,000,000) ≈ 3. So most lookups touch < 4 nodes (I/Os).

B+ Tree vs B-tree — Key Differences

They are related, but here are distinguishing points:

  • Data placement: B-tree stores keys and records (or pointers) in every node (internal and leaves). B+ tree stores data only in leaves (internal nodes store keys for routing only).
  • Range queries: B+ tree is typically faster for range scans because leaves are linked and contain all records sorted; B-tree may spread records across internal nodes making range scans less sequential.
  • Space utilization: B+ internal nodes can be denser (only keys) so higher fan-out; resulting height usually smaller in B+ trees for the same dataset and node size.
  • Implementation simplicity for databases: B+ trees are commonly used in databases and filesystems because they make sequential scans and block-based storage efficient.

Variants and Practical Considerations

Many practical systems use variants or optimizations:

  • Prefix compression in internal nodes to save space when keys have long common prefixes (e.g., strings).
  • Variable-length keys support: nodes store offsets or compact representations to pack more keys per page.
  • Bulk loading: build a B+ tree efficiently by sorting keys and filling leaves fully before creating internal nodes; much faster than successive inserts.
  • Concurrency control and locking: latch coupling (crabbing) or lock-free variants to allow concurrent readers/writers. Typical approach: fine-grained locks on nodes, careful ordering to avoid deadlocks, or optimistic concurrency control.
  • Checksum and crash recovery: many DBMSes add checksums and write-ahead logs (WAL) to ensure atomic updates and durability of splits/merges.

Implementation Tips and Practical Guidance

When implementing a B+ tree, consider these pragmatic details:

  • Define clearly: maximum keys per node, minimum occupancy, and whether the order is measured by keys or children.
  • Choose an in-node search algorithm: linear scan for very small node sizes, binary search for moderate sizes, or small SIMD-accelerated search if keys are numeric and performance-critical.
  • Node layout on disk: allocate nodes as pages with fixed-size headers (node type, num keys, sibling pointers, parent pointer optional). Keep pointers as page IDs rather than memory addresses.
  • Splitting policy: choose whether to do equal split or bias splits (may impact tree shape slightly). For leaf splits, typically split roughly in half and promote the first key of the right sibling.
  • Deletion policy: implement redistribute before merge where possible; merging reduces tree size but may be more expensive if done too aggressively.
  • Buffer management: keep frequently accessed internal levels in memory to avoid repeated disk reads — often the top few internal nodes remain cached.
  • Testing: include unit tests for edge cases (root splits, root becoming empty, merging at different levels, duplicates, bulk load, restart after crash if persistence involved).

Advanced Topics and Considerations

A few advanced subjects for deeper study:

  • Latch-free and lock-free B+ trees: research papers propose multi-word CAS and versioned nodes to enable concurrent updates without global locks.
  • Adaptive fan-out and compression: adapt node capacities based on workload to trade memory vs I/O.
  • Cache-conscious designs (B+-Tree variants): e.g., B-tree variants tuned for CPU caches (CSB+-trees) pack child pointers contiguously to leverage prefetching.
  • Hybrid structures: combine trie-like prefix routing with B+ trees for efficient string keys.
  • Secondary indexes: B+ trees can be used for secondary indexes mapping keys to record IDs; duplicates are handled by storing lists of pointers or allowing repeated keys in leaves.

Recap: Why Use a B+ Tree?

  • Efficient disk-based indexing with low height and few I/Os per lookup.
  • Excellent support for range scans and ordered iteration thanks to linked leaves.
  • Balanced tree guarantees — consistent performance for lookups, inserts, and deletes.
  • Flexible node size design allows tuning to block size or cache lines for performance.
