Circular Linked List — Comprehensive Explanation

An in-depth, pedagogical guide to the concept, structure, algorithms, edge cases, and practical notes on circular linked lists. This page emphasizes thorough textual explanation and algorithmic reasoning. A single, simple demonstration is available at the end.

Definition & Types

A circular linked list is a linked list in which nodes are connected to form a circle: the last node points back to the first node. Unlike linear linked lists, there is no null (or None) pointer at the tail. Circular linked lists come in a few common variants:

  • Singly circular linked list: each node has a single pointer/reference to the next node; the last node's next points to the head.
  • Doubly circular linked list: each node has pointers to both the next and previous nodes; both directions wrap around so the head's previous points to the tail and the tail's next points to the head.
  • Circular sentinel (dummy) node: some implementations maintain a sentinel node so that insertion and deletion logic is simplified because the list is never truly empty — the sentinel always exists and points to itself when empty.

Why use a circular list? It provides easy wrap-around traversal and can simplify some operations where rotation is required. It is especially useful when you need to cycle through elements repeatedly without restarting traversal explicitly.

Typical node for a singly circular list:

  value;      // data payload
  next;       // pointer/reference to another Node

Key Properties

  • No null at tail: the last node's next points back to a node (commonly the head).
  • Traversal must be aware of stopping condition to avoid infinite loops — typically stop when you revisit a starting node.
  • Operations at both ends (beginning and end) can be done efficiently if you maintain a tail pointer: insert at beginning and end can be O(1) with careful pointer updates.
  • With a sentinel node, even empty lists behave uniformly and operations are simpler.

Important: Many algorithms for linear lists must be slightly adapted when using circular lists because there is no natural null end marker.

Traversal & Visiting All Nodes (Step-by-step)

Goal: visit each node exactly once and stop. A typical pattern:

  1. If the list is empty (no head/tail), stop — there are no nodes.
  2. Pick a start node (commonly the head).
  3. Process the start node.
  4. Move to current = current.next and repeat processing until current == start again; then stop.

Careful: If you don't check for revisiting the start, traversal becomes infinite. When implementing, guard against an empty list first.

// pseudocode: traverse and print values
if head == null:
  return
start = head
current = start
do:
  visit(current)
  current = current.next
while current != start

When using a pointer to the tail (last node) instead of head, you can set start = tail.next (which is the head).

Insertion Algorithms (Step-by-step)

This section covers common insertions: at beginning, at end, and after a given node. We'll show step-by-step reasoning and pseudocode.

Insert at beginning (no tail maintained)

Case: only have a pointer to head (which may be null).

  1. Create a new node n with desired value.
  2. If head is null: set n.next = n (node points to itself) and head = n.
  3. Otherwise: find the tail by iterating until node.next == head; set tail.next = n; set n.next = head; set head = n.
// insertAtBeginning(head, value) -> returns new head
n = new Node(value)
if head == null:
  n.next = n
  return n    // new head
// find tail
tail = head
while tail.next != head:
  tail = tail.next
tail.next = n
n.next = head
return n

Insert at beginning (with tail pointer)

If you maintain a tail pointer (null if empty), insertion at beginning becomes O(1).

  1. Create new node n.
  2. If tail is null: set n.next = n and tail = n.
  3. Otherwise: set n.next = tail.next (head) and tail.next = n. The new node is the new head but tail remains the same.
// tail != null points to last node
n = new Node(value)
if tail == null:
  n.next = n
  tail = n
else:
  n.next = tail.next
  tail.next = n
// head is tail.next

Insert at end

Two main approaches:

  • Without tail pointer: find the tail (node whose next is head) and insert similar to previous, O(n).
  • With tail pointer: O(1). Insert new node after tail and update tail to this new node.
// with tail pointer
n = new Node(value)
if tail == null:
  n.next = n
  tail = n
else:
  n.next = tail.next   // new node points to head
  tail.next = n        // old tail points to new node
  tail = n             // update tail

Insert after a given node

Assuming you already have a pointer/reference to node p:

  1. Create new node n.
  2. Set n.next = p.next; p.next = n.
  3. If p was tail and you maintain tail, update tail = n.
n = new Node(value)
n.next = p.next
p.next = n
if p == tail:
  tail = n

Deletion Algorithms (Step-by-step)

Common deletions: delete head, delete tail, delete a specific node with given value or pointer. We must carefully handle empty list and single-node list.

Delete head (no tail pointer)

  1. If head is null: list empty — nothing to do.
  2. If head.next == head: list has one node; free it and set head = null.
  3. Otherwise: find tail (node whose next == head). Let newHead = head.next. Set tail.next = newHead. Free the old head and set head = newHead.
// deleteHead(head) -> returns new head
if head == null:
  return null
if head.next == head:
  free(head)
  return null
// find tail
tail = head
while tail.next != head:
  tail = tail.next
newHead = head.next
tail.next = newHead
free(head)
return newHead

Delete head (with tail pointer)

  1. If tail is null: empty — nothing to do.
  2. If tail.next == tail: only one node — free it and set tail = null.
  3. Otherwise: head = tail.next; remove head by setting tail.next = head.next; free old head.
// tail -> last node
if tail == null:
  return
head = tail.next
if head == tail:
  free(head)
  tail = null
else:
  tail.next = head.next
  free(head)

Delete a node after node p

  1. If p is null: nothing to do.
  2. Let toDelete = p.next. If toDelete == p (single-node special case), handle accordingly.
  3. Set p.next = toDelete.next. If toDelete == tail, update tail = p.
  4. Free toDelete.
toDelete = p.next
if toDelete == p:
  // only one node
  free(toDelete)
  tail = null
else:
  p.next = toDelete.next
  if toDelete == tail:
    tail = p
  free(toDelete)

Delete by value (first occurrence)

  1. If list empty: done.
  2. Use two pointers: prev and curr. Initialize prev = tail (if available) and curr = head.
  3. Traverse until curr.value == target or you return to head.
  4. If found: set prev.next = curr.next; adjust tail if needed; free curr.
// deleteValue(tail, value)
if tail == null:
  return
prev = tail
curr = tail.next   // head
do:
  if curr.value == value:
    if curr == prev:   // single node
      free(curr); tail = null; return
    prev.next = curr.next
    if curr == tail:
      tail = prev
    free(curr)
    return
  prev = curr
  curr = curr.next
while curr != tail.next

Searching & Computing Size

Search is similar to traversal: start at head (tail.next if using tail), iterate until you either find the value or return to start.

To compute size:

  1. If list empty: size = 0.
  2. Start at head; set count = 1 and traverse until you return to head, incrementing for each node encountered.
// size(tail)
if tail == null:
  return 0
count = 1
curr = tail.next
while curr.next != tail.next:
  count += 1
  curr = curr.next
return count

Note: computing size is O(n) each time unless you maintain a separate counter.

Cycle Detection & Breaking Cycles

Although a circular linked list is by definition cyclic, in general linked lists you may need to detect cycles. Two common techniques:

Floyd's Tortoise and Hare (fast/slow)

Concept: advance two pointers at different speeds; if they meet, a cycle exists. To find the cycle's starting node and length is also possible.

// detect cycle (for general singly linked list)
slow = head
fast = head
while fast != null and fast.next != null:
  slow = slow.next
  fast = fast.next.next
  if slow == fast:
    // cycle detected
    break
if fast == null or fast.next == null:
  // no cycle
else:
  // meeting point; to find start: reset slow = head; move both 1 step until they meet
  slow = head
  while slow != fast:
    slow = slow.next
    fast = fast.next
  // slow (or fast) is cycle start

Breaking a cycle intentionally

If you have a circular list and want to convert it to a linear list:

  1. If list empty: nothing to do.
  2. If list has one node: set node.next = null.
  3. Otherwise: find tail (node whose next is head) and set tail.next = null.

In environments with garbage collection, be mindful of references to allow collection. In manual memory-managed languages, free nodes as appropriate.

Josephus Problem — Example Use of Circular Lists

The Josephus problem is a classic application of circular lists: people arranged in a circle; counting proceeds and every k-th person is eliminated until only one remains.

High-level algorithm using a circular list:

  1. Build a circular list of n nodes representing people.
  2. Keep a pointer curr initially at tail (or head); to eliminate k-th node, advance curr k-1 times then remove curr.next.
  3. Repeat until one node remains; that node is the survivor.
// josephus(n, k) returns survivor's index (1-based)
create circular list of nodes 1..n, let tail point to last node
while tail != tail.next:      // more than one node
  // move k-1 steps from tail so tail.next (head of counting) is the node to remove
  for i in 1..(k-1):
    tail = tail.next
  // remove tail.next
  toRemove = tail.next
  tail.next = toRemove.next
  free(toRemove)
return tail.value

This algorithm runs in O(n*k) naively. There is also a mathematical closed-form O(n) recurrence solution for the survivor.

Edge cases (must guard against these in code)

  • Empty list (head or tail null).
  • Single-node list where node.next == node.
  • Operations that assume a null terminator (wrong for circular lists).
  • Infinite loops if traversal stopping condition is wrong.
  • Concurrent modifications (multi-threading) — need synchronization.

Common pitfalls

  • Using a while (curr != null) traversal condition — never true for a circular list.
  • Forgetting to update the tail pointer after insert/delete — leads to incorrect head references.
  • Accidental memory leaks in manual memory management when nodes are no longer referenced but still point to each other in a cycle; explicitly break cycles if needed before freeing structures.

Pseudocode & Short Implementations

Below are concise examples in common languages (illustrative, not complete with memory management details).

Python (singly circular with tail)

class Node:
    def __init__(self, value):
        self.value = value
        self.next = None

class CircularList:
    def __init__(self):
        self.tail = None

    def insert_end(self, value):
        n = Node(value)
        if not self.tail:
            n.next = n
            self.tail = n
        else:
            n.next = self.tail.next
            self.tail.next = n
            self.tail = n

    def traverse(self):
        if not self.tail: return
        curr = self.tail.next
        start = curr
        while True:
            print(curr.value)
            curr = curr.next
            if curr == start:
                break

C (singly circular with explicit pointers)

typedef struct Node {
  int value;
  struct Node* next;
} Node;

/* insert at end with tail pointer */
void insert_end(Node** tailRef, int v) {
  Node* n = malloc(sizeof(Node));
  n->value = v;
  if (*tailRef == NULL) {
    n->next = n;
    *tailRef = n;
  } else {
    n->next = (*tailRef)->next;
    (*tailRef)->next = n;
    *tailRef = n;
  }
}

The exact memory handling differs by language, but pointer-update patterns are the same.

Applications, Practical Notes, Exercises & Answers

Applications

  • Round-robin scheduling (tasks are represented in a circular list and repeatedly cycled).
  • Implementing a fixed-buffer that naturally wraps around.
  • Games and simulations where a circular structure models a loop (e.g., players around a table).
  • Algorithms like Josephus or certain graph traversals use circular lists conceptually.

Practical notes

  • Prefer maintaining a tail pointer for constant-time insertions at both ends and simpler edge-case handling.
  • Consider sentinel (dummy) node if you want to avoid many special cases. The sentinel points to itself when empty; its next is head.
  • When converting between circular and linear representations, explicitly handle pointer update to avoid leaking or dangling pointers.

Exercises (small)

  1. Given a circular list with n nodes, write pseudocode to rotate the list left by k positions.
  2. Design an algorithm to split a circular list into two halves (if odd, first list has one more node).
  3. Given a singly linked list that may or may not be circular, write steps to determine whether it is circular and, if so, return the node where the cycle begins.

Exercise Answers (concise)

1 — rotate left by k: If you maintain tail, you can move tail forward k % n steps; new head = tail.next afterwards. This is O(k) or O(1) if you can compute advancement using stored structure index.

2 — split into halves: Use slow/fast pointers starting at head: advance fast by 2 and slow by 1 until fast reaches head or head.prev equivalent; slow will be at midpoint; split by adjusting next pointers so two circular lists result. Carefully handle even/odd lengths.

3 — detect & return cycle start: Use Floyd's algorithm: when slow == fast, reset slow to head and advance both one step at a time until they meet; that meeting node is cycle start.

Mini-challenge: Why is a circular list useful for implementing a queue with fixed capacity? (Hint: think about wrap-around and using an array vs. linked nodes.)

Answer: For a fixed-capacity queue you often use a circular buffer (array). A circular linked list provides natural wrap-around for pointers; when capacity is dynamically sized or nodes are reused, circular linkage models the circular buffer behavior without needing modulo arithmetic on indices.
