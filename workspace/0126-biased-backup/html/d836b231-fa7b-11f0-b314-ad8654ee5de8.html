Backtracking — A Comprehensive Textual Guide

An in-depth explanation of the backtracking paradigm, algorithmic patterns, examples, and guidance for practical use. This page is primarily textual and educational; a single simple demonstration is available below.

What is Backtracking?

Backtracking is a general algorithmic technique for solving constraint satisfaction and combinatorial search problems. It systematically explores the space of candidate solutions by building them incrementally and abandoning ("backtracking" from) candidates as soon as it determines they cannot lead to a valid complete solution.

Think of backtracking as depth-first search of the solution space with early pruning when partial solutions violate constraints.

Core Concepts and Terminology

  • State: A partial assignment or configuration (e.g., first k positions filled in a permutation).
  • Choice/Decision: A single action that extends the state (e.g., place a number at index i).
  • Constraint: A rule that partial or complete states must satisfy.
  • Solution: A complete state satisfying all constraints.
  • State-space tree: A tree where nodes are states and edges are choices; leaves are complete states.
  • Pruning: Rejecting a branch early because no extension can lead to a solution.

Generic Backtracking Template (Pseudocode)

Below is the canonical recursive description. It emphasizes structure over language-specific detail.

// backtrack(state):
//   if state is a complete solution:
//     record or output solution
//     return (possibly continue searching for more solutions)
//   for each choice that extends state and does not violate constraints:
//     make the choice (modify state)
//     backtrack(new state)
//     undo the choice (restore state)

Key points: (1) choices are explored one-by-one; (2) the "undo" step is essential and often implemented by recursion stack or explicit revert operations.

How Backtracking Works — An Intuitive Narrative

Imagine you must place a sequence of items in positions. For each position, you try possible items that are compatible with already placed ones. If a chosen item later leads to a conflict or fails to lead to a full solution, you remove it and try the next choice. This is like exploring a maze: you follow a path until you hit a dead end, then backtrack to the last branch point and try a different path.

Step-by-step

  1. Start with an initial empty state (no choices made).
  2. Consider the first decision point: enumerate feasible choices.
  3. For each choice, extend the state and recursively proceed to the next decision point.
  4. If you reach a full valid solution, record it (or return it).
  5. If you reach a state that violates constraints or leaves no choices, undo the last choice and try the next one.
  6. Continue until all branches have been explored (or until you find the desired number of solutions).

When to Use Backtracking

Backtracking is appropriate when:

  • You must search a discrete combinatorial space (permutations, subsets, assignments).
  • Constraints allow early pruning of many partial solutions.
  • There is no obvious polynomial-time algorithm and the search space is manageable with pruning.

Common applications: constraint satisfaction problems (CSPs), puzzles (Sudoku, crosswords), combinatorial generation (permutations, combinations), graph coloring, N-Queens, exact cover.

Pruning and Heuristics

Effective backtracking relies on pruning and good ordering of choices.

  • Early pruning: Evaluate constraints as soon as possible to discard invalid partial states.
  • Variable/choice ordering: Choose the next decision that is most constrained (minimum remaining values heuristic) to reduce branching.
  • Value ordering: Try values more likely to succeed first (heuristic-driven).
  • Forward checking: After making a choice, temporarily eliminate incompatible possibilities from future variables and detect dead ends early.
  • Constraint propagation: More aggressive than forward checking (examples: arc consistency / AC-3), it tightens domains of variables across constraints.

Complexity

Worst-case time complexity of backtracking is exponential in the size of the input — often proportional to the size of the search tree. However, with effective pruning the practical runtime can be dramatically better.

Space complexity is typically proportional to recursion depth (the size of the partial solution), not the total number of nodes in the search tree.

Implementation Details & Practical Tips

  • Represent the partial state compactly (arrays, bitmasks) so making/undoing choices is cheap.
  • Avoid copying large structures at every recursive step; instead mutate and revert, or use incremental updates.
  • Implement fast constraint checks. For example, in N-Queens maintain arrays for columns and diagonals rather than scanning the whole board.
  • When generating unique permutations with duplicates, sort the input and skip duplicate choices when appropriate.
  • Consider iterative backtracking (explicit stack) when recursion depth is problematic or for certain iterative interfaces.

Classic Example: N-Queens (Detailed Walkthrough)

Problem statement: Place N queens on an N×N chessboard so that no two queens attack each other (no two in same row, column or diagonal). Standard backtracking sets one queen per row and chooses a column for each row.

Pseudocode (one-queen-per-row approach)

function solveNQueens(row):
  if row == N:
    record solution
    return
  for col in 0..N-1:
    if column[col] free and diag1[row+col] free and diag2[row-col+N-1] free:
      place queen at (row,col)
      mark column[col], diag1[row+col], diag2[row-col+N-1] as occupied
      solveNQueens(row+1)
      remove queen at (row,col) and unmark
      // continue trying next col

Other Example Problems and Notes

Permutations

To generate all permutations of an array, choices at position i are any element not yet used. The state stores a partial permutation and a boolean used[] array. Avoid duplicates by skipping equal elements when already considered at the same recursion level. 

permute(prefix):
  if prefix.length == n: record prefix
  for each element e not used:
    mark e used; append e to prefix
    permute(prefix)
    remove e from prefix; mark e unused

Subsets (Power Set)

Each element has two choices: include or exclude. This yields 2^n leaves. Useful for exhaustive checks when n <= ~25 with pruning for constraints. 

subset(index, current):
  if index == n: record current
  else:
    // exclude
    subset(index+1, current)
    // include
    current.add(a[index])
    subset(index+1, current)
    current.removeLast()

Subset Sum / Knapsack (decision variant)

Consider branch-and-bound: compute upper bounds to prune branches where total achievable sum can't reach target. Order items by value/weight ratio for better pruning.

Advanced Variations and Optimizations

  • Iterative Deepening: Combine depth-first search with increasing limits to control depth or cost (useful when solution depth is unknown).
  • Meet-in-the-middle: Split problem when full search is expensive and combine partial results.
  • Dancing Links (DLX): An efficient technique by Donald Knuth to implement exact cover backtracking with fast removal/restoration of constraints.
  • Memoization and DP pruning: For some decision problems you can memoize failed partial states to avoid repeating work.

Common Mistakes and Pitfalls

  • Forgetting to undo state changes (leads to incorrect results).
  • Copying large structures naively at each step—prefer mutation+undo or sparse copies.
  • Not pruning early enough (search explodes).
  • Generating duplicates (e.g., permutations with repeated elements) without proper checks.

Checklist for Designing a Backtracking Solution

  1. Define state clearly (what partial information do you maintain?).
  2. Define choices available at each step.
  3. Define constraints and how to check them cheaply.
  4. Decide how to record solutions (first found or all solutions?).
  5. Consider ordering and pruning heuristics to reduce search.
  6. Implement make/undo operations carefully and test on small instances.
