Compiler — Complete Conceptual Guide

A thorough, text-first explanation of compilers: theory, phases, algorithms, examples, and a minimal demonstration.

What is a Compiler?

A compiler is a program that translates source code written in a high-level programming language (the source language) into another form (commonly machine code, assembly, or an intermediate representation). The goal is to transform human-readable instructions into an efficient, executable form that can be run by hardware or a virtual machine.

A compiler does more than simple text replacement: it performs lexical and syntactic analysis, verifies semantics (type checking, name resolution), optimizes code, and generates target-specific instructions. Compilers bridge abstract programming concepts and concrete machine behavior.

Major Phases of a Compiler (Overview)

  1. Lexical Analysis (Scanning): Convert a stream of characters into a stream of tokens (identifiers, numbers, operators).
  2. Syntax Analysis (Parsing): Build a parse tree or abstract syntax tree (AST) from tokens, checking grammar rules.
  3. Semantic Analysis: Enforce language semantics—type checking, scope resolution, symbol table building, and annotating the AST.
  4. Intermediate Code Generation: Produce a machine-independent representation such as three-address code or bytecode.
  5. Optimization: Improve the intermediate code for performance, size, or power without changing semantics.
  6. Code Generation: Translate optimized intermediate code into target machine instructions or bytecode.
  7. Assembly, Linking, and Loading: Assemble, link multiple modules, and prepare a runnable executable with proper runtime layout.

1. Lexical Analysis

Lexical analysis groups input characters into tokens. Tokens have types (IDENTIFIER, NUMBER, PLUS, KEYWORD) and attributes (like the identifier name or numeric value).

Key concepts

  • Regular expressions define token patterns.
  • Lexers are implemented by constructing finite automata (NFAs then DFAs) from regexes and running the DFA on the input.
  • Longest-match and maximal munch rules resolve ambiguities (e.g., ">=" vs ">" then "=").

Basic algorithm (high level)

  1. Read input characters.
  2. Use DFA to find the longest sequence that matches a token pattern.
  3. If matched, emit token and repeat; otherwise report a lexical error.

Implementation tip: lexical analyzers can be generated automatically by tools such as lex/flex or handcrafted for specific languages.

2. Syntax Analysis (Parsing)

The parser consumes tokens and checks whether they conform to the grammar of the language, producing a parse tree or AST.

Grammar types

  • Context-free grammars (CFGs) express language syntax.
  • Common parser strategies: top-down (LL) and bottom-up (LR).

Top-down (LL) parsing

LL parsers read input left-to-right and produce a leftmost derivation. LL(1) uses a single lookahead token and a parse table. Recursive-descent is a common handwritten LL parser.

Bottom-up (LR) parsing

LR parsers build a rightmost derivation in reverse. Variants such as SLR, LALR, and canonical LR differ by construction of parsing tables. LR parsers use shift and reduce actions on a stack to recognize grammar constructs.

Example: parsing a simple expression

Given grammar: E → E + T | T; T → T * F | F; F → (E) | id

The parser converts token sequence id + id * id into a tree that reflects operator precedence (multiplication higher than addition).

3. Semantic Analysis

Semantic analysis ensures the AST obeys language rules beyond syntax. This phase performs:

  • Type checking and type inference.
  • Scope and binding resolution using symbol tables (globals, local scopes).
  • Checking for semantic errors (undefined variables, incorrect number of arguments).
  • Augmenting the AST with type information and other annotations used by later phases.

Symbol tables

Symbol tables map names to declarations and attributes (type, storage class, location). They are typically organized per scope using chained hash tables or a stack of tables.

Type systems and inference

Static typed languages check types at compile-time, often using unification (e.g., Hindley-Milner) for inference or constraint solving. Dynamic languages may perform less at compile time and defer checks to runtime.

4. Intermediate Representations (IR)

IR is a lower-level, machine-independent representation that the compiler uses for optimizations and transformations. Common IRs:

  • Three-address code (quadruples): op a b result (e.g., t1 = a + b)
  • Static Single Assignment (SSA): each variable assigned once; easier for many optimizations.
  • Control-flow graphs (CFGs): basic blocks connected by control flow edges.

Three-address code example

t1 = b * 2
t2 = a + t1
c = t2

This breaks down complex expressions for clearer analysis and emission of target code.

5. Optimization Techniques

Optimizations transform IR to improve performance or reduce size while preserving semantics.

Local optimizations

  • Constant folding: evaluate constant expressions at compile time (2+3 → 5).
  • Copy propagation: replace variables that are mere copies with original values.
  • Peephole optimization: local instruction sequences replaced with more efficient ones.

Global optimizations

  • Common subexpression elimination: reuse results of repeated computations.
  • Dead code elimination: remove code whose results are never used.
  • Loop-invariant code motion: move computations out of loops when safe.

Data-flow analysis

Data-flow frameworks compute properties (like liveness or reaching definitions) by solving equations over the CFG using iterative algorithms. A typical iterative solver uses a worklist: 

// pseudocode for generic iterative data-flow
initialize IN[n], OUT[n] for each block n
worklist = {all blocks}
while worklist not empty:
  remove block n
  oldOUT = OUT[n]
  IN[n] = meet({OUT[pred] for pred in predecessors(n)})
  OUT[n] = transfer(n, IN[n])
  if OUT[n] changed:
    add successors(n) to worklist

The "meet" operation depends on the analysis (union for liveness, intersection for available expressions).

SSA (Static Single Assignment)

Converting IR to SSA simplifies many optimizations. SSA requires inserting phi-functions at join points for variables with multiple definitions. Dominance frontiers help compute where to place phis.

6. Code Generation

Code generation maps IR to target machine instructions. This includes instruction selection, register allocation, and instruction scheduling.

Instruction selection

Use pattern matching on expression trees/DAGs to choose efficient target instructions. Modern compilers use table-driven or tree-pattern matching algorithms.

Register allocation

Registers are limited. Two common strategies:

  • Graph coloring: construct an interference graph where nodes are variables and edges indicate simultaneously-live variables. Coloring assigns registers; spilling occurs when coloring fails.
  • Linear scan: simpler, faster approach often used in JITs. It allocates registers by scanning intervals and spilling when necessary.

Calling conventions and activation records

Compilers must generate code that obeys the platform's calling convention: how arguments are passed (registers or stack), how return values are passed, which registers must be preserved, and the layout of activation records (stack frames).

7. Linking, Loading, and Runtime

Linking resolves external references across translation units and combines object files into an executable. The loader places program segments into memory and prepares initial execution context.

Memory layout

  • Text: code segment
  • Data: global/static initialized data
  • BSS: uninitialized global data
  • Heap: dynamic allocations
  • Stack: function frames and local variables

Runtime systems

Managed languages (Java, .NET) include runtime components like garbage collectors, class loaders, and JIT compilers that interact with the compiled code at execution time.

Common Algorithms and Pseudocode (Concise)

Lexical analysis via DFA

// Pre-built DFA transition table
state = start
for each character c in input:
  state = transition[state][c]
  if state is accepting:
    mark potential token end
if no accepting state found at current position:
  report lexical error

Shift-reduce parsing (very high-level)

stack = [0]  // states
while true:
  action = parseTable[stack.top][lookahead]
  if action is shift s:
    stack.push( token ); stack.push(s); consume lookahead
  else if action is reduce by production A → β:
    pop 2 * |β| symbols; t = goto[stack.top][A]; stack.push(A); stack.push(t)
  else if action is accept: success
  else: error

Graph coloring register allocation (outline)

  1. Build interference graph from liveness analysis.
  2. Simplify: remove nodes with degree < K (K = number of registers), push them on a stack.
  3. If no such node, spill: choose a node, remove it, mark for spilling.
  4. Assign colors by popping stack and choosing a register not used by neighbors; spilled nodes require memory loads/stores.

Errors and Recovery

Compilers must detect and report errors with useful messages. Techniques for error recovery include panic-mode recovery (skip tokens until a statement boundary), phrase-level recovery (insert/delete tokens), and global correction algorithms (find minimal edits).

Good diagnostics are often as important as correctness — telling the programmer what went wrong and where saves time.

Practical Considerations and Tools

  • Lexer and parser generators: lex/flex, yacc/bison, ANTLR (generates LL(*) parsers), Menhir (OCaml).
  • IR and optimization frameworks: LLVM, GCC's GIMPLE/RTL, MLIR for multi-level IRs.
  • Debugging tools: produce intermediate dumps (tokens, AST, CFG), use test suites and formal verification where necessary.

Further Topics (Advanced)

  • Just-In-Time (JIT) compilation: compile at runtime with profiling feedback.
  • Profile-guided optimization (PGO): use runtime data to specialize code.
  • Speculative optimization and deoptimization: assume hot-path behavior and revert when assumptions violate.
  • Formal methods for compiler correctness: translation validation, verified compilers (e.g., CompCert).

Quick glossary

  • Token: lexical unit.
  • AST: abstract syntax tree, simplified parse tree.
  • CFG: control-flow graph.
  • SSA: static single assignment form.
  • IR: intermediate representation.
  • CFG (grammar): context-free grammar.

Quick algorithms at a glance

  • Regex → NFA → DFA: Thompson's construction and subset construction.
  • LR parse table construction: items, closures, goto + conflict resolution.
  • Data-flow: worklist solver with meet operators.
  • Register allocation: simplify/spill/assign coloring or linear-scan intervals.
