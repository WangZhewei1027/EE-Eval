Greedy Algorithms — Comprehensive Explanation

A thorough, educational narrative covering the greedy strategy, key properties, classic algorithms, proof techniques, counterexamples, and step-by-step reasoning.

Overview: What is a Greedy Algorithm?

A greedy algorithm builds a solution piece by piece, always choosing the next piece that offers the most apparent immediate benefit (according to some criterion). The algorithm never revisits choices: once a choice is made, it is final. This simplicity often yields fast and easy-to-implement solutions.

Core idea (informal)

To solve a problem, repeatedly choose a locally optimal option with the hope that this leads to a globally optimal solution. Examples of "local" choices: pick the largest or smallest remaining item, pick the interval that finishes earliest, pick the shortest edge, etc.

When Greedy Works — Key Properties

  • Greedy-choice property: A globally optimal solution can be reached by making a greedy choice (i.e., a locally optimal choice) and then solving the remaining subproblem optimally.
  • Optimal substructure: An optimal solution to the problem contains optimal solutions to subproblems.

These two properties are necessary (informally) to justify a greedy approach. Verifying them rigorously is often the core of correctness proofs.

Typical proof technique: exchange argument

The exchange argument shows that any optimal solution can be transformed (by swapping or replacing parts) into a solution that begins with the greedy choice, without decreasing optimality. This establishes correctness for many greedy algorithms.

Greedy Strategy: Step-by-Step Narrative

  1. Characterize a greedy choice: decide what local choice you will make at each step.
  2. Prove that a greedy choice is safe: show that there exists an optimal solution that begins with that choice.
  3. Design the remainder of the algorithm: after picking the greedy choice, define the subproblem and solve it recursively or iteratively.
  4. Analyze complexity and edge cases.

Designing greedy algorithms is often more art than mechanical procedure: intuition, trial-and-error, and verifying the greedy-choice and exchange arguments are central.

Classic Greedy Algorithms (with explanations)

1) Activity Selection (Interval Scheduling — maximize number of non-overlapping intervals)

Problem statement: Given activities, each with a start time s_i and finish time f_i, select the maximum-size subset of pairwise non-overlapping activities.

Greedy choice: Always pick the activity that finishes earliest among the remaining activities. Intuition: picking an activity that finishes earliest leaves as much room as possible for the remaining activities. 

Pseudocode (iterative):
1. Sort activities by finish time f (ascending).
2. Let selected = empty list.
3. Let last_finish = -infinity.
4. For each activity a in sorted order:
     if a.start >= last_finish:
         add a to selected
         last_finish = a.finish
5. Return selected

Correctness hint: Use an exchange argument. Take an optimal solution and replace its earliest-finishing activity with the greedy earliest-finishing activity — such a replacement does not reduce the number of activities and preserves feasibility.

2) Fractional Knapsack (maximize value with capacity, items divisible)

Problem statement: Given items with values v_i and weights w_i and knapsack capacity W, maximize total value; fractional amounts of items are allowed.

Greedy choice: Take items in decreasing order of value density (v_i / w_i). If the current item fits entirely, take it; otherwise take a fractional part to fill the knapsack. 

Pseudocode:
1. For each item compute density = v_i / w_i.
2. Sort items by density descending.
3. remaining = W; total_value = 0
4. For item in sorted:
     if item.weight <= remaining:
         take whole item; remaining -= item.weight; total_value += item.value
     else:
         take fraction = remaining / item.weight
         total_value += item.value * fraction
         break
5. Return total_value

Correctness: A standard greedy-exchange or continuous optimization argument shows this is optimal when fractions are allowed.

3) Huffman Coding (minimize weighted path length of a prefix code)

Problem statement: Given symbol frequencies, build a binary prefix code that minimizes the weighted sum of code lengths (expected code length).

Greedy choice: Repeatedly merge the two least-frequent symbols/trees into a new tree node whose frequency is the sum. Place the merged tree back into the pool and repeat until one tree remains. 

High-level algorithm:
1. Put all symbols (with frequencies) into a min-priority queue.
2. While queue size > 1:
     a = extractMin()
     b = extractMin()
     create node with freq = a.freq + b.freq, children = (a,b)
     insert node back
3. Remaining node is the Huffman tree.

Correctness: Exchange arguments and induction show that combining the two least frequent symbols is safe.

4) Minimum Spanning Tree (Prim's and Kruskal's)

Both algorithms use greedy local choices:

  • Prim's: start from any vertex, repeatedly add the cheapest edge that connects the growing tree to a new vertex.
  • Kruskal's: sort edges by weight and add edges in increasing order if they don't create a cycle (use union-find).

The cut property and cycle property formalize why adding the cheapest acceptable edge is safe.

5) Shortest Paths with Nonnegative Weights (Dijkstra's algorithm)

Greedy view: Repeatedly settle (fix) the vertex with minimum tentative distance from the source; once settled, its distance is final. This is a greedy choice: pick the closest unsettled vertex next.

Dijkstra requires nonnegative edge weights for correctness.

When Greedy Fails — Counterexamples

0/1 Knapsack

Problem statement: items are indivisible; select subset to maximize value under weight limit. Greedy by density can fail.

Example:

  • Item A: weight 10, value 60 (density 6)
  • Item B: weight 20, value 100 (density 5)
  • Item C: weight 30, value 120 (density 4)

Capacity W = 50. Greedy picks A (10,60), then B (20,100) then cannot pick C fully. Total value = 160. Optimal picks B and C (20+30=50) total value = 220. Thus greedy fails.

Coin change (non-canonical systems)

Problem statement: given coin denominations, minimize the number of coins to make amount N; greedy picks largest coin first.

Example (greedy fails): Denominations = {1, 3, 4}, amount = 6.

Greedy: pick 4 (remaining 2) -> then 1 + 1 => total 3 coins. Optimal: 3 + 3 => 2 coins.

Proof Approaches and Formal Tools

Exchange (swap) argument

Show any optimal solution can be transformed to one that starts with the greedy choice by swapping parts. This proves the greedy choice is safe.

Induction

Base case: trivial small instance. Assume the algorithm works on smaller sizes. Show that after making the greedy choice, the remaining subproblem satisfies the induction hypothesis.

Matroid theory (general characterization)

For a family of feasible sets that forms a matroid, the greedy algorithm that selects elements by decreasing weight (taking each element if it keeps feasibility) finds a maximum-weight independent set. This is a deep sufficient condition for greedy to be optimal in abstract settings.

Complexity and Implementation Tips

  • Sorting is common. If algorithm sorts, complexity often includes O(n log n).
  • Use appropriate data structures: priority queues (heaps), union-find, sorting, or simple linear scans depending on needs.
  • Carefully handle ties in sorting (define deterministic tie-breakers when helpful for proofs).
  • When correctness is unclear, attempt: (a) small counterexamples, (b) exchange arguments, (c) formal reduction to known optimal cases.

Quick Reference

Key checklist when considering a greedy solution:
  • Is there a natural local choice?
  • Can you give an exchange argument or induction to justify it?
  • Does the problem have optimal substructure?
  • Are there known counterexamples (try small instances)?
