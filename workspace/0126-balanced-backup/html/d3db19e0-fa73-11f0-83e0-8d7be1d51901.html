Quick Concepts

  • Type system: rules that classify values (number, string, boolean, object, etc.) and limit how they combine.
  • Static vs Dynamic: static systems check types at compile/time (e.g. C, Rust, TypeScript static checks), dynamic systems check at runtime (e.g. JavaScript, Python).
  • Strong vs Weak: strong typing prevents unexpected conversions (few implicit coercions); weak typing does many implicit conversions.
  • Nominal vs Structural: nominal systems use names (types are distinct by declaration), structural systems use shape (duck typing).
  • Type inference: automatically deducing types (e.g. let x = 1 implies x: number).

Examples & Notes

Strong typing: prevents implicit conversions. Example (pseudo):
function add(x: number, y: number) { return x + y } add("5", 2) // compile-time error
Weak typing (JS): '5' - 2 === 3 because '5' coercion -> number; '5' + 2 === '52' because + also concatenates when string appears. Nominal typing (e.g., Java, C#): types are distinct by name. Structural typing (e.g., Go interfaces, TypeScript) compares members. Type inference saves annotations: compiler deduces types from usage (Hindley-Milner in ML/Haskell is powerful). Soundness: A sound type system guarantees that well-typed programs don't go wrong at runtime (no unexpected type errors). Unsoundness (like unsafe casts) may let runtime errors through despite passing the type checker.
This single-file demo is educational. It contains simplified "toy" checkers to illustrate concepts; real compilers/type systems are far more complex.
