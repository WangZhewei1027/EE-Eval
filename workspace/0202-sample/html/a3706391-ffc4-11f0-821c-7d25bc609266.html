Binary Search Tree (BST): Comprehensive Explanation

A Binary Search Tree (BST) is a special kind of binary tree that maintains its elements in an ordered fashion to enable efficient search, insertion, and deletion operations. BSTs play a fundamental role in computer science, particularly in algorithms and data structure design.

1. Definition and Basic Properties

A Binary Search Tree is a binary tree satisfying the following properties for each node:

  • Node Key: Each node contains a comparable key (and possibly associated data).
  • Left Subtree Property: The key of every node in the left subtree is less than the node's key.
  • Right Subtree Property: The key of every node in the right subtree is greater than the node's key.
  • No Duplicate Keys: Typically, BSTs do not allow duplicate keys to preserve strict ordering.
    If duplicates are allowed, a convention (e.g., duplicates go to the left or right subtree) must be consistently applied.

This strict ordering gives BSTs their search efficiency.

Note: The term "binary" means each node has at most two children: left and right.

2. Why Use a BST?

The BST structure provides:

  • Dynamic data storage: Elements can be inserted and removed efficiently.
  • Quick search: Like a binary search on a sorted array, search operations can be performed in O(h) time, where h is the height of the tree.
  • Ordered data: In-order traversal of BST yields the keys in sorted order.

Time Complexities

Operation Average Case Worst Case
Search O(log n) O(n)
Insertion O(log n) O(n)
Deletion O(log n) O(n)

Note: The worst case occurs when the tree becomes skewed (like a linked list).

3. Structural Components

Each BST Node typically contains:

  • key: The value stored for ordering.
  • left: Pointer/reference to the left child node (or null if none).
  • right: Pointer/reference to the right child node (or null if none).

The root node is the top-most node with no parent.

4. Fundamental BST Operations Explained

4.1 Searching in a BST

To search for a key K:

  1. Start at the root node.
  2. If the node is null, key not found.
  3. If K equals the node’s key, search successful.
  4. If K is less than the node’s key, search recursively in the left subtree.
  5. If K is greater than the node’s key, search recursively in the right subtree.

This method efficiently narrows down the search space at every step, like binary search on an array.

Search Algorithm (Recursive Pseudocode):

function search(node, K)
  if node is null
    return null  // Key not found
  if K == node.key
    return node  // Key found
  else if K < node.key
    return search(node.left, K)
  else
    return search(node.right, K)

4.2 Inserting into a BST

Insertion adds a new key while maintaining BST ordering.

Algorithm to insert key K:

  1. Start from the root.
  2. If current node is null, place new node here.
  3. If K < current node's key, recursively insert in left subtree.
  4. If K > current node's key, recursively insert in right subtree.
  5. If the key already exists, typically ignore or handle duplicates according to defined policy.

Insertion Algorithm (Simplified Recursive Pseudocode):

function insert(node, K)
  if node is null
    return new Node(K)  // Insert here
  if K < node.key
    node.left = insert(node.left, K)
  else if K > node.key
    node.right = insert(node.right, K)
  return node

4.3 Deleting from a BST

Deletion is more nuanced because the BST must remain ordered after node removal.

Consider the node to be deleted has key K. There are three cases:

  1. Node is a leaf (no children): Simply remove it by setting parent's pointer to null.
  2. Node has one child: Replace the node with its single child.
  3. Node has two children: Find the node's in-order successor (the smallest key in the right subtree) or in-order predecessor (largest key in the left subtree), copy its key to the node, then delete the successor/predecessor node. This ensures BST property remains intact.

Deletion Algorithm Steps (Conceptual Summary):

function delete(node, K)
  if node is null
    return null
  if K < node.key
    node.left = delete(node.left, K)
  else if K > node.key
    node.right = delete(node.right, K)
  else
    // Node with key K found
    if node.left is null
      return node.right
    else if node.right is null
      return node.left
    else
      temp = minValueNode(node.right)  // Find in-order successor
      node.key = temp.key
      node.right = delete(node.right, temp.key)
  return node

Where minValueNode(node) finds the node with minimum key in subtree rooted at node:

function minValueNode(node)
  current = node
  while current.left is not null
    current = current.left
  return current

5. Traversals in a BST

Traversal means visiting all nodes in some order. Important BST traversal orders are:

  • In-order (Left, Root, Right): Visits nodes in ascending order of their keys.
  • Pre-order (Root, Left, Right): Useful for tasks like creating a copy of the tree.
  • Post-order (Left, Right, Root): Useful for deleting nodes, freeing memory, or evaluating tree-based expressions.

In-order Traversal Algorithm (Recursive Pseudocode):

function inorder(node)
  if node is not null
    inorder(node.left)
    print(node.key)
    inorder(node.right)

Example:

Consider inserting keys into BST in this order: 50, 30, 70, 20, 40, 60, 80

The BST will look like:


         50
       /    \
     30      70
    /  \    /  \
  20   40 60   80

In-order traversal will print: 20 30 40 50 60 70 80

6. Additional Concepts

6.1 Balanced vs. Unbalanced BSTs

A BST is balanced if the height difference between left and right subtrees of any node is small (commonly, ≤ 1). Balanced BSTs guarantee O(log n) height and thus efficient operations.

Unbalanced BSTs can degrade into linked lists with height O(n), causing operations to degrade to linear time.

Self-balancing BSTs (like AVL trees, Red-Black trees) automatically maintain balance during insertions and deletions.

6.2 Real-World Use Cases

  • Implementing dynamic sets or dictionaries supporting efficient search, add, and remove.
  • Database indexing.
  • Priority or sorted collections supporting quick access to ordered data.
  • Symbol tables in compilers/interpreters.
