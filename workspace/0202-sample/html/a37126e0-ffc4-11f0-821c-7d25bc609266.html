Dijkstra’s Algorithm: A Comprehensive Explanation

In the field of computer science and graph theory, Dijkstra’s algorithm stands as one of the most foundational and important algorithms for solving shortest path problems on weighted graphs. Developed by the Dutch computer scientist Edsger W. Dijkstra in 1956 and published in 1959, this algorithm efficiently finds the shortest paths between nodes in a graph, which may represent, for example, road networks, communication networks, or any system of interconnected points.

Definition

Dijkstra’s Algorithm is a greedy graph search algorithm that finds the shortest path from a single source vertex to all other vertices in a weighted graph with non-negative edge weights.

Context and Applicability

Consider a graph G = (V, E) where:

  • V is the set of vertices (or nodes).
  • E is the set of edges connecting pairs of vertices.

Each edge (u, v) ∈ E is associated with a non-negative weight w(u, v) representing the cost, distance, or time to travel from u to v. The goal of Dijkstra's algorithm is to determine the minimum-cost paths from a source vertex s ∈ V to every other vertex in the graph.

This problem appears in many domains:

  • Finding the quickest driving route on a map.
  • Network packet routing to minimize latency.
  • Planning delivery or logistics routes.
  • Robot navigation and pathfinding in grid-like spaces.

Key Assumptions

Dijkstra’s algorithm requires that all edge weights be non-negative. This is crucial because the algorithm assumes that once a shortest path to a vertex is found, it does not need to be revisited. Negative edges can cause this assumption to fail, and in such cases, other algorithms like the Bellman-Ford algorithm are used.

Intuition Behind the Algorithm

At its core, Dijkstra’s algorithm incrementally builds the set of vertices with known shortest path distances from the source by always selecting the "closest" unexplored vertex and updating path distances through it. By repeating this process, the algorithm "radiates out" from the source and discovers the shortest paths greedily.

Imagine standing at the source vertex and “spreading out” like ripples, calculating the shortest distances to neighboring nodes, then new neighbors, and so on — each time picking the next closest vertex where the path cost is minimal.

The Algorithm: Step-by-Step Description

  1. Initialization:
    • Assign to every vertex a tentative distance value: set it to zero for our source vertex and to infinity (∞) for all other vertices.
    • Mark all vertices unvisited. Create a set of all the unvisited vertices called the unvisited set.
  2. Visit the Unvisited Vertex with the Smallest Tentative Distance:
    • Initially, this is the source vertex, since its tentative distance is zero.
  3. For the Current Vertex, Consider All of its Unvisited Neighbors:
    • Calculate their tentative distances through the current vertex.
    • Compare the newly calculated tentative distance to the current assigned value and assign the smaller one. This means if the path through the current vertex is shorter, update the distance.
  4. After Considering All Neighbors of the Current Vertex, Mark the Current Vertex as Visited:
    • A visited vertex will not be checked again.
  5. Repeat Steps 2-4 Until All Vertices Have Been Visited:
    • When the destination vertex’s shortest path is found or all vertices are visited, the algorithm terminates.

Detailed Explanation of Core Concepts

Tentative Distance

The tentative distance to a vertex represents the length of the currently known shortest path from the source to that vertex. It may be updated multiple times when a shorter path is discovered via a neighbor's relaxation.

Relaxation

Relaxation is a key operation in which the algorithm tries to improve the shortest path estimate to each neighbor of the current node by checking if a path through the current vertex is shorter than previously known paths. Concretely: 

if (dist[current] + w(current, neighbor) < dist[neighbor]) {
    dist[neighbor] = dist[current] + w(current, neighbor);
}

Priority Queue (Min-PQ)

Though the explanation is textual here, in practical implementations, a priority queue is used to efficiently select the next vertex with the smallest tentative distance. This speeds up the lookup from O(V) to O(log V) per extraction. For educational clarity, no code demonstrations of data structures are shown here.

Example: Step-by-Step Without Visualization

Consider the following weighted graph (the numbers represent edge weights): 

Vertices: A, B, C, D, E

Edges:
A - B : 4
A - C : 2
B - C : 1
B - D : 5
C - D : 8
C - E : 10
D - E : 2

We want to find the shortest paths from A to all other vertices.

  1. Initialize distances:
    • dist(A) = 0 (source)
    • dist(B,C,D,E) = ∞
  2. Visit A (tentative dist 0):
    • Update neighbors:
      • dist(B) = min(∞, 0 + 4) = 4
      • dist(C) = min(∞, 0 + 2) = 2
  3. Mark A visited.
  4. Next, visit vertex with smallest dist: C (2):
    • Update neighbors:
      • dist(B) = min(4, 2 + 1) = 3
      • dist(D) = min(∞, 2 + 8) = 10
      • dist(E) = min(∞, 2 + 10) = 12
  5. Mark C visited.
  6. Next, B (3):
    • Update neighbors:
      • dist(D) = min(10, 3 + 5) = 8
  7. Mark B visited.
  8. Next, D (8):
    • Update neighbors:
      • dist(E) = min(12, 8 + 2) = 10
  9. Mark D visited.
  10. Finally, visit E (10). No neighbors to update.
  11. Mark E visited.

Resulting shortest path distances from A:

  • dist(A) = 0
  • dist(B) = 3
  • dist(C) = 2
  • dist(D) = 8
  • dist(E) = 10

This means the shortest path from A to B has a total weight of 3, the path to C is 2, and so forth.

Summary of Properties

  • Completeness: Dijkstra’s algorithm will always find a shortest path if one exists.
  • Time Complexity: Using a simple array for distance storage, the complexity is O(V²). Using a priority queue with adjacency lists improves this to O((V + E) log V).
  • Space Complexity: Primarily to store graph representation and distance arrays, usually O(V + E).
  • Edge Weights: Must be non-negative for correctness.

Formal Pseudocode

procedure Dijkstra(Graph, source):
    create vertex set Q

    for each vertex v in Graph:
        dist[v] ← infinity
        prev[v] ← undefined
        add v to Q
    dist[source] ← 0

    while Q is not empty:
        u ← vertex in Q with min dist[u]
        remove u from Q

        for each neighbor v of u:
            alt ← dist[u] + length(u, v)
            if alt < dist[v]:
                dist[v] ← alt
                prev[v] ← u

    return dist[], prev[]

Interpretation of Results

The dist[] array contains the shortest distances from the source to each vertex, while the prev[] array allows reconstructing the actual path taken by backtracking from any destination vertex to the source.

Brief Remarks on Variants

There are several variants and optimizations of the algorithm, including:

  • Bidirectional Dijkstra: Runs simultaneous searches from source and target to speed up shortest path computation.
  • Using Fibonacci Heaps: Improves theoretical performance but with more complex implementation.
  • A* Search: Extends Dijkstra by incorporating heuristics for faster directed search in pathfinding.
